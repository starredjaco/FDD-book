# 附录 C：驱动程序开发者的硬件概念

## 如何使用本附录

主要章节教授 FreeBSD 的驱动程序面向端：如何编写 softc、如何声明 `cdev`、如何设置中断、如何将 PCI BAR 漫游到 `bus_space` 句柄、如何用 `bus_dma(9)` 驯服 DMA 引擎。在每个这些主题之下，有一层安静的书本一直使用但不总是命名的硬件知识。物理地址真正是什么。为什么 CPU 不能简单地通过普通指针读取设备寄存器。为什么 PCI 和 I2C 感觉像完全不同的世界，尽管两者都在 CPU 和外设之间传递字节。为什么中断不是函数调用。为什么 DMA 从上到下改变您编写驱动程序的方式。

本附录就是那个安静层，写下来了。它是概念指南，不是电子学教科书。它由驱动程序作者为编写正确代码需要相信的机器内容塑造，并故意不包含完整的计算机架构教学。如果您曾经编写过 `bus_space_read_4` 调用并想知道标签和句柄真正代表什么，或设置过 MSI-X 向量并想知道通过哪些线发送什么信号，或调试过在您的桌子上工作但在不同板上失败的 DMA 传输，您来对了地方。

## 内核中的内存

误解内存的驱动程序将编写看起来完美但在真实硬件上崩溃的代码。内核中的内存与 C 教程中的内存不同。至少有两个地址空间在起作用（物理和虚拟），有硬件强制执行的约束（DMA 寻址限制、页面边界），有即使是普通读取也有副作用的区域（设备寄存器）。本部分构建本附录中每个后续主题依赖的心理模型。

### 物理内存 vs 虚拟内存

**它是什么。** 物理地址是内存控制器用于选择 RAM 中真实位置的数字，或 I/O 结构用于选择外设上寄存器的数字。虚拟地址是运行线程在解引用指针时看到的数字；内存管理单元（MMU）使用内核维护的页表即时将其转换为物理地址。

**这对驱动程序编写者为何重要。** 驱动程序中的每个指针都是虚拟的。CPU 执行您的 C 代码时永远不会看到物理地址。内核和 `pmap(9)` 层维护页表，以便内核虚拟地址映射到内核决定放置相应页面的物理内存中的任何位置。"知道"物理地址的驱动程序不能简单地将其转换为 `(void *)` 并解引用；转换可能不存在、权限可能禁止它，或者属性可能对您需要的访问类型错误。

硬件在镜子的另一边。外设设备位于物理地址空间的某处（其寄存器是物理地址，当它执行 DMA 时，它在总线上发出物理地址）。设备永远不会解引用虚拟指针。如果您的驱动程序将内核虚拟地址交给设备并期望它读取那里的数据，设备将读取一些不相关的物理位置，您将花很长时间弄清楚为什么。

**这在 FreeBSD 中如何显示。** 两个地方。首先，`bus_space(9)` 编码寄存器访问的转换：标签描述物理地址空间的类型，句柄描述内核已设置的虚拟映射，`bus_space_read_*` 和 `bus_space_write_*` 访问器使用句柄通过该映射到达设备。其次，`bus_dma(9)` 仲裁 DMA：`bus_dmamap_load` 调用遍历缓冲区并生成设备应使用的总线地址列表，这不一定是虚拟地址或物理地址，因为 IOMMU 可能介于其间。

**常见陷阱。** 将固件或 BAR 寄存器返回的物理地址转换为指针并直接从中读取。在某些架构和某些构建上，这可以编译甚至可能返回某些内容，但它永远不正确。始终通过 `bus_space`、`pmap_mapdev` 或已建立的 FreeBSD API 映射。

**本书讲解位置。** 第 16 章建立区别并展示为什么原始指针算术对设备内存不够。第 21 章在 DMA 使物理端可见时回到它。

**接下来读什么。** `bus_space(9)`、`pmap(9)` 和 `/usr/src/sys/kern/subr_bus_dma.c` 的早期部分。

### MMU、页表和设备映射

**它是什么。** MMU 是使用页表树将虚拟地址转换为物理地址的硬件单元。每个页表项保存物理页号和一小组属性位：可读、可写、可执行、可缓存、用户可访问。同一物理页可以以不同属性映射到多个虚拟地址；同一虚拟地址可以在不同时间指向不同的物理页。

**这对驱动程序编写者为何重要。** 设备内存不是普通 RAM，CPU 必须知道这一点。寄存器可能自己更改（设备完成传输时设置的状态位），读取可能有副作用（读取即清除的标志），写入可能需要以特定顺序变得可见。如果 CPU 像缓存 RAM 中的整数那样缓存设备寄存器，代码将永远读取陈旧值。MMU 通过让内核以不同属性映射设备区域来解决这个问题：通常不缓存、强排序，有时写组合。驱动程序通常不手动选择这些属性。`bus_space` 和 `bus_alloc_resource` 为您做这件事。

**这在 FreeBSD 中如何显示。** 当 `bus_alloc_resource_any(dev, SYS_RES_MEMORY, &rid, RF_ACTIVE)` 返回时，内核已经以 MMIO 的正确属性映射了设备区域。您返回的 `struct resource *` 携带访问器使用的 `bus_space_tag_t` 和 `bus_space_handle_t` 对。您永远不应该自己操作底层页表。实际上访问是两行：

```c
uint32_t val = bus_space_read_4(rman_get_bustag(res),
    rman_get_bushandle(res), STATUS_REG_OFFSET);
```

每个 `bus_space_read_4` / `bus_space_write_4` 调用通过内核为您设置的 MMIO 映射进行；没有指针解引用，没有缓存加载。

**常见陷阱。** 混合同一物理区域的可缓存和不可缓存视图。用 `bus_dmamem_alloc` 分配然后转换为不同虚拟映射的 DMA 缓冲区将给您相同字节的两个不一致视图。使用分配器返回的映射，而不是您在旁边构建的映射。

**本书讲解位置。** 第 16 章在解释为什么 `bus_space` 抽象存在时间接触 MMU。第 21 章在 DMA 一致性成为主题时使其显式。

**接下来读什么。** `pmap(9)`、`bus_space(9)` 和 `/usr/src/sys/vm/vm_page.h` 了解内核的物理页表示（`vm_page_t`）。

### 为什么硬件不能简单地看到内核指针

**它是什么。** 陈述前两小节最重要后果的简短方式。内核生活在虚拟内存中。外设生活在物理（有时是 I/O 虚拟）内存中。内核指针对设备没有意义。

**这对驱动程序编写者为何重要。** 每次给设备内存地址时，必须给它总线地址，而不是虚拟指针。总线地址是设备看到的；它可能是物理地址，或者可能是 IOMMU 重映射的 I/O 虚拟地址。您不自己计算它。加载映射时 DMA 基础设施为您计算它。

**这在 FreeBSD 中如何显示。** `bus_dmamap_load` 接受虚拟缓冲区并生成 `bus_dma_segment_t` 条目，其 `ds_addr` 是设备必须使用的地址。您将该值复制到您即将交给硬件的描述符中。当您发现自己想将 `vtophys` 的输出或原始 BAR 值传递给设备时，停下来改用 `bus_dma`。

**常见陷阱。** 将 `(uint64_t)buffer` 写入设备描述符，因为它在一台机器上的玩具测试中工作。它将在内核地址与总线地址不同的任何机器上失败，这是大多数真实机器。

**本书讲解位置。** 第 21 章。整章都是关于遵守这个分离的。

**接下来读什么。** `bus_dma(9)` 和下面的 DMA 部分。

### 一页纸的内核内存分配

内核提供三种主要的内存获取方式，每种都有不同的目的。驱动程序经常使用它们，选择错误的是典型的初学者错误。这里的重点是概念差异；完整 API 参考在附录 A 中。

- **`malloc(9)`** 是通用分配器。它返回的内核虚拟内存完全适合 softc 结构、列表、您自己记账的命令缓冲区以及任何其他保留在驱动程序数据中的内容。它不会自动物理连续，也不会自动适合 DMA。将其用于控制数据。

- **`uma(9)`** 是区域分配器。它是驱动程序经常分配和释放的固定大小对象的快速缓存，例如每请求结构或每包状态。它是通用分配的专门化；它不改变物理与虚拟的故事。

- **`contigmalloc(9)`** 返回物理连续范围，带有对齐和上限地址限制选项。驱动程序仅当真正需要单个连续物理块且 `bus_dma` 层不是更好的选择时才直接调用它。在现代驱动程序中，`bus_dma` 几乎总是更好的选择。调用形态本身就说明了：

  ```c
  void *buf = contigmalloc(size, M_DEVBUF, M_WAITOK | M_ZERO,
      0, BUS_SPACE_MAXADDR, PAGE_SIZE, 0);
  ```

  `low`、`high`、`alignment` 和 `boundary` 参数正是 `bus_dma_tag_create` 调用会表达的 DMA 约束，这就是为什么 `bus_dma(9)` 几乎总是编码它们的更好位置。

- **`bus_dmamem_alloc(9)`** 是 DMA 感知分配器。它返回在给定标签下可用于 DMA 的缓冲区，正确对齐、可选一致，并与您可以加载的 `bus_dmamap_t` 配对。这几乎是每个 DMA 场景中您想要的调用。

**这对驱动程序编写者为何重要。** 要问自己的问题始终是"谁将读取或写入此内存？"。如果只有 CPU 触摸它，`malloc(9)` 就可以。如果设备需要读取或写入它，您想要在表达设备寻址约束的 `bus_dma(9)` 标签下使用 `bus_dmamem_alloc(9)`。混合两者是整个属驱动程序错误的来源。

**常见陷阱。** 用 `malloc(9)` 分配缓冲区，将其虚拟地址交给设备，并想知道为什么设备读取垃圾。修复很少是修补症状；而是通过 `bus_dma(9)` 重写分配。

**本书讲解位置。** 第 5 章在内核 C 上下文中介绍 `malloc(9)`，从第 7 章开始的驱动程序章节重复使用它来获取 softc 和记账内存。第 21 章介绍 `bus_dmamem_alloc`。附录 A 列出完整 API 和标志。

**接下来读什么。** `malloc(9)`、`contigmalloc(9)`、`uma(9)`、`bus_dma(9)`。

### IOMMU 和总线地址

**它是什么。** IOMMU 是位于设备和主内存之间的硬件单元（有时是 `amdvi` 或 `intel-iommu` 单元；在 ARM 上是 SMMU），将设备发出的地址转换为物理地址。它是 MMU 的设备端模拟。

**这对驱动程序编写者为何重要。** 当存在 IOMMU 时，设备在总线上使用的地址（总线地址或 I/O 虚拟地址）与底层内存的物理地址不同。IOMMU 让内核给每个设备一个受限的内存视图，这对安全和隔离有好处，它让内核在设备否则需要连续物理页的地方使用分散-聚集。

**这在 FreeBSD 中如何显示。** 对于大多数驱动程序是透明的。当内核使用 `IOMMU` 选项和架构后端之一（amd64 上 `/usr/src/sys/x86/iommu/` 下的 Intel 和 AMD 驱动程序，arm64 上 `/usr/src/sys/arm64/iommu/` 下的 SMMU 后端）构建时，`/usr/src/sys/dev/iommu/` 中的 `busdma_iommu` 层自动桥接 `bus_dma(9)` 和 IOMMU，而不更改驱动程序的代码。您仍然分配标签、创建映射、加载映射并读取每个段的 `ds_addr`；您获得的数字是设备应使用的地址，无论是否有 IOMMU。您不需要知道是否存在 IOMMU。

**常见陷阱。** 假设 `ds_addr` 是物理地址并以这种方式记录它。在 IOMMU 系统上，该数字是 I/O 虚拟地址，不会匹配您在 `/dev/mem` 中看到的任何内容。正确的心理模型是"设备使用的地址"，不是"物理地址"。

**本书讲解位置。** 第 21 章简要提及 IOMMU 集成，并坚持 `bus_dma` 隐藏它的常见情况。

**接下来读什么。** `bus_dma(9)`、`/usr/src/sys/dev/iommu/busdma_iommu.c` 了解桥接代码，以及 `/usr/src/sys/x86/iommu/` 或 `/usr/src/sys/arm64/iommu/` 了解每架构后端。

### 一段话的内存屏障

在现代 CPU 上，内存操作可能以与代码发出它们的顺序不同的顺序对其他代理（其他 CPU、其他设备）变得可见。对于普通 RAM，内核的锁定原语为您隐藏这一点。但对于设备寄存器，排序很重要：在写入"数据"寄存器之前写入"启动"寄存器会产生与相反顺序不同的结果。内核为双方提供显式屏障。对于寄存器访问，`bus_space_barrier(tag, handle, offset, length, flags)` 是正确的工具；对于 DMA 缓冲区，`bus_dmamap_sync(tag, map, op)` 是正确的工具。您稍后会在本附录中再次遇到两者。目前，内化规则：当两次访问的顺序对硬件重要时，用屏障说明。不要仅依赖代码顺序。

## 总线和接口

总线是在 CPU 和外设之间携带命令、地址和数据的物理和逻辑路径。FreeBSD 在 Newbus 框架中将每个总线视为专门的树：总线有枚举其子设备的驱动程序，每个子设备是 `device_t`，每个子设备又可以是更多子设备的总线。"总线"的概念在 Newbus 级别是统一的。"特定总线的电气、协议和枚举规则"概念根本不统一，驱动程序作者必须理解他们正在为哪一种编写。

本部分给出本书关心的四个总线的心理模型：PCI（和 PCIe）、USB、I2C 和 SPI。它们的章节在后面；这是您首先需要的共享词汇表。

### 驱动程序相关级别的 PCI 和 PCIe

**它是什么。** PCI 是将外设附加到 CPU 复合体的规范总线。经典 PCI 是并行和共享的；现代 PCI Express 是一组点对点串行通道，交换结构将其组织成树。从驱动程序的角度来看，电气差异几乎是不可见的：您仍然看到配置空间，您仍然看到基地址寄存器，您仍然看到中断，您仍然编写探测和附加代码。除非另有说明，本书中的"PCI"意味着"PCI 和 PCIe 一起"。

PCI 设备由总线:设备:功能（B:D:F）元组和固件和内核填充的配置空间标识。配置空间保存供应商和设备标识符、描述这是什么类型外设的类代码、描述设备内存和 I/O 窗口的基地址寄存器（BAR），以及广播可选功能如 MSI、MSI-X 和电源管理的能力结构链表。

**这对驱动程序编写者为何重要。** 每个 PCI 驱动程序都做大致相同的舞蹈。在 `probe` 期间读取 `pci_get_vendor(dev)` 和 `pci_get_device(dev)`，将它们与驱动程序支持的设备表进行比较，匹配时返回 `BUS_PROBE_*` 值。在 `attach` 中，用 `bus_alloc_resource_any(dev, SYS_RES_MEMORY, &rid, RF_ACTIVE)` 分配每个 BAR，用 `rman_get_bustag` 和 `rman_get_bushandle` 拉出 `bus_space_tag_t` 和 `bus_space_handle_t`，用 `bus_alloc_resource_any(dev, SYS_RES_IRQ, ...)` 分配中断（或对消息信号中断使用 `pci_alloc_msix`/`pci_alloc_msi`），用 `bus_setup_intr` 连接处理程序，您就开张了。

能力是 PCI 驱动程序遇到特定设备系列功能的地方。`pci_find_cap(dev, PCIY_MSIX, &offset)` 找到 MSI-X 能力；`pci_find_extcap(dev, PCIZ_AER, &offset)` 找到高级错误报告的扩展能力。驱动程序通常只遍历它关心的能力。

**这在 FreeBSD 中如何显示。** PCI 总线驱动程序位于 `/usr/src/sys/dev/pci/` 下。关键头文件是 `/usr/src/sys/dev/pci/pcireg.h` 用于寄存器常量，`/usr/src/sys/dev/pci/pcivar.h` 用于客户端驱动程序调用的函数。您将编写的每个 PCI 驱动程序都以 `device_probe_t` 和 `device_attach_t` 开始，文件底部有 `DRIVER_MODULE(name, pci, driver, 0, 0)`。最后两个参数是每模块事件处理程序及其参数，通常保留为 `0, 0`（或 `NULL, NULL`）；旧书提到的单独 `devclass_t` 槽在几个版本前已从宏中删除。

**常见陷阱。** 忘记在期望设备 DMA 之前调用 `pci_enable_busmaster(dev)`。芯片可以生活在安静状态，其中 MMIO 工作而 DMA 静默地什么都不做。在 `attach` 中启用总线主控是通常序列的一部分。

**本书讲解位置。** 第 18 章是 PCI 驱动程序编写的完整章节。第 20 章用 MSI-X 扩展它。

**接下来读什么。** `pci(4)`、`pci(9)` 和紧凑的真实示例如 `/usr/src/sys/dev/uart/uart_bus_pci.c`。PCI 总线本身在 `/usr/src/sys/dev/pci/pci.c` 中实现。

### 驱动程序相关级别的 USB

**它是什么。** USB 是主机控制的分层总线，其中主机控制器通过集线器树与设备对话。与 PCI 不同，总线严格分层且类似轮询：主机调度每个事务。设备不会在没有被询问的情况下在总线上"说话"。USB 速度等级（低速、全速、高速、超速）共享相同的概念模型；协议细节改变。

USB 设备公开一组端点。每个端点是具有特定传输类型的单向数据通道：控制、批量、中断或同步。控制是每个设备都有的设置通道。批量是高吞吐量，无时间保证。中断是小、周期性、低延迟（鼠标、键盘、传感器）。同步是实时流（音频、视频），以可靠性换取时间。

**这对驱动程序编写者为何重要。** FreeBSD USB 驱动程序附加到父设备是 USB 总线而不是 PCI 总线的 `device_t`。识别模型不同：代替配置空间中的供应商/设备 ID 和类代码，USB 驱动程序匹配 USB 供应商 ID（VID）、产品 ID（PID）、设备类、子类和协议。FreeBSD USB 栈为您针对这些字段的结构化探测。在运行时，USB 驱动程序通过 USB 框架打开特定端点并发出生命周期由主机控制器管理的传输；您不像 PCI 驱动程序那样写入寄存器。

**这在 FreeBSD 中如何显示。** USB 栈位于 `/usr/src/sys/dev/usb/` 下。类驱动程序使用 `usbd_transfer_setup` 描述它想要的端点，使用 `usbd_transfer_start` 启动传输，并使用回调函数处理完成。传输类型常量位于 `/usr/src/sys/dev/usb/usb.h`（`UE_CONTROL`、`UE_ISOCHRONOUS`、`UE_BULK`、`UE_INTERRUPT`）。

**常见陷阱。** 以寄存器和中断的方式思考。USB 设备驱动程序是围绕传输完成回调的事件驱动，而不是设备寄存器读取。面向寄存器的思维方式导致与框架对抗而不是使用它。

**本书讲解位置。** 第 26 章是 USB 和串行驱动程序的完整章节。本附录是概念级入门；该章节是您的第一个 USB 驱动程序生活的地方。

**接下来读什么。** `usb(4)`、`usbdi(9)` 和 `/usr/src/sys/dev/usb/` 下的头文件。

### 驱动程序相关级别的 I2C

**它是什么。** I2C（内部集成电路）是一种慢速、双线、主从总线，设计用于连接板上的低带宽外设：温度传感器、电源管理控制器、EEPROM、小型显示器。每个从设备有一个 7 位或 10 位地址。主设备通过断言启动条件、发送地址、发送或接收字节并断言停止条件来启动每个事务。

速度通常为 100 kHz（标准）、400 kHz（快速）、1 MHz（快速加），以及更新设备上的更高变体。总线支持多个主设备，但大多数嵌入式系统使用单个主设备；在这种情况下，驱动程序只需担心主设备到从设备的交互。

**这对驱动程序编写者为何重要。** FreeBSD 中的 I2C 驱动程序有两种风格。I2C *控制器*驱动程序实现特定控制器芯片的主设备端，并作为 I2C 总线挂在 Newbus 树下。I2C *设备*驱动程序是使用 `iicbus` 层与从设备对话而不知道控制器细节的客户端。您要求 `iicbus_transfer` 执行一系列简短的 `struct iic_msg` 操作，`iicbus` 层负责仲裁和时序。

客户端驱动程序中没有要分配的中断或 BAR。没有 DMA。每个传输短且阻塞，通常几个字节；长读取被分成一系列传输。

**这在 FreeBSD 中如何显示。** 框架位于 `/usr/src/sys/dev/iicbus/` 下。客户端接口以 `iicbus_transfer(dev, msgs, nmsgs)` 和 `struct iic_msg` 类型为中心。FreeBSD 在嵌入式平台上通过 FDT 发现 I2C 总线，在公开 I2C 控制器的 x86 笔记本电脑上通过 ACPI 发现。

**常见陷阱。** 编写忙等待的 I2C 驱动程序。即使在 1 MHz，传输也是几十微秒；通过轮询循环固定 CPU 的驱动程序浪费资源。`iicbus` 层为您处理等待。

**本书讲解位置。** 本书不专门为 I2C 设立完整章节；本附录是主要概念处理，嵌入式章节（第 32 章）在设备树绑定的上下文中重新讨论它。对于您可以从头到尾阅读的具体客户端，打开 `/usr/src/sys/dev/iicbus/icee.c`。

**接下来读什么。** `iicbus(4)`、`/usr/src/sys/dev/iicbus/iicbus.h` 和示例客户端如 `/usr/src/sys/dev/iicbus/icee.c`（EEPROM 驱动程序）。

### 驱动程序相关级别的 SPI

**它是什么。** SPI（串行外设接口）是一种简单、快速、全双工、主从总线。它本身没有寻址；每个从设备有专用的片选线，主设备断言该线以开始事务。四根线是典型的：SCLK（时钟）、MOSI（主出从入）、MISO（主入从出）和 SS/CS（从选择）。速度通常范围从几 MHz 到几十 MHz。

与 I2C 不同，SPI 除了电气协议之外没有协议：主设备在 MOSI 上移出位，同时在 MISO 上移入位。这些位意味着什么完全取决于设备。

**这对驱动程序编写者为何重要。** 与 I2C 一样，FreeBSD 将控制器与客户端分开。控制器驱动程序知道特定芯片中的 SPI 主设备如何编程；客户端驱动程序使用 `spibus` 接口发送和接收字节而不知道控制器。客户端使用描述单个事务的 `struct spi_command` 调用 `spibus_transfer` 或 `spibus_transfer_ext`，包括要断言的片选和要移位的缓冲区。

**这在 FreeBSD 中如何显示。** 框架位于 `/usr/src/sys/dev/spibus/` 下。典型的客户端驱动程序在 Newbus 中挂在 `spibus` 父设备下，并使用与任何其他设备相同的探测/附加形态。`/usr/src/sys/dev/spibus/spigen.c` 是向用户空间公开原始 SPI 的通用字符设备客户端；阅读它是了解框架客户端侧的好方法。

**常见陷阱。** 期望 SPI 携带任何内置帧的数据。它不携带。设备数据手册告诉您字节意味着什么；总线什么都不告诉您。

**本书讲解位置。** 本书不专门为 SPI 设立完整章节；本附录是主要概念处理，嵌入式章节（第 32 章）在设备树绑定的上下文中重新讨论它。对于您可以从头到尾阅读的具体示例，打开 `/usr/src/sys/dev/spibus/spigen.c`。

**接下来读什么。** `/usr/src/sys/dev/spibus/spibus.c`、`/usr/src/sys/dev/spibus/spigen.c` 和您正在驱动的任何 SPI 外设的设备数据手册。

### 四种总线的紧凑比较

小型参考通常比长描述更容易。此表沿实际影响驱动程序设计的轴线比较总线。这不是完整的电气比较。

| 方面 | PCI / PCIe | USB | I2C | SPI |
| :-- | :-- | :-- | :-- | :-- |
| 拓扑 | 树（PCIe 交换机） | 严格主机/集线器树 | 多点，寻址 | 多点，片选 |
| 典型速度 | 每通道 GB/s | KB/s 到 GB/s | 100 kHz - 1 MHz | 1 MHz - 几十 MHz |
| 识别 | 供应商/设备 + 类 | VID/PID + 类 | 7 或 10 位从地址 | 无（片选） |
| 协议帧 | PCI TLP，不可见 | 包 + 传输类型 | 启动/地址/数据/停止 | 原始位 |
| 中断 | INTx 或 MSI / MSI-X | 传输回调 | 罕见；总线轮询 | 罕见；总线轮询 |
| DMA | 是，设备驱动 | 主机控制器做 | 否 | 否 |
| 驱动程序角色 | 寄存器级 | 框架客户端 | `iicbus` 客户端 | `spibus` 客户端 |
| 枚举 | 固件 + `pci(4)` | 集线器/设备描述符 | FDT / ACPI / 手工接线 | FDT / ACPI / 手工接线 |

您应该带走的模式是 PCI 和 USB 足够复杂需要自己的枚举栈和事务机制，而 I2C 和 SPI 足够简单，短事务结构就是整个 API。驱动程序作者的工作负载在该分割的每一侧看起来都不同。

## 中断

中断是设备告诉 CPU 发生了什么的方式。没有中断，驱动程序将不得不轮询，消耗 CPU 周期来发现硬件已经知道的事件。有了中断，CPU 做其他工作直到硬件发出信号；然后内核在狭窄、定义良好的上下文中分派处理程序。本节解释该信号实际上是什么、常见形式的区别以及为什么中断纪律是驱动程序工作中最苛刻的技能之一。

### 中断实际上是什么

**它是什么。** 在硬件级别，中断是设备为请求 CPU 注意而引发的信号。在经典并行总线上，信号是设备拉高或拉低的线。在现代 PCIe 上，信号是中断控制器识别的携带小有效载荷的内存写入事务。在这两种情况下，中断控制器将信号映射到 CPU 向量，CPU 保存刚好足够的状态以切换上下文，内核的低级入口代码决定调用哪个驱动程序。

驱动程序看到的是函数调用：中断处理程序在小栈上运行，在特殊调度上下文中，通常禁用抢占或高度受限。硬件模型是"设备引发了一条线"；软件模型是"内核运行了您的回调"。

**这对驱动程序编写者为何重要。** 中断不是普通函数调用。它们异步发生，可能在不同于您的驱动程序正在运行的 CPU 上，并且可以在几乎任何代码路径中间到达。这就是为什么中断处理程序必须短、必须避免睡眠、必须不在没有在该上下文中安全的锁的情况下触摸共享状态。驱动程序学习的大部分关于锁定、排序和延迟工作的纪律存在都是因为这个事实。

**这在 FreeBSD 中如何显示。** 中断通过 `intr_event(9)` 到达驱动程序。驱动程序调用 `bus_setup_intr(dev, irq_resource, flags, filter, ithread, arg, &cookie)` 注册最多两个回调：首先在中断上下文中运行的*过滤器*，只有自旋锁可用，以及可选地作为内核线程运行并可以使用睡眠锁的 *ithread*（中断线程）。过滤器的返回值决定接下来发生什么：`FILTER_HANDLED`（完全处理，在控制器处确认）、`FILTER_STRAY`（不是我的，什么都不做）或 `FILTER_SCHEDULE_THREAD`（ithread 应该现在运行）。这些常量位于 `/usr/src/sys/sys/bus.h`。

**常见陷阱。** 在过滤器中做真正的工作。过滤器在那里决定所有权、如果可能快速确认设备、以及要么处理极少量的工作要么调度线程。长处理属于 ithread 或 taskqueue，不属于过滤器。

**本书讲解位置。** 第 19 章是中断的专门章节。第 20 章添加 MSI 和 MSI-X。第 14 章（taskqueue）解释如何将繁重工作延迟出中断路径。

**接下来读什么。** `intr_event(9)`、`bus_setup_intr(9)` 和 `/usr/src/sys/sys/bus.h` 中 `FILTER_` 和 `INTR_` 常量周围的过滤器/ithread 文档。

### 边沿触发 vs 电平触发

**它是什么。** 硬件信号可以表示"有新东西"的两种方式。

- **边沿触发。** 设备断言转换（上升或下降沿）以发出事件信号。如果驱动程序没有记录发生了中断，事件就会丢失。边沿是一个时刻，不是状态。
- **电平触发。** 设备保持线（或信号）活动，只要条件持续。中断控制器持续触发，直到驱动程序确认设备并取消断言线。电平是一个状态，不是一个时刻。

**这对驱动程序编写者为何重要。** 边沿触发中断是不宽容的：如果驱动程序有竞争，第二个边沿在第一个正在处理而未被记录时到达，第二个事件就被遗忘。电平触发中断是自愈的：如果您返回时线仍然断言，控制器将再次触发。传统 PCI INTx 线是电平触发和共享的，这就是为什么驱动程序处理程序必须读取设备的状态寄存器并在声明之前检查"此中断"是否真的是它们的。MSI 和 MSI-X 实际上是类似边沿的：每条消息是一个离散事件。

**这在 FreeBSD 中如何显示。** 区别主要在您的驱动程序之下处理。带有 `SYS_RES_IRQ` 和 rid `0` 的传统 INTx 注册为您获取电平触发、共享中断；`pci_alloc_msi` 和 `pci_alloc_msix` 为您提供每向量且不与不相关驱动程序共享的消息信号中断。您的过滤器仍然必须表现良好；但共享关注消失了。

**常见陷阱。** 编写在不实际检查设备的情况下返回 `FILTER_HANDLED` 的过滤器。在电平触发 INTx 上，这会挂起系统，因为线保持断言，控制器永远触发。

**本书讲解位置。** 第 19 章教授传统情况；第 20 章教授消息信号情况。

**接下来读什么。** `/usr/src/sys/dev/pci/pci.c` 中 `pci_alloc_msi` 周围的传统中断讨论，以及第 19 章中的过滤器示例。

### 没有 MSDN 的 MSI 和 MSI-X

**它是什么。** 消息信号中断。代替断言共享中断线，PCI Express 设备向中断控制器监视的特殊地址发送小型内存写入事务。写入的有效载荷识别哪个中断已触发。MSI 支持每设备少量向量（通常最多 32 个）；MSI-X 支持数千个，每个都有自己的地址和数据，以及每向量 CPU 亲和性。

**这对驱动程序编写者为何重要。** 当驱动程序使用 MSI 或 MSI-X 而不是传统 INTx 时，两件事改变。首先，中断不再共享，所以过滤器不需要与不相关的驱动程序共存（它仍然必须确认事件来自它自己的硬件）。其次，每个设备可以有多个中断。NIC 可以每个接收队列一个中断、每个发送队列另一个、管理路径另一个，您可以将每个绑定到不同的 CPU。这改变了您如何构建驱动程序：每个队列有自己的 softc 子结构、自己的锁和自己的处理程序。

**这在 FreeBSD 中如何显示。** `pci_msi_count(dev)` 和 `pci_msix_count(dev)` 告诉您有多少向量可用。`pci_alloc_msi(dev, &count)` 和 `pci_alloc_msix(dev, &count)` 预留它们；调用用实际分配的数量更新 `count`。然后您使用 `rid = 1, 2, ..., count` 的 `bus_alloc_resource_any(dev, SYS_RES_IRQ, &rid, RF_ACTIVE)` 分配每个向量。CPU 亲和性用 `bus_bind_intr(dev, res, cpu_id)` 设置。拆除镜像序列：每向量 `bus_teardown_intr` 和 `bus_release_resource`，然后单个 `pci_release_msi(dev)`。

**常见陷阱。** 请求太多向量并假设系统会给它们全部。`pci_alloc_msix` 可能返回比请求更少的，即使在广告充足的硬件上；驱动程序必须使其队列结构适应实际分配的数量。回退梯子（MSI-X，然后 MSI，然后 INTx）是标准形态。

**本书讲解位置。** 第 20 章。

**接下来读什么。** `pci(9)` 了解函数列表，`/usr/src/sys/dev/pci/pci.c` 了解实现。

### 为什么中断纪律重要

**它是什么。** 在中断下保持驱动程序正确的规则集。其中大部分来自三个事实：过滤器运行时可用的原语很少、过滤器可以与驱动程序中的每个其他路径竞争、延迟确认产生楔入系统。

**核心规则。**

- 保持过滤器短且无分配。没有 `malloc`、没有睡眠、没有可能睡眠的调用、没有复杂逻辑。
- 在返回 `FILTER_HANDLED` 之前在设备处确认，或确保设备继续断言直到您的 ithread 运行。
- 使用自旋锁（`MTX_SPIN`）保护过滤器和 CPU 路径共享的状态。睡眠锁不能在过滤器中获取。
- 将真正的工作延迟到 ithread、taskqueue 或 callout。过滤器的工作是将数据移出关键路径。
- 计算您的中断。每个向量的 `sysctl` 计数器很便宜，通常是驱动程序行为异常时的第一个线索。

**常见陷阱。** 在可以从过滤器重新进入的路径上持有睡眠锁。拥有睡眠互斥锁的线程不能被同一 CPU 上的中断抢占；内核的 `witness(9)` 诊断将在生产之前捕获其中许多，但仅当实际执行该场景时。

**本书讲解位置。** 第 11 章一般介绍纪律；第 19 章使其对真实硬件具体化。

## DMA

DMA（直接内存访问）是外设如何在没有 CPU 复制每个字节的情况下读取或写入内存。它存在是因为替代方案（CPU 读取寄存器、将字节放入 RAM 并循环）对于任何现代设备来说太慢了。DMA 也是驱动程序中最硬件感知的部分，因为这是驱动程序必须说总线的地址语言而不是内核的虚拟指针语言的地方。不遵守 DMA 纪律的驱动程序将产生几乎无法从用户空间诊断的数据损坏。

### DMA 是什么以及为什么它存在

**它是什么。** 设备在自己的控制下使用驱动程序预先交给它的总线地址从系统内存读取或写入的传输。CPU 设置传输并继续其他工作；设备通常通过引发中断发出完成信号。数据移动本身期间没有每字节 CPU 参与。

**这对驱动程序编写者为何重要。** 关于如何分配、加载、同步和释放内存的一切在 DMA 进入画面时都会改变。您不再独占拥有缓冲区；设备在传输期间拥有它。您必须告诉内核正在读取或写入缓冲区的哪些部分以及何时，以便正确处理缓存一致性、对齐和地址重映射。

**这在 FreeBSD 中如何显示。** `bus_dma(9)` 是整个机制。驱动程序作者的工作流程在每个支持 DMA 的设备上是一致的：

1. 在 `attach` 中创建 `bus_dma_tag_t`，描述设备的寻址约束（对齐、边界、`lowaddr`、`highaddr`、最大段大小、段数等）。标签捕获设备的真相。
2. 使用 `bus_dmamem_alloc(tag, &vaddr, flags, &map)` 分配支持 DMA 的缓冲区，或分配普通缓冲区并稍后加载它。
3. 使用 `bus_dmamap_load(tag, map, vaddr, length, callback, arg, flags)` 加载缓冲区。回调接收一个或多个 `bus_dma_segment_t` 条目，其 `ds_addr` 字段是交给设备的总线地址。
4. 在设备读取之前，`bus_dmamap_sync(tag, map, BUS_DMASYNC_PREWRITE)`。在 CPU 读取设备写入的内容之前，`bus_dmamap_sync(tag, map, BUS_DMASYNC_POSTREAD)`。标志在 `/usr/src/sys/sys/bus_dma.h` 中定义。
5. 传输完成时，`bus_dmamap_unload(tag, map)` 释放映射；`bus_dmamem_free(tag, vaddr, map)` 释放缓冲区；`bus_dma_tag_destroy(tag)` 释放标签（通常在 `detach` 中）。

**常见陷阱。** 忘记 `PREWRITE` 或 `POSTREAD` 同步。在一致性架构上（例如带 IOMMU 的 amd64），驱动程序通常没有它也能工作；在非一致性架构上（某些 ARM 系统），它静默损坏数据。每次都编写同步调用，即使您在宽容平台上开发。

**本书讲解位置。** 第 21 章是全长的专门处理。

**接下来读什么。** `bus_dma(9)` 和 `/usr/src/sys/kern/subr_bus_dma.c`。

### 心理模型：标签、映射、段、同步

图表形状的简短复习。这些部分像这样配合：

```text
+----------------------------------------------------------+
| 设备约束  ->  bus_dma_tag_t     (制作一次)    |
+----------------------------------------------------------+
         |
         v
+----------------------------------------------------------+
| 内核内存中的缓冲区  ->  虚拟地址             |
+----------------------------------------------------------+
         |
         v  bus_dmamap_load()
+----------------------------------------------------------+
| bus_dmamap_t  ->  一个或多个 bus_dma_segment_t 条目  |
|                    (ds_addr, ds_len)                     |
+----------------------------------------------------------+
         |
         v  bus_dmamap_sync(PREWRITE) / PREREAD
+----------------------------------------------------------+
| 设备在 ds_addr 读取或写入内存                 |
+----------------------------------------------------------+
         |
         v  bus_dmamap_sync(POSTWRITE) / POSTREAD
+----------------------------------------------------------+
| CPU 读取缓冲区；内核知道排序有效     |
+----------------------------------------------------------+
```

每个箭头都是一个责任。标签表达设备可以容忍什么。映射表达设备将实际触摸的一组特定段。同步调用括起设备对缓冲区的使用。错过其中任何一个，传输就是未定义的。

### 为什么同步和映射重要

**它是什么。** 两个关注点共同解释为什么 `bus_dma` 不仅仅是地址翻译器。

- **映射。** 设备需要总线地址，而不是虚拟地址。在 IOMMU 系统上，映射是真正的 I/O 虚拟范围；在非 IOMMU 系统上，它是物理地址，如果设备无法到达内存实际存在的位置，可能通过弹跳缓冲区。驱动程序不选择哪个；`bus_dma` 根据标签选择。
- **同步。** CPU 有缓存。设备可能有自己的。内核有排序规则。同步调用将驱动程序的意图（"我即将将此缓冲区交给设备读取"）转换为架构需要的任何缓存刷新、失效或内存屏障。

**这对驱动程序编写者为何重要。** 跳过同步是 DMA 驱动程序中微妙数据损坏的最常见单一来源。代码似乎可以工作，因为 CPU 缓存碰巧被其他活动刷新；然后内核更新稍微改变缓存行为，驱动程序开始失败。每次都编写每个同步调用是唯一可靠的习惯。

**这在 FreeBSD 中如何显示。** 四个同步标志是 `BUS_DMASYNC_PREREAD`、`BUS_DMASYNC_PREWRITE`、`BUS_DMASYNC_POSTREAD` 和 `BUS_DMASYNC_POSTWRITE`（`/usr/src/sys/sys/bus_dma.h` 中的值 1、4、2、8）。"PRE"在设备触摸缓冲区之前运行；"POST"在之后运行。"READ"和"WRITE"两半遵循 `bus_dma(9)` 中的措辞：`PREREAD` 是"设备即将更新主机内存，CPU 稍后将读取写入的内容"；`PREWRITE` 是"CPU 已更新主机内存，设备即将访问它"。在驱动程序通常考虑的方向对中，`PREWRITE` 覆盖 CPU 写入缓冲区然后设备读取的情况，`PREREAD` 覆盖设备写入缓冲区然后 CPU 读取的情况。

**常见陷阱。** 将标志名称当作描述设备自己动作的方式阅读。如果设备即将写入内存，CPU 稍后将读取设备写入的内容，您首先调用 `BUS_DMASYNC_PREREAD`（因为 CPU 稍后将读取此内存），而不是 `BUS_DMASYNC_PREWRITE`。标志对跟踪 CPU 参与的内存操作，而不是设备碰巧在线上移动数据的方向。

**本书讲解位置。** 第 21 章在同步语义上花费大量篇幅，因为它们很容易出错。

**接下来读什么。** `bus_dma(9)` 和 `/usr/src/sys/sys/bus_dma.h` 中的头注释。

### DMA 缓冲区 vs 控制缓冲区

**它是什么。** 保持内存部分和 DMA 部分同步的最终区别。并非每个支持 DMA 的驱动程序中的缓冲区都是 DMA 缓冲区。

- **DMA 缓冲区**是设备读取或写入的缓冲区。它们通过 `bus_dma(9)`：用 `bus_dmamem_alloc` 或 `malloc` 加 `bus_dmamap_load` 分配、加载、同步、卸载。
- **控制缓冲区**是只有 CPU 触摸的缓冲区：环索引、每请求记账、驱动程序检查但硬件不检查的命令结构。它们生活在普通 `malloc(9)` 内存中。

**这对驱动程序编写者为何重要。** 两种缓冲区通常存在于同一驱动程序中，通常在同一 softc 内并排。保持它们直接使代码更容易审查。设备读取的指针应明确标记和分配；设备永远看不到的指针应同样明确标记和分配。混淆两者通常产生难以重现的错误。

**本书讲解位置。** 第 21 章在漫游完整描述符环驱动程序时重新讨论分割。

### DMA 同步快速检查

| 您即将... | 调用这个 |
| :-- | :-- |
| 将缓冲区交给设备供设备*读取* | `bus_dmamap_sync(..., BUS_DMASYNC_PREWRITE)` |

## 结语：如何持续将硬件概念与驱动程序代码联系起来

本附录中的硬件概念不是与驱动程序编写分开的主题。它们中的每一个都出现在代码中。读者将一遍又一遍使用的序列始终相同：

1. 设备存在于物理地址空间中。其寄存器位于物理地址。
2. 内核为驱动程序提供以正确属性映射这些寄存器的 `bus_space` 句柄。
3. 设备在事件上引发中断，驱动程序通过过滤器（快、小、自旋锁安全）和可选的 ithread（较慢、睡眠锁安全）处理它们。
4. 当设备需要移动数据时，驱动程序使用 `bus_dma(9)` 生成设备可以使用的总线地址，并用同步调用括起每次传输。
5. 驱动程序永远不会将内核虚拟指针交给设备，永远不会将 `ds_addr` 当作 CPU 可以直接读取的物理地址。

如果您记住该序列，这里的硬件概念就不再是单独的主题，而成为您正在编写的代码的运行评论。当您即将转换指针并将其交给硬件时，停下来将动作翻译成序列：这在哪个地址空间，谁将读取它，该视图需要什么属性。序列永远不会改变。细节随每个设备系列改变。

三个加强模型的习惯。

第一个是在阅读代码之前阅读真实驱动程序的头注释。大多数 FreeBSD 驱动程序以解释锁定纪律、MSI-X 结构或环布局的块开头。该注释存在是因为代码本身无法传达它。当注释说"接收环使用每队列锁；过滤器确认设备并调度 taskqueue"时，仔细阅读它。它是该驱动程序的硬件到软件桥的压缩地图。

第二个是每次命名总线。当您阅读不熟悉的驱动程序时，找到 `DRIVER_MODULE` 行。该行中的总线名称（`pci`、`usb`、`iicbus`、`spibus`、`acpi`、`simplebus`）在您阅读单个函数之前告诉您应用哪个硬件模型。附加到 `pci` 的驱动程序是面向寄存器和中断驱动的；附加到 `iicbus` 的驱动程序是面向事务和在总线级别轮询的。当您知道哪个总线在它后面时，相同的 C 代码读取方式不同。

第三个是在您的驱动程序笔记旁边保持简短的伴侣表。`examples/appendices/appendix-c-hardware-concepts-for-driver-developers/` 下的文件正是为此设计的。当您打开新数据手册时可以瞥一眼的总线比较速查表。当您怀疑缺少同步时可以漫游的 DMA 心理模型检查表。可以用您今天正在使用的特定标签和句柄对注释的物理与虚拟图。教学在本附录中；应用在这些表中。

这样，书的硬件端就有了一个整合的家。章节持续教学；附录持续命名；示例持续提醒。当读者关闭本附录并打开真实驱动程序时，词汇表就准备好了。
