# 附录 B：系统编程算法与逻辑

## 如何使用本附录

主要章节教授驱动程序作者日常使用的原语。除了这些原语，还有第二层知识，本书假设但不总是命名：几乎在您将阅读的每个驱动程序中都出现的小型数据结构目录、控制流形态和推理模式。虚拟内存子系统中的红黑树、softc 一侧的待处理请求 `STAILQ`、展开 `attach` 例程的 `goto out` 梯子、驱动协议握手的枚举状态开关。这些想法单独看都不难。让它们感觉难的是第一次在真实内核代码中遇到它们，模式是隐式的，名称不在页面上。

本附录就是那个缺失的名称。它是一本简短、实用的野外指南，介绍 FreeBSD 内核和驱动程序代码中重复出现的算法模式，组织方式让您在看到模式时能够识别它，在必须编写时能够决定哪个模式适合。它不是计算机科学教科书，也不是已经教授您将最常使用的特定片段的章节的替代品。它提供的是一个中间层：足够的模式识别让您自信地阅读驱动程序代码，足够的心理建模让空白页面凝视您时选择正确的结构，足够的注意事项意识避免每个驱动程序作者最终都会犯一次的错误。

### 您将在这里找到什么

本附录按问题系列组织，而不是按抽象分类。每个系列收集几个相关模式，具有一致的简短结构：

- **它是什么。** 一两句话。
- **驱动程序为什么使用它。** 内核或驱动程序代码中的具体角色。
- **何时使用。** 简洁的决策提示。
- **何时不使用。** 同一决策的另一面。
- **核心操作。** 您需要肌肉记忆的少数名称。
- **常见陷阱。** 真正花费人们时间的错误。
- **本书讲解位置。** 指回主要章节的指针。
- **接下来读什么。** 手册页或现实的源代码树示例。

当模式有稳定的接口的真实 FreeBSD 实现时，条目直接使用这些名称。当模式是通用的，条目仍然基于 FreeBSD 驱动程序实际如何应用它。

### 本附录不是什么

它不是算法的第一课。如果您从未遇到过链表或状态机，本附录会感觉太压缩；先读第 4 和第 5 章，然后再回来。它也不是深度理论参考。您在这里几乎找不到渐近分析，因为驱动程序作者很少在纸上选择 O(log n) 和 O(n)。他们选择的模式，权衡主要是关于锁定、缓存行为和不变量。最后，它不是附录 A（涵盖 API）、附录 C（涵盖硬件概念）或附录 E（涵盖内核子系统）的替代品。如果您想要的模式指向那里，去那里然后再回来。

## 内核中的数据结构

没有状态的驱动程序很少见。几乎每个非平凡驱动程序都保存某种集合：待处理请求、打开句柄、每 CPU 统计、每通道配置、每客户端上下文。问题是哪种集合形态匹配访问模式。内核提供了一个头文件数据结构的小型工具包，您应该默认使用它而不是自己滚动。每个都解决不同的问题；混淆它们是驱动程序设计中最常见的类别错误之一。

### `<sys/queue.h>` 系列

**它是什么。** 一系列侵入式链表宏。您定义头和条目，将条目嵌入元素结构中，宏为您提供插入、删除和遍历，无需为列表节点进行堆分配。

头文件定义了四种风格：

- `SLIST` 是单向链表。最便宜，仅向前，任意元素的 O(n) 删除。
- `LIST` 是双向链表。向前和向后遍历，任意元素的 O(1) 删除。
- `STAILQ` 是单向尾队列。仅向前，但在尾部快速插入。经典 FIFO。
- `TAILQ` 是双向尾队列。您可以合理想要的每个操作，代价是每个元素两个指针。

**驱动程序为什么使用它们。** 树中几乎每个每驱动程序集合都是其中之一。待处理命令、打开文件句柄、注册回调、子设备。它们可预测、在列表级别无分配，宏使代码读起来几乎像伪代码。

**何时使用**当您有一个您已经自己分配的对象集合，并且您想要标准列表操作而不发明它们。侵入式设计意味着对象可以同时存在于多个列表中；那是特性，不是错误。

**何时不使用**当您需要按键有序查找时。列表是 O(n) 搜索。改用树或哈希表。

**在四个中选择。**

- 除非有理由不这样做，否则从 `TAILQ` 开始。它是最常见和最灵活的。
- 当您知道只像队列一样推送和弹出时降到 `STAILQ`。您每个元素节省一个指针。
- 当您想要双向链接行为但不需要尾指针时使用 `LIST`。例如，独立元素的无序集合。
- 当每个字节都很重要且列表很短或遍历本来是线性的时候使用 `SLIST`。热结构中的快速路径有时证明它是合理的。

**核心操作。** 宏在风格之间是统一的。插入：`TAILQ_INSERT_HEAD`、`TAILQ_INSERT_TAIL`、`TAILQ_INSERT_BEFORE`、`TAILQ_INSERT_AFTER`。删除：`TAILQ_REMOVE`。遍历：`TAILQ_FOREACH`，以及当您可能删除当前元素时的 `_SAFE` 变体。相同操作存在 `LIST_`、`STAILQ_` 和 `SLIST_` 前缀。字段存储是结构中嵌入的 `TAILQ_ENTRY(type)`。

**常见陷阱。**

- 在删除元素时使用非 `_SAFE` foreach。普通 `TAILQ_FOREACH(var, head, field)` 展开为在每次迭代结束时读取 `TAILQ_NEXT(var, field)` 的 `for` 循环。如果主体释放了 `var`，下一步解引用已释放的内存。`TAILQ_FOREACH_SAFE(var, head, field, tmp)` 在主体运行之前将下一个指针缓存到 `tmp` 中，并且当循环主体可能删除 `var` 时总是正确的答案。
- 忘记列表是侵入式的。同一元素不能通过同一条目字段一次位于两个 `TAILQ` 中。如果您需要那样，定义两个具有不同名称的 `TAILQ_ENTRY` 字段。
- 在同一头上混合风格。在 `TAILQ_HEAD` 上的 `SLIST_INSERT_HEAD` 可以编译但产生静默损坏。保持宏与头类型一致。

**本书讲解位置。** 在第 5 章中作为内核 C 方言的一部分简要介绍，并在本书后面部分的驱动程序需要内部集合时重新讨论。

**接下来读什么。** `queue(3)` 获取完整宏目录，以及任何保持待处理工作的真实驱动程序。`/usr/src/sys/net/if_tuntap.c` 大量使用这些习语。

### 通过 `<sys/tree.h>` 的红黑树

**它是什么。** 一组在您的类型中内联生成自平衡二叉搜索树的宏。FreeBSD 的实现是秩平衡的，但它公开历史 `RB_` 前缀，并且在所有实际目的中表现得像红黑树。

**驱动程序为什么使用它们。** 当您需要按键 O(log n) 有序查找且键不是小整数时，红黑树是默认选择。虚拟内存映射、某些子系统中的调度器运行队列，以及命名对象的每子系统注册表都使用此头文件。

**何时使用**当您有一个按键索引的对象集合、集合增长到数百或数千条目，并且您关心有序遍历以及查找时。

**何时不使用**当集合很小（十几个元素）或键空间小且密集时。在这两种情况下，缓存热的数组上的线性扫描通常胜过树，因为指针追逐的常数因子远高于顺序读取。

**核心操作。** `RB_HEAD(name, type)` 定义头类型，`RB_ENTRY(type)` 定义嵌入节点字段，`RB_PROTOTYPE` 和 `RB_GENERATE` 实例化函数系列（插入、删除、查找、最小、最大、下一个、上一个、foreach）。您还编写一个比较函数，像 `strcmp(3)` 那样返回负数、零或正数。同一头文件中的变体 `SPLAY_` 系列在引用局部性重要时提供伸展树。

**常见陷阱。**

- 在头文件中调用 `RB_GENERATE`。这会在链接时产生多个定义。将 `RB_GENERATE` 保持在恰好一个 `.c` 文件中，将 `RB_PROTOTYPE` 保持在声明类型的头文件中。
- 忘记树是侵入式的。元素不能通过同一条目存在于两棵树中，从错误的树删除会静默损坏两者。
- 在插入期间持有指针。平衡旋转不移动节点，但比较器必须一致；如果您的键在节点位于树中时更改，您已使搜索结构无效，每个后续查找都是未定义的。

**本书讲解位置。** 在本书中不作为专门模式教授。读者通过示例在附录 E 漫游虚拟内存子系统以及第 7 部分的高级驱动程序需要有序查找时遇到它。

**接下来读什么。** `tree(3)` 获取宏目录，以及任何保持命名事物有序集合的子系统。

### 通过 `<net/radix.h>` 的基数树

**它是什么。** 内核特定的基数（或 Patricia）树，主要由路由表用于将地址与前缀匹配。基数树与红黑树的不同之处在于它根据前缀长度而不是总顺序键，这正是最长前缀匹配所需要的。头文件位于 `/usr/src/sys/net/radix.h`，与网络代码的其余部分一起，而不是 `/usr/src/sys/sys/` 下。

**驱动程序为什么使用它们。** 很少直接使用。基数树存在是因为网络栈需要它，并且机制专用于该用途。驱动程序几乎从不创建自己的基数树。您可能作为网络相关代码中的调用者遇到一个，或者您可能在兼容层中看到相关的 LinuxKPI 基数树 shim。

**何时使用**当您真正需要在可变长度键上进行前缀匹配时。路由、ACL 表、基于前缀的策略。

**何时不使用**当您实际需要的是按键精确查找时。改用 `<sys/tree.h>`。

**核心操作。** `rn_inithead`、`rn_addroute`、`rn_delete`、`rn_lookup`、`rn_match`、`rn_walktree`。这些足够专业化，路由子系统是权威参考。

**常见陷阱。** 头文件和例程与路由表抽象紧密耦合。尝试为非路由数据结构重用基数树几乎总是一个错误；抽象会以不舒服的方式泄漏。`/usr/src/sys/compat/linuxkpi/common/include/linux/radix-tree.h` 中的 LinuxKPI 基数树是不同的结构，不应与本机树混淆。

**本书讲解位置。** 仅在附录 E 中与路由表一起提及。本附录命名它以便您可以在路过时识别它。

**接下来读什么。** `/usr/src/sys/net/` 下的路由表代码。

### 通过 `<sys/bitstring.h>` 的位图和位串

**它是什么。** 由机器字数组支持的紧凑位集抽象。您分配 N 位的位串，然后设置、清除、测试和扫描单个位。

**驱动程序为什么使用它们。** 任何时候您有一个密集的有限宇宙并希望跟踪每个元素的布尔状态。固定硬件描述符池的空闲槽位图、中断分配、次设备号、每通道启用标志。位图每个元素使用一位，并支持高效的首个空闲搜索。

**何时使用**当宇宙是密集的、预先知道的，并且您跟踪每个元素的布尔状态时。当您需要"找到第一个空闲槽"或"设置了多少个"时，位图特别有效。

**何时不使用**当宇宙稀疏时。32 位键空间中几千个已用槽号的集合不属于位图；使用哈希表或树。

**核心操作。** `bit_alloc`（内核分配器变体）和 `bit_decl`（栈）、`bit_set`、`bit_clear`、`bit_test`、`bit_nset`、`bit_nclear` 用于范围、`bit_ffs` 用于首个已设置、`bit_ffc` 用于首个已清除、`bit_foreach` 用于迭代。大小用 `bitstr_size(nbits)` 计算。

**常见陷阱。**

- 忘记位索引是基于零的，以及 `bit_ffs` / `bit_ffc` 宏不返回值；它们将找到的索引写入 `*_resultp` 并在没有匹配时将其设置为 `-1`。在使用索引之前始终检查 `if (result < 0)`。
- 并发修改下的竞争条件。`bitstring(3)` 不是原子的；如果两个上下文可以并发设置或清除位，它们需要锁或协调移交。
- 分配错误的大小。使用 `bitstr_size(nbits)` 而不是手动进行除法。

**本书讲解位置。** 未直接教授。需要空闲槽位图的驱动程序将使用此头文件，此条目是指针。

**接下来读什么。** `bitstring(3)`。

### 使用 `hashinit(9)` 的哈希表

**它是什么。** 从 `<sys/queue.h>` 列表构建哈希表的小型辅助模式。`hashinit(9)` 分配 2 的幂大小的 `LIST_HEAD` 桶数组，并返回数组和掩码。您的代码将键哈希为 32 位值，与掩码进行 AND 运算，并遍历结果桶。

**驱动程序为什么使用它们。** 当您需要按键无序查找且集合足够大以至于线性扫描太慢时。文件系统名称缓存、每进程打开文件表，以及任何维护可哈希标识符键的注册表的驱动程序都使用此形态。

**何时使用**当集合大、查找频繁、排序不重要时。哈希表在平均情况查找上胜过树，并且更容易粗粒度锁定（每桶一个锁，或者如果写入罕见则整个表一个锁）。

**何时不使用**当集合很小（树或列表更简单）或您需要有序遍历时（`RB_FOREACH` 是答案，不是哈希）。

**核心操作。** `void *hashinit(int elements, struct malloc_type *type, u_long *hashmask);` 返回桶数组并将掩码写入 `*hashmask`。每个桶是 `LIST_HEAD`，所以您使用通常的 `LIST_` 宏插入和迭代。`hashdestroy` 拆除它。`hashinit_flags` 接受额外的标志字（`HASH_WAITOK` 或 `HASH_NOWAIT`），当您需要非睡眠变体时；`hashinit` 本身等效于 `HASH_WAITOK`。`phashinit` 和 `phashinit_flags` 当您更喜欢模运算而不是掩码时为您提供素数大小的表，并通过输出指针写入选择的大小（不是掩码）。小型 `<sys/hash.h>` 头文件提供基本哈希函数如 `hash32_buf` 和 `hash32_str`，以及 `jenkins_hash` 和 `murmur3_32_hash`；Fowler-Noll-Vo 系列位于 `<sys/fnv_hash.h>` 中。

**常见陷阱。**

- 直接哈希指针。指针低位通常对齐，产生糟糕的桶分布。对内容或移位指针使用适当的哈希函数。
- 假设 `hashinit` 使用您请求的大小。它向下舍入到最接近的 2 的幂并返回该大小的掩码。请求 100 个元素产生 64 桶表，而不是 100。
- 忘记锁的故事。裸哈希表不是线程安全的。预先决定您想要单个锁、每桶锁还是更花哨的东西，并在 softc 中记录它。

**本书讲解位置。** 在维护每客户端注册表的第 7 部分驱动程序中简要引用。此条目为您需要自己构建时提供脚手架。

**接下来读什么。** `hashinit(9)`，以及 `/usr/src/sys/kern/subr_hash.c` 中的实现。

## 循环和环形缓冲区

环出现在调解硬件和软件的驱动程序中无处不在。NIC 分发包传输的描述符环。UART 在接收它们的中断和消耗它们的进程之间缓冲字符。命令驱动程序将请求批处理到环中供硬件引擎拾取。形状总是相同的：固定大小数组加上回绕的生产者索引和消费者索引。变化的是谁与谁并发、使用哪些原语以及环是否需要无锁。

### 单生产者单消费者环形缓冲区

**它是什么。** 固定大小的循环缓冲区，其中恰好一个上下文生产，恰好一个上下文消费。生产者在写入后推进 `head`（或 `tail`）索引；消费者在读取后推进 `tail`（或 `head`）索引；两者都不会写入对方的索引。

**驱动程序为什么使用它们。** 中断处理程序生产数据（接收的字符、完成事件），线程消费它。那是典型的 SPSC 形态。它构造上是无锁的，因为没有两个上下文修改同一变量。

**何时使用**当您恰好有一个生产者和恰好一个消费者，并且您想避免锁时。字符设备环形缓冲区是经典案例。

**何时不使用**当多个线程可以生产，或多个线程可以消费时。添加第二个生产者破坏不变量；您需要锁，或需要 `buf_ring(9)`。

**核心操作和不变量。** 通常的不变量是：

- `head` 仅由生产者推进；`tail` 仅由消费者推进。
- `(head - tail) mod capacity` 是已占用槽的数量。
- `capacity - (head - tail) mod capacity` 是空闲槽的数量（减一，取决于编码）。
- 当 `head == tail` 时缓冲区为空。
- 当 `(head + 1) mod capacity == tail` 时缓冲区已满。牺牲的一个槽是您区分满和空的方式。

替代编码使用单独的 `count` 变量或使用带掩码的容量模的自由运行索引（`head & mask`、`tail & mask`）。自由运行索引使满与空成为有符号比较，并避免牺牲的槽。

**常见陷阱。**

- 没有一个槽约定就混淆满和空。每个环设计都必须有明确的答案；选择一个并写在文件顶部。
- 没有正确排序内存。在现代 CPU 上，写入可能无序变得可见。生产者必须确保缓冲区内容在更新的 `head` 之前对消费者可见。在内核中，这意味着显式内存屏障（生产者上的 `atomic_thread_fence_rel`，消费者上的 `atomic_thread_fence_acq`），或使用原子存储释放和加载获取变体。
- 当代码未强制执行时假设 `capacity` 是 2 的幂。掩码需要 2 的幂；模不需要。代码和不变量必须一致。

**本书讲解位置。** 第 10 章为 `myfirst` 驱动程序的字符设备构建循环缓冲区，并从第一原则教授头/尾/满/空纪律。在第 11 和 12 章中当环遇到同步时重新讨论。

**接下来读什么。** 伴侣 `examples/` 树中第 10 章的循环缓冲区示例，以及任何使用环桥接中断和进程上下文的小型字符设备驱动程序。

### 用于 MPMC 环的 `buf_ring(9)`

**它是什么。** FreeBSD 的内核环形缓冲区库。`buf_ring` 提供多生产者多消费者环，具有无锁入队和单消费者及多消费者出队路径。它被 iflib、网络驱动程序和树中其他地方大量使用。

<!-- remove after Chapter 11 revision is durable -->
参见第 11 章了解介绍，参见 `buf_ring(9)` 手册页了解权威并发契约。

**驱动程序为什么使用它们。** 当多个上下文可以入队（多个 CPU 将包推送到同一传输队列）或多个上下文可以出队（多个工作线程排空同一工作队列）时，您无法逃脱 SPSC 不变量。`buf_ring` 为您执行原子比较并交换工作，并将内存排序细节隐藏在稳定接口后面。

**何时使用**当您有多个生产者或多个消费者，并且您想要无锁路径时。网络相关驱动程序是典型情况。

**何时不使用**当您真正有一个生产者和一个消费者时。SPSC 环更简单、更便宜、更容易推理。

**核心操作。** 设置和拆除时的 `buf_ring_alloc(count, type, flags, lock)` 和 `buf_ring_free`。`buf_ring_enqueue(br, buf)` 入队，成功时返回 `0` 或满时返回 `ENOBUFS`。`buf_ring_dequeue_mc(br)` 在多个消费者下安全出队；当调用者保证单消费者语义时 `buf_ring_dequeue_sc(br)` 更快。`buf_ring_peek`、`buf_ring_count`、`buf_ring_empty`、`buf_ring_full` 为您提供不移除的检查。

**常见陷阱。**

- 一次从多个消费者调用 `buf_ring_dequeue_sc`。SC 路径假设只有一个消费者在运行；违反它会静默损坏尾索引。
- 将 `ENOBUFS` 视为硬错误。`buf_ring_enqueue` 返回 `ENOBUFS` 是普通的反压信号；调用者必须重试、丢弃或在较慢路径上排队。
- 忘记 `buf_ring_count` 和 `buf_ring_empty` 是建议性的。另一个 CPU 可能在您检查后立即入队或出队。围绕最终一致性而不是瞬时真理设计调用者。

**本书讲解位置。** 在第 10 章中命名为教学 SPSC 环的生产级对应物。当驱动程序示例需要真实环语义时在第 5 部分网络章节中使用。

**接下来读什么。** `buf_ring(9)`，以及 `/usr/src/sys/sys/buf_ring.h` 中的头文件本身。对于真实用户，浏览 iflib 或 `/usr/src/sys/dev/` 下的任何高性能 NIC 驱动程序。

### 回绕算术和满与空问题

每个环都必须区分满和空。两种状态满足 `head == tail` 或相距一个槽，您选择的编码决定使用哪个比较。两个常见约定值得命名，以便您可以在代码中发现它们。

- **牺牲槽。** 环保存 `capacity - 1` 个可用槽。空是 `head == tail`。满是 `(head + 1) mod capacity == tail`。简单、便宜、适用于任何容量。
- **自由运行索引。** `head` 和 `tail` 计算自环创建以来的每个操作，永不回绕。数组访问的模索引是 `head & mask`（这要求 `capacity` 是 2 的幂）。占用计数是 `head - tail` 作为无符号算术；满是 `occupied == capacity`；空是 `occupied == 0`。优点是您永远不会牺牲槽；代价是您必须小心编写 `(uint32_t)(head - tail)`。

两种约定都是正确的。选择一个，使其成为文件顶部的一句话注释。知道环形缓冲区的读者会知道要寻找什么；不知道的读者会有注释。

## 排序和搜索

驱动程序排序和搜索的频率低于列表、环或树。但当它们这样做时，小型工具包几乎涵盖每种情况，知道哪个工具合适可以节省时间。

### 内核内 `qsort` 和 `bsearch`

**它是什么。** 内核提供标准库中熟悉的 `qsort` 和 `bsearch`，在 `/usr/src/sys/sys/libkern.h` 中声明。线程安全变体 `qsort_r` 将用户上下文指针传递给比较器。

**何时使用**当您需要一次或偶尔在原位排序数组，或需要通过排序数组进行二分搜索时。设备 ID 表、兼容性匹配和排序快照都适合这里。

**何时不使用**当集合是长期存在的且频繁更改时。在这种情况下，您想要树，而不是排序搜索对。还要避免在热路径上使用 `qsort`：比较器的函数调用开销对于小数组占主导地位。

**核心操作。** `qsort(base, nmemb, size, cmp)` 和 `qsort_r(base, nmemb, size, cmp, thunk)` 用于排序；`bsearch(key, base, nmemb, size, cmp)` 用于搜索。比较器以 `strcmp(3)` 风格返回负数、零或正数。这三个都在 `libkern.h` 中声明，并拥有第 3 节手册页（`qsort(3)`、`bsearch(3)`）。

**常见陷阱。**

- 不稳定的比较。`qsort` 不保证稳定。如果相等键之间的顺序重要，向比较器添加决胜局。
- 当字段很宽时传递使用减法测试的比较器（`a->x - b->x`）。整数溢出可能使负数看起来像正数。使用显式比较（`a->x < b->x ? -1 : a->x > b->x ? 1 : 0`）。

**本书讲解位置。** 在第 4 章（C 之旅）和第 7 部分中引用，其中特定驱动程序在附加时对表进行一次排序。

**接下来读什么。** `qsort(3)`、`bsearch(3)`。

### 设备 ID 表上的二分搜索

**它是什么。** `bsearch` 的专门应用。驱动程序维护 `(vendor, device)` 或兼容字符串键的排序表，探测路径进行二分搜索以决定硬件是否是它支持的。

**驱动程序为什么使用它。** PCI 驱动程序可能匹配数十个设备 ID。在附加时扫描表是可以的；线性地进行也行。当表增长时使用 `bsearch` 进行是一个小型、可读、快速的改进。

**何时使用**当 ID 表长于少数条目且按键排序很容易时。您可以通过静态声明排序表在编译时排序，或在模块加载时使用 `qsort` 排序一次。

**何时不使用**当匹配不是简单的键比较时。PCI 匹配有时涉及子类和编程接口字段；直接表达匹配逻辑，而不是将其扭曲到比较器中。

**常见陷阱。** 在运行时排序然后依赖排序进行 ABI 兼容性。如果另一段代码按索引读取表，更改顺序会破坏它。当您可以时将表声明为 `const` 并在构建时排序。

**本书讲解位置。** 在第 18 章（PCI）和第 6 章的设备匹配讨论中引用。

**接下来读什么。** 任何 PCI 驱动程序的 ID 表和探测路径。`/usr/src/sys/dev/uart/uart_bus_pci.c` 是一个可读的示例。

### 线性搜索何时获胜

**它是什么。** 提醒对于小型集合，缓存热数组上的线性搜索是您可以编写的最快查找。CPU 缓存奖励顺序读取并惩罚指针追逐。

**何时使用**当集合少于几十个条目时。在实践中，对二十个项目进行线性扫描比任何树查找都快，因为树遍历需要多次缓存未命中，而线性扫描只需要一次。

**何时不使用**当集合无限制增长时，或当热路径延迟要求严格且最坏情况比平均更重要时。

**常见陷阱。** 优化错误的一面。在测量之前不要引入树来治愈"线性搜索很慢"。真正的成本通常在其他地方。

## 状态机和协议处理

许多驱动程序是伪装的状态机。USB 设备探测、附加、空闲、恢复、挂起、分离。网络驱动程序经历链路断开、链路接通、协商、稳定状态、错误恢复。当状态转换复杂时，使状态显式立即得到回报。当它们简单时，枚举和开关通常可以完成工作。

### 显式状态枚举 vs 隐式标志位

**它是什么。** 表示状态的两种方式。枚举说"此设备正好处于 N 个状态之一"。标志字说"此设备具有 N 个独立布尔值的任意组合"。

**何时使用枚举**当条件互斥且命名转换重要时。链路是 `LINK_DOWN`、`LINK_NEGOTIATING` 或 `LINK_UP` 之一。命令是 `CMD_IDLE`、`CMD_PENDING`、`CMD_RUNNING` 或 `CMD_DONE` 之一。枚举强制每个转换都是显式的；您不会意外地同时处于两个状态。

**何时使用标志位**当条件独立且可以组合时。"中断启用"、"自动协商允许"、"混杂模式"。在同一标志字中混合互斥和独立条件是常见错误；将它们分成枚举用于前者，标志字用于后者。

**常见陷阱。**

- 将状态机编码为位标志，然后必须决定"同时设置 `CMD_PENDING` 和 `CMD_DONE`"意味着什么。那永远不会意味着任何有用的东西。切换到枚举。
- 将独立布尔值编码为单个打包状态。您将糟糕地重新发明按位 OR。

### 基于开关的 FSM

**它是什么。** 最简单的显式状态机：状态的枚举、接受事件的函数中的开关，以及每个情况更新状态并执行操作的主体。

**何时使用**当状态空间小（可能十个状态）且转换浅时。开关易于阅读、易于扩展，如果您启用正确的警告标志，编译器会捕获缺失的情况。

**何时不使用**当开关增长到数百行或同一转换逻辑在多个状态中重复时。那就是转向表驱动 FSM 的信号。

**常见陷阱。**

- 忘记状态。使用 `-Wswitch-enum` 编译，以便编译器告诉您。
- 在开关内部进行非平凡工作。保持开关作为调度器；将真正的工作委托给命名辅助函数。

**本书讲解位置。** 该模式自然出现在许多驱动程序示例中。第 5 章以其最简单的形式介绍它。

### 表驱动 FSM

**它是什么。** 状态机表示为按 `(state, event)` 索引的二维表。每个单元格保存下一状态，通常还有在该转换上运行的操作的函数指针。

**何时使用**当 FSM 有许多状态和许多事件，并且您希望转换逻辑是数据而不是代码时。可视化矩阵通常比追逐巨型开关更容易。协议栈、总线枚举逻辑和挂起恢复状态机是经典用途。

**何时不使用**当 FSM 小时。五状态开关比五乘五矩阵更容易阅读。

**核心操作。** 定义状态枚举、事件枚举和包含 `next_state` 和 `action_fn` 的转换结构。将表声明为静态常量二维数组。驱动程序用 `table[current_state][event]` 步进机器。

**常见陷阱。**

- 没有检查的空操作指针。要么表必须密集（每个单元格都是有效转换），要么调度器必须将空操作视为"忽略"，并且约定必须一致。
- 没有明确纪律的情况下在操作函数中嵌入锁。状态转换是临界区；决定调用者持有锁还是调度器获取它，并坚持这一点。

### 函数指针分发

**它是什么。** 完整状态表的较轻替代方案。驱动程序的当前"模式"是一组处理函数的指针；切换状态就像重新分配指针一样简单。

**何时使用**当"状态"实际上是一组驱动程序根据模式不同执行的方法时。链路接通时的接收路径与仍在协商的链路不同。正在引导其固件的设备与已在运行的设备以不同方式分发命令。

**何时不使用**当状态转换是临时的或众多时。函数指针交换是粗粒度操作；为少数命名良好的模式保留它使代码诚实。

**常见陷阱。**

- 与交换竞争。如果一个 CPU 通过指针调用而另一个正在替换它，调用可能分派到陈旧代码。交换需要排序。在内核中，这通常意味着锁或通过 `epoch(9)` 进行旧指针引用的任何状态结构的 RCU 风格延迟释放。
- 孤立资源。当您交换函数指针表时，确保旧模式拥有的资源干净地移交或释放。交换本身不会释放任何东西。

### 重入和部分完成

**它是什么。** 状态机设计关注而不是数据结构。真实驱动程序以交错顺序接收事件：驱动程序正在处理 ioctl 中途时中断触发；触发它的命令仍在发布时完成到达；分离与打开竞争。状态机必须容忍这一点。

**设计原则。**

- 使每个转换成为单个原子更新。读取当前状态、计算下一状态、用锁或原子提交。在提交状态之前不要执行操作；如果崩溃或提前返回，状态必须匹配现实。
- 将"做什么"与"谁运行它"分开。状态转换决定需要发生什么；taskqueue 或工作线程实际执行工作。这样转换可以在昂贵工作开始之前完成，第二个调用者看到一致状态。
- 为每个需要时间的转换有一个"瞬态"状态。正在发送的命令既不是 `IDLE` 也不是 `RUNNING`；它是 `SENDING`。正在关闭的链路既不是 `UP` 也不是 `DOWN`；它是 `GOING_DOWN`。瞬态状态为并发调用者提供等待或重试的东西。

**常见陷阱。** 假设事件是序列化的。它们不是。中断可以随时到达，分离可以与打开竞争，用户空间关闭可以与设备内部拆除竞争。构建状态机，使得从每个状态处理每个事件，即使处理只是"拒绝"或"推迟"。

**本书讲解位置。** 在第 11 章（同步）、第 14 章（深入中断）和第 5 部分中当真实网络驱动程序遇到这些情况时重新讨论。

## 错误处理模式

正确处理快乐路径但错误处理错误的驱动程序将损坏内存、泄漏资源或使内核 panic。驱动程序代码中的良好错误处理几乎总是关于结构，而不是聪明。内核有一个强大的、统一的习语，本节命名该习语。

### `goto out` 习语

**它是什么。** 规范的内核清理模式。在函数顶部，将每个资源声明为 `NULL`。在每次获取时，检查结果；失败时，`goto fail_N`，其中 `fail_N` 是一个标签，释放到目前为止获取的确切资源。在函数末尾，成功路径返回；清理梯子以获取的反向顺序运行，并从每个标签下落到最终返回。

**驱动程序为什么使用它。** 因为它有效。梯子使不可能忘记释放，释放顺序在视觉上与获取顺序的反向相同。该模式在 FreeBSD 树中无处不在。

**何时使用**当函数获取多个资源时。三个通常是梯子比嵌套 `if` 更干净的阈值。

**何时不使用**当函数简单时。单个获取、单个释放和单个错误路径不需要梯子；具有两个返回的直接函数更清晰。

**形态。**

```c
int
my_attach(device_t dev)
{
    struct my_softc *sc;
    struct resource *mem = NULL;
    void *ih = NULL;
    int error, rid;

    sc = device_get_softc(dev);
    sc->dev = dev;

    rid = 0;
    mem = bus_alloc_resource_any(dev, SYS_RES_MEMORY, &rid, RF_ACTIVE);
    if (mem == NULL) {
        device_printf(dev, "cannot allocate memory window\n");
        error = ENXIO;
        goto fail_mem;
    }

    error = bus_setup_intr(dev, sc->irq, INTR_TYPE_MISC | INTR_MPSAFE,
        NULL, my_intr, sc, &ih);
    if (error != 0) {
        device_printf(dev, "cannot setup interrupt: %d\n", error);
        goto fail_intr;
    }

    sc->mem = mem;
    sc->ih  = ih;
    return (0);

fail_intr:
    bus_release_resource(dev, SYS_RES_MEMORY, rid, mem);
fail_mem:
    return (error);
}
```

形状是重要的。每个标签释放恰好在其与函数开始之间获取的资源。下落是有意的；标签按人类从函数顶部阅读它们的顺序堆叠。

**变体。**

- 带`条件检查`的单个 `out:` 标签（`if (mem != NULL) bus_release_resource(...)`）适用于短函数。对于长函数，编号标签变体更容易审计，因为它直接编码顺序。
- 某些代码命名标签为 `err_mem`、`err_intr` 而不是 `fail_mem`、`fail_intr`。前缀不重要；文件内的一致性重要。

**常见陷阱。**

- 以错误顺序释放。始终以分配的反向顺序释放。梯子的视觉顺序是提醒，不是证明；添加新资源时查看标签。
- 将指针初始化为 `NULL` 以外的其他东西。如果梯子有条件释放，条件必须可靠。未初始化的指针是未定义的；释放未初始化的内存是 panic。
- 在成功路径上以部分状态返回。末尾的成功路径应该将完全所有权移交给调用者或返回错误并展开。没有第三种选择。

**本书讲解位置。** 第 5 章明确介绍该模式，本书从第 7 章开始一致使用它。

**接下来读什么。** `/usr/src/sys/dev/` 下的真实附加函数。几乎任何驱动程序都显示该模式在行动。

### 返回码约定

**它是什么。** 内核约定：整数返回码，其中 `0` 表示成功，正 errno 值表示失败。`1` 不是错误；它是一个真值返回，在 `EPERM`（也是 `1`）旁边会令人困惑。成功时始终返回 `0`。

**核心约定。**

- 成功时 `0`，失败时正 errno（`EINVAL`、`ENOMEM`、`ENXIO`、`EIO`、`EBUSY`、`EAGAIN`）。
- 切勿返回负错误码。那是 Linux 约定，不匹配 FreeBSD 的其余部分。
- 向上原样传播错误码，除非您可以更具体。
- 当函数返回指针时，约定是失败时返回 `NULL`，如果调用者需要区分原因，则设置单独的 `int`。
- 探测例程遵循不同的规则：成功时返回 `BUS_PROBE_DEFAULT` 及其类似物，不匹配时返回 `ENXIO`。完整列表见附录 A。

**常见陷阱。**

- 从内核函数返回 `-1`。那是用户空间习语；不要在内核代码中这样做。
- 重载单个返回值以表示"错误"和"资源计数"。如果需要两者，请使用单独的输出指针参数。
- 吞掉错误。来自低级调用的错误码是契约；要么处理它，要么传播它。失败的 `copyin` 后返回 `0` 是内核 panic 的诞生方式。

**本书讲解位置。** 再次是第 5 章，与清理梯子一起。

### 资源获取和释放顺序

**它是什么。** 纪律说：您在 attach 中获取的每个资源必须在 detach 中以反向顺序释放。您获取的每个锁必须沿每个退出路径释放。您添加的每个引用必须在丢弃最后一个之前释放。上面的清理梯子是这个原则的局部表达；整个驱动程序是全局表达。

**核心原则。**

- Attach 以顺序 A、B、C 获取。Detach 以顺序 C、B、A 释放。两者是镜像。
- attach 内部的每个错误路径以反向展开迄今为止已获取的内容。那就是梯子。
- 锁保持尽可能短的时间，沿每个退出释放，包括错误路径。展开内存但忘记互斥锁的 `goto` 梯子仍然是错误的。
- 所有权应该是局部的。一个函数获取；同一函数释放，或者它显式地将所有权移交给调用者。"泄漏"所有权到参数副作用的函数几乎总是等待发生的错误。

**本书讲解位置。** 第 7 章为第一个驱动程序介绍镜像纪律，之后的每个驱动程序章节重复它。

## 并发模式

两个线程触摸相同状态是内核错误诞生的地方。附录 A 中的同步原语是工具；本节收集很好使用这些工具的模式。这里的目标是模式识别，而不是完整的并发课程。

### 使用条件变量的生产者-消费者

**它是什么。** 一个上下文生产数据（填充缓冲区、发布命令、接收字符），另一个消费它。由互斥锁保护的共享状态，以及当状态为空时消费者等待的条件变量。

**心理模型。**

- 互斥锁保护共享状态。这是不可商量的。
- 消费者在互斥锁下检查状态。如果没有什么可做的，它通过 `cv_wait` 在条件变量上睡眠。`cv_wait` 原子地在睡眠时放下互斥锁并在返回时重新获取它。
- 生产者也在修改状态时持有互斥锁，然后发出条件变量信号。
- 消费者在唤醒后重新检查状态，因为虚假唤醒和共享信号是可能的。检查始终是循环，不是 `if`。

**形态。**

```c
/* 生产者。 */
mtx_lock(&sc->lock);
put_into_buffer(sc, item);
cv_signal(&sc->cv);
mtx_unlock(&sc->lock);

/* 消费者。 */
mtx_lock(&sc->lock);
while (buffer_is_empty(sc))
    cv_wait(&sc->cv, &sc->lock);
item = take_from_buffer(sc);
mtx_unlock(&sc->lock);
```

**常见陷阱。**

- 在 `cv_wait` 周围使用 `if` 而不是 `while`。虚假唤醒是真实的；始终重新检查。
- 不持有锁就发出信号。FreeBSD 允许它，但如果在持有锁时发出信号，几乎总是更容易推理。
- 当您意思是 `cv_broadcast` 时使用 `cv_signal`。`cv_signal` 唤醒恰好一个等待者；如果多个等待者等待不同的前置条件，只有一个会被解除阻塞，可能是错误的那个。
- 在自旋锁下睡眠。`cv_wait` 是可睡眠的；您需要普通互斥锁（`MTX_DEF`），而不是自旋互斥锁（`MTX_SPIN`）。

**本书讲解位置。** 第 10 章围绕循环缓冲区隐式构建模式；第 11 和 12 章形式化它。

### 读者-写者：`rmlock(9)` vs `sx(9)`

**它是什么。** 两种不同的读写锁，具有不同的代价配置。

- `sx(9)` 是可睡眠的共享-独占锁。读者阻塞写者；写者阻塞读者。双方都可能在临界区内部睡眠。
- `rmlock(9)` 是读多锁，具有极便宜的读者路径（本质上是每 CPU 指示器）和昂贵得多的写者路径（写者必须等待每个读者排空）。

**在它们之间选择。**

- 如果读取频繁、写入罕见、且读者路径不得睡眠，选择 `rmlock`。
- 如果读取频繁、写入偶尔、且读者路径需要睡眠（例如，它调用 `copyout`），选择 `sx`。
- 如果读写大致平衡或临界区短，两者都不对。使用普通互斥锁。

**核心操作。**

- `sx`：`sx_slock`、`sx_sunlock` 用于共享；`sx_xlock`、`sx_xunlock` 用于独占；`sx_try_slock`、`sx_try_xlock`。
- `rmlock`：`rm_rlock(rm, tracker)`、`rm_runlock(rm, tracker)` 用于读者（每个读者需要自己的 `struct rm_priotracker`，通常在栈上）；`rm_wlock(rm)`、`rm_wunlock(rm)` 用于写者。

**常见陷阱。**

- 为平衡工作负载使用 `rmlock`。写者路径确实很慢；仅当读取压倒写入时使用它。
- 让 `rmlock` 读者睡眠而未使用 `RM_SLEEPABLE` 初始化。默认 `rmlock` 禁止在读路径中睡眠。
- 将共享锁升级为独占锁。`sx` 和 `rmlock` 都不支持无锁升级。放下共享锁，获取独占锁，并重新检查状态。

**本书讲解位置。** 第 12 章。

### 使用 `refcount(9)` 的引用计数

**它是什么。** 内核认可的模式，用于计算仍使用对象的上下文数量。原子计数器、`refcount_acquire` 增加它、`refcount_release` 减少它并在计数达到零时返回 `true`（以便调用者可以释放）。

**驱动程序为什么使用它。** 每当对象可以比产生它的操作更长时。正在关闭的 `cdev`，而读取正在进行；其分离与 ioctl 竞争的 softc；传递给硬件的缓冲区必须不被释放直到硬件完成。

**何时使用**当您有共享所有权且没有单一位置可以可靠地释放对象时。如果一段代码明确拥有对象，跳过引用计数并在那里释放它。

**核心操作。** `refcount_init(count, initial)`、`refcount_acquire(count)` 返回先前值、`refcount_release(count)` 如果计数达到零则返回 `true`（调用者必须释放）。有 `_if_gt` 和 `_if_not_last` 变体用于条件获取，以及 `refcount_load` 用于只读检查。

**常见陷阱。**

- 为真正所有权层次结构的资源使用 `refcount`。引用计数用于真正共享的所有权，而不是关于谁释放什么的懒惰。
- 在释放之前释放。`refcount_release` 返回 `true` 是您释放的许可；在该分支内进行，而不是之前。
- 将引用计数与锁混合。引用计数操作是无锁的，但"增加引用计数然后解引用"是竞争，如果另一个线程可能在两步之间丢弃最后一个引用。通常的修复是在决定获取时持有锁。

**本书讲解位置。** 在附录 A 中提及，并发上下文通过第 11 和 12 章构建。

## 结语：如何在真实代码中识别这些模式

在他人驱动程序中识别模式与自己编写模式是不同的技能。容易的部分是发现 API：`TAILQ_FOREACH`、`buf_ring_enqueue`、`goto fail_mem`。更难的部分是当作者根本没有使用 API 时发现模式，因为模式是纯控制流。

三个习惯有帮助。

第一个是询问作者试图保留什么不变量。环形缓冲区保留"中断和线程永不写入同一索引"。状态机保留"我们总是恰好处于一种状态"。清理梯子保留"每个资源恰好释放一次，以反向顺序"。当您阅读新代码时，首先命名不变量给其余代码一个附加的地方。

第二个是阅读文件顶部的注释。大多数 FreeBSD 驱动程序包含一个简短的块注释，解释它们使用的锁定纪律、状态机或环约定。该注释正是因为读者无法单独从代码中恢复它而存在。在阅读代码之前阅读它，当代码让您惊讶时再次阅读它。

第三个是让本附录中的模式殖民您的词汇表。当您发现自己写"环形缓冲区"时，能够说出它是 SPSC 还是 MPMC、索引是牺牲槽还是自由运行、屏障属于哪里。当您发现自己写"状态机"时，能够命名状态、事件和瞬态状态。当您发现自己写"清理路径"时，能够以获取的反向顺序列出资源。一旦这些词自动出现在脑海中，附录就变得有用了。

从这里您可以向几个方向移动。附录 A 有上面命名的每个原语的详细 API 条目。附录 C 为正好位于这些软件模式之下的硬件端模式（DMA 环、描述符数组和一致性纪律）奠定基础。附录 E 涵盖许多这些模式诞生于其中的内核子系统。本书的每一章都有实验室，这些模式出现在可工作的驱动程序中，以便下次您在野外遇到一个时，您不会减速就能识别它。这就是模式识别真正是：继续阅读的信心。
