# 调试与追踪

## 读者指导与学习成果

第22章以一个能够经受挂起、恢复和关机的驱动程序结束了第四部分。版本为`1.5-power`的`myfirst`驱动程序附加到PCI设备，分配MSI-X向量，运行DMA管道，处理完整的kobj电源管理契约，通过sysctl暴露计数器，并通过回归测试脚本测试附加、分离、挂起、恢复和运行时PM。对于一个能够启动、运行、挂起、唤醒并最终卸载的驱动程序来说，这些机制已经完成。该驱动程序还缺少的是当出现问题时告诉开发者内部发生了什么的能力。

第23章添加了这种能力。第五部分的标题是*调试、工具与实战*，第23章通过教授可观测性和诊断规范来开启这一部分，这些规范将一个你刚刚编写的驱动程序转变为一个你可以维护多年的驱动程序。读者将在本章学习内核空间调试与用户态调试有何不同，如何有意识地而非习惯性地使用`printf()`和`device_printf()`，如何从日志消息被写入的那一刻到用户使用`dmesg`读取的那一刻来跟踪日志消息，如何构建一个能够捕获普通内核遗漏错误的调试内核，如何要求DTrace在不重新构建任何东西的情况下显示驱动程序的实时行为，如何使用`ktrace`和`kdump`跟踪用户程序通过驱动程序接口进入内核的过程，如何在症状首次出现时识别常见驱动程序错误的特征模式，以及如何重构驱动程序的调试支持使其保持有用而不将源代码变成一堆散乱的`printf`调用。

第23章的范围正是这些。它教授FreeBSD驱动程序作者在实验工作、回归测试和响应错误报告中使用的日常内核调试工具包。它教授足够的DTrace来检测驱动程序并测量其行为。它教授内核调试选项背后的思维模型，以便读者可以为眼前的问题选择正确的选项集。它展示了这些工具在源代码树中的位置以及如何直接阅读其文档。它不教授事后内核崩溃分析的深层机制，因为该材料属于本书后期的专门主题。它不教授驱动程序架构或与特定子系统的集成，因为第24章专门讲述这些。它不教授高级性能调优或大规模性能分析，因为第六部分深入涵盖硬件感知调优。第23章引入的规范是后续章节假设的基础：你无法调优你无法观测的东西，你也无法修复你无法复现的东西。

第五部分是驱动程序赢得将工作原型与生产软件区分开来的品质的地方。第22章使驱动程序能够经受电源状态的改变。第23章使驱动程序能够告诉你它在做什么。第24章将使驱动程序与用户实际通过程序访问的内核子系统干净地集成。第25章将使驱动程序在压力下具有弹性。每一章都增加一种品质。第23章增加了可观测性，这恰好是每个后续主题所依赖的品质。

### 为什么内核调试需要单独一章

在编写第一行代码之前，理解为什么DTrace、`dmesg`和调试内核选项需要整整一章是有帮助的。你可能已经在想为什么这比调试用户程序更难：添加一些`printf`语句，在`gdb`下运行，单步执行代码，完成。这种思维模型在用户空间是合理的，但在内核空间是灾难性的，原因有三。

第一个原因是**驱动程序错误就是内核错误**。当用户态程序有错误时，操作系统保护机器的其余部分：内核杀死进程，shell打印消息，用户重试。这种保护在内核内部不适用。一个错误的指针、一个遗漏的锁、一个未释放的分配、中断处理程序与任务之间的竞态，都可能使整个系统崩溃。在测试中崩溃的驱动程序会使运行中的虚拟机崩溃。泄漏内存的驱动程序最终会耗尽机器内存。破坏内存的驱动程序会破坏恰好位于错误旁边的任何数据。第23章之所以存在，是因为这些失败看起来和行为都不像用户态失败，而且找到它们的工具看起来和行为也不像用户态工具。

第二个原因是**可见性有限且代价高昂**。在用户态调试器中，程序为你停止。你可以单步执行、打印变量、设置条件断点、使用`rr`或`gdb --record`倒转时钟。在内核中，所有这些在运行时通常都不可用，除非有特殊准备。你不能像停止进程那样停止运行中的内核，因为内核也是运行其他所有东西的东西，包括键盘、终端、网络和磁盘。你添加的每一点可见性都有代价：`printf`便宜但慢；SDT探针快但必须编译进去并启用；DTrace脚本灵活但在内核本身运行并触及它触发的每个探针。第23章之所以存在，是因为你必须谨慎地花费可见性预算，而了解每个工具的代价是学会如何使用它的一部分。

第三个原因是**反馈循环更长**。在用户态，编写-编译-运行周期是几秒钟。在内核工作中，在最佳条件下周期相同，但错误的后果是重启或回滚虚拟机快照。在用户态一次就能复现的错误在内核态可能需要十次挂起和恢复才能让驱动程序的内部计数器溢出并出现症状。在`gdb`下每次运行都能复现的错误可能只在无负载内核、特定负载或特定CPU上复现。第23章之所以存在，是因为内核调试只有在反馈循环尽可能短的情况下才能忍受，而短循环需要为每类错误选择正确的工具组合。

第23章通过以`myfirst`驱动程序为实例，具体地教授这三个现实，从而赢得了它的位置。完成第23章的读者可以为任何FreeBSD驱动程序添加规范的、可切换的调试输出；无需帮助即可阅读和解释`dmesg`、`/var/log/messages`和DTrace输出；构建调试内核并知道每个选项能带来什么；首次出现症状时找到并修复常见驱动程序错误类；并将模糊的报告（"我的机器在驱动程序加载时有时会挂起"）转化为可复现的测试用例、假设、修复和回归测试。

### 第22章结束时的驱动程序状态

在开始之前需要验证几个先决条件。第23章扩展了第22章第4阶段结束时产生的驱动程序，标记为版本`1.5-power`。如果以下任何项目不确定，请返回第22章并在开始本章之前修复它，因为调试主题假设驱动程序有一个稳定的调试基线。

- 你的驱动程序干净编译并在`kldstat -v`中标识为`1.5-power`。
- 驱动程序分配一个或三个MSI-X向量，注册过滤加任务中断管道，将每个向量绑定到CPU，并在附加期间打印中断横幅。
- 驱动程序分配`bus_dma`标签和4 KB DMA缓冲区，通过`dev.myfirst.N.dma_bus_addr`暴露总线地址，并在分离时清理DMA资源。
- 驱动程序实现`DEVICE_SUSPEND`、`DEVICE_RESUME`和`DEVICE_SHUTDOWN`。`devctl suspend myfirst0`后跟`devctl resume myfirst0`干净地成功，后续的DMA传输无误运行。
- 挂起、恢复、关机和运行时PM转换的计数器作为sysctl出现在`dev.myfirst.N.`下。
- `HARDWARE.md`、`LOCKING.md`、`SIMULATION.md`、`PCI.md`、`INTERRUPTS.md`、`MSIX.md`、`DMA.md`和`POWER.md`在你的工作树中是最新的。
- `INVARIANTS`、`WITNESS`、`WITNESS_SKIPSPIN`、`DDB`、`KDB`和`KDB_UNATTENDED`在你的测试内核中启用。

这就是第23章扩展的驱动程序。添加在源代码行数上不多但在可维护性上很重要：一个新的`myfirst_debug.c`文件、一个匹配的`myfirst_debug.h`头文件、一组带有编译时和运行时切换的日志宏、一组静态定义的DTrace探针点、一个详细模式sysctl旋钮、一组区分正常操作和异常事件的调试计数器、版本升级到`1.6-debug`、一个解释子系统的`DEBUG.md`文档，以及一个小型的辅助脚本集合，将原始日志和追踪输出转化为人类可读的内容。

### 你将学到什么

到本章结束时，你将能够：

- 用具体术语描述内核调试与用户态调试的不同之处，并解释这种差异如何塑造本章其余部分的工具包。
- 在驱动程序代码中正确使用`printf()`、`device_printf()`、`device_log()`和`log()`，理解何时使用每个以及FreeBSD中的日志优先级级别意味着什么。
- 读取和过滤`dmesg`，跟踪内核消息缓冲区的生命周期，关联启动时和运行时的消息，并知道你想要的消息何时在`dmesg`中，何时在`/var/log/messages`中，何时两者都有。
- 使用`DDB`、`KDB`、`KDB_UNATTENDED`、`INVARIANTS`、`WITNESS`及相关选项构建自定义调试内核，并知道每个选项在运行时实际做什么、代价是什么以及何时需要它。
- 在测试期间有意进入`ddb`，运行基本检查命令（`show`、`ps`、`bt`、`trace`、`show alllocks`、`show pcpu`、`show mbufs`），干净地离开`ddb`，并理解你在每种情况下看到的内容。
- 理解内核崩溃转储是什么，`dumpdev`和`savecore(8)`如何产生一个，以及`kgdb`如何打开转储进行事后分析，达到足以打开一个并检查几帧的水平。
- 解释DTrace是什么，加载`dtraceall`模块，列出内核暴露的提供者，编写追踪内核函数的简短脚本，并使用`fbt`、`syscall`、`sched`、`io`、`vfs`和`sdt`提供者解决常见驱动程序问题。
- 使用`SDT_PROVIDER_DEFINE`、`SDT_PROBE_DEFINE`和`SDT_PROBE`宏将SDT（静态定义追踪）探针添加到驱动程序，并使用`dtrace -n 'myfirst:::entry'`观察它们。
- 对与驱动程序对话的用户程序运行`ktrace`，使用`kdump`读取输出，并将`open`、`ioctl`、`read`和`write`等系统调用映射回驱动程序的入口点。
- 识别驱动程序错误的特征模式：在`vmstat -m`中可见的未释放分配，`WITNESS`报告的锁顺序违规，BAR末尾之外的`bus_space`访问，传输前后遗漏的`bus_dmamap_sync`调用，`DEVICE_DETACH`运行后对设备的访问，以及每个产生的症状。
- 将驱动程序的调试支持重构为专用的`myfirst_debug.c`和`myfirst_debug.h`对，带有运行时可切换的详细输出、用于函数级追踪的ENTRY和EXIT宏、用于结构化问题报告的ERROR和WARN宏、用于外部追踪的SDT探针，以及用于调试控制的sysctl子树。
- 阅读FreeBSD源代码树中真实的驱动程序调试代码，如`/usr/src/sys/dev/ath/if_ath_debug.h`中的`DPRINTF`系列，`/usr/src/sys/dev/virtio/block/virtio_blk.c`中的`bootverbose`用法，`/usr/src/sys/dev/virtio/virtqueue.c`中的SDT探针，以及`/usr/src/sys/dev/iwm/if_iwm.c`等驱动程序中的调试sysctl。

列表很长，因为调试本身就是一系列技能。每个项目都是狭窄且可教学的。本章的工作是组合。

### 本章不涵盖的内容

几个相邻主题被明确推迟，以便第23章专注于日常调试规范。

- **深入的`kgdb`脚本和Python扩展**。本章展示如何打开崩溃转储并检查几帧。在`kgdb`之上构建Python助手、使用用户定义命令遍历内核数据结构以及脚本化转储分析属于后续的高级章节。
- **`pmcstat`和硬件性能计数器**。这些是性能调优工具，不是狭义上的调试工具。第六部分会回到它们。
- **超出`WITNESS`的锁性能分析**。`options LOCK_PROFILING`路径和`lockstat` DTrace提供者在上下文中被提及，但完整的争用分析调优工作流程属于第六部分的性能章节。
- **详细的`KTR`事件追踪**。本章在与`ktrace`对比时提及`KTR`，以便读者不会混淆两者，但重量级的内核内事件缓冲区及其循环日志工作流程是高级材料，很少需要驱动程序关注。
- **限速日志框架**。第25章深入涵盖日志礼仪、静态计数器和基于时间的限速。第23章非正式地介绍限速需求并移交。
- **`bhyve debug`后端、GDB服务器模式和实时内核级断点**。这些是可能的，偶尔有用，但属于高级小众领域，不属于第一个调试章节。
- **错误注入和故障测试框架**。`fail(9)`、`fail_point(9)`及相关工具在上下文中简要介绍，但留给第25章和后续测试材料。

守住这些界限使第23章成为关于如何发现错误的章节，而不是关于内核曾经尝试帮助你发现错误的所有地方的章节。

### 预计时间投入

- **仅阅读**：四到五个小时。第23章的概念比第21章或第22章轻，但它们连接到很多工具，首次通过工具概览需要时间。
- **阅读加输入示例**：分两到三次进行十到十二小时。驱动程序分三个阶段演进：首先是基于宏的日志和详细模式，然后是SDT探针，最后是重构到自己的文件对。每个阶段都很短，每个阶段都有一个简短的实验来确认它工作。
- **阅读加所有实验和挑战**：分三到四次进行十五到十八小时。实验包括有意进行的`ddb`会话、测量驱动程序延迟的DTrace练习、`ktrace`/`kdump`演练、读者必须用`vmstat -m`找到的有意内存泄漏，以及读者必须在`WITNESS`下复现的有意锁顺序违规。

第4节和第5节在词汇量方面最密集。如果`ddb`命令或DTrace语法在首次通过时感觉晦涩，那是正常的。停下来，在模拟驱动程序上运行匹配的练习，当形状稳定后再回来。调试是具体实践比进一步阅读更能稳定想法的技能之一。

### 先决条件

在开始本章之前，确认：

- 你的驱动程序源代码与第22章第4阶段（`1.5-power`）匹配。起点假设每个第22章原语：`device_suspend`、`device_resume`和`device_shutdown`方法；softc的`suspended`标志和电源计数器；POWER.md文档；以及回归测试脚本。第23章建立在所有这些之上。
- 你的实验机器运行FreeBSD 14.3，磁盘上有`/usr/src`并与运行内核匹配。
- 已构建、安装并正常启动带有`INVARIANTS`、`WITNESS`、`WITNESS_SKIPSPIN`、`DDB`、`KDB`、`KDB_UNATTENDED`、`KDTRACE_HOOKS`和`DDB_CTF`的调试内核。这些正是第22章已经要求的选项；本章第4节详细解释每一个，以便你知道它们做什么。
- `bhyve(8)`或`qemu-system-x86_64`可用。第23章的实验有意在虚拟机中运行。几个实验可能导致内核崩溃（这就是课程），而虚拟机快照是保持反馈循环短的正确工具。
- 以下用户空间命令在你的路径中：`dmesg`、`sysctl`、`kldstat`、`kldload`、`kldunload`、`devctl`、`vmstat`、`ktrace`、`kdump`、`dtrace`、`procstat`、`savecore`、`kgdb`。
- 你已在`1.5-power`状态快照了虚拟机并清楚地命名了快照。本章的几个实验有意使内核崩溃；你希望能够在一分钟内回滚到已知良好状态。

如果以上任何项目不稳定，现在就修复它。与本书中的许多主题不同，调试是一种奖励准备并惩罚走捷径的技能。没有快照就开始的实验会以一小时的恢复结束；没有`KDTRACE_HOOKS`就开始的DTrace练习会静默失败。

### 如何从本章获得最大收益

五个习惯在本章比之前任何章节都更有价值。

第一，将`/usr/src/sys/kern/subr_prf.c`、`/usr/src/sys/kern/kern_ktrace.c`、`/usr/src/sys/kern/subr_witness.c`和`/usr/src/sys/sys/sdt.h`加入书签。第一个定义内核的`printf`、`log`、`vprintf`和`tprintf`实现，是对调用它们时发生什么的权威回答。第二个是`ktrace`实现。第三个是`WITNESS`，第23章严重依赖的锁顺序验证器。第四个是SDT探针宏。在相应章节开始时各读一次是为流畅度做的最有效的事情。这些文件都不长；最长的是几千行，大部分是注释。

第二，保持三个真实驱动程序示例在手边：`/usr/src/sys/dev/ath/if_ath_debug.h`、`/usr/src/sys/dev/virtio/virtqueue.c`和`/usr/src/sys/dev/iwm/if_iwm.c`。第一个说明了带有编译时和运行时切换的经典内核`DPRINTF`宏模式。第二个包含真实的SDT探针。第三个显示驱动程序如何通过`device_get_sysctl_tree()`连接调试sysctl。第23章在适当的时候指向每一个。现在读一遍，不要试图记忆，为章节其余部分提供具体的锚点来挂载想法。

## 第1节：为什么内核调试具有挑战性

在介绍工具之前，先介绍思维方式。第1节阐明了内核调试与读者可能已经了解的内容有何不同。编写过shell脚本、小型C程序甚至多线程应用服务器的读者可能使用过`printf`、`gdb`、`strace`、`valgrind`、`lldb`或浏览器的开发者工具。这些工具在用户空间非常出色，它们鼓励的思维模型在用户空间是合理的。在内核空间，思维模型需要转变。第1节描述了这种转变，命名了驱动程序作者不断遇到的具体困难，并为后续逐个工具的工作奠定了基础。

### 内核代码的本质

用最简单的说法，驱动程序是在内核地址空间中运行的C代码。这一事实驱动了内核调试与用户态调试之间的大部分差异。

内核地址空间包含一切：内核本身运行的代码、内核用来管理进程、文件、内存和设备的数据结构、系统上每个进程的页表、每个待处理网络数据包的缓冲区、每个磁盘请求的队列，以及每个设备映射寄存器后面的内存。驱动程序持有的每个指针都可能触及内核的每个其他部分。没有用户空间程序所享有的那种分离，即进程只能触及自己的地址空间，操作系统强制执行边界。

后果是尖锐的：驱动程序中的一个错误指针可能破坏任何内核数据结构，而不仅仅是驱动程序自己的。一个在softc末尾写入一个字节的驱动程序可能恰好写入内核的进程表，或文件系统的空闲块位图，或另一个驱动程序的中断向量表。这种破坏通常不会立即产生崩溃；它会在五分钟后当其他代码读取驱动程序静默损坏的结构时产生崩溃。原因和症状之间的这种延迟是内核错误比用户态错误感觉更难的最常见原因。你不是在错误指针发生的那一刻捕获它；你是捕获到另一段代码在混乱中绊倒。

内核代码也在没有通常安全网的情况下运行。在用户空间，解引用空指针会产生段错误，内核杀死进程，用户看到消息。在内核空间，解引用空指针会产生页面错误，内核本身出错，错误处理程序决定系统是否可以继续。如果页面错误发生在会让内核处于不一致状态的地方（在中断处理程序内、在临界区内、在`malloc`内），内核会崩溃以防止进一步损坏。在启用了`INVARIANTS`的开发内核上，内核比发布内核更积极地崩溃，因为权衡正是重点：在开发中大声地捕获错误，而不是让它静默传播。

也没有用户程序拥有的标准输出流。用户空间C程序调用`printf`，输出出现在终端上，因为C运行时将调用链接到一个最终写入shell连接到终端的文件描述符的链。在内核中，没有shell，没有用户意义上的终端，也没有C运行时。当驱动程序调用`printf`时，它触及内核自己的`printf`实现，后者写入内核消息缓冲区，并根据设置写入系统控制台。第2节详细介绍了这一点；现在的要点是内核的日志管道不是用户程序使用的管道，假设熟悉会导致后续困惑。一个"只是调用`printf`"的驱动程序调用的是与用户态程序调用的不同的函数，具有不同的语义和不同的输出路径。

### 内核空间调试的风险

在用户空间，错误的调试更改的后果通常很小。打印错误变量的`printf`是一个烦恼。在运行十亿次循环内的`printf`会减慢程序但不会破坏任何东西。在调试语句中误用`malloc`会泄漏几个字节，进程最终退出，泄漏随之消失。

在内核空间，错误的调试更改的后果可能大得多。错误路径上的`printf`可能使内核死锁，如果printf的路径需要调用上下文已经持有的锁。在中断处理程序中调用可能睡眠的代码的`printf`会在`INVARIANTS`下崩溃。持有自旋锁时调用睡眠分配器（`malloc(..., M_WAITOK)`而不是`M_NOWAIT`）的调试消息会让`WITNESS`非常不高兴，如果启用了`WITNESS_KDB`，会将机器丢入内核调试器。用于调试目的但没有适当同步的全局计数器本身就是错误的来源。这些都是驱动程序作者犯的真实错误；每一个都是本章将教授的课程。

细微之处在于调试代码严格来说是代码，它必须与驱动程序中的任何其他代码一样正确。"我以后再清理"在内核上下文中是不安全的。匆忙的调试更改可能引入比原始错误更难找到的新错误。第23章教授避免这个陷阱的习惯：使用禁用时编译掉的宏；使用内核自己的日志工具而不是发明一个；包装条件块，以便详细模式代码仅在详细模式开启时付费；并以与生产代码相同的谨慎程度测试调试代码。

另一个风险是观察性的：添加调试输出可能会改变你试图观察的行为。没有调试输出时可靠复现的竞态条件可能在你添加`printf`调用时完全消失，因为printf通过锁序列化并恰好隐藏了竞态窗口。性能错误在检测后可能看起来不同，因为检测本身增加了代价。内核开发者称之为内核调试的海森堡问题。本章的工具部分选择是因为它们最小化了这个问题：DTrace、SDT探针和KTR在不活动时几乎没有开销；`device_printf`虽然冗长但一致。在代码中散布`printf("HERE\n")`总是可用的，但作为一个习惯，它的代价大于帮助。

### 与用户态调试的区别

内核空间和用户空间调试之间的差异值得逐个工具地阐明。

用户可以在`gdb`下运行程序、设置断点并观看程序停止。你不能直接对运行中的内核这样做，因为暂停的内核意味着暂停的系统。粗略的等效方法是对崩溃转储使用`kgdb`，它让你检查崩溃时刻的内核状态，但不能单步向前。另一个粗略的等效方法是`ddb`，即内核内调试器，它让你暂停活动内核但作为内核本身的一部分运行，有自己的有限命令集。两者都有用；两者的行为都不像活动进程上的`gdb`。

用户可以在Linux上使用`strace`或在FreeBSD上使用`truss`观察程序的系统调用。FreeBSD的`ktrace`是最接近的等效方法；它将进程的系统调用、信号、namei事件、I/O和上下文切换的跟踪记录到`kdump`可以读取的文件中。但`ktrace`看到的是驱动程序接口的用户侧，而不是驱动程序的内部。它可以显示用户进程调用了`ioctl(fd, CMD_FOO, ...)`，但不能显示驱动程序中`ioctl`实现内部发生了什么。第6节专门花时间讲解`ktrace`正是因为它是显示边界最干净的工具。

用户可以使用`valgrind`监视程序中的内存错误和泄漏。内核没有直接的等效工具。最接近的模拟是用于按类型汇总内核分配的`vmstat -m`、用于锁顺序错误的`WITNESS`选项、用于开发者写入代码的一致性检查的`INVARIANTS`，以及用于更重内存调试分配器的`MEMGUARD`。第7节介绍了实际模式。

用户可以在Linux上使用`perf`或在FreeBSD上使用`gprof`对程序进行性能分析。FreeBSD提供`pmcstat`和DTrace用于性能分析。DTrace对于驱动程序开发者提出的问题来说是更灵活的工具，第23章专注于DTrace而不是`pmcstat`。

用户通常可以重新运行程序来查看错误是否再次发生。内核开发者通常不能；崩溃的内核必须重启，只在负载下出现或依赖于时序的错误可能需要数小时才能复现。对这种困难的回应是将可复现性构建到过程中：快照虚拟机、脚本化测试用例、记录一切。本章的每个工具都是为了支持这种回应而存在。

最后一个区别是文化上的。内核开发社区围绕调试构建了一个特定的词汇表，值得自己学习。诸如"锁顺序"、"临界区"、"抢占禁用"、"epoch"、"巨锁"、"可睡眠上下文"、"不可睡眠上下文"和"中断上下文"等术语在FreeBSD中都有精确的含义，正确使用它们是精通内核调试的一部分。第23章在使用时会介绍它们；最后的词汇表收集了反复出现的术语。

### 受控、增量更改的重要性

在用户空间，当程序行为异常时，第一直觉通常是同时更改几件事，看看行为是否改变。在内核空间，这是一个失败的策略。每次更改都可能是新的错误。每次重建都需要虚拟机重启。每个测试周期都消耗时间。在两次编译之间更改五件事的驱动程序开发者会花几个小时困惑于哪个更改导致了新症状，因为没有单个更改是孤立的。

正确的模式是**受控、增量更改**。做一个更改。重建。测试。观察效果。记录发生了什么。根据证据决定下一个更改。继续。这每次更改较慢，但每次有效修复较快，因为你永远不必在五个修复中的一个实际引入新错误时撤销组合更改。这种规范第一次感觉是人为的。经过十几次会话后，它不再感觉人为，开始感觉像是取得进展的唯一方式。

一个相关的规范是**始终从已知良好状态开始**。不要在尚未验证的状态下调试驱动程序。不要通过应用六个本地更改然后运行测试来开始会话。提交你的基线，验证它工作，然后才开始新工作。如果新工作出错，`git checkout .`让你回到已知良好状态，你可以重新开始。没有这个习惯的驱动程序作者在每个调试会话中花一半时间想知道是基线还是新更改有问题。

第三个规范是**修复前先复现**。无法复现的错误无法证明已修复。调试会话的第一个工作不是修复错误；而是产生一个简短、可靠的方法来触发错误，理想情况下是在脚本中。一旦触发可靠，修复就很简单：应用候选，重新运行触发，观察错误是否消失。没有可靠的触发，你就是在猜测。

第四个规范是**小型、脚本化的复现用例**。花费一小时修复的错误与花费一周修复的错误之间的区别通常在于复现脚本的大小。你可以用五行shell复现的错误是可以用`git bisect`二分查找的错误。你只能在三小时的复杂使用后复现的错误是你甚至不敢开始调试的错误。花时间缩小复现范围是值得的时间。

第23章的实验有意设计得很小。崩溃用三行触发。DTrace脚本五行。`ktrace`演示使用一行C程序。这不是偶然的：这是良好驱动程序调试在任何规模上使用的相同模式。最小复现和最小更改可以扩展到真实错误。

### 思维转变

开发者从用户空间转向内核空间时改变的不仅仅是工具；还有假设。在用户空间，假设内核会捕获你的错误、运行时会为你清理、最坏情况通常是可以重启的崩溃程序，这是合理的。在内核空间，这些假设是错误的。内核什么也不捕获；运行时就是你正在编写的东西；最坏情况是让一切崩溃的panic。

这不是害怕的理由。这是小心的理由。小心在这里有特定含义：驱动程序作者编写的代码应该能够被其他人在六个月后仅从日志和源代码调试。小心的代码使用清晰的函数名。小心的代码以有用的详细级别记录它所做的事情。小心的代码检查自己的不变量。小心的代码的结构使得当它崩溃时，崩溃是明显的。小心的代码不试图聪明；聪明的代码是你无法在压力下调试的代码。

第23章教授内核调试的工具，但它实际教授的习惯是小心。章节中的每个工具在被小心的开发者应用于小心的代码时最有用。同样的工具应用于鲁莽的代码产生的信息是你无法解释的。重点不是记住DTrace语法或`ddb`命令；重点是发展基于证据、小步骤、针对良好基线、有良好日志、以任何合作者都能跟随的方式进行调试的习惯。

### 第1节总结

第1节确立了问题的形状。内核代码在单个共享地址空间中运行，没有用户态依赖的安全网，具有损坏整个系统的失败模式。内核调试工具与它们的用户态对应物不同，代价更高，交互性更低。受控、增量、可复现更改的规范不是可选的；它是唯一有效的策略。所需的思维转变是真实的，但它是可学习的，是本章其余部分工具站立的基础。

第2节从每个驱动程序最终使用的最简单、最通用的工具开始介绍工具概览：内核的`printf`系列。



## 第2节：使用`printf()`和`device_printf()`记录日志

内核的`printf`系列是工具包中最谦逊的调试工具。它也是使用最广泛的，差距很大。`/usr/src/sys/dev`中的几乎每个FreeBSD驱动程序都在某处调用`device_printf`。用户见过的几乎每条启动消息都是由内核`printf`写入的。在DTrace之前，在`ddb`之前，在SDT探针之前，在调试内核之前，驱动程序作者就会使用`device_printf`，因为它总是可用的，它总是安全的（受本节将阐明的限制条件约束），而且地球上每个其他驱动程序作者都理解它。

本节教授如何有意地使用这个函数系列。目标不仅是展示如何调用`device_printf`，因为那部分很简单。目标是建立将日志记录作为驱动程序设计特性而非随时间积累且从未清理的散乱调试注释集合的习惯。

### 内核上下文中`printf()`的基础

FreeBSD的内核`printf`位于`/usr/src/sys/kern/subr_prf.c`。其签名是熟悉的：

```c
int printf(const char *fmt, ...);
```

它接受格式字符串和可变参数列表，并将格式化文本写入内核消息缓冲区。缓冲区是一个内存循环区域，在现代FreeBSD系统上通常约为192 KB，每条内核消息都写入其中。内核本身不保留永久日志；用户空间`syslogd(8)`守护进程是读取缓冲区并将选定行写入`/var/log/messages`的东西。第3节涵盖完整的管道。

格式字符串理解用户态`printf`理解的大多数转换：`%d`、`%u`、`%x`、`%s`、`%c`、`%p`等。它理解一些内核特有的额外功能，如用于打印字节缓冲区十六进制转储的`%D`（带有分隔符参数）。它不理解浮点转换如`%f`和`%g`，因为内核默认不使用浮点数学，转换代码被有意省略。在内核内尝试打印`double`或`float`几乎总是错误，编译器通常会对此发出警告。

内核的`printf`与用户态的安全属性不同。用户态`printf`可以分配、可以在C运行时内部持有锁、可以在I/O上阻塞。内核的`printf`不能分配；不能阻塞；必须几乎可以从任何上下文安全调用。`subr_prf.c`中的实现故意简单：它格式化为一个小的栈缓冲区，将结果复制到消息缓冲区，在适当时唤醒`syslogd`，然后返回。这种简单性是`printf`成为通用回退工具的部分原因。

有一些上下文即使是内核的`printf`也不安全。例如，快速中断处理程序在内核稳定控制台状态之前运行，从快速中断内部进行的`printf`可能在某些平台上产生挂起或死锁。持有`printf`路径也试图获取的锁的路径也可能死锁。在实践中，驱动程序作者通过仅从普通驱动程序入口点（attach、detach、ioctl、read、write、suspend、resume）和中断任务（过滤加任务中断管道的延迟部分，如第19章介绍）使用`printf`，而不是从过滤器本身使用，来避免这个问题。`myfirst`驱动程序已经遵循这种模式；第23章保留了它。

### 首选用法：用于结构化、设备感知日志记录的`device_printf()`

原始`printf`是通用的。驱动程序作者可以做得更好。内核提供`device_printf`，在`/usr/src/sys/sys/bus.h`中声明：

```c
int device_printf(device_t dev, const char *fmt, ...);
```

该函数为每条消息添加设备名称前缀，通常是驱动程序名称连接单元号：`myfirst0: `、`em1: `、`virtio_blk0: `。这个前缀不是装饰性的。它是多设备系统内核日志中最重要的信息。没有它，在有八张网卡和三个存储控制器的机器中，像"interrupt received"这样的消息是无用的。有了它，"myfirst0: interrupt received"在任何读取日志的工具中立即可定位。

规则很简单：**在驱动程序代码中，优先使用`device_printf`而不是`printf`**。驱动程序中每个有`device_t`在作用域内的地方，使用`device_printf`。每个没有`device_t`的地方（例如全局初始化函数），使用带有显式前缀的`printf`：

```c
printf("myfirst: module loaded\n");
```

这是使内核日志在整个系统中可读的模式。这也是几乎每个FreeBSD驱动程序遵循的模式。打开`/usr/src/sys/dev/`下的任何文件，模式在第一个attach函数中就可见。

`myfirst`驱动程序在大多数地方已经使用`device_printf`，因为第18章在驱动程序首次成为PCI设备时引入了这个约定。本章第8节将`device_printf`包装在添加严重性、类别和详细模式切换的宏中，但底层函数保持不变。第23章没有任何东西替换`device_printf`；本章是对它的阐述。

### `device_log()`和日志优先级级别

`device_printf`是简单的：它以内核的默认日志优先级（在syslog术语中是LOG_INFO）写入消息。有时驱动程序想以更高的优先级写入，以标记严重问题，或以更低的优先级写入，以标记通常应该被过滤掉的例行信息消息。

FreeBSD 14为此公开了`device_log`，同样在`/usr/src/sys/sys/bus.h`中：

```c
int device_log(device_t dev, int pri, const char *fmt, ...);
```

`pri`参数是来自`/usr/src/sys/sys/syslog.h`的syslog优先级常量之一：

- `LOG_EMERG` (0)：系统不可用
- `LOG_ALERT` (1)：必须立即采取行动
- `LOG_CRIT` (2)：严重情况
- `LOG_ERR` (3)：错误情况
- `LOG_WARNING` (4)：警告情况
- `LOG_NOTICE` (5)：正常但重要的情况
- `LOG_INFO` (6)：信息性
- `LOG_DEBUG` (7)：调试级别消息

对于驱动程序工作，实践中重要的四个优先级是`LOG_ERR`、`LOG_WARNING`、`LOG_NOTICE`和`LOG_INFO`。驱动程序很少需要`LOG_EMERG`、`LOG_ALERT`或`LOG_CRIT`，因为大多数驱动程序失败仅限于驱动程序，而不是整个系统。`LOG_DEBUG`在主动调试期间有用；第23章将其用于详细模式输出。

底层的`log(9)`函数同样在`subr_prf.c`中，更简单：`void log(int pri, const char *fmt, ...)`。`device_log`包装`log`并添加设备名称前缀。

`printf`和`log`之间的实际区别在于，当内核配置保守时输出去向何处。默认情况下，FreeBSD将`LOG_NOTICE`及以上级别的所有内容发送到控制台；`LOG_INFO`及以下级别进入内核消息缓冲区但不进入控制台。这在两个地方很重要。首先，在启动期间：高优先级消息在启动期间在屏幕上可见；`LOG_INFO`消息不可见。其次，在运行期间：生产系统上的严重错误应该出现在控制台上（以便监视`messages`尾部tail的系统管理员立即看到），而例行探测或附加消息不应该杂乱屏幕。

第23章中的`myfirst`驱动程序使用此约定：

- 探测和附加成功：`device_printf`（即`LOG_INFO`）。
- 挂起、恢复和关机转换：`device_printf`（例行）。
- 来自硬件的错误、意外中断计数、DMA失败和失败的设备状态：`device_log(dev, LOG_ERR, ...)`或`device_log(dev, LOG_WARNING, ...)`。
- 详细模式输出（由sysctl启用）：调试宏内的`device_log(dev, LOG_DEBUG, ...)`。

### 记录什么和不记录什么

知道不记录什么与知道记录什么同样重要。输出过多的驱动程序会杂乱`dmesg`、浪费消息缓冲区空间，并在任何人读取之前将较旧的信息消息推出循环缓冲区。输出过少的驱动程序在出问题时静默，给读者没有任何可用的信息。

第23章中`myfirst`驱动程序的经验法则是：

**始终在`device_printf`级别（LOG_INFO）记录：**

- 附加时的一行横幅，说明驱动程序名称、版本和分配的任何关键资源（中断类型、MSI-X向量计数、DMA缓冲区大小）。读者应该能够一目了然地看到驱动程序做了什么。
- 分离时的一行注释。分离通常是静默的，但记录其完成在驱动程序在卸载时挂起且读者需要知道分离至少已开始时很有用。
- 每次挂起、恢复和关机时的一行注释。电源转换很重要，读者通常想将错误与之前的转换相关联。

**始终使用带有`LOG_ERR`或`LOG_WARNING`的`device_log`记录：**

- 分配资源的任何失败（中断、DMA标签、DMA缓冲区、互斥锁）。错误路径应该始终说明失败了什么。
- 驱动程序检测到的任何硬件错误：设备未预期的响应、超时、指示硬件已故障的寄存器值。
- 驱动程序捕获的任何状态不一致：未启动传输的DMA完成、发现设备已挂起的挂起、发现设备仍持有客户端引用的分离。

**永远不要在正常稳态路径中记录：**

- 每次中断。即使设备安静，在实际负载下几秒钟内记录每次中断的驱动程序也会淹没日志。使用DTrace或计数器来观察每次中断。
- 每次DMA传输。同样的原因。
- 每次`read`或`write`调用。同样的原因。
- 生产中内部函数的进入和退出。使用详细模式宏（第8节），以便开发者可以打开或关闭进入/退出追踪。

**在详细模式下记录（由第8节的宏和调试sysctl启用）：**

- 关键函数的函数进入和退出，带有简短的参数摘要。
- 带有中断状态寄存器值的单个中断事件。
- DMA传输提交和完成。
- sysctl和ioctl操作及其返回的值。

"始终"和"详细"之间的界限是一个判断调用，不同的驱动程序有不同的划分。`/usr/src/sys/dev/re/if_re.c`在生产中记录很少，并使用`re_debug` sysctl启用详细输出。`/usr/src/sys/dev/virtio/block/virtio_blk.c`使用`bootverbose`来控制附加时的额外详细信息。`/usr/src/sys/dev/ath/if_ath_debug.h`定义了一个复杂的每个子系统掩码，可以选择性地启用不同类别的输出。第23章的`myfirst`驱动程序采用了更简单的模式，对于教学目的来说已经足够，读者开始处理真实驱动程序时可以扩展它。

### 追踪函数流而不刷屏

一种诱人但产生弊大于利的调试习惯是在每个函数的开始和结束编写`device_printf`。想法是正确的：如果你能看到调用序列，你通常可以推断正在发生什么。实践是错误的，因为序列产生压倒性的输出量，输出通过printf路径序列化，这会改变被测代码的时序。

正确的模式是使用**由sysctl控制的详细模式**。定义一对宏：

```c
#define MYF_LOG_ENTRY(sc)       \
    do {                        \
        if ((sc)->debug_verbose) \
            device_printf((sc)->dev, "ENTRY: %s\n", __func__); \
    } while (0)

#define MYF_LOG_EXIT(sc, err)   \
    do {                        \
        if ((sc)->debug_verbose) \
            device_printf((sc)->dev, "EXIT: %s err=%d\n", __func__, (err)); \
    } while (0)
```

然后，在驱动程序想要追踪的每个函数顶部，调用`MYF_LOG_ENTRY(sc);`。在每次返回时，调用`MYF_LOG_EXIT(sc, err);`。当`debug_verbose`标志为零（默认值）时，宏编译为一个从未采取的分支，基本上没有代价。当标志为1（在运行时通过`sysctl dev.myfirst.0.debug_verbose=1`启用）时，宏产生任何驱动程序开发者都可以读取的追踪。

上面显示的布尔标志故意简单。它引入了运行时控制旋钮的想法，而不要求读者同时学习新的布局。本章第8节将这个习惯用法充实为一个完整的宏组，用32位位掩码替换单个布尔（以便操作员可以仅启用中断追踪，或仅启用I/O追踪，而不是一次启用所有），将整个装置移动到`myfirst_debug.h`中，并在旁边添加SDT探针。现在，要内化的模式是追踪级别的输出应该**默认关闭**且**运行时可切换**。编译时调试标志（旧的`#ifdef DEBUG`风格）是较差的第二选择：你必须重建模块才能更改它们。运行时切换让开发者在需要的那一刻在运行系统上启用详细模式，在收集到证据的那一刻禁用它。

### 将日志重定向到`dmesg`和`/var/log/messages`

内核的`printf`写入内核消息缓冲区。该缓冲区是用户运行`dmesg(8)`时读取的内容。第3节详细介绍`dmesg`；简短版本是缓冲区是循环的，因此旧消息最终会被新消息覆盖，因为缓冲区会填满。

`syslogd(8)`守护进程从缓冲区读取新消息，并将它们写入适当的syslog文件，通常是`/var/log/messages`。优先级和文件之间的映射在`/etc/syslog.conf`中配置。默认情况下，`LOG_INFO`或更高级别的消息进入`/var/log/messages`；`LOG_DEBUG`消息被丢弃，除非显式启用。`/var/log/messages`由`newsyslog(8)`根据`/etc/newsyslog.conf`中的规则轮转；该轮转产生读者在长时间运行系统的日志目录中找到的`messages.0.gz`、`messages.1.gz`等。

实际后果是驱动程序的日志输出同时存在于两个地方。最近的消息在内核缓冲区中（可通过`dmesg`访问）和在`/var/log/messages`中（可通过任何文件读取工具访问）。旧消息可能已经从缓冲区中老化但仍保留在`/var/log/messages`或其档案中。非常旧的消息可能已经从`/var/log/messages`轮转到`messages.N.gz`档案中，可以用`zcat`、`zgrep`或`zless`读取。第3节详细介绍这些工具。

这里有一个微妙的细节值得一提，因为它会绊倒新的驱动程序作者：**内核产生的`LOG_DEBUG`消息默认不写入`/var/log/messages`**。如果读者的调试输出是`LOG_DEBUG`并且他们无法在日志文件中找到它，原因通常是`/etc/syslog.conf`，而不是驱动程序。快速测试：运行驱动程序，然后运行`dmesg | grep myfirst`。如果消息出现在`dmesg`中但不在`/var/log/messages`中，则消息已产生但被syslog过滤掉。在`/etc/syslog.conf`中启用`LOG_DEBUG`（如果syslog被配置为关心内核调试消息）是一个修复；使用`device_printf`或`device_log(dev, LOG_INFO, ...)`是更常见的修复，因为它通过使用更高的优先级绕过了问题。

### 具体演练：`myfirst`驱动程序中的日志记录

第23章开始时的`myfirst`驱动程序在明显位置已经有日志行：attach打印横幅，detach确认完成，suspend和resume记录它们的转换，DMA错误记录警告。第23章的第1阶段以三种有针对性的方式升级它。

首先，驱动程序内部每个有`device_t`在作用域内的`printf`都变成`device_printf`。快速grep发现大约八行仍在模块的`MOD_LOAD`和`MOD_UNLOAD`处理程序中使用普通`printf`。模块级别的保持为`printf`（它们没有`device_t`），但它们使用字面`"myfirst: "`前缀，以便读者仍然可以在日志中找到它们。在任何地方调用没有前缀的`printf`的驱动程序是一个产生不可找到日志行的驱动程序。

其次，每个错误路径变成显式优先级的`device_log`：

```c
/* 之前 */
if (error) {
    device_printf(dev, "bus_dma_tag_create failed: %d\n", error);
    return (error);
}

/* 之后 */
if (error) {
    device_log(dev, LOG_ERR, "bus_dma_tag_create failed: %d\n", error);
    return (error);
}
```

更改很小，在大多数测试内核中行为相同。价值在于优先级。现在，监视`LOG_ERR`消息的生产系统将看到驱动程序的错误；在`LOG_INFO`过滤的开发环境将看到所有内容。

第三，引入了一种新的"罕见但重要"状态更改日志记录模式。具体来说，第23章驱动程序在读者可能想知道但不想每次中断都记录的三个事件上以`LOG_NOTICE`记录：附加后第一次成功的DMA传输、从空闲检测转换到运行时挂起、以及挂起后第一次成功的恢复。这些不是错误，但也不是例行；它们是开发人员阅读日志时希望看到突出显示的事件。

第1阶段的确切代码更改在本节末尾的动手练习中列出，相应的文件在`examples/part-05/ch23-debug/stage1-logging/`下。

### 关于`uprintf`和`tprintf`的说明

`subr_prf.c`中出现两个驱动程序作者偶尔看到但很少使用的兄弟函数。

`uprintf(const char *fmt, ...)`将消息写入当前用户进程的控制终端（如果有），*而不是*写入内核消息缓冲区。它在驱动程序的`ioctl`处理程序想要产生只有运行命令的用户看到的错误消息而不污染系统日志的罕见情况下有用。在实践中，大多数驱动程序通过通常的`errno`机制返回错误代码，不使用`uprintf`。`/usr/src/sys/kern/`树中有少量调用者，主要是在内核想警告用户关于他们自己的行动的地方（例如安全相关事件）。

`tprintf(struct proc *p, int pri, const char *fmt, ...)`写入特定进程的终端。它更加罕见。第23章不直接使用这两个函数。第25章日志记录材料将在面向用户的诊断消息的上下文中回到它们。

### 格式化约定

一些小的格式化习惯使日志输出更容易阅读：

1. **每行以`\n`结束**。内核的`printf`不添加换行符。没有换行符的消息会在`dmesg`中与下一条消息连在一起，产生像`myfirst0: interrupt receivedmyfirst0: interrupt received`这样几乎无法视觉解析的输出。

2. **寄存器值使用十六进制，计数器使用十进制**。`status=0x80a1`是设备寄存器；`count=27`是计数器。在单个驱动程序中混合这两种约定会产生难以一目了然阅读的输出。`%x`和`%d`转换是保持它们视觉上区分的廉价方法。

3. **将变量值放在右边，而不是左边**。`myfirst0: dma transfer failed (err=%d)`比`myfirst0: %d was the err from dma transfer failed`更易读。人类读者从左到右解析英语，将变量数据放在行尾匹配该解析。

4. **为重复的消息类别使用短标签**。设备前缀后面的内容开头的`INFO:`、`WARN:`、`ERR:`、`DEBUG:`将日志转化为grep可以过滤的东西。第8节在其宏中使用此约定。

5. **不要在消息中包含时间戳**。内核和syslog添加自己的时间戳；手动时间戳产生重复信息和视觉噪音。

6. **不要在生产消息中包含函数名**。函数名在详细模式输出中有用（其中进入/退出宏自动打印它们），但在每条生产消息中是浪费的。像`myfirst0: DMA completed`这样的消息比`myfirst0: myfirst_dma_completion_handler: DMA completed`更易读。如果你想要函数名，打开详细模式。

7. **匹配驱动程序的风格**。如果驱动程序的其余部分使用大写"DMA"，在每条日志行中使用大写"DMA"。如果其余部分使用小写"dma"，匹配它。一致性比混合大小写更容易grep。

这些都是小习惯，但它们在日志成为读者无法交互访问的系统上错误的唯一证据时得到了回报。

### 练习：在驱动程序的关键点添加`device_printf()`

第2节的动手配套是一个简短的练习。读者应该：

1. 依次打开`myfirst_pci.c`、`myfirst_sim.c`、`myfirst_intr.c`、`myfirst_dma.c`和`myfirst_power.c`。

2. 对于每个有`device_t`或softc（持有`device_t`）在作用域内的`printf`调用，将其替换为`device_printf(dev, ...)`。

3. 对于每个错误路径消息，如果条件是真正的错误，将调用更改为`device_log(dev, LOG_ERR, ...)`，如果条件是可恢复的，更改为`device_log(dev, LOG_WARNING, ...)`。例行信息消息保持使用`device_printf`。

4. 对于附加横幅，添加一行命名驱动程序版本：`device_printf(dev, "myfirst PCI driver %s attached\n", MYFIRST_VERSION);`。在`myfirst_pci.c`顶部将`MYFIRST_VERSION`定义为字符串字面量。第23章将在第8节的最终重构中将其从`"1.5-power"`升级到`"1.6-debug"`；现在，保持为`"1.5-power-stage1"`。

5. 重建模块，加载它，并验证`dmesg | grep myfirst0`显示附加横幅。生成一个人为的DMA失败（模拟器的`sim_force_dma_error` sysctl是一种方法；第17章介绍了这个钩子），并验证错误消息以`LOG_ERR`优先级出现。

6. 提交更改，消息如"Chapter 23 Section 2: use device_printf consistently"。

读者应该随意对照`examples/part-05/ch23-debug/stage1-logging/myfirst_pci.c`和其他stage1文件检查他们的工作，但学习发生在输入中，而不是检查中。

### 第2节总结

第2节确立了日志记录的基本规则。当有`device_t`时使用`device_printf`。当优先级重要时使用带有显式优先级的`device_log`。记录足够的日志使其可读，不要记录太多使其变成噪音。例行信息路由到`LOG_INFO`，真正的错误路由到`LOG_ERR`，详细追踪路由到由sysctl控制的`LOG_DEBUG`。

第3节现在从驱动程序写入的那一刻到用户读取的那一刻跟踪日志输出，通过`dmesg`、内核消息缓冲区、`/var/log/messages`和syslog管道。驱动程序的工作在它调用`device_printf`时结束；读者用来查看驱动程序说了什么的工具从那里开始。




## 第3节：使用`dmesg`和Syslog进行诊断

第2节以驱动程序的`device_printf`调用结束。第3节从另一端开始：用户运行`dmesg`、打开`/var/log/messages`或通过syslog档案grep的那一刻。中间是内核的消息缓冲区、`syslogd`守护进程、`newsyslog`轮转器，以及一组悄然决定哪些消息出现在哪里的配置文件。理解管道是驱动程序作者不再担心"为什么我的消息不出现？"并开始调试实际驱动程序的方式。

### 内核消息缓冲区

驱动程序产生的每条内核消息首先进入**内核消息缓冲区**。这是内核在启动时分配的单一循环内存区域，在amd64 FreeBSD 14上默认大小约为192 KB。大小由`msgbufsize`可调参数设置；当前大小可以这样查看：

```sh
sysctl kern.msgbufsize
```

缓冲区是"循环的"，意思是当它填满时，新消息会覆盖最旧的消息。这对于寻找旧错误的读者很重要：如果你在五小时前重启，产生了一堆驱动程序日志输出，现在才去寻找，输出可能已经从缓冲区中滚出了。安全的假设是缓冲区在安静系统上保留几天的例行消息，在繁忙系统上保留几小时的消息。

因为缓冲区是单一的内存块，读取它代价很小。`dmesg(8)`命令通过调用`sysctl(8)`函数读取`kern.msgbuf`并打印内容来工作。`dmesg`二进制文件很小；其源代码在`/usr/src/sbin/dmesg/dmesg.c`，如果读者好奇，值得快速一看。

缓冲区的启动时部分比运行时部分更仔细地保留。在启动期间，内核将每条自动配置消息、每个驱动程序的附加横幅、每个硬件检测注释写入缓冲区。`/var/run/dmesg.boot`文件由`/etc/rc.d/dmesg`启动脚本在启动序列完成时生成，在运行时输出开始覆盖它之前捕获缓冲区的内容。在运行系统已经长时间将启动消息从活动缓冲区推出后，读取`dmesg.boot`是查看启动消息的方式：

```sh
cat /var/run/dmesg.boot | grep myfirst
```

对于驱动程序作者，`dmesg.boot`是在需要数小时后回答"我的驱动程序在启动时附加了吗？"的方式。活动`dmesg`是回答"从那时起驱动程序说了什么？"的方式。

### 实践中的`dmesg`

常见的调用值得记忆。

不带参数的`dmesg`从最旧到最新打印当前内核缓冲区内容。在运行了一段时间的系统上，这是很多输出：

```sh
dmesg
```

通过管道到`less`滚动，或通过管道到`grep`过滤：

```sh
dmesg | less
dmesg | grep myfirst
dmesg | grep -i error
dmesg | tail -50
```

`dmesg -a`打印所有消息，包括内核配置为从正常输出中抑制的优先级的消息。在某些配置上，这会显示额外的低优先级详细信息。在大多数配置上，输出与普通`dmesg`相同。

`dmesg -M /var/crash/vmcore.0 -N /var/crash/kernel.0`从保存的崩溃转储中读取消息缓冲区，而不是从活动内核。这是读者查看内核崩溃前产生的最后几条消息的方式，当崩溃本身阻止正常日志路径运行时。第4节的`kgdb`材料会回到这点。

`dmesg -c`形式（在某些系统上读取后清除缓冲区）在FreeBSD内核公开缓冲区的方式中不可用，因为缓冲区不会通过读取而排空；读取产生副本，活动缓冲区继续接收新消息。从Linux迁移的用户有时会期望`dmesg -c`，会感到惊讶。

在驱动程序开发期间的一个有用习惯是在每次模块加载、每次测试和每个有趣事件后运行`dmesg | grep myfirst | tail`。重复建立了"驱动程序刚才说了什么？"的反射，在意外发生时很有价值。

### `/var/log/messages`中的永久日志

内核消息缓冲区是易失的；重启会清空它。对于任何更长期的内容，FreeBSD使用`syslogd(8)`，标准的UNIX syslog守护进程，它在新内核产生消息时读取它们并将选定的消息写入磁盘上的文件。

几乎每个FreeBSD系统上的规范文件是`/var/log/messages`。此文件接收大多数`LOG_INFO`或更高级别的内核消息，以及大多数用户态守护进程消息。优先级和目标文件之间的映射在`/etc/syslog.conf`中。相关摘录看起来像：

```text
*.notice;kern.debug;lpr.info;mail.crit;news.err		/var/log/messages
```

语法是"设施.优先级"对。`kern.debug`意味着"优先级`LOG_DEBUG`或更高级别的所有内核消息"。`*.notice`意味着"优先级`LOG_NOTICE`或更高级别的所有设施"。读者应该知道这个语法存在；当特定类型的消息需要发送到特定文件时，理解它是有用的。

因为`/var/log/messages`是普通文本文件，读者工具包中的每个工具都可以用于它：

```sh
tail -f /var/log/messages
tail -50 /var/log/messages
grep myfirst /var/log/messages
less /var/log/messages
wc -l /var/log/messages
```

在驱动程序开发期间特别有用：

```sh
tail -f /var/log/messages | grep myfirst
```

这给出了仅驱动程序输出的实时流，这是大多数调试会话实际需要的。

当消息老化时，它们移动到`/var/log/messages.0`，然后`.1`，然后`.N.gz`，随着轮转发生。轮转由`newsyslog(8)`完成，它由`cron`的周期性作业调用并从`/etc/newsyslog.conf`读取规则。`/var/log/messages`的典型规则在文件达到大小阈值时轮转，保留一些压缩档案，并重启`syslogd`以重新打开新文件。

要搜索档案，`zgrep`是工具：

```sh
zgrep myfirst /var/log/messages.*.gz
```

或者，对于包括当前文件的整个历史：

```sh
grep myfirst /var/log/messages && zgrep myfirst /var/log/messages.*.gz
```

在寻找几天前发生的错误的读者会例行使用这些命令。

### 控制台输出与缓冲输出

一个细微差别首次遇到时会困惑许多驱动程序作者。内核有两个消息受众：内存缓冲区和系统控制台。控制台是笔记本电脑上的物理终端（或虚拟机中的VGA控制台，或无头机器中的串行控制台）。消息可以去一个、另一个或两者，取决于其优先级和当前控制台日志级别。

控制台日志级别由`kern.consmsgbuf_size`（控制台使用多少内存）和更重要的是`kern.msgbuflock`和控制台优先级过滤器控制。最简单的旋钮是：

```sh
sysctl -d kern.log_console_level
```

在FreeBSD上，高优先级消息（LOG_WARNING及以上）始终显示在控制台上。较低优先级的消息被缓冲但不显示。这就是为什么崩溃消息无论运行级别如何都会出现在控制台上：它的优先级是`LOG_CRIT`或更高，内核的控制台输出路径绕过通常的缓冲。

对于驱动程序作者，实际后果是`device_log(dev, LOG_ERR, ...)`比`device_printf(dev, ...)`更容易被人类注意到，因为前者到达控制台而后者只进入缓冲区。第23章第2节指出了这一点；第3节通过机制加强了它。

产生大量LOG_ERR消息的驱动程序会淹没控制台，在极端情况下会减慢系统，因为控制台输出路径并不便宜。限速是解决方案；第25章正确教授它。现在，教训是优先级很重要。

### 实践中的日志审查

开发期间读取驱动程序日志的常见模式如下：

1. 加载模块。
2. 运行测试。
3. 卸载模块。
4. 将`dmesg`输出复制到文件进行分析。

shell形式的序列：

```sh
# 干净状态
sudo kldunload myfirst 2>/dev/null
sudo kldload ./myfirst.ko

# 测试
dev_major=$(stat -f '%Hr' /dev/myfirst0)
echo "Running driver test..."
some_test_program

# 捕获
dmesg | tail -200 > /tmp/myfirst-test-$(date +%Y%m%d-%H%M%S).log

# 再次干净状态
sudo kldunload myfirst

# 现在悠闲地检查日志
less /tmp/myfirst-test-*.log
```

这个模式鼓励的几个习惯值得明确说明：

**每次先干净状态。** 在加载你想测试的模块之前卸载模块的任何现有实例。否则你可能意外运行旧的编译版本，并想知道为什么新代码没有预期的效果。卸载也使`dmesg`缓冲区更可读，因为你的新会话从已知状态开始。

**捕获每个有趣的会话。** 你保存到文件的日志是可以diff、grep、分享并在下周返回的日志。你留在活动缓冲区中的日志可能在你返回时已经被推出。

**在文件名中使用时间戳。** 代码片段中的`date +%Y%m%d-%H%M%S`约定产生像`myfirst-test-20260419-143027.log`这样的文件名。当你从一天的测试中积累了十几个日志文件时，这是找到你想要的那个的唯一方法。

**将`tail -N`缩小到合理窗口。** 默认是10行，对于驱动程序测试几乎永远不够。200到500通常是正确的大小：长到足以捕获完整测试周期，短到可以一次读完。如果你的测试产生更多输出，要么你的测试太大，要么你的日志太冗长。

### 过滤输出

`grep`工具链是你的朋友。一些模式经常出现，值得建立肌肉记忆。

**仅显示驱动程序的输出：**

```sh
dmesg | grep '^myfirst'
```

`^myfirst`锚点仅匹配以设备名称开头的行。没有锚点，你也会匹配文本中恰好包含"myfirst"的行（例如，关于模块文件描述符的内核消息）。

**仅显示错误：**

```sh
dmesg | grep -iE 'error|fail|warn|fault'
```

不区分大小写，使用匹配几个常见错误单词的扩展正则表达式。

**显示特定间隔：**

```sh
awk '/START OF TEST/,/END OF TEST/' /var/log/messages
```

这依赖于读者的测试工具在边界处打印"START OF TEST"和"END OF TEST"标记。这样做的脚本是其日志容易切片的脚本。

**仅提取时间戳和第一个单词：**

```sh
awk '{print $1, $2, $3, $5}' /var/log/messages | grep myfirst
```

用于繁忙日志的高级扫描。

**统计每种消息类型的数量：**

```sh
dmesg | grep myfirst | awk '{print $2}' | sort | uniq -c | sort -rn
```

这按第二个单词分组统计消息，这（使用第2节的约定）通常是类别，如INFO、WARN、ERR或DEBUG。用于发现"为什么我的驱动程序产生30,000条DEBUG消息和2条INFO消息？"

### 永久配置：`/etc/syslog.conf`和`/etc/newsyslog.conf`

对于大多数驱动程序工作，默认syslog配置就可以了。第23章不要求读者更改它。但知道这两个文件存在并大致了解它们做什么是有价值的。

`/etc/syslog.conf`在启动时和`SIGHUP`时由`syslogd`读取。它包含将"设施和优先级"映射到"目标"的规则。设施是生产者类型（`kern`、`auth`、`mail`、`daemon`、`local0`到`local7`），优先级是syslog优先级级别。驱动程序的消息始终具有设施`kern`（因为它们来自内核）和驱动程序指定的优先级。

读者读完第23章后的一个有用实验是在`/etc/syslog.conf`中添加这样的行：

```text
kern.debug					/var/log/myfirst-debug.log
```

然后touch文件并用`service syslogd reload`重新加载syslogd。现在每条内核debug级别消息都进入单独的文件。如果读者然后在`myfirst`驱动程序上启用详细模式（第8节的sysctl），单独的文件仅捕获详细输出，主`messages`文件保持干净。这是值得知道的技巧；它让读者启用深度详细日志而不让`/var/log/messages`淹没一切。

`/etc/newsyslog.conf`在轮转日志时由`newsyslog`读取。典型规则看起来像：

```text
/var/log/messages    644  5    100    *     JC
```

字段从左到右：文件路径、模式、保留的轮转数、最大大小（KB）、轮转时间、标志。对于读者刚才添加的`myfirst-debug.log`文件，适当的newsyslog规则防止日志无限增长：

```text
/var/log/myfirst-debug.log  644  3  500  *  JC
```

同样，这不是章节要求；这是值得知道的模式。

### 跨启动周期关联消息

常见的调试任务之一是"驱动程序在上次启动后工作但现在不工作"。读者想将上次启动发生的事情与现在发生的事情进行比较。两条路径：

**路径一：** 读取捕获启动时消息的`/var/run/dmesg.boot`，并与当前`dmesg`比较：

```sh
diff <(grep myfirst /var/run/dmesg.boot) <(dmesg | grep myfirst)
```

这准确显示启动后哪些消息是新的。用于发现运行时意外。

**路径二：** grep `/var/log/messages`的档案：

```sh
zgrep myfirst /var/log/messages.*.gz | head -100
```

这显示过去几天驱动程序的历史，跨重启。用于发现"它从周二开始失败"之类的模式。

在附加时记录版本的驱动程序使这更容易。如果`dmesg.boot`显示`myfirst0: myfirst PCI driver 1.5-power attached`而当前启动显示`myfirst0: myfirst PCI driver 1.6-debug attached`，读者立即知道两个是不同版本，它们之间发生了代码更改。第23章第1阶段练习正是出于这个原因添加了这条附加时版本行。

### 练习：生成已知的驱动程序消息并确认它出现

第3节的一个小型动手配套。读者应该：

1. 从第2节练习重建并加载`1.5-power-stage1`驱动程序。确认附加横幅出现在`dmesg`中。

2. 触发模拟器的`sim_force_dma_error=1` sysctl使下一次DMA传输失败。运行DMA传输。确认错误消息同时出现在`dmesg`和`/var/log/messages`中。

3. 产生一条详细信息行：将（尚不存在的）debug_verbose sysctl设置为1。由于sysctl尚不存在，此步骤将失败，读者应记下它失败的方式。这是故意的；第8节添加sysctl。

4. 对于确实出现的行，使用`awk`或`grep`仅提取带有`myfirst0:`前缀的行，并统计它们。

5. 通过在`/etc/syslog.conf`中添加`kern.debug`行来创建`/var/log/myfirst-debug.log`文件，如上所述。重新加载`syslogd`。确认文件已创建。确认尚无消息出现在其中（因为`myfirst`驱动程序尚未产生`LOG_DEBUG`输出）。记录在第8节返回此步骤。

练习很短。它的价值是读者现在实际上在真实驱动程序输出上使用了`dmesg`、`/var/log/messages`、`grep`和`syslog.conf`。后续章节将假设这些工具是熟悉的。

### 第3节总结

第3节从驱动程序调用`device_printf`或`device_log`的那一刻跟踪到用户看到输出的那一刻。消息首先进入内核的循环消息缓冲区，可通过`dmesg`访问。启动时快照保留在`/var/run/dmesg.boot`中。`syslogd`守护进程读取新消息并根据`/etc/syslog.conf`中的规则将它们写入`/var/log/messages`。`newsyslog`守护进程根据`/etc/newsyslog.conf`轮转文件。高优先级消息的控制台输出是自动的，低优先级消息是受控的。

有了清晰的日志管道，第4节转向调试基础的另一个必要部分：调试内核。错误总是静默崩溃且不为日志产生任何内容的驱动程序是无法调试的驱动程序。调试内核使这些静默错误变得响亮。




## 第4节：内核调试构建和选项

调试内核不是不同的内核；它是同一个内核，带有额外的检查、额外的调试辅助，以及发布内核没有的一小组编译功能。驱动程序源代码不变。用户空间工具不变。改变的是运行中的内核现在捕获更多自己的错误，在崩溃发生时保留更多上下文，并提供更多检查自身的方式。

本节是本章最长的一节，因为调试内核是本章其余部分站立的基础。每个后续工具都依赖于正确设置选项。DTrace依赖于`KDTRACE_HOOKS`。`ddb`依赖于`DDB`和`KDB`。`INVARIANTS`捕获否则会产生静默内存损坏的错误。`WITNESS`捕获否则会在数周后产生间歇性崩溃的锁顺序违规。每个选项都有代价、好处和适当的上下文；第4节详细介绍它们。

### 核心调试选项

FreeBSD的内核通过内核配置文件配置，如`/usr/src/sys/amd64/conf/GENERIC`。配置文件是启用或禁用编译时功能的`options`行列表。调试内核是通过从`GENERIC`（或其他基线配置）开始、添加调试选项、重建、安装和启动结果来产生的。

驱动程序调试相关的选项分为四组：内核调试器、一致性检查、锁调试基础设施和追踪基础设施。

**内核调试器组。** 这些启用内核内调试器`ddb`及其支持的基础设施。

- `options KDB`：启用内核调试器框架。这是允许任何后端（`DDB`、通过串行的`GDB`等）附加的保护伞选项。
- `options DDB`：启用内核内DDB调试器。DDB是大多数驱动程序作者使用的后端。它在内核内运行并理解内核数据结构。
- `options DDB_CTF`：启用将CTF（紧凑C类型格式）数据加载到内核中，这让`ddb`和DTrace打印有关内核结构的类型感知信息。需要在配置中有`makeoptions WITH_CTF=1`。
- `options DDB_NUMSYM`：启用DDB内的符号编号查找。用于查看特定地址。
- `options KDB_UNATTENDED`：崩溃时，不暂停进入`ddb`。相反，内核转储核心（如果配置了`dumpdev`），重启并继续。这是实验室虚拟机的正确设置，读者不想每次崩溃都丢失内核到调试器会话。
- `options KDB_TRACE`：任何进入调试器（崩溃或其他）时，自动打印堆栈跟踪。几乎每个诊断会话都节省一步。
- `options BREAK_TO_DEBUGGER`：允许串行或USB控制台上的中断进入`ddb`。在无头测试系统上有用。
- `options ALT_BREAK_TO_DEBUGGER`：相同效果的替代组合键。

典型的调试内核配置包括所有这些。amd64上的`GENERIC`内核已经启用了`KDB`、`KDB_TRACE`和`DDB`；读者启用的额外选项是`KDB_UNATTENDED`、`INVARIANTS`、`INVARIANT_SUPPORT`、`WITNESS`等。

**一致性检查组。** 这些在内核代码中启用额外的运行时断言。

- `options INVARIANTS`：在内核中启用断言。几乎每个`/usr/src/sys/*`文件都包含`KASSERT`调用，这些`KASSERT`调用仅在启用`INVARIANTS`时才执行某些操作。没有`INVARIANTS`，`KASSERT(ptr != NULL, ("oops"))`编译为无。有了`INVARIANTS`，它在失败时使内核崩溃。代价是可衡量的（繁忙系统上可能是10%的内核CPU时间），但对于驱动程序开发通常可以忽略不计，因为内核每次操作花费的时间比使用它的应用程序少。好处是驱动程序否则会静默损坏数据的错误在发生时被捕获。
- `options INVARIANT_SUPPORT`：`INVARIANTS`本身使用的机制。如果启用`INVARIANTS`则需要；如果任何可加载模块是用`INVARIANTS`构建的，也需要单独启用。忘记这会在模块加载时产生"未解析符号"错误。
- `options DIAGNOSTIC`：轻量级额外检查，不如`INVARIANTS`激进。一些子系统将其用于足够便宜以至于可以在生产中运行但太慢不适合紧凑循环的检查。

**锁调试组。** 这些启用验证驱动程序同步规范的基础设施。

- `options WITNESS`：启用锁顺序验证器。WITNESS跟踪内核持有的每个锁，记录获取锁的顺序，并在线程以如果另一个线程以相反方式持有它们可能死锁的顺序获取锁时大声抱怨。有锁顺序错误的驱动程序会在第一次以错误顺序执行时产生WITNESS报告，即使没有实际死锁发生。这是一个巨大的调试胜利，因为它在实际死锁表现之前捕获错误。
- `options WITNESS_SKIPSPIN`：跳过自旋锁的锁顺序检查。自旋锁有自己的一般检查器不适合评估的约束，检查器可能对它们产生误报。启用`WITNESS_SKIPSPIN`保持检查器对常见互斥锁情况有用。
- `options WITNESS_KDB`：锁顺序违规时，立即进入`ddb`而不是仅记录。激进；适合手动`ddb`会话容易的虚拟机。
- `options DEBUG_LOCKS`：通用锁API的额外调试（与WITNESS分开）。捕获未初始化锁的使用、错误上下文中持有的锁和相关问题。
- `options LOCK_PROFILING`：让`lockstat`测量锁争用的检测。严格来说不是调试选项（它是性能分析选项），代价高；仅在寻找争用时使用，不要默认使用。
- `options DEBUG_VFS_LOCKS`：VFS特定的锁调试。仅对文件系统驱动程序工作有用。

**追踪基础设施组。** 这些启用DTrace和KTR框架。

- `options KDTRACE_HOOKS`：内核范围的DTrace钩子。没有这个，DTrace没有可附加的东西。即使没有DTrace脚本运行，开销也很轻，因为钩子主要是无操作函数指针槽。
- `options KDTRACE_FRAME`：DTrace用于`stack()`操作的反栈帧信息。在DTrace脚本内需要有意义的堆栈跟踪。
- `makeoptions WITH_CTF=1`：一个`makeoptions`（不是`options`）行，启用内核的CTF生成。CTF是DTrace和DDB用来理解内核结构布局的类型数据。
- `options KTR`：内核内事件环形缓冲区追踪器。与`ktrace`不同；KTR是一个高性能内核事件缓冲区，主要对底层内核黑客有用。大多数驱动程序作者不需要`KTR`，可以关闭它；第23章仅为了正确命名而提及它。
- `options KTR_ENTRIES`、`KTR_COMPILE`、`KTR_MASK`：KTR配置旋钮。完整列表见`/usr/src/sys/conf/NOTES`。

对于第23章，读者需要`KDB`、`DDB`、`DDB_CTF`、`KDB_UNATTENDED`、`KDB_TRACE`、`INVARIANTS`、`INVARIANT_SUPPORT`、`WITNESS`、`WITNESS_SKIPSPIN`、`KDTRACE_HOOKS`和`makeoptions DEBUG=-g`用于符号。如果读者遵循了第22章的先决条件，这些已经启用。

### 构建调试内核

构建自定义内核是标准的FreeBSD操作。工作流程：

1. 将`GENERIC`配置复制为新名称：

   ```sh
   cd /usr/src/sys/amd64/conf
   sudo cp GENERIC MYDEBUG
   ```

2. 编辑`MYDEBUG`以添加调试选项。最小添加：

   ```text
   ident MYDEBUG

   options INVARIANTS
   options INVARIANT_SUPPORT
   options WITNESS
   options WITNESS_SKIPSPIN
   options KDB_UNATTENDED

   makeoptions DEBUG=-g
   makeoptions WITH_CTF=1
   ```

   `DDB`、`KDB`、`KDTRACE_HOOKS`和`DDB_CTF`已经在FreeBSD 14的amd64上的`GENERIC`中，所以不需要重新添加。如果你在非x86架构上，检查相应的`conf/GENERIC`文件以查看已经包含什么。

   一个经常混淆的小点：`/usr/src/sys/amd64/conf/`保存内核*配置源文件*。一些教程提到的较旧的`/usr/src/sys/amd64/compile/`目录是`config(8)`历史上填充的构建输出目录；在现代FreeBSD系统上，实际构建产品位于`/usr/obj/usr/src/amd64.amd64/sys/<KERNCONF>/`下，`compile/`路径不再是编辑或查找源代码的地方。

3. 构建内核：

   ```sh
   cd /usr/src
   sudo make -j4 buildkernel KERNCONF=MYDEBUG
   ```

   在合理的虚拟机上，这需要十到二十分钟。在较慢的虚拟机上，半小时。构建是调试工作流程中最长的单个步骤，但它每个配置更改运行一次，而不是每次测试运行一次。

4. 安装新内核：

   ```sh
   sudo make installkernel KERNCONF=MYDEBUG
   ```

   这将内核文件复制到`/boot/kernel`并将以前的内核备份到`/boot/kernel.old`。如果新内核无法启动，读者可以从引导加载程序的菜单中选择`kernel.old`并继续。

5. 重启。

6. 验证：

   ```sh
   uname -a
   ```

   输出应在内核的标识字符串中提及`MYDEBUG`。

7. 确认调试选项活动：

   ```sh
   sysctl debug.witness.watch
   sysctl kern.witness
   ```

   在启用`WITNESS`的内核上，这些产生输出。在非`WITNESS`内核上，sysctl不存在。

从"想要调试内核"到"运行调试内核"的总时间通常不到一小时。投入的时间在第一次调试会话内就会收回，因为调试内核是捕获错误的内核。

### 何时使用`INVARIANTS`

`INVARIANTS`是驱动程序开发者最重要的单个调试选项。它也是最被误解的。

当内核程序员编写`KASSERT(condition, ("message %d", val))`时，意图是：这个条件在这个点必须为真；如果它曾经为假，说明发生了代码无法恢复的错误。没有`INVARIANTS`，`KASSERT`编译为无；不检查条件。有了`INVARIANTS`，`KASSERT`评估条件并在失败时调用`panic`，打印消息。

考虑来自`myfirst`驱动程序的具体例子。DMA提交路径（第21章）持有缓冲区锁，设置进行中标志，并调用`bus_dmamap_sync`。在没有等待完成的情况下两次调用提交路径的错误是双重提交错误，它可能损坏DMA。提交路径顶部的`KASSERT`捕获它：

```c
static int
myfirst_dma_submit(struct myfirst_softc *sc, ...)
{
    KASSERT(!sc->dma_in_flight,
        ("myfirst_dma_submit: previous transfer still in flight"));
    ...
}
```

没有`INVARIANTS`，这行什么也不做。有了`INVARIANTS`，第一次错误在第一个完成之前提交第二个传输，内核会以该消息崩溃。读者查看回溯，看到双重提交，并修复错误。

这就是`INVARIANTS`价值的形状：它将静默损坏变成响亮的崩溃。权衡是，不编写`KASSERT`的驱动程序作者从`INVARIANTS`得不到任何好处；该选项仅在它运行的代码有断言时才有用。第1阶段的`myfirst`驱动程序有一些；第7节添加更多；第8节将它们移动到调试头文件。

`KASSERT`旁边还有其他断言助手：

- `MPASS(expr)`：`KASSERT(expr, (...))`的紧凑形式。扩展为带有编译器提供消息的相同检查。当条件是不言自明时有用。
- `VNASSERT(expr, vp, msg)`：专门用于vnode的断言。驱动程序作者很少直接使用它。
- `atomic_testandset_int`、`atomic_cmpset_int`：不是断言助手，而是驱动程序同步中具有自己调试感知变体的常见操作。

在其上下文中阅读`KASSERT`：它既是文档辅助也是调试辅助。查看不熟悉函数源的读者在顶部找到`KASSERT`并立即知道代码期望的不变量。随时间积累`KASSERT`的驱动程序以普通散文无法匹配的方式变成自文档化的。

### 何时使用`WITNESS`和`DEBUG_LOCKS`

`WITNESS`是驱动程序调试对的另一半。`INVARIANTS`捕获逻辑错误，`WITNESS`捕获同步错误。

基本思想：每个锁都有一个标识符，内核记录每个线程持有的锁集合，当线程尝试获取锁时，`WITNESS`检查获取顺序是否与内核历史中的先前获取一致。如果线程A获取锁X然后锁Y，后来线程B获取锁Y然后锁X，两个线程形成潜在的死锁：如果A持有X等待Y而B持有Y等待X，两者都无法继续。`WITNESS`在线程第二次尝试以错误顺序获取锁时标记不一致，即使实际死锁在测试期间没有发生。

`WITNESS`投诉看起来像：

```text
lock order reversal:
 1st 0xfffff80001a23200 myfirst_sc (myfirst_sc) @ /usr/src/sys/dev/myfirst/myfirst_pci.c:523
 2nd 0xfffff80001a23240 some_other_lock (some_other_lock) @ /some/other/file.c:789
witness_order_list_add: lock order reversal
```

消息标识两个锁、它们的地址、获取它们的文件和行号，以及逆转。驱动程序作者阅读这个并立即知道在哪里查找。

`WITNESS`与其他调试选项交互。`WITNESS_KDB`在违规时进入`ddb`而不是仅记录。`WITNESS_SKIPSPIN`跳过自旋锁，它们有不同的规则（它们不能睡眠，它们有优先级处理，顺序规范略有不同）。`DEBUG_LOCKS`在`WITNESS`之上添加额外检查：检测双重解锁、错误线程的解锁、未初始化互斥锁的使用。对于驱动程序开发，启用所有三个（`WITNESS`、`WITNESS_SKIPSPIN`、`DEBUG_LOCKS`）是默认推荐。

`WITNESS`的代价是可衡量的。每次锁获取增加几十条指令和一个缓存行的内存触及。在驱动程序繁重的工作负载上，这可能比非调试内核慢20或30%。对于开发和测试，这是可接受的。对于生产内核，`WITNESS`通常关闭。

第7节的常见错误演练包括一个`WITNESS`实验：读者有意在`myfirst`驱动程序中引入锁顺序违规，重建，重新加载，触发路径，并观察`WITNESS`捕获它。该实验是内化工具如何工作的最快方式。

### 何时使用`DEBUG_MEMGUARD`和`MEMGUARD`

`MEMGUARD`是一个更重的内存调试分配器。它在两侧带有保护页分配内存，检测写入超过分配边界，并检测释放后使用。它通过`options DEBUG_MEMGUARD`启用并通过`sysctl vm.memguard.desc`在运行时配置。

MEMGUARD代价高：由于保护页，它使用的内存远多于普通分配器，其分配路径更慢。正确使用是有针对性的：你为驱动程序大量使用的特定malloc类型启用它，而不是为整个内核。如果`myfirst`驱动程序有一个malloc类型`M_MYFIRST`，读者可以仅为该类型启用MEMGUARD：

```sh
sysctl vm.memguard.desc="M_MYFIRST"
```

重启后（MEMGUARD需要在启动时初始化其区域），`M_MYFIRST`内存的分配将通过MEMGUARD进行。超过分配末尾的写入立即崩溃。释放后使用立即崩溃。读者可以运行测试并让内核在驱动程序内存中捕获堆错误，而不减慢系统其余部分。

第23章不大量使用MEMGUARD，因为`myfirst`驱动程序的内存模式很简单。第7节将其作为驱动程序进行更复杂分配的工具提及。对于最终编写具有动态队列、缓冲池或长期分配的驱动程序的读者，MEMGUARD值得回来。

### 权衡：性能与调试可见性

首次学习选项时的诱惑是启用一切。这是开发期间的正确选择。这是生产的错误选择。

启用`INVARIANTS`、`WITNESS`、`DEBUG_LOCKS`、`KDTRACE_HOOKS`、`DDB`和`MEMGUARD`的内核明显比`GENERIC`慢：根据机器正在做什么，可能整体慢20%，锁繁重工作负载慢50%。在开发工作站上，慢20%是可以的。在生产服务器上，慢20%意味着需要多20%的硬件来做同样的工作。

调试内核和生产内核之间的区别在实践中是鲜明的。开发者运行调试内核，因为他们关心发现错误。用户运行发布内核，因为他们关心性能。在`WITNESS`和`INVARIANTS`下正确的驱动程序是正确的，句号；那些选项不改变驱动程序行为，它们只检查它。在调试内核下通过`WITNESS`实验室运行的驱动程序也会在发布内核下通过。这是使模式工作的不变量：你针对严格内核开发并针对宽松内核部署，严格内核的额外检查是保持宽松内核正确的东西。

对于读者，实际建议是：始终在你用于驱动程序工作的虚拟机上运行调试内核。不要为日常开发切换到发布内核。驱动程序错误的反馈比性能代价值得一百倍。当你准备在生产风格内核上测试驱动程序（用于时序测量、集成测试）时，为该特定测试启动发布内核，确认正确行为，然后回到调试内核。

### `ddb`命令概览

DDB是内核内调试器。当内核在编译了`DDB`的系统上崩溃时，系统不会立即重启，而是进入控制台上的调试器提示符：

```text
KDB: enter: panic
[ thread pid 42 tid 100049 ]
Stopped at:  kdb_enter+0x3b:  movq    $0,kdb_why
db>
```

在那个提示符下，读者可以检查内核状态。命令简洁直接。完整导览属于单独的参考；以下是驱动程序作者最常用的列表。

**检查命令：**

- `bt`或`backtrace`：显示当前线程的回溯。这是读者在每次崩溃时运行的第一条命令。
- `ps`：显示进程表。用于查看正在运行什么。
- `show thread <tid>`：显示特定线程的详细信息。
- `show proc <pid>`：显示特定进程的详细信息。
- `show pcpu`：显示每CPU状态，包括每个CPU上的当前线程。
- `show allpcpu`：一次显示所有每CPU状态。
- `show lockchain <addr>`：显示阻塞在特定锁上的线程链。
- `show alllocks`：显示当前持有的所有锁，跨所有线程。用于诊断明显的死锁。
- `show mbufs`：显示mbuf统计信息，如果涉及网络栈。
- `show malloc`：显示内核malloc统计信息，按类型分组。用于在崩溃前发现泄漏内存的驱动程序。
- `show registers`：显示当前CPU的寄存器。

**导航命令：**

- `x/i <addr>`：在地址处反汇编。用于查看页面错误发生的指令。
- `x/xu <addr>`：将内存转储为十六进制字节。
- `x/sz <addr>`：将内存转储为以null结尾的字符串。

**控制命令：**

- `continue`：恢复执行。如果内核足够稳定可以进入`ddb`，这有时可以让机器正常继续。通常更安全的选择是`panic`。
- `reset`：立即重启机器。
- `panic`：强制崩溃，如果设置了`dumpdev`则触发崩溃转储。
- `call <func>`：按名称调用内核函数。高级；在驱动程序调试中很少有用。

**线程遍历命令：**

- `show all procs`：列出所有进程。
- `show sleepq`：显示睡眠队列。
- `show turnstile`：显示turnstile状态（用于阻塞锁）。
- `show sema`：显示信号量。

驱动程序作者不需要记住这个列表。模式是：当你进入`ddb`时，先运行`bt`，然后`ps`，然后`show alllocks`，然后`show mbufs`，然后`show malloc`。这五条命令捕获了大多数驱动程序崩溃所需的80%信息。

一个有用的习惯是为前几次`ddb`会话打印命令并放在键盘旁边。第一次会话感觉尴尬，因为肌肉记忆还没有形成。到第三次会话，命令就流畅了。

### `kgdb`简介

`ddb`是活动调试器。`kgdb`是事后调试器。当内核在配置了`dumpdev`的情况下崩溃时，崩溃例程将完整内核内存映像写入转储设备。在下一次启动时，`savecore(8)`将转储从转储设备复制到`/var/crash`。读者可以然后用`kgdb`打开它：

```sh
sudo kgdb /boot/kernel/kernel.debug /var/crash/vmcore.0
```

（注意：`kernel.debug`是内核的调试符号版本，在内核用`makeoptions DEBUG=-g`构建时产生。没有`DEBUG=-g`，`kgdb`仍然可以打开转储但产生不太有用的输出。）

在`kgdb`内部，读者有接近普通`gdb`界面的东西。命令相同：`bt`、`frame N`、`info locals`、`print variable`、`list`等。区别是机器没有运行；内核在崩溃时刻永远暂停，每次检查都是针对那个冻结状态。你可以沿栈向上走、查看数据结构、跟随指针并推理导致崩溃的原因。你不能向前单步执行。

崩溃上的典型`kgdb`会话看起来像这样：

```text
(kgdb) bt
#0  doadump (...)
#1  kern_reboot (...)
#2  vpanic (...)
#3  panic (...)
#4  witness_checkorder (...)
...

(kgdb) frame 10
#10 myfirst_pci_detach (dev=0xfffff80001a23000) at myfirst_pci.c:789
789            mtx_destroy(&sc->lock);

(kgdb) list
784         mtx_lock(&sc->lock);
785         sc->detaching = true;
786         /* wait for all users */
787         while (sc->refs > 0)
788             cv_wait(&sc->cv, &sc->lock);
789            mtx_destroy(&sc->lock);
790         return (0);
791     }

(kgdb) print sc->refs
$1 = 0

(kgdb) print sc->lock
$2 = {
   ...
}
```

读者遍历堆栈找到驱动程序帧，查看驱动程序在做什么，并识别错误。在这个例子中，驱动程序正在销毁仍然锁定的互斥锁，因为第789行调用`mtx_destroy`而没有解锁。错误立即可见。

第23章不大量使用`kgdb`，因为实验设计为可从`ddb`和`dmesg`诊断。第7节在常见错误演练中为需要事后检查的错误类提及`kgdb`。想要更多的读者应该阅读`kgdb(1)`手册页并在已知崩溃上实验。

### 练习：构建调试内核并用`kgdb`确认符号存在

一个使本节落地的小练习。

1. 如果尚未完成，按照上述方式构建并安装`MYDEBUG`内核，带有`INVARIANTS`、`WITNESS`、`WITNESS_SKIPSPIN`和`makeoptions DEBUG=-g`。重启。

2. 确认正在运行的内核是调试构建：

   ```sh
   uname -v | grep MYDEBUG
   sysctl debug.witness.watch
   ```

3. 配置转储设备。这通常是交换分区：

   ```sh
   sudo dumpon /dev/adaNpY
   ```

   其中`adaNpY`是你的交换分区。在典型的虚拟机上，这是`ada0p3`。

4. 触发有意崩溃以产生崩溃转储。最安全的方式是以root身份运行`sysctl debug.kdb.panic=1`：

   ```sh
   sudo sysctl debug.kdb.panic=1
   ```

   系统进入`ddb`（因为这个练习没有设置`KDB_UNATTENDED`；如果设置了，系统写入转储并重启）。运行`bt`、`ps`和`show alllocks`，然后输入`panic`让转储继续。系统写入转储设备并重启。

5. 重启后，验证`savecore(8)`已复制转储：

   ```sh
   ls /var/crash
   ```

   你应该看到`vmcore.0`、`info.0`和`kernel.0`。

6. 用`kgdb`打开转储：

   ```sh
   sudo kgdb /var/crash/kernel.0 /var/crash/vmcore.0
   ```

7. 在`kgdb`内部，运行`bt`。你应该看到崩溃堆栈。用`frame N`向上走几帧。确认源行和函数参数可见；这就是调试符号带来的东西。

8. 用`quit`退出`kgdb`。

练习不是关于修复错误；它是关于证明调试基础设施工作。从这点开始，读者造成的任何真实崩溃都是完全可调试的：崩溃被保留，符号存在，调试器打开转储。这就是基础。没有它，调试主要是猜测。

### 第4节总结

第4节将读者的内核转变为调试内核。这些选项启用断言、锁顺序检查、DTrace钩子、调试符号和内核内调试器。代价是较慢的系统；好处是系统大声捕获自己的错误。`ddb`是活动检查工具；`kgdb`是事后检查工具。两者都与读者现在运行的相同调试内核一起工作。

有了日志管道（第3节）和调试内核（第4节），读者有了每个后续工具假设的基础设施。第5节介绍DTrace，将调试内核转变为实时追踪和测量平台的工具。




## 第5节：使用DTrace进行实时内核检查

DTrace是FreeBSD调试工具包的手术刀。当`printf`是钝器而`ddb`是锤子时，DTrace是手术刀：它让你在不修改源、重建内核或停止系统的情况下，对内核的实时行为提出具体、尖锐的问题并获得答案。学会DTrace的驱动程序作者达到了一种新的生产力水平：说"我想知道驱动程序的中断多久触发一次"并在十秒钟内得到答案而不更改一行代码的能力。

本节以驱动程序作者需要的详细程度介绍DTrace。它不试图成为DTrace参考；优秀的`dtrace(1)`手册页、在线提供的DTrace指南和书籍*Illumos DTrace Toolkit*是更深入的参考。第5节给读者的是足够的DTrace来检测`myfirst`驱动程序、测量其行为并理解DTrace能做什么和不能做什么。

### DTrace是什么？

DTrace是一个**动态追踪框架**。"动态"这个词很重要：它不需要内核已经构建了每个追踪点的知识；不需要运行代码被修补；不需要读者重建和重启。你加载DTrace模块，请求探针，框架将检测附加到运行中的内核代码。当检测代码运行时，检测触发，DTrace记录事件。当你完成后，你分离，检测被移除。

框架起源于Solaris并移植到FreeBSD。它位于`/usr/src/cddl/`下（源代码树的CDDL许可部分），因为它继承了Solaris许可。用户空间工具是`dtrace(1)`；内核端基础设施是一组可以按需加载和卸载的模块。

DTrace有三个组织概念：**提供者**、**探针**和**动作**。

**提供者**是追踪点的来源。FreeBSD附带的提供者示例：`fbt`（函数边界追踪，每个内核函数的入口和退出各一个探针）、`syscall`（每个系统调用入口和返回各一个探针）、`sched`（调度器事件）、`io`（块I/O事件）、`vfs`（文件系统事件）、`proc`（进程事件）、`vm`（虚拟内存事件）、`sdt`（静态定义追踪，用于编译到内核中的探针）和`lockstat`（锁操作）。一些提供者（`fbt`、`syscall`）一旦模块加载就始终可用；其他（`sdt`）仅在内核代码显式定义探针时存在。

**探针**是读者可以追踪的特定点。探针有四部分名称，格式为`provider:module:function:name`。例如，`fbt:kernel:myfirst_pci_attach:entry`是`kernel`模块（主内核二进制，不是可加载模块）中`fbt`提供者在函数`myfirst_pci_attach`上的入口探针。通配符`*`匹配任何段：`fbt::myfirst_*:entry`匹配任何模块中以`myfirst_`开头的每个函数在函数入口处。

**动作**是探针触发时DTrace做的事情。最简单的动作是`{ trace(...) }`，它记录参数。更有趣的动作聚合：`@counts[probefunc] = count()`统计每个函数的探针触发了多少次。D语言（DTrace的脚本语言）类似C但更安全：它没有循环、没有函数调用、没有内存分配，并在严格安全约束下在内核中运行。

DTrace的力量来自组合：你启用特定探针，附加特定动作，仅观察你关心的事件，你不触及源代码就做到了这一切。

### 启用DTrace支持

第4节介绍了DTrace需要的内核选项：`options KDTRACE_HOOKS`、`options KDTRACE_FRAME`、`options DDB_CTF`和`makeoptions WITH_CTF=1`。如果读者用这些选项构建了`MYDEBUG`内核，DTrace准备就绪。

用户空间端需要加载DTrace模块。总括模块是`dtraceall`，它拉入所有提供者：

```sh
sudo kldload dtraceall
```

或者，可以加载单个提供者：

```sh
sudo kldload dtrace          # 核心框架
sudo kldload dtraceall       # 所有提供者（开发推荐）
sudo kldload fbt             # 仅函数边界追踪
sudo kldload systrace        # 仅syscall追踪
```

对于开发，`dtraceall`是最简单的选择：读者可能想要的任何提供者都可用。

确认DTrace工作：

```sh
sudo dtrace -l | head -20
```

这列出DTrace知道的前二十个探针。在加载了`dtraceall`的默认FreeBSD 14内核上，输出包括数十万个探针，大多数来自`fbt`提供者（每个内核函数有入口和返回探针）。快速计数：

```sh
sudo dtrace -l | wc -l
```

典型数字是40,000到80,000个探针，取决于加载了哪些模块。

仅列出`myfirst`驱动程序函数中的探针：

```sh
sudo dtrace -l -n 'fbt::myfirst_*:'
```

如果驱动程序已加载，这产生每个`myfirst_*`函数的入口和返回探针列表。如果驱动程序未加载，列表为空：`fbt`探针仅对内核已加载的代码存在。

### 编写简单的DTrace脚本

最简单的DTrace调用追踪单个探针并在它触发时打印：

```sh
sudo dtrace -n 'fbt::myfirst_pci_attach:entry'
```

语法是`provider:module:function:name`。`fbt::myfirst_pci_attach:entry`中的双冒号使用默认空模块（匹配任何模块）。DTrace每个探针触发打印一行：

```text
CPU     ID                    FUNCTION:NAME
  2  28791         myfirst_pci_attach:entry
```

这确认函数在CPU 2上被调用，内部探针ID为28791。对于一次性"这个函数运行了吗？"问题，这已经足够。

稍微更有用的脚本追踪入口和返回：

```sh
sudo dtrace -n 'fbt::myfirst_pci_attach:entry,fbt::myfirst_pci_attach:return'
```

每行显示入口和返回何时触发。两者一起告诉读者函数运行并返回而没有崩溃。

要一次追踪许多函数，使用通配符：

```sh
sudo dtrace -n 'fbt::myfirst_*:entry'
```

每个`myfirst_*`函数的入口产生一行。对于调试附加路径的读者，这是实时查看调用序列的廉价方式。

要捕获参数，添加D语言动作：

```sh
sudo dtrace -n 'fbt::myfirst_pci_suspend:entry { printf("%d: suspending %s", timestamp, stringof(args[0]->name)); }'
```

`args[0]`指的是被探函数的第一个参数。对于FBT探针，`args[0]`是函数的第一个参数。对于`myfirst_pci_suspend`，第一个参数是`device_t`，`args[0]->name`（如果FreeBSD的device_t布局有DTrace可访问的`name`字段）显示设备名称。

注意：`device_t`实际上是指向不透明结构的指针，DTrace只能在加载了CTF数据时跟随类型感知的指针链。这就是为什么`options DDB_CTF`和`makeoptions WITH_CTF=1`重要：没有它们，DTrace将`args[0]`视为整数，无法有意义地解引用它。有了它们，DTrace知道`args[0]`是`device_t`（即`struct _device *`），可以遍历结构。

更有趣的脚本统计每个`myfirst_*`函数触发多少次：

```sh
sudo dtrace -n 'fbt::myfirst_*:entry { @counts[probefunc] = count(); }'
```

让它在驱动程序被练习时运行一分钟。按Ctrl-C。DTrace打印直方图：

```text
  myfirst_pci_attach                                                1
  myfirst_pci_detach                                                1
  myfirst_pci_suspend                                              42
  myfirst_pci_resume                                               42
  myfirst_intr_filter                                           10342
  myfirst_rx_task                                                5120
  myfirst_dma_submit                                             1024
```

读者一眼就能看到驱动程序的调用分布：两次attach/detach、四十二次suspend/resume循环、一万次中断过滤器、五千次rx任务、一千次DMA提交。任何意外的数字都是调查的信号。如果读者期望中断和任务一一对应，却看到二对一，就有错误。

### 测量函数延迟

DTrace最有用的习惯之一是测量函数花费多长时间。模式是在入口时打时间戳，在返回时打时间戳，然后相减：

```sh
sudo dtrace -n '
fbt::myfirst_pci_suspend:entry { self->start = timestamp; }
fbt::myfirst_pci_suspend:return /self->start/ {
    @times = quantize(timestamp - self->start);
    self->start = 0;
}'
```

第一个子句将入口时间戳存储在线程局部变量`self->start`中。第二个子句，以`self->start`非零为前提，计算差值，将其添加到quantize聚合（对数直方图），并清除变量。让它在读者练习挂起循环时运行，按Ctrl-C，DTrace打印直方图：

```text
           value  ------------- Distribution ------------- count
            1024 |                                         0
            2048 |@@@@                                     4
            4096 |@@@@@@@@@@@@@@@@@@@@@@@@@@@@             28
            8192 |@@@@@@@@                                 8
           16384 |@@                                       2
           32768 |                                         0
```

数字以纳秒为单位。这说：大多数suspend调用花费在4,096到8,192纳秒之间（4到8微秒），尾部达到16,384纳秒（16微秒）。对于应该快速的suspend调用，这是告诉读者驱动程序的规范是否工作的测量类型。

对于不同的见解，使用`avg()`或`max()`聚合：

```sh
sudo dtrace -n '
fbt::myfirst_pci_suspend:entry { self->start = timestamp; }
fbt::myfirst_pci_suspend:return /self->start/ {
    @avg = avg(timestamp - self->start);
    @max = max(timestamp - self->start);
    self->start = 0;
}'
```

这最后打印平均和最大suspend时间。用于汇总报告。

### 使用`syscall`提供者

`syscall`提供者显示每个系统调用。这对于驱动程序工作特别有用，因为用户空间对驱动程序的调用（open、read、write、ioctl）首先通过syscall层。监视特定文件描述符上系统调用的脚本是查看用户程序如何使用驱动程序的轻量级方式：

```sh
sudo dtrace -n 'syscall::ioctl:entry /execname == "myftest"/ { printf("%s ioctl on fd %d", execname, arg0); }'
```

这打印名为`myftest`的进程发出的每个`ioctl`系统调用。`execname`变量是一个内置的DTrace变量，保存当前进程的可执行文件名。在此脚本运行时运行测试程序`myftest`（一个练习驱动程序的小型用户程序）给出了程序发出的每个ioctl的实时视图。

### 使用`sched`提供者

`sched`提供者跟踪调度器事件：线程切换、唤醒、入队。它对于理解内核中的并发，包括驱动程序中断处理有用：

```sh
sudo dtrace -n 'sched:::on-cpu /execname == "myftest"/ { @[cpu] = count(); }'
```

这统计每个CPU上`myftest`程序被调度到该CPU的次数。用于理解调度器是将程序保持在一个核心上还是跨核心跳动，这会影响驱动程序的缓存行为。

### 使用`io`提供者

`io`提供者显示块I/O事件。与`myfirst`（通用PCI驱动程序）不直接相关，但值得知道。显示每个磁盘I/O的脚本：

```sh
sudo dtrace -n 'io:::start { printf("%s %d", execname, args[0]->bio_bcount); }'
```

对于存储驱动程序，`io`是必不可少的。对于`myfirst`它是外围的。第23章不深入探索它。

### 静态定义追踪（SDT）

`fbt`提供者检测每个函数的入口和返回。这很灵活但粗糙：你得到每个函数的入口/返回探针，而你经常想在函数内部特定有趣点的探针。

SDT（静态定义追踪）解决了这个问题。SDT让驱动程序作者在源代码中的特定位置添加命名探针，DTrace使它们可见。当DTrace没有监视时，探针基本上免费：它们编译为单个无操作指令。当DTrace附加时，无操作被替换为触发探针的陷阱。

SDT机制位于`/usr/src/sys/sys/sdt.h`。基本宏是：

- `SDT_PROVIDER_DEFINE(name)`：声明提供者。通常每个驱动程序一次，在主源文件中。
- `SDT_PROBE_DEFINEn(provider, module, function, name, "arg1-type", "arg2-type", ...)`：声明带有N个类型参数的探针。
- `SDT_PROBEn(provider, module, function, name, arg1, arg2, ...)`：以给定参数触发探针。

`myfirst`驱动程序的最小示例。在`myfirst_pci.c`顶部附近：

```c
#include <sys/sdt.h>

SDT_PROVIDER_DEFINE(myfirst);
SDT_PROBE_DEFINE2(myfirst, , attach, entry,
    "struct myfirst_softc *", "device_t");
SDT_PROBE_DEFINE2(myfirst, , attach, return,
    "struct myfirst_softc *", "int");
SDT_PROBE_DEFINE3(myfirst, , dma, submit,
    "struct myfirst_softc *", "bus_addr_t", "size_t");
SDT_PROBE_DEFINE2(myfirst, , dma, complete,
    "struct myfirst_softc *", "int");
```

然后在驱动程序代码中：

```c
int
myfirst_pci_attach(device_t dev)
{
    struct myfirst_softc *sc = device_get_softc(dev);
    int error;

    SDT_PROBE2(myfirst, , attach, entry, sc, dev);
    ...
    error = ...;  /* real attach code */
    ...
    SDT_PROBE2(myfirst, , attach, return, sc, error);
    return (error);
}
```

从用户空间，读者现在可以通过DTrace观察`myfirst`探针：

```sh
sudo dtrace -n 'myfirst:::' | head
```

这显示每个myfirst探针。仅监视DMA事件：

```sh
sudo dtrace -n 'myfirst:::dma-submit'
sudo dtrace -n 'myfirst:::dma-complete'
```

按DMA大小统计提交：

```sh
sudo dtrace -n 'myfirst:::dma-submit { @[args[2]] = count(); }'
```

这最后打印追踪期间DMA传输大小的直方图。对于驱动程序调试会话，那是将猜测转化为证据的数据类型。

第23章的第2阶段正是向`myfirst`驱动程序添加这些探针。第5节末尾的练习贯穿添加过程。相应的文件位于`examples/part-05/ch23-debug/stage2-sdt/`。

使用SDT的真实驱动程序值得研究。`/usr/src/sys/dev/virtio/virtqueue.c`定义：

```c
SDT_PROVIDER_DEFINE(virtqueue);
SDT_PROBE_DEFINE6(virtqueue, , enqueue_segments, entry, "struct virtqueue *",
    ...);
SDT_PROBE_DEFINE1(virtqueue, , enqueue_segments, return, "uint16_t");
```

并在`virtqueue_enqueue`中触发它们：

```c
SDT_PROBE6(virtqueue, , enqueue_segments, entry, vq, desc, head_idx, ...);
...
SDT_PROBE1(virtqueue, , enqueue_segments, return, idx);
```

读者可以在带有virtio-backed设备的系统上（虚拟机中常见）运行`sudo dtrace -l -n 'virtqueue:::' `并立即看到这些探针。追踪它们给出了用`printf`不可能复现的virtqueue活动实时视图。

### 驱动程序工作有用的提供者

驱动程序作者最常用的提供者总结：

- **`fbt`**：函数边界追踪。每个内核函数有入口和返回探针。最适合"这个函数运行了吗？"和"它看到了什么参数？"。
- **`sdt`**：静态定义追踪。驱动程序显式添加的探针。最适合在特定点观察驱动程序特定事件。
- **`syscall`**：系统调用追踪。最适合看用户程序要求内核做什么。
- **`sched`**：调度器事件。最适合理解并发和CPU使用。
- **`io`**：块I/O事件。最适合存储工作。
- **`vfs`**：文件系统事件。最适合文件系统驱动程序工作。
- **`proc`**：进程生命周期事件。最适合进程创建、退出和信号交付。
- **`lockstat`**：锁操作。最适合锁争用分析；活动时代价高。
- **`priv`**：特权相关事件。用于安全审计。

大多数驱动程序工作大量使用`fbt`、`sdt`和`syscall`，偶尔触及其他。

### 常见DTrace陷阱

几个绊倒新DTrace用户的模式。

**缓冲区太小。** DTrace有每CPU缓冲区；如果脚本触发探针比DTrace能排空它们的速度快，事件会丢弃。读者看到像`dtrace: X drops on CPU N`的消息。修复通常是使前提更紧（触发更少），或用`-b 16m`增加缓冲区大小。

**忘记清除线程局部变量。** 在延迟测量模式中，脚本在入口设置`self->start`并在返回检查`self->start != 0`。如果读者忘记在使用后清除它（`self->start = 0;`），变量累积状态，后续追踪会困惑。始终配对设置和清除。

**当`printf()`更清晰时使用`trace()`。** `trace(x)`以难以阅读的简洁格式打印值。`printf("x=%d", x)`通常更好。

**作为非特权用户运行。** DTrace需要root。作为普通用户运行`dtrace`会以权限错误失败。始终使用`sudo dtrace`。

**不存在的探针。** 如果探针名称打错或模块未加载，`dtrace`打印错误。修复是在尝试使用之前用`dtrace -l`确认探针存在。

**`stringof()`陷阱。** 打印内核字符串指针时，读者经常需要`stringof(ptr)`，不是`ptr`，因为DTrace不自动跟随内核指针。忘记这会产生垃圾输出。

### 练习：使用DTrace测量驱动程序读取操作花费多长时间

第5节的动手配套。

1. 确保`MYDEBUG`内核正在运行且`dtraceall`已加载。

2. 编写一个从`/dev/myfirst0`读取一百次的用户程序。程序可以很简单：

   ```c
   #include <fcntl.h>
   #include <unistd.h>
   int main(void) {
       int fd = open("/dev/myfirst0", O_RDWR);
       char buf[4096];
       for (int i = 0; i < 100; i++)
           read(fd, buf, sizeof(buf));
       close(fd);
       return 0;
   }
   ```

   用`cc -o myftest myftest.c`编译。

3. 在一个终端中，在驱动程序的`read`路径上运行DTrace延迟脚本。假设驱动程序的读取函数是`myfirst_read`：

   ```sh
   sudo dtrace -n '
   fbt::myfirst_read:entry { self->start = timestamp; }
   fbt::myfirst_read:return /self->start/ {
       @times = quantize(timestamp - self->start);
       self->start = 0;
   }'
   ```

4. 在另一个终端中，运行用户程序：

   ```sh
   ./myftest
   ```

5. 在DTrace脚本上按Ctrl-C。读取直方图。大多数读取应该在低微秒范围内（几千纳秒）。

6. 截屏或保存输出。与你的预期比较。如果读取花费比预期更长，那是调试会话的开始。

可选：在`fbt::myfirst_intr_filter:entry`上运行相同的脚本并观察中断延迟分布。

### 第5节总结

第5节介绍了DTrace作为调试工具包的手术刀。DTrace是动态的：探针附加到运行代码而不需要重建。它是安全的：D语言没有循环或内存分配。它是广泛的：提供者暴露内核中数十万个观察点。对于驱动程序工作，`fbt`（每个函数的自动探针）和`sdt`（特定点的自定义探针）的组合给了读者在不修改源代码的情况下回答几乎所有"驱动程序在做什么？"问题的能力。

DTrace不活动时的代价基本上为零：探针在内核中作为无操作指令或函数指针槽存在。活动时的代价取决于脚本做什么：简单计数是便宜的，复杂直方图更昂贵，每个探针都打印的脚本可能淹没缓冲区。理解权衡是编写有效DTrace脚本的一部分。

有了日志（第2节）、日志管道（第3节）、调试内核（第4节）和DTrace（第5节），读者有四个互补的内核行为视图。第6节添加驱动程序工作的最后一个必要视图：`ktrace`，显示用户程序通过驱动程序接口进入内核时在做什么的工具。


## 第6节：使用ktrace和kdump追踪内核活动

第5节介绍了DTrace，它从内核内部观察驱动程序。第6节转向另一个方向：从用户程序的角度观察驱动程序接口。`ktrace`工具记录进程对内核发出的每一个系统调用，`kdump`将这些记录转换为可读的文本。对于驱动程序作者，这种用户侧视角是不可或缺的，它回答了"用户程序到底在请求什么？"的问题。

### 什么是ktrace？

`ktrace`是FreeBSD的用户空间追踪工具。当内核编译时启用了`options KTRACE`（`GENERIC`内核中默认启用），内核可以在任意进程上记录以下事件：

- 每个系统调用的入口和返回，包括参数和返回值。
- 传递给进程的每个信号。
- 每个`namei`（路径名查找）事件。
- 传递给read或write的每个I/O缓冲区。
- 上下文切换（粗粒度）。

追踪记录被写入二进制文件（默认为`ktrace.out`）。要读取它，用户运行`kdump(1)`，它将二进制追踪记录转换为人类可读的文本。

`ktrace`是一个简单的工具，比DTrace古老得多，某些接口显示出其年代感。但对于"这个进程让内核做什么？"这个核心问题，它是理想的。

实现在`/usr/src/sys/kern/kern_ktrace.c`，用户侧头文件在`/usr/src/sys/sys/ktrace.h`。用户空间工具位于`/usr/src/usr.bin/ktrace/`和`/usr/src/usr.bin/kdump/`。

内核需要`options KTRACE`来支持`ktrace`。此选项在每种架构的`GENERIC`中默认启用；读者不需要重建内核即可使用`ktrace`。

### 启动追踪

最简单的调用通过PID附加到正在运行的进程：

```sh
sudo ktrace -p 12345
```

这告诉内核开始记录PID为12345的进程。内核将事件写入当前目录的`ktrace.out`。完成后，停止追踪：

```sh
sudo ktrace -C
```

或者更精确地，停止追踪特定PID：

```sh
sudo ktrace -c -p 12345
```

要从头开始运行一个带追踪的命令：

```sh
sudo ktrace /path/to/program arg1 arg2
```

这启动程序并从第一个系统调用开始追踪。程序退出后，追踪文件保留供读者检查。

有用的标志：

- `-d`：也追踪后代进程。如果程序fork，子进程也会被追踪。
- `-i`：在`exec`之间继承追踪。如果程序`exec`另一个程序，追踪继续。没有`-i`，追踪在exec处停止。
- `-t <mask>`：选择要记录的事件类型。掩码是一组字母代码，如`cnios`（调用、namei、I/O、信号）。默认是大多数类型。
- `-f <file>`：写入指定文件而不是`ktrace.out`。
- `-C`：清除所有追踪（停止所有追踪）。

常见的组合：

```sh
sudo ktrace -di -f mytrace.out -p 12345
```

追踪该进程、所有后代、跨越exec，写入`mytrace.out`。

### 使用`kdump`读取输出

二进制追踪文件本身是无用的。`kdump`将其转换：

```sh
kdump -f mytrace.out
```

输出如下：

```text
 12345 myftest  CALL  open(0x8004120a0,0x0)
 12345 myftest  NAMI  "/dev/myfirst0"
 12345 myftest  RET   open 3
 12345 myftest  CALL  ioctl(0x3,0x20006601,0x7fffffffe8c0)
 12345 myftest  RET   ioctl 0
 12345 myftest  CALL  read(0x3,0x7fffffffe900,0x1000)
 12345 myftest  GIO   fd 3 read 4096 bytes
                "... data bytes ..."
 12345 myftest  RET   read 4096/0x1000
```

每行有一个PID、进程名、事件类型和事件特定数据：

- `CALL`行显示系统调用入口，参数为十六进制数。
- `NAMI`行显示路径名查找，带有解析后的字符串。
- `RET`行显示系统调用返回及其值。
- `GIO`行显示通用I/O缓冲区，后跟缓冲区内容的转储。
- `PSIG`行显示信号传递。

输出很密集，但经过练习后可读。驱动程序工作的常见模式：

1. 在`ktrace`下运行用户程序。
2. 查找`/dev/myfirst0`的`open`。注意内核返回的文件描述符。
3. 将后续行筛选到该fd上的操作：`ioctl`、`read`、`write`、`close`。
4. 记录确切的参数，特别是`ioctl`，其命令编号编码了驱动程序被要求执行的操作。

例如，上面的`ioctl(0x3,0x20006601,0x7fffffffe8c0)`行显示程序在fd 3上发出ioctl，命令为`0x20006601`。该命令是`_IO`/`_IOR`/`_IOW`/`_IOWR`编码；低16位是命令号，高位是方向和大小。调试ioctl错误的驱动程序作者可以查看此命令，将其与驱动程序头文件中的定义比较，确认程序正在发送驱动程序期望的命令。

### 有用的`kdump`筛选器

`kdump`接受几个筛选输出的标志：

- `-t <mask>`：按类型筛选，与`ktrace -t`使用相同的掩码。
- `-E`：显示距前一事件的经过时间。
- `-R`：显示相对时间戳而不是绝对时间戳。
- `-l`：在每行显示PID和进程名。
- `-d`：十六进制转储I/O缓冲区（`GIO`行默认启用）。
- `-H`：更可读的十六进制转储。

常见的优化：

```sh
kdump -f mytrace.out -E -t cnri
```

显示经过时间、系统调用入口/返回加上namei加上I/O。

管道传给`grep`也很有用：

```sh
kdump -f mytrace.out | grep -E 'myfirst|ioctl|open|read|write|close' | less
```

这只显示与驱动程序相关的行。

### 查找与驱动程序相关的系统调用

用户程序通过五个主要系统调用与驱动程序通信：`open`、`ioctl`、`read`、`write`和`close`。追踪用户程序的驱动程序作者会关注这些：

- **`open("/dev/myfirst0", ...)`**：程序打开设备。返回值是后续系统调用使用的文件描述符。
- **`ioctl(fd, cmd, arg)`**：程序发出设备特定的命令。`cmd`编码操作；`arg`通常是指向结构的指针。
- **`read(fd, buf, size)`**：程序将`size`字节读入`buf`。
- **`write(fd, buf, size)`**：程序从`buf`写入`size`字节。
- **`close(fd)`**：程序关闭描述符。

按顺序显示这些的追踪记录是驱动程序接口用户侧视图。程序传递错误大小给`read`的错误在追踪中立即显示为`read(fd, buf, 0)`（零大小）。程序忘记调用`close`的错误显示为退出前缺少`close`调用。程序使用未定义的`ioctl`命令的错误显示为`cmd`参数中的异常十六进制值。

### 比较`ktrace`和DTrace

这两个工具在某些方面重叠，在其他方面互补。粗略指南：

在以下情况使用`ktrace`：

- 你想查看用户程序在做什么，从程序的角度。
- 你想要一个可以保存和以后查看的记录。
- 你想要最少的设置（不需要重建内核，不需要编写脚本）。
- 你想捕获系统调用的参数，包括指针目标字符串和I/O缓冲区。

在以下情况使用DTrace：

- 你想查看内核内部发生了什么。
- 你想测量延迟、分布或争用。
- 你想跨多个事件聚合数据。
- 你需要在不修改程序或在其下运行包装器的情况下追踪。

对于大多数驱动程序工作，读者会两者都用。`ktrace`回答"程序在请求什么？"。DTrace回答"驱动程序在如何处理它？"。两者的结合才是完整的画面。

### 关于`KTR`与`ktrace`的说明

名称会迷惑新的驱动程序作者。`ktrace`（小写）是上述用户空间追踪工具。`KTR`（大写）是内核内事件追踪框架，通过`options KTR`启用。它们是不同的。

`KTR`是一个高性能的内核事件环形缓冲区，供内核开发者调试内核本身。驱动程序作者可以在代码中添加`CTR`（category-0、category-1等）宏，通过`ddb`的`show ktr`命令观察它们。实际上，`KTR`被少数核心内核子系统（调度器、锁定、某些设备子系统）使用，个别驱动程序很少使用。第23章不再进一步讨论`KTR`；好奇的读者可以阅读`/usr/src/sys/sys/ktr.h`了解宏定义，阅读`/usr/src/sys/conf/NOTES`了解配置选项。

### 练习：追踪访问驱动程序的简单用户程序

第6节的动手配套练习。

1. 编写一个小用户程序`myftest.c`，打开`/dev/myfirst0`，发出一个ioctl，读4 KB，写4 KB，然后关闭。最简单的版本：

   ```c
   #include <sys/ioctl.h>
   #include <fcntl.h>
   #include <unistd.h>
   #include <stdio.h>

   int
   main(void)
   {
       int fd, error;
       char buf[4096];

       fd = open("/dev/myfirst0", O_RDWR);
       if (fd < 0) { perror("open"); return 1; }

       error = ioctl(fd, 0x20006601 /* placeholder */, NULL);
       printf("ioctl: %d\n", error);

       error = read(fd, buf, sizeof(buf));
       printf("read: %d\n", error);

       error = write(fd, buf, sizeof(buf));
       printf("write: %d\n", error);

       close(fd);
       return 0;
   }
   ```

   编译：`cc -o myftest myftest.c`。

2. 追踪程序：

   ```sh
   sudo ktrace -di -f mytrace.out ./myftest
   ```

3. 读取追踪：

   ```sh
   kdump -f mytrace.out | less
   ```

4. 查找与`/dev/myfirst0`相关的行：

   ```sh
   kdump -f mytrace.out | grep -E 'myfirst0|CALL|RET|NAMI' | less
   ```

5. 确认每个系统调用都在那里，参数符合预期。对于ioctl，手动解码命令号：`0x20006601`表示方向IOC_OUT（高字节0x20），大小0（接下来的14位），类型'f'（0x66），命令01。定义了`MYFIRST_IOC_RESET = _IO('f', 1)`的驱动程序会看到完全相同的命令。

6. （可选）在第二个终端中用DTrace运行程序，统计驱动程序函数：

   ```sh
   sudo dtrace -n 'fbt::myfirst_*:entry { @[probefunc] = count(); }' &
   ./myftest
   ```

   将DTrace计数与ktrace序列比较。读者应该看到每个用户系统调用触发了特定的驱动程序函数：`open` → `myfirst_open`，`ioctl` → `myfirst_ioctl`，`read` → `myfirst_read`，等等。

该练习将工具建立在具体、实际的追踪中。每位驱动程序作者在其职业生涯的余下时间里都会在工具包中使用这种模式：用`ktrace`进行用户侧追踪，用DTrace进行内核侧追踪，通过时间和PID关联。

### 第6节总结

第6节介绍了`ktrace`和`kdump`，这对用户侧追踪工具补充了DTrace。`ktrace`记录进程发出的系统调用；`kdump`将记录转换为文本；两者的结合从用户程序的角度显示驱动程序的接口。对于驱动程序作者，这种视角对于调试源自驱动程序边界用户侧的错误至关重要。

有了日志、日志分析、调试内核、DTrace和`ktrace`在手，读者有了处理大多数驱动程序调试问题的五个工具。第7节转向这些工具最常发现的错误类别，每类错误的特征症状以及最佳诊断工具。


## 第7节：诊断常见驱动程序错误

前六节构建了一个工具箱：日志、日志检查、调试内核、DTrace和`ktrace`。第7节使用该工具箱处理读者实际会遇到的问题。每位有经验的FreeBSD驱动程序作者至少都遇到过这些错误中的每一种。本节的目的是描述症状、解释根本原因，并将每类错误匹配到最高效诊断它的工具。

下面列出的错误都不奇特。它们源于普通的编码错误、对内核环境的不完全理解，或在一个平台上成立但在另一个平台上不成立的假设。学会识别症状的读者将节省数小时的挫败，因为修复的第一步永远是正确分类。

### 7.1 内存泄漏：安静的膨胀

内核模块中的内存泄漏比用户程序中的更难检测。没有透明运行的`valgrind`等价物，内核在模块卸载时不会退出，因此泄漏即使在驱动程序消失后仍继续消耗内存。对于长期运行的系统，小泄漏是致命的：它分配的池无限增长，其他子系统开始失败，最终机器耗尽内核内存并崩溃。

特征症状很简单：一个`malloc`池增长但永不收缩。暴露该症状的工具是`vmstat -m`：

```sh
vmstat -m | head -1
vmstat -m | grep -E 'Type|myfirst'
```

输出如下：

```text
         Type InUse MemUse Requests  Size(s)
      myfirst    12     3K       48  256
```

重要的三列是`InUse`、`MemUse`和`Requests`。`InUse`是该池当前未完成分配的数量。`MemUse`是这些分配占用的总内存，以KB为单位。`Requests`是自池创建以来进行的分配总数，包括后来释放的那些。

健康的池在稳定负载下`InUse`大致保持恒定。泄漏的池`InUse`无限增长。

要确认泄漏，运行锻炼驱动程序的工作负载，然后每隔几分钟取两次`vmstat -m`快照：

```sh
vmstat -m | grep myfirst
# 运行工作负载5分钟
vmstat -m | grep myfirst
```

如果第一个显示`InUse=12`而第二个显示`InUse=4800`，且工作负载不应增加四千个分配，则驱动程序正在泄漏。

第二步是识别有问题的路径。每个`malloc(9)`调用与一个预期的`free(9)`配对。泄漏在于任何分配而不释放的路径。在驱动程序中搜索`malloc(..., M_MYFIRST, ...)`和`free(..., M_MYFIRST)`，验证每个分配在每个退出路径上都有匹配的释放。

驱动程序代码中最常见的泄漏模式：

1. **错误路径泄漏。** 分配成功，后续步骤失败，错误返回跳过了`free`。修复是使用`goto fail;`标签的单个清理路径，如第18章所述。

2. **条件泄漏。** 内存仅在设置了标志时释放，有时标志未设置。修复是无条件释放或更仔细地跟踪所有权。

3. **忘记上下文析构器。** 对象按打开文件描述符分配并存储在`si_drv1`或类似位置，但`d_close`处理程序未释放它。修复是将`d_close`（或`cdev`销毁回调）视为`d_open`的对称对应。

4. **定时器或任务队列泄漏。** 任务被调度，其清理例程为延迟处理分配内存，但任务在运行前被取消，分配的缓冲区从未释放。

识别路径后，添加缺失的`free`，重新加载模块，重新运行工作负载，用`vmstat -m`确认`InUse`现在稳定。添加少量日志通常有用：分配路径和释放路径中的`device_printf`快速揭示`dmesg`中分配与释放的比例。对于使用`DPRINTF`宏的驱动程序，专用的`MYF_DBG_MEM`类使内存追踪可选且可在运行时配置，这正是第8节将为`myfirst`驱动程序实现的内容。

DTrace也可以帮助这里，特别是在启用了`KDTRACE_HOOKS`的内核中。简单的一行脚本统计驱动程序的`malloc`和`free`调用并显示任何不平衡：

```sh
dtrace -n 'fbt::malloc:entry /execname == "kernel"/ { @["malloc"] = count(); }
           fbt::free:entry   /execname == "kernel"/ { @["free"]   = count(); }'
```

更针对驱动程序的方法是在驱动程序本身添加DTrace探针，每个分配一个、每个释放一个，让DTrace统计比例。这是第8节为`myfirst`驱动程序演示的技术。

### 7.2 竞态条件：罕见但破坏性强的错误

竞态条件是最难重现且最容易遗漏的驱动程序错误。竞态发生在两个线程在没有正确同步的情况下访问同一状态，结果行为取决于两个线程的相对时序。轻负载下竞态可能永远不会触发；重负载下每秒触发数百次。

竞态条件的症状变化很大：
- 在不可预测位置的偶发崩溃
- 数据损坏（不应存在的值）
- 偶发的锁定相关断言失败，如"mutex myfirst_lock not owned"
- 看似不可能的栈，两个线程似乎在同一个互斥锁保护的区域内

没有单一症状证明是竞态，但任何仅在高负载下出现的错误模式都应引起怀疑。

FreeBSD发现竞态最有效的单一工具是`WITNESS`，如第4节介绍。编译了`options WITNESS`的内核检查每个锁操作，并在任何违反声明的锁顺序、在持有睡眠锁时尝试获取自旋锁、在持有自旋锁时调用睡眠函数等许多相关错误时崩溃。

当`WITNESS`崩溃或打印违规时，它生成一个栈追踪，显示：
- 正在获取哪个锁
- 已持有哪些锁
- 获取会违反的声明顺序
- 两个栈（当前锁和冲突锁）

修复通常是以下之一：

1. **重新排序获取。** 如果代码先取锁A再取锁B，而另一路径先取B再取A，其中之一必须改变。
2. **添加缺失的锁。** 如果状态在没有互斥锁的情况下被访问，在访问周围添加适当的`MTX_DEF`互斥锁。
3. **删除冗余锁。** 有时两个锁保护重叠状态，可以删除一个。
4. **切换锁类型。** 自旋锁和睡眠锁有不同的规则。需要睡眠的区域必须使用睡眠锁。从中断上下文调用的区域通常必须使用自旋锁。

没有触发`WITNESS`违规的驱动程序并非没有竞态，因为`WITNESS`只捕获锁顺序问题。在没有锁的情况下访问状态的竞态需要不同的方法：仔细阅读代码、在可疑点添加DTrace探针确认时序，偶尔在关键区域撒上`mtx_assert(&sc->sc_mtx, MA_OWNED)`断言。`INVARIANTS`启用这些断言，这是开发期间运行调试内核的另一个原因。

在`WITNESS`、`INVARIANTS`、`mtx_assert`和怀疑争用的路径上的DTrace计数器之间，大多数竞态可以在一两小时内缩小范围。幸免于这套武器的竞态很少，几乎总是涉及无锁数据结构、原子操作或内存排序假设，需要与资深工程师仔细审查。

### 7.3 bus_space和bus_dma误用

新驱动程序作者经常遇到共享独特症状系列的总线访问错误：

- 读取返回0xFF、0xFFFFFFFF或其他可疑常数
- 写入似乎成功但设备没有反应
- 机器工作几秒或几分钟后挂起或崩溃
- 在一种架构（amd64）上行为正确，在另一种（arm64、riscv64）上错误

每一点都指向`bus_space`或`bus_dma`误用。

第一个错误是跳过句柄。`bus_space_read_4()`接受从attach期间分配的资源上的`rman_get_bustag()`和`rman_get_bushandle()`获得的`bus_space_tag_t`和`bus_space_handle_t`。使用寄存器的原始物理地址或通过资源的直接转换获得的指针，绕过了平台的总线抽象。在amd64上程序可能看似工作（内核以一种容忍此方式映射MMIO区域），但在arm64上寄存器访问失败。

修复是机械性的：总是通过`bus_space_read_N()`/`bus_space_write_N()`或更新的`bus_read_N()`/`bus_write_N()`基于资源的辅助函数，永远不要解引用设备内存的原始指针。

第二个错误是大小不正确。`bus_space_read_4`读取32位值。如果寄存器是16位且代码读取4字节，它也会读取相邻寄存器，相邻寄存器的值现在出现在返回值的第16到31位。更糟的是，某些硬件不容忍不匹配的大小并以错误响应。修复是对每个寄存器的宽度使用正确的`read_1`、`read_2`、`read_4`或`read_8`变体，如设备数据手册所述。

第三个错误是偏移不正确。`bus_space`句柄指向映射区域的基地；偏移被加上以计算寄存器地址。偏移中的拼写错误读取不同的寄存器。例如，读取偏移`0x18`而不是`0x10`产生意外值，驱动程序的后续逻辑基于错误读数。修复是在头文件中将每个偏移定义为命名常量，引用常量而不是数字：`#define MYFIRST_REG_STATUS 0x10`、`#define MYFIRST_REG_CONFIG 0x14`等等。

对于DMA，最常见的误用是在设备仍在读取或写入缓冲区时释放或重用它。特征症状是间歇性数据损坏，有时仅在高负载下发生。原因是缺少`bus_dmamap_sync`调用、在描述符仍在排队时释放`bus_dmamem`缓冲区，或`bus_dmamap_sync`中方向不正确。

诊断方法是记录每个DMA映射、同步和取消映射操作，然后与描述符环的状态交叉检查日志。驱动程序DMA路径上的DTrace一行脚本通常足以发现错误：

```sh
dtrace -n 'fbt::myfirst_dma_sync:entry { printf("%s dir=%d", probefunc, arg1); }'
```

关于DMA规则和陷阱的完整论述，第21章是参考。调试章节的作用是帮助读者识别症状并缩小原因。

### 7.4 分离后使用

当驱动程序的分离路径返回、驱动程序的softc被释放，但内核的其他部分仍持有对释放内存的引用时，会出现更微妙的一类错误。例子包括：

- 在`bus_release_resource`之后但处理程序被拆除之前到达的中断
- 在分离后触发的callout，解引用释放的softc
- 在卸载时仍在运行的任务触发的驱动程序中的DTrace探针
- 用户进程保持打开的字符设备节点，在驱动程序卸载时接收I/O

症状几乎总是致命的：在分离后到达的驱动程序代码中的页面错误、栈损坏的内核崩溃，或崩溃前内核消息缓冲区中的虚假数据。

修复有几个组成部分，分离路径必须按特定顺序实现每个：

1. 首先，阻止新进入：在softc中设置标志（`sc->sc_detaching = 1`），或获取所有入口点检查的写锁，使新调用看到驱动程序正在消失。

2. 等待进行中的调用者完成。`bus_teardown_intr`排空中断处理程序。`callout_drain`等待待处理的callout完成。`taskqueue_drain`排空任何延迟任务。`destroy_dev_sched_cb`等待打开的文件描述符关闭。

3. 只有在排空每个外部调用者后，才释放驱动程序在attach中分配的资源：释放内存、释放IRQ、释放内存资源、销毁互斥锁。

原则很简单：分离是attach的镜像，attach中分配的每个资源必须在分离中以相反顺序释放。违反此原则产生分离后使用错误。

选择的调试工具是调试内核本身。启用`DEBUG_MEMGUARD`的内核可以配置为毒化释放的内存，因此对释放的softc的访问产生立即页面错误和清晰的栈，而不是可能需要数小时才显现的微妙损坏。

### 7.5 中断处理程序错误

刚接触内核的驱动程序作者有时将中断处理程序视为普通函数，并犯常见但有害的错误：

- 从过滤器级中断处理程序调用睡眠函数（`malloc(..., M_WAITOK, ...)`、`tsleep`、睡眠锁的`mtx_lock`、`copyin`、`uiomove`）
- 尝试长时间持有睡眠锁
- 在中断处理程序内而不是`taskqueue`中读取或写入大块数据
- 不确认中断，使处理程序永远循环触发

每一点都有独特的症状。在中断上下文中睡眠产生`WITNESS`违规，或在发布内核中产生调度器状态的静默损坏。长时间持有睡眠锁导致其他线程自旋或睡眠，延迟膨胀。在处理程序中做繁重工作阻塞后续中断并降低整个系统的响应性。不确认中断使CPU永远在100%处理中断上。

治疗总是一样的：使中断处理程序简短且原子。它应该：
1. 读取状态寄存器确定原因。
2. 通过写入状态寄存器确认中断。
3. 将任何实质性工作交给`taskqueue`或`ithread`，如第19章所述。
4. 返回。

复杂处理、内存分配和长时间运行的操作都属于延迟路径，而不是处理程序本身。

DTrace对诊断中断处理程序性能特别有用。`intr`提供者，或处理程序入口上的`fbt`探针，可以测量每次调用花费多长时间：

```sh
dtrace -n 'fbt::myfirst_intr:entry { self->t = timestamp; }
           fbt::myfirst_intr:return /self->t/ {
               @ = quantize(timestamp - self->t); self->t = 0; }'
```

健康的处理程序在几微秒内返回。如果量化显示调用在数百微秒或毫秒范围内，处理程序做了太多工作应该重构。

### 7.6 生命周期顺序错误

驱动程序的生命周期经历probe、attach、open、close、detach和unload。每个方法有关于它可以做什么和之前必须发生什么的规则。违反规则产生特征错误：

- 在probe期间调用`bus_alloc_resource_any`（必须在attach中发生）产生部分分配和probe顺序逻辑的混乱
- 在probe中做实质性工作显著减慢启动，因为同一probe为每个候选设备运行
- 在硬件初始化之前创建`cdev`让用户进程打开设备并在未初始化状态上接收I/O
- 在分离中销毁`cdev`而其他线程仍保持打开会损坏devfs状态
- 在模块unload而不是detach中做重要工作使每次attach分配的资源未释放

治疗是让每个生命周期方法专注于其任务：

- **probe**只识别设备并返回`BUS_PROBE_DEFAULT`或错误；不分配、不注册
- **attach**分配资源、初始化softc、设置中断、创建`cdev`、注册设备
- **detach**精确地以相反顺序逆转attach
- **open/close**管理每文件状态而不触及设备级资源
- **unload**只执行模块级清理；每设备清理属于detach

当驱动程序遵循此结构时，生命周期错误很少。当偏离时，它们常见且痛苦。

### 7.7 复制到/来自用户错误

`copyin(9)`、`copyout(9)`和相关`fueword`/`suword`函数跨用户/内核边界传输数据。它们受虚拟内存系统保护：如果用户空间地址无效，复制返回`EFAULT`而不是崩溃。然而，此保护仅在通过这些函数进行复制时适用。直接解引用用户空间指针的驱动程序在用户指针错误时会立即崩溃，这在实际部署中大多数时候会发生。

症状是页面错误指令中带有用户空间地址的崩溃，栈指向驱动程序的read/write/ioctl路径。

修复是强制性的：每次用户数据跨越边界，使用`copyin`/`copyout`或通过`struct uio`的`uiomove`。永远不要将用户指针转换为内核指针并解引用它。此规则是绝对的。

对于ioctl，`d_ioctl`处理程序接收内核指针，因为内核已从用户空间复制了固定大小的参数。但如果ioctl的参数是包含指向更大用户缓冲区的指针的结构，该嵌入指针仍是用户空间的，需要`copyin`才能安全访问它指向的缓冲区。

### 7.8 模块级错误

有些错误影响整个模块而不是单个设备：

- 模块无法加载：加载器在`dmesg`中打印错误；阅读并修复症状（缺失依赖、符号解析失败、版本不匹配）
- 模块无法卸载：通常意味着设备仍附加。在`kldunload`之前分离所有设备（`devctl detach myfirst0`）
- 模块卸载但留下垃圾：卸载路径（`module_t`事件处理程序，或`DRIVER_MODULE_ORDERED`卸载）未逆转模块加载。通过阅读卸载路径并使其与加载对称来修复。

调试工具简单：`dmesg | tail`显示加载/卸载消息，`kldstat -v`显示模块状态，`vmstat -m`显示模块的malloc池是否在卸载时被排空。

### 7.9 调试检查清单

这里是将工具匹配到症状的紧凑检查清单。读者可以在驱动程序开发期间将其放在手边。

| 症状                                    | 第一个工具                    | 第二个工具          |
|----------------------------------------|-----------------------------|----------------------|
| `dmesg`行错误或缺失                    | `dmesg`, `/var/log/messages` | 添加`device_printf` |
| `InUse`在`vmstat -m`中增长             | `vmstat -m`                 | DTrace malloc计数  |
| 锁顺序崩溃                             | `WITNESS`追踪              | `ddb> show locks`   |
| 负载下偶发损坏                         | `WITNESS`, `INVARIANTS`      | DTrace时序        |
| 寄存器读取返回0xFFFFFFFF                | 检查`bus_space`句柄   | 数据手册、偏移    |
| 设备使用时机器挂起                     | DTrace `fbt::myfirst_*`     | `ddb> bt`           |
| 分离后崩溃                             | `DEBUG_MEMGUARD`             | 审计分离顺序  |
| 驱动程序负载下系统慢                   | DTrace `intr`提供者      | 缩短处理程序      |
| ioctl命令错误                          | `ktrace` / `kdump`           | 解码命令      |
| 用户空间错误地址崩溃                   | 检查`copyin`/`copyout`   | 审计ioctl指针 |
| 模块无法卸载                  | `kldstat -v`                | `devctl detach`     |

表格并不详尽，但覆盖了绝大多数实际驱动程序错误。有了它和第1到6节的工具，读者有了大多数现场诊断的结构化方法。

### 第7节总结

第7节将日志、日志分析、调试内核、DTrace和`ktrace`的工具箱用于读者最可能遇到的错误：内存泄漏、竞态条件、总线访问错误、分离后使用、中断错误、生命周期错误、用户/内核边界错误和模块级问题。对每类错误，本节描述了症状、根本原因和最高效诊断它的工具。末尾的检查清单使映射显式化，未来在键盘上的工作应该让读者第一次而不是第三次就能选择正确的工具。

剩下的缺口是驱动程序本身。到目前为止在本章中，`myfirst`主要被用作外部工具的目标。第8节通过重构驱动程序以暴露自己的追踪点和调试详细度旋钮来填补该缺口，使仪器化成为代码的有机部分而不是事后想法。版本号升级到`1.6-debug`，驱动程序获得`myfirst_debug.h`和将伴随本书其余部分的结构。


## 第8节：使用追踪点重构和版本化

本章前七节是关于学习使用FreeBSD中已有的工具：`printf`、`dmesg`、调试内核、DTrace、`ktrace`，以及阅读其输出的纪律。第8节转向内部。从第22章到达的`myfirst`驱动程序版本为`1.5-power`，没有自己的真正调试基础设施。每条日志语句都是无条件的。没有操作员可以旋转的旋钮，没有在出现问题时请求更详细输出的sysctl，也没有可用于DTrace的静态追踪点。第8节修复了这个缺口。

本节的工作在代码行数上很少，但在影响上很大。到本节末尾，驱动程序将拥有：

1. 一个`myfirst_debug.h`头文件，定义`DPRINTF`宏、详细度位和SDT追踪点。
2. 一个sysctl树（`dev.myfirst.0.debug`），在运行时设置详细度。
3. 在整个驱动程序中一致应用的入口-退出-错误追踪模式。
4. 三个SDT提供者探针，暴露open、close和I/O事件。
5. 示例树中的`DEBUG.md`文档，描述如何配置和读取新输出。
6. 在`myfirst_version`和`MODULE_VERSION`中记录的到`1.6-debug`的版本升级。

生成的驱动程序将作为本书剩余章节的平台。第5、6和7部分中添加的每个新子系统都将接入此框架，使追踪基础设施随驱动程序一起增长，而不是在最后才附加。

### 8.1 为什么需要调试头文件

较大的FreeBSD驱动程序有专门用于调试基础设施的头文件。该模式在`/usr/src/sys/dev/ath/if_ath_debug.h`、`/usr/src/sys/dev/bwn/if_bwn_debug.h`、`/usr/src/sys/dev/iwn/if_iwn_debug.h`和许多其他文件中可见。这些头文件各自：

1. 定义一个检查softc中详细度位掩码的`DPRINTF`宏
2. 声明一组详细度类，如`#define MYF_DBG_INIT 0x0001`、`MYF_DBG_OPEN 0x0002`等
3. 可选地声明映射到驱动程序功能边界的SDT探针

将其放在头文件中有两个好处。首先，调试基础设施是一个独立的关注点，与功能代码干净地分离。其次，当读者想了解驱动程序的追踪故事时，头文件是要阅读的唯一文件。该模式成熟、广泛使用，是`myfirst`在本节将采用的。

头文件保存在`examples/part-05/ch23-debug/stage3-refactor/myfirst_debug.h`下，驱动程序源文件在顶部包含它：

```c
#include "myfirst_debug.h"
```

驱动程序源中的配套更改是增量的：现有的`device_printf`调用可以保留，新调用通过`DPRINTF`宏添加。

### 8.2 声明调试类

第一步是定义详细度类。类是32位掩码中的单个位。驱动程序为每个功能区域保留一个位。对于`myfirst`，八个类在此阶段绰绰有余：

```c
/* myfirst_debug.h */
#ifndef _MYFIRST_DEBUG_H_
#define _MYFIRST_DEBUG_H_

#include <sys/sdt.h>

#define MYF_DBG_INIT    0x00000001  /* probe/attach/detach */
#define MYF_DBG_OPEN    0x00000002  /* open/close生命周期 */
#define MYF_DBG_IO      0x00000004  /* read/write路径 */
#define MYF_DBG_IOCTL   0x00000008  /* ioctl处理 */
#define MYF_DBG_INTR    0x00000010  /* 中断处理程序 */
#define MYF_DBG_DMA     0x00000020  /* DMA映射/同步 */
#define MYF_DBG_PWR     0x00000040  /* 电源管理事件 */
#define MYF_DBG_MEM     0x00000080  /* alloc/free追踪 */

#define MYF_DBG_ANY     0xFFFFFFFF
#define MYF_DBG_NONE    0x00000000

#endif /* _MYFIRST_DEBUG_H_ */
```

值`MYF_DBG_NONE`是默认值：完全没有调试输出。`MYF_DBG_ANY`启用每个类，在开发期间有用。典型的操作员配置可能只启用`MYF_DBG_INIT | MYF_DBG_OPEN`以获取生命周期事件而不会有每次I/O的噪音。

每个位声明为单个十六进制数字或一对，因此操作员可以用简单值设置掩码：`sysctl dev.myfirst.0.debug=0x3`启用初始化和open追踪。注释名称使每个位的用途清晰。

### 8.3 DPRINTF宏

接下来是根据掩码控制日志记录的宏：

```c
#ifdef _KERNEL
#define DPRINTF(sc, m, ...) do {                                        \
        if ((sc)->sc_debug & (m))                                        \
                device_printf((sc)->sc_dev, __VA_ARGS__);                \
} while (0)
#endif
```

宏接受三个参数：softc指针、位掩码和格式字符串加可变参数。它展开为`sc->sc_debug & m`的测试，如果测试成功则调用`device_printf`。`do { ... } while (0)`模式是多语句宏在`if`/`else`上下文中必须表现为单个语句的标准惯用法。

位清除时`DPRINTF`调用的成本是从`sc->sc_debug`的单次加载、一次按位与和一个分支。该分支几乎总是被预测为"不采取"，因此实际成本可以忽略不计。驱动程序可以在整个代码中自由撒`DPRINTF`调用，禁用调试时用户不付出任何代价。

当位设置时，调用变成普通的`device_printf`，像任何其他内核日志一样出现在`dmesg`中。行为与通常的日志路径没有区别，只是日志记录现在是有条件的。

### 8.4 在softc中添加`sc_debug`

softc增加一个新字段，一个`uint32_t sc_debug`。其值由sysctl操作，attach函数注册该sysctl。来自`myfirst_debug.c`的相关摘录：

```c
struct myfirst_softc {
        device_t        sc_dev;
        struct mtx      sc_mtx;
        struct cdev    *sc_cdev;
        uint32_t        sc_debug;       /* 调试详细度掩码 */
        /* 其他字段与1.5-power相同 */
};
```

attach函数将该字段初始化为零并注册sysctl：

```c
sc->sc_debug = 0;
sysctl_ctx_init(&sc->sc_sysctl_ctx);
sc->sc_sysctl_tree = SYSCTL_ADD_NODE(&sc->sc_sysctl_ctx,
    SYSCTL_CHILDREN(device_get_sysctl_tree(sc->sc_dev)),
    OID_AUTO, "debug",
    CTLFLAG_RW, 0, "debug verbosity tree");

SYSCTL_ADD_U32(&sc->sc_sysctl_ctx,
    SYSCTL_CHILDREN(sc->sc_sysctl_tree),
    OID_AUTO, "mask",
    CTLFLAG_RW, &sc->sc_debug, 0, "debug class bitmask");
```

attach之后，sysctl出现为：

```sh
sysctl dev.myfirst.0.debug.mask
```

操作员可以随意更改：

```sh
sysctl dev.myfirst.0.debug.mask=0x3     # 启用 INIT + OPEN
sysctl dev.myfirst.0.debug.mask=0xFFFFFFFF   # 启用所有类
sysctl dev.myfirst.0.debug.mask=0        # 禁用
```

分离例程像现在一样销毁sysctl上下文：

```c
sysctl_ctx_free(&sc->sc_sysctl_ctx);
```

一个细节很重要：将sysctl字段放在softc的顶部附近会使其位于第一个缓存行中，在DPRINTF期间其访问成本最低。这是一个小的性能要点，但本书对原则性编码的强调应该注意它。

### 8.5 入口/退出/错误模式

有了`DPRINTF`，驱动程序的函数可以使用一致的追踪模式。每个重要函数记录入口、退出（或错误）和任何有趣的中间状态。对于像`myfirst_open`这样的小函数，看起来像这样：

```c
static int
myfirst_open(struct cdev *dev, int flags, int devtype, struct thread *td)
{
        struct myfirst_softc *sc = dev->si_drv1;
        int error = 0;

        DPRINTF(sc, MYF_DBG_OPEN, "open: pid=%d uid=%d flags=0x%x\n",
            td->td_proc->p_pid, td->td_ucred->cr_uid, flags);

        mtx_lock(&sc->sc_mtx);
        if (sc->sc_open_count >= MYFIRST_MAX_OPENS) {
                error = EBUSY;
                goto out;
        }
        sc->sc_open_count++;
out:
        mtx_unlock(&sc->sc_mtx);

        if (error != 0)
                DPRINTF(sc, MYF_DBG_OPEN, "open failed: error=%d\n", error);
        else
                DPRINTF(sc, MYF_DBG_OPEN, "open ok: count=%d\n",
                    sc->sc_open_count);

        return (error);
}
```

三个原则指导该模式：

1. 入口日志显示谁调用了以及使用了什么参数。对于open，这是`pid`、`uid`和flags。
2. 退出日志显示结果：要么是"failed: error=N"，要么是带有任何相关状态的"ok: ..."。
3. 中间状态在重要时被记录。对于open，新的open计数是有用的。

close、read、write和ioctl的模式相同。每个函数有两三个`DPRINTF`调用包围工作。

这种纪律在第一个错误出现时就产生回报。没有该模式，开发者必须追溯地添加日志，猜测问题在哪里。有了该模式，打开单个位（`sysctl dev.myfirst.0.debug.mask=0x2`用于`MYF_DBG_OPEN`）产生每次open和close的完整追踪，包括参数和结果。诊断时间从数小时降至数分钟。

### 8.6 添加SDT探针

静态`DPRINTF`宏在`dmesg`中生成人类可读的消息。SDT探针产生DTrace可以聚合、筛选和计时的机器可读事件。两者各有其位。驱动程序为三个最有趣的事件声明SDT探针，有心的用户可以随意附加自定义脚本。

在`myfirst_debug.h`中，探针声明如下：

```c
#include <sys/sdt.h>

SDT_PROVIDER_DECLARE(myfirst);
SDT_PROBE_DECLARE(myfirst, , , open);
SDT_PROBE_DECLARE(myfirst, , , close);
SDT_PROBE_DECLARE(myfirst, , , io);
```

匹配的定义位于驱动程序源文件（`myfirst_debug.c`）中：

```c
#include <sys/sdt.h>

SDT_PROVIDER_DEFINE(myfirst);
SDT_PROBE_DEFINE2(myfirst, , , open,
    "struct myfirst_softc *", "int");
SDT_PROBE_DEFINE2(myfirst, , , close,
    "struct myfirst_softc *", "int");
SDT_PROBE_DEFINE4(myfirst, , , io,
    "struct myfirst_softc *", "int", "size_t", "off_t");
```

`DEFINE`宏将探针注册到内核的SDT基础设施。`DEFINEn`接受`n`个参数，每个都有描述DTrace脚本将作为`arg0`、`arg1`等接收的内容的C风格类型字符串。

驱动程序然后在适当的位置触发探针：

```c
/* 在myfirst_open中 */
SDT_PROBE2(myfirst, , , open, sc, flags);

/* 在myfirst_close中 */
SDT_PROBE2(myfirst, , , close, sc, flags);

/* 在myfirst_read或myfirst_write中 */
SDT_PROBE4(myfirst, , , io, sc, is_write, (size_t)uio->uio_resid, uio->uio_offset);
```

没有脚本附加的探针成本是围绕无操作的单个分支，与位清除的`DPRINTF`相同的可忽略成本。当DTrace脚本确实附加时，分支被采取，参数被记录，脚本将它们视为`args[0]`、`args[1]`等。

有了这些探针，读者现在可以运行如下脚本：

```sh
dtrace -n 'myfirst::: { @[probename] = count(); }'
```

这统计每个`myfirst`探针事件。要查看每秒传输的字节数：

```sh
dtrace -n 'myfirst:::io { @["bytes"] = sum(arg2); }'
```

或追踪每个带PID和flags的open：

```sh
dtrace -n 'myfirst:::open { printf("open pid=%d flags=0x%x", pid, arg1); }'
```

驱动程序现在以两种形式暴露其行为：通过`DPRINTF`的人类可读形式和通过SDT的机器可解析形式。操作员可以为工作选择正确的形式。

本书中每个改变驱动程序行为的章节都升级其版本。规则是：驱动程序中的版本字符串与示例树README和章节文本中显示的版本匹配。

在第22章中，驱动程序达到了`1.5-power`。第23章第8节将其带到`1.6-debug`。`myfirst_debug.c`中的更改：

```c
/*
```


这确认了模块身份和调试基础设施。

## 动手实验

本章的实验形成渐进式序列。每个实验强化第1到6节中的特定工具，最后一个实验应用第8节的重构，使读者最终拥有一个已仪器化并准备好迎接本书其余部分的驱动程序。

这些实验都不长。全部五个可以在一个晚上完成，尽管将它们分散到两三个会话给读者更多时间消化每个工具。

### 实验23.1：第一次DDB会话

**目标：** 有意进入内核调试器，检查驱动程序的状态，并安全退出。

**前提条件：** 按第4节所述构建的调试内核，带有`options KDB`和`options DDB`。加载了第22章的`myfirst`驱动程序（版本`1.5-power`）。

**步骤：**

1. 确认调试内核正在运行：

   ```sh
   sysctl kern.version
   sysctl debug.kdb
   ```

   第二个命令应显示DDB在可用后端中。

2. 加载驱动程序并确认设备节点：

   ```sh
   sudo kldload myfirst
   ls /dev/myfirst0
   ```

3. 用键盘中断进入DDB。在系统控制台上，在VGA控制台按`Ctrl-Alt-Esc`，或在串行控制台发送BREAK字符。出现提示：

   ```
   KDB: enter: manual entry
   [thread pid 42 tid 100024 ]
   Stopped at      kdb_enter+0x37: movq    $0,0x158e4fa(%rip)
   db>
   ```

4. 显示设备树：

   ```
   db> show devmap
   ```

   在输出中定位`myfirst0`。

5. 打印当前线程的回溯：

   ```
   db> bt
   ```

6. 打印所有进程：

   ```
   db> ps
   ```

   确认读者自己的shell在列表中可见。

7. 继续内核运行：

   ```
   db> continue
   ```

   系统恢复正常运行。

**观察要点：**

- 内核在断点处干净地停止。
- 所有设备、进程和内核线程都是可检查的。
- `continue`命令使系统返回正常，没有任何副作用。

在实验日志本"第23章，实验1"下记录花费的时间和任何笔记。读者应该明白DDB在有意识使用时是安全的工具。

### 实验23.2：使用DTrace测量驱动程序

**目标：** 使用DTrace在简单工作负载下测量`myfirst`驱动程序的open、close和I/O速率。

**前提条件：** 加载了带有`options KDTRACE_HOOKS`和`makeoptions WITH_CTF=1`的内核。`myfirst`驱动程序在`1.5-power`版本加载（此阶段尚无SDT探针；实验仅使用`fbt`）。

**步骤：**

1. 确认DTrace工作：

   ```sh
   sudo dtrace -l | head -5
   ```

2. 启动统计每个`myfirst`函数入口的DTrace脚本：

   ```sh
   sudo dtrace -n 'fbt::myfirst_*:entry { @[probefunc] = count(); }'
   ```

3. 在第二个终端中，锻炼驱动程序：

   ```sh
   for i in $(seq 1 100); do cat /dev/myfirst0 > /dev/null; done
   ```

4. 停止DTrace脚本（在第一个终端按Ctrl-C）并读取计数：

   ```
     myfirst_open                                             100
     myfirst_close                                            100
     myfirst_read                                             100
   ```

5. 现在测量`myfirst_read`中花费的时间：

   ```sh
   sudo dtrace -n 'fbt::myfirst_read:entry { self->t = timestamp; }
                   fbt::myfirst_read:return /self->t/ {
                       @ = quantize(timestamp - self->t); self->t = 0; }'
   ```

6. 用1000次读操作锻炼驱动程序，停止DTrace，读取量化结果。

**观察要点：**

- 每次`cat`产生一次open、一次read和一次close，与计数匹配。
- 每次读取的时间很短（微秒级）并集中在量化的一个桶中。
- 未对驱动程序做任何修改；测量是非侵入性的。

### 实验23.3：使用ktrace进行用户侧追踪

**目标：** 追踪锻炼`myfirst`驱动程序的用户程序并验证系统调用序列。

**前提条件：** 驱动程序已加载。第6.4节的测试程序`myftest.c`已编译。

**步骤：**

1. 编译测试程序（来自第6.4节）：

   ```sh
   cc -o myftest myftest.c
   ```

2. 在`ktrace`下运行它：

   ```sh
   sudo ktrace -di -f mytrace.out ./myftest
   ```

3. 转储追踪：

   ```sh
   kdump -f mytrace.out | less
   ```

4. 定位与驱动程序相关的系统调用：

   ```sh
   kdump -f mytrace.out | grep -E 'myfirst0|CALL|RET'
   ```

5. 将追踪与实验23.2中驱动程序的DTrace计数相关联。每个用户侧系统调用应匹配一个或多个内核侧函数入口。

**观察要点：**

- 用户程序的驱动程序视图在追踪中是完整的。
- 参数、返回值和错误码都可见。
- 用户侧追踪补充了DTrace的内核侧视图。

### 实验23.4：使用vmstat -m发现内存泄漏

**目标：** 引入一个故意泄漏，用`vmstat -m`检测它，并修复它。

**前提条件：** 读者控制下的驱动程序源代码。已编译可加载模块。

**步骤：**

1. 修改`myfirst_open`分配一个小缓冲区并存储在`si_drv2`中，不在`myfirst_close`中释放：

   ```c
   /* 在myfirst_open中 */
   dev->si_drv2 = malloc(128, M_MYFIRST, M_WAITOK | M_ZERO);
   ```

2. 构建并加载驱动程序。

3. 运行多次打开和关闭设备的工作负载：

   ```sh
   for i in $(seq 1 1000); do cat /dev/myfirst0 > /dev/null; done
   ```

4. 检查内存池：

   ```sh
   vmstat -m | grep myfirst
   ```

   `InUse`列应显示1000（或更多），`MemUse`应显示大约128 KB。

5. 通过在`myfirst_close`中添加`free(dev->si_drv2, M_MYFIRST);`修复泄漏。

6. 构建并重新加载驱动程序。再次运行工作负载。

7. 检查池：

   ```sh
   vmstat -m | grep myfirst
   ```

   工作负载后`InUse`现在应稳定在零附近。

**观察要点：**

- `vmstat -m`输出立即暴露了泄漏。
- 没有专业工具，泄漏会在内核耗尽内存前一直不被注意。
- 修复在识别症状后是机械性的且简单。

实验后不要将泄漏诱导代码留在驱动程序中。在继续之前恢复到干净版本。

### 实验23.5：安装1.6-debug重构

**目标：** 将第8节的重构应用到`myfirst`驱动程序并确认新基础设施工作。

**前提条件：** 读者控制下第22章的驱动程序源代码。能够构建内核模块的开发环境。

**步骤：**

1. 创建完全如第8.2节所示的`myfirst_debug.h`，以及第8.6节的SDT声明。

2. 更新softc添加`uint32_t sc_debug;`。

3. 在`myfirst_attach`中注册sysctl，如第8.4节所示。

4. 将无条件的`device_printf`调用替换为适当的`DPRINTF(sc, MYF_DBG_<class>, ...)`调用。

5. 在`myfirst_debug.c`（或`myfirst.c`）中定义SDT提供者和探针，并在open、close和read/write中触发它们。

6. 将`myfirst_version`升级到`1.6-debug`，`MODULE_VERSION`升级到16。

7. 构建：

   ```sh
   cd /path/to/myfirst-source
   make clean && make
   ```

8. 重新加载：

   ```sh
   sudo kldunload myfirst
   sudo kldload ./myfirst.ko
   ```

9. 确认模块以新版本加载：

   ```sh
   kldstat -v | grep myfirst
   ```

10. 确认sysctl存在：

    ```sh
    sysctl dev.myfirst.0.debug.mask
    ```

11. 启用完整详细度并打开设备：

    ```sh
    sudo sysctl dev.myfirst.0.debug.mask=0xFFFFFFFF
    cat /dev/myfirst0 > /dev/null
    dmesg | tail
    ```

    预期输出包括如下行：

    ```
    myfirst0: open: pid=1234 uid=1001 flags=0x0
    myfirst0: open ok: count=1
    myfirst0: read: size=4096 off=0
    ```

12. 禁用详细度并确认消息停止：

    ```sh
    sudo sysctl dev.myfirst.0.debug.mask=0
    cat /dev/myfirst0 > /dev/null
    dmesg | tail
    ```

13. 将DTrace附加到SDT探针：

    ```sh
    sudo dtrace -n 'myfirst::: { @[probename] = count(); }'
    ```

    在第二个终端中，锻炼驱动程序：

    ```sh
    for i in $(seq 1 100); do cat /dev/myfirst0 > /dev/null; done
    ```

    停止DTrace并确认探针已触发。

**观察要点：**

- 运行时详细度控制是即时和响应的。
- DPRINTF路径产生人类可读消息；SDT探针产生机器可解析事件。
- 两种形式互补且独立。
- 两者默认关闭，因此生产用户不付出运行时成本。

驱动程序现在处于`1.6-debug`版本，拥有支持后续每个章节的基础设施。在日志本中记录升级。

## 挑战练习

以下练习建立在章节材料之上。它们足够开放，每个读者会得出略有不同的答案。花时间；逐步推进；在每个步骤使用本章的工具。

### 挑战23.1：真实驱动程序中的SDT探针

从`/usr/src/sys/dev/`中选择一个已包含SDT探针的驱动程序。好的候选包括`virtqueue.c`、某些`ath`文件或`if_re.c`。

对于选择的驱动程序：

1. 用`dtrace -l -P <provider>`列出其探针。
2. 编写一行脚本按探针名称统计事件。
3. 编写第二行脚本按一个探针参数（例如包长度或命令号）聚合。
4. 用三句话解释驱动程序作者通过添加探针获得了什么。

### 挑战23.2：自定义DTrace脚本

编写一个DTrace脚本，它：

1. 附加到`myfirst` SDT探针（在实验23.5中添加）。
2. 按pid跟踪每个打开文件描述符的生命周期。
3. 在fd关闭时打印摘要，显示：pid、多少次read、多少次write、总字节数、open和close之间的经过时间。

脚本应不超过50行D代码。通过运行一个小用户程序（多次打开、读取、写入和关闭`/dev/myfirst0`）来测试它。

### 挑战23.3：WITNESS实验

修改驱动程序包含一个故意的锁排序错误。例如，添加两个互斥锁`sc_mtx_a`和`sc_mtx_b`，安排一个代码路径先取A再取B，而另一路径先取B再取A。

用`options WITNESS`构建，加载，并触发错误。捕获产生的内核崩溃或WITNESS输出。用三句话描述WITNESS显示了什么、为什么它捕获了该问题、以及修复方案是什么。

确保在练习后恢复错误。不要留下损坏的驱动程序加载。

### 挑战23.4：扩展错误分类

在`myfirst`驱动程序中选择任意四个错误路径（例如，`myfirst_open`中的`EBUSY`路径、`myfirst_ioctl`中的`EINVAL`路径等）。对每个路径：

1. 识别在现实世界中会产生此错误的根本原因。
2. 添加一个`DPRINTF(sc, MYF_DBG_<class>, ...)`，清楚地捕获原因。
3. 在`DEBUG.md`中写一个简短说明，解释错误发生时在`dmesg`中要查找什么。

目标是使驱动程序中的每个错误都是自解释的，无需读者交叉引用源代码。

### 挑战23.5：启动时调试

修改驱动程序使其仅在`bootverbose`设置时记录attach时的版本和硬件详情。效果应为：正常启动显示一行，详细启动（通过`boot -v`请求）显示详细配置。

阅读`/usr/src/sys/sys/systm.h`了解`bootverbose`的声明，阅读`/usr/src/sys/kern/subr_boot.c`了解示例。用两句话描述FreeBSD使用`bootverbose`的目的，以及为什么对于早期启动的特定情况，它比驱动程序特定的详细标志更好。

## 故障排除和常见错误

每位读者至少会遇到以下一些问题。每个问题都记录了症状、可能的原因和修复路径。

**"我的kldload失败，显示`link_elf: symbol X undefined`。"**

- 原因：模块依赖于不在运行内核中的内核符号。通常意味着内核比模块源代码旧，或者模块针对不同的内核构建编译。
- 修复：从同一源代码树使用相同的编译标志重建内核和模块。确认`uname -a`和模块的构建目录匹配。

**"DTrace说`invalid probe specifier`。"**

- 原因：探针名称拼写错误，或提供者/模块/函数字段在运行时不存在。
- 修复：运行`dtrace -l | grep <provider>`列出可用探针，从列表中选择名称。记住通配符必须匹配现有名称。

**"WITNESS在启动时崩溃，显示`WITNESS_CHECKORDER`。"**

- 原因：早期驱动程序（通常是第三方的）违反声明的锁顺序。在标准内核下违规是静默的；在WITNESS下导致立即崩溃。
- 修复：临时禁用有问题的驱动程序，或在启动时设置`debug.witness.watch=0`禁用检查，或以修正的锁顺序重建有问题的驱动程序。

**"DPRINTF消息不出现在`dmesg`中。"**

- 原因：`sc_debug`中的对应位未设置。
- 修复：用`sysctl dev.myfirst.0.debug.mask`确认sysctl。设置位：`sysctl dev.myfirst.0.debug.mask=0xFF`。重试。

**"DTrace说`probe description myfirst::: matched 0 probes`。"**

- 原因：要么驱动程序未加载，要么CTF未生成（SDT探针已声明但对DTrace不可见）。
- 修复：运行`kldstat | grep myfirst`确认模块已加载。如果已加载，用`makeoptions WITH_CTF=1`重建内核并重启。

**"修复后内存泄漏仍然存在。"**

- 原因：有多个泄漏。第一次修复处理了一个路径，但另一个路径仍在泄漏。
- 修复：重新检查`vmstat -m`数字。如果仍在增长，追踪分配器：在每个`malloc`和`free`调用处添加`device_printf`，重新加载，锻炼驱动程序，并统计出现次数。

**"驱动程序卸载了，但`vmstat -m`仍显示非零InUse。"**

- 原因：卸载路径未释放所有内存。这通常是因为某些东西由open分配但未被close释放，且close路径在分离前从未被调用。
- 修复：验证使用了`destroy_dev_sched_cb`在`detach`继续之前等待未完成的open。每个按open分配的`si_drv1`或`si_drv2`缓冲区必须在close或分离排序的回调中释放。

**"我的`fbt`探针名称有误导性。"**

- 原因：某些驱动程序使用内联函数或静态辅助函数，编译器可能已内联消除。编译后的二进制只显示包含函数。
- 修复：用`-O0`编译模块或给辅助函数添加`__noinline`。这仅对调试有用；发布构建应使用正常优化级别。

**"ktrace记录太长无法阅读。"**

- 原因：默认ktrace一次捕获许多事件类。
- 修复：用`-t`标志限制到特定类：`ktrace -t c ./myftest`只捕获系统调用，不捕获I/O或NAMI。

**"内核启动但完全不打印驱动程序消息。"**

- 原因：要么驱动程序未构建到内核中，要么驱动程序的probe返回了非零错误。
- 修复：`kldstat -v | grep myfirst`确认模块已加载。`dmesg | grep -i 'probe'`找到probe消息。如果模块已加载但驱动程序未附加，检查`device_set_desc`和probe函数的返回值。


## 附录：FreeBSD源码树中的真实调试模式

第7节和第8节中的模式基于生产驱动程序已使用的约定。检查源码树的读者会在各处看到这些约定的变体。本附录给出简短导览。

**`if_ath_debug.h`**（`/usr/src/sys/dev/ath/if_ath_debug.h`）定义了大约40个详细度类，覆盖重置、中断、RX、TX、信标、速率控制、信道选择等等，每个表示为64位掩码中的一个位。类名遵循`ATH_DEBUG_<subsystem>`模式（例如`ATH_DEBUG_RECV`、`ATH_DEBUG_XMIT`、`ATH_DEBUG_RESET`、`ATH_DEBUG_BEACON`），`DPRINTF`宏遵循标准模式。驱动程序在源代码的数百个位置使用`ATH_DEBUG_<class>`，为操作员提供对追踪输出的细粒度控制。

**`virtqueue.c`**（`/usr/src/sys/dev/virtio/virtqueue.c`）大量使用SDT探针。顶部的单个`SDT_PROVIDER_DEFINE(virtqueue)`声明提供者，后跟每个重要事件的十几个`SDT_PROBE_DEFINEn`调用：入队、出队、通知等。驱动程序在热路径中触发探针，没有脚本附加时探针是无操作。像`dtrace -n 'virtqueue::: { @[probename] = count(); }'`这样的DTrace脚本立即给出virtio设备活动的可见性。

**`if_re.c`**（`/usr/src/sys/dev/re/if_re.c`）混合使用`device_printf`、`printf`和`if_printf`。`device_printf`调用出现在attach和detach处，设备身份有用。`if_printf`出现在包路径中，接口身份重要。`printf`出现在共享辅助函数中，两者都不可用。职责划分是务实的，不是教条。

**`uart_core.c`**（`/usr/src/sys/dev/uart/uart_core.c`）定义了`UART_DBG_LEVEL`和一系列检查级别并通过`printf`记录的`UART_DBG`宏。级别通过`options UART_POLL_FREQ`和`options UART_DEV_TOLERANCE_PCT`等在编译时设置。设计是静态的：调试决策在构建时而非运行时固定。对于必须在非常早期的启动（sysctl尚不可用）时运行的驱动程序，这是正确的选择。

**`random_harvestq.c`**（`/usr/src/sys/dev/random/random_harvestq.c`）大量使用WITNESS。文件中的每个锁在每个依赖该锁的函数头部都有`mtx_assert(&lock, MA_OWNED)`检查。构建中启用WITNESS时，这些断言捕获任何忘记获取锁的调用者。在发布构建中，断言编译掉，零成本。

在整个源码树中，模式是一致的：驱动程序通过静态调试日志、SDT探针和断言宏的组合暴露其内部。这些在禁用时运行时不花任何成本，启用时提供丰富可见性。在自己的驱动程序中采用此模式的读者编写的代码从第一天调试到生产监控都能优雅扩展。

## 附录：驱动程序调试工作案例研究

本章的材料在针对具体错误演练时最为有用。本附录演练三个现实案例研究。每个从读者可能遇到的症状开始，遵循本章工具支持的诊断步骤，到达根本原因并记录修复。三个案例一起覆盖了驱动程序错误的三大类别：日志路径中的正确性错误、锁顺序错误和中断路径中的性能错误。

案例以叙述而非枯燥的命令序列编写，因为调试是一个叙述过程。经验丰富的开发者不遵循单一算法；他们阅读追踪、形成假设、测试它、改进它、迭代。这些演练试图保留那种质感，同时保持每步可重现。

### 案例研究1：缺失的消息

**症状。** 读者在`myfirst_open`中添加了一行`DPRINTF(sc, MYF_DBG_OPEN, "open ok: count=%d\n", sc->sc_open_count)`。他们设置`sysctl dev.myfirst.0.debug.mask=0x2`以启用`MYF_DBG_OPEN`，打开设备，检查`dmesg`。什么都没出现。

**第一个假设：位是错的。** 读者再次检查掩码。`MYF_DBG_OPEN`是`0x02`，所以设置`mask=0x2`应该启用它。假设是错的。

**第二个假设：设备未被打开。** 读者运行`cat /dev/myfirst0 > /dev/null`并检查`dmesg`。仍然什么都没有。

**第三个假设：sysctl实际未设置该字段。** 读者回读sysctl：

```sh
sysctl dev.myfirst.0.debug.mask
```

输出显示`0x0`。写入未生效。这是真正的问题。

**缩小范围。** 读者检查attach代码并发现：

```c
SYSCTL_ADD_U32(&sc->sc_sysctl_ctx,
    SYSCTL_CHILDREN(sc->sc_sysctl_tree),
    OID_AUTO, "mask",
    CTLFLAG_RD, &sc->sc_debug, 0, "debug class bitmask");
```

注意：`CTLFLAG_RD`（只读），不是`CTLFLAG_RW`。sysctl被声明为只读，所以`sysctl`命令看起来成功但实际上静默拒绝了写入。

**修复。** 将标志改为`CTLFLAG_RW`：

```c
SYSCTL_ADD_U32(&sc->sc_sysctl_ctx,
    SYSCTL_CHILDREN(sc->sc_sysctl_tree),
    OID_AUTO, "mask",
    CTLFLAG_RW, &sc->sc_debug, 0, "debug class bitmask");
```

重建、重新加载、设置掩码、打开设备，消息出现了。

**经验教训。** 三个工具在一分钟内找到了这个错误：`dmesg`显示了症状（缺失消息），`sysctl`暴露了真实状态（写入未生效），驱动程序源使原因可见（`CTLFLAG_RD`）。缺少其中任何一个，读者可能浪费十分钟在错误的假设上。工具不能替代思考；它们提供思考所需的证据。

sysctl回读通常是有用的纪律。每当驱动程序值应该是可写的，通过回读来测试写入。这一个习惯将捕获至少咬过每个FreeBSD开发者一次的一类错误。

### 案例研究2：间歇性崩溃

**症状。** 在轻负载下驱动程序完美工作。在压力工作负载（例如，一百个同时读取者）下，内核崩溃：

```text
panic: mutex myfirst_lock recursed on non-recursive mutex myfirst
```

崩溃是间歇性的：有时工作负载运行五分钟才崩溃，有时立即崩溃。读者确认使用了带有`options WITNESS`和`options INVARIANTS`的调试内核。

**第一步：阅读栈。** 崩溃消息包含栈追踪。相关行指向`myfirst_read`作为递归位置，`myfirst_read`本身已在栈的更高处。不知何故，`myfirst_read`在前一次`myfirst_read`调用仍在执行时被调用，两个调用都在同一线程上。

**假设。** `myfirst_read`内部的某物正在触发二次`myfirst_read`调用。最可能的候选是`uiomove`，它调用用户程序的页面错误路径，在非常不寻常的情况下可能回调到设备I/O。

**验证。** 读者添加两个`DPRINTF`调用，一个在入口处，一个在`uiomove`之前，各自捕获当前线程：

```c
DPRINTF(sc, MYF_DBG_IO, "read entry: tid=%d\n", curthread->td_tid);
DPRINTF(sc, MYF_DBG_IO, "read about to uiomove: tid=%d\n", curthread->td_tid);
```

重建、重新加载、运行工作负载。内核日志在任何"about to uiomove"之前显示同一线程ID的两个"read entry"消息，确认了假设。

**诊断。** `myfirst_read`中的`mtx_lock(&sc->sc_mtx)`调用在`uiomove`期间被持有。`uiomove`目标恰好在`/dev/myfirst0`的内存映射区域中，因为测试工作负载使用了`mmap`输出。`uiomove`触发的页面错误重新进入`myfirst_read`来服务错误，这尝试第二次获取`sc->sc_mtx`。

**修复。** 锁应在`uiomove`之前释放。一般来说，睡眠锁绝不能在可能页面错误的函数中持有。

```c
/* 有错误的版本 */
mtx_lock(&sc->sc_mtx);
error = uiomove(sc->sc_buffer, sc->sc_bufsize, uio);
mtx_unlock(&sc->sc_mtx);

/* 修复后的版本 */
mtx_lock(&sc->sc_mtx);
/* 快照缓冲区以便释放锁 */
tmp_buffer = sc->sc_buffer;
tmp_bufsize = sc->sc_bufsize;
mtx_unlock(&sc->sc_mtx);
error = uiomove(tmp_buffer, tmp_bufsize, uio);
```

重建、重新加载、运行压力工作负载。崩溃消失了。

**经验教训。** `WITNESS`没有直接捕获这个，因为这是递归而不是锁顺序违规。但`INVARIANTS`将递归变成干净的崩溃而不是静默损坏，DPRINTF追踪使时间线可见。"不要在`uiomove`或任何可能睡眠的函数中持有睡眠锁"这条规则在`locking(9)`中有说明，但在实践中很容易忘记。驱动程序代码中的每次睡眠锁获取都值得短暂检查"被包裹的调用会睡眠或页面错误吗？"。

### 案例研究3：慢中断

**症状。** 驱动程序工作正常，但设备活动时系统迟钝。Shell响应性下降，交互应用程序卡顿，`top`显示高系统CPU。

**第一步：确认驱动程序是原因。** 读者卸载驱动程序并比较：

```sh
# 卸载前
top -aC1 -bn1 | head -20
sudo kldunload myfirst
# 卸载后
top -aC1 -bn1 | head -20
```

如果迟钝在驱动程序卸载后消失，驱动程序是来源。

**第二步：识别昂贵的代码路径。** DTrace的`profile`提供者以固定速率采样内核并显示时间花在哪里：

```sh
sudo dtrace -n 'profile-1001hz /arg0/ { @[stack(10)] = count(); } tick-10s { exit(0); }'
```

在工作负载的十秒后，输出显示最热的栈。如果`myfirst_intr`主导样本，中断处理程序太重。

**第三步：测量每次中断花费多长时间。**

```sh
sudo dtrace -n 'fbt::myfirst_intr:entry { self->t = timestamp; }
                fbt::myfirst_intr:return /self->t/ {
                    @ = quantize(timestamp - self->t); self->t = 0; }'
```

健康的处理程序在10微秒（10 000纳秒）内完成。如果量化显示大多数调用在100 000到1 000 000纳秒范围（100微秒到1毫秒），处理程序做了太多工作。

**第四步：阅读处理程序代码。** 读者检查`myfirst_intr`并发现：

```c
static void
myfirst_intr(void *arg)
{
        struct myfirst_softc *sc = arg;
        struct mbuf *m;

        mtx_lock(&sc->sc_mtx);
        m = m_getcl(M_NOWAIT, MT_DATA, M_PKTHDR);
        if (m != NULL) {
                myfirst_process_data(sc, m);
                m_freem(m);
        }
        mtx_unlock(&sc->sc_mtx);
}
```

处理程序做真正的工作：它分配mbuf、处理数据并释放mbuf。`myfirst_process_data`包括几个包的工作，每个都有自己的分配和处理。在负载下，每次中断可以占用CPU数百微秒。

**修复。** 繁重工作移到taskqueue。处理程序本身做绝对最少的事：确认中断并信号taskqueue。

```c
static void
myfirst_intr(void *arg)
{
        struct myfirst_softc *sc = arg;

        /* 确认中断 */
        bus_write_4(sc->sc_res, MYFIRST_REG_INTR_STATUS, 0);
        /* 调度延迟处理 */
        taskqueue_enqueue(sc->sc_tq, &sc->sc_process_task);
}

static void
myfirst_process_task(void *arg, int pending)
{
        struct myfirst_softc *sc = arg;
        struct mbuf *m;

        mtx_lock(&sc->sc_mtx);
        m = m_getcl(M_NOWAIT, MT_DATA, M_PKTHDR);
        if (m != NULL) {
                myfirst_process_data(sc, m);
                m_freem(m);
        }
        mtx_unlock(&sc->sc_mtx);
}
```

taskqueue在`attach`中设置，如第19章所述。

重建、重新加载、运行工作负载。Shell响应性恢复，`top`显示驱动程序贡献正常量的系统CPU，`myfirst_intr`的延迟量化现在集中在1到10微秒桶。

**经验教训。** 三次DTrace测量指导了修复：`profile-1001hz`定位热代码，`fbt::myfirst_intr:entry/return`测量处理程序持续时间，最终修复后测量确认了改进。诊断期间未修改源代码；只有在理解了错误后才进行修改。

中断处理程序必须简短这条规则对驱动程序作者来说很熟悉，但执行不是自动的。调试构建不会因慢处理程序而崩溃。捕获此类错误的唯一方法是测量，DTrace是正确的工具。成熟的驱动程序团队在每个发布周期中运行`fbt::<driver>_intr:entry/return`性能分析；他们的处理程序保持精简，因为测量使任何偏离可见。

### 案例研究4：消失的设备

**症状。** 在`kldunload myfirst`之后，第二次`kldload myfirst`有时在`devfs_ioctl`中因页面错误崩溃，栈指向旧的`myfirst_ioctl`。崩溃不是每次都发生，但在负载下（例如，如果守护进程在卸载期间保持`/dev/myfirst0`打开）它始终是致命的。

**第一步：阅读崩溃信息。** 栈显示`devfs_ioctl -> myfirst_ioctl -> (address not found)`。"address not found"是出错指令。这是函数代码的释放后使用：内核试图调用一个内存已被取消映射的函数。

**假设。** 驱动程序的卸载路径未等待打开设备的进程。该进程在卸载和重新加载之间发出了ioctl；ioctl分派到现已取消映射的`myfirst_ioctl`。

**验证。** 读者检查`myfirst_detach`并发现：

```c
static int
myfirst_detach(device_t dev)
{
        struct myfirst_softc *sc = device_get_softc(dev);

        destroy_dev(sc->sc_cdev);
        bus_release_resource(dev, SYS_RES_MEMORY, 0, sc->sc_mem);
        mtx_destroy(&sc->sc_mtx);
        return (0);
}
```

`destroy_dev`调用是问题：它立即销毁设备节点，但节点上正在进行的ioctl可能仍在执行。当`destroy_dev`返回时，驱动程序认为释放资源是安全的。但ioctl在那些资源消失时仍在分派。

**修复。** 使用`destroy_dev_sched_cb`（或等效地，`destroy_dev_sched`然后等待完成）推迟实际销毁直到没有线程在设备的方法内部。

```c
static int
myfirst_detach(device_t dev)
{
        struct myfirst_softc *sc = device_get_softc(dev);

        /* 告诉devfs在所有调用者完成后销毁节点 */
        destroy_dev_sched_cb(sc->sc_cdev, myfirst_detach_cb, sc);
        /* 返回成功；清理在回调中发生 */
        return (0);
}

static void
myfirst_detach_cb(void *arg)
{
        struct myfirst_softc *sc = arg;

        bus_release_resource(sc->sc_dev, SYS_RES_MEMORY, 0, sc->sc_mem);
        mtx_destroy(&sc->sc_mtx);
}
```

detach立即返回；实际资源释放在回调中发生，回调被调度为仅在`cdev`方法内的每个调用者退出后才触发。

重建、在有问题的工作负载下反复重新加载。崩溃消失了。

**经验教训。** 这是第7.4节描述的经典分离后使用错误。调试内核的`DEBUG_MEMGUARD`选项可以通过毒化释放的内存更早捕获它，将潜在损坏变成下次调用时的立即页面错误。修复使用了正是为此情况存在的FreeBSD原语（`destroy_dev_sched_cb`）。阅读`cdev(9)`中的`cdev`管理文档是防止新代码中出现此错误的练习。

### 案例研究5：损坏的输出

**症状。** 用户程序从`/dev/myfirst0`读取期望特定模式（比如序列`0x01, 0x02, 0x03, ...`）。大多数时候模式正确。偶尔，在重负载系统上，模式偏离：一个或两个字节错误，或序列偏移。没有崩溃，没有错误码，只有微妙错误的数据。

**第一步：可靠重现。** 读者编写一个小测试程序，重复读取并与预期模式比较。在负载下，不匹配大约在万分之一读取中发生。读者确认卸载驱动程序使用简单`/dev/zero`测试不会产生不匹配，这排除了用户程序作为错误来源。

**假设1：缓冲区上的竞态。** 如果两个线程并发读取，且缓冲区在没有锁定的情况下共享，它们可能相互损坏。读者检查驱动程序并发现：

```c
static int
myfirst_read(struct cdev *dev, struct uio *uio, int ioflag)
{
        struct myfirst_softc *sc = dev->si_drv1;

        /* 将模式生成到sc->sc_buffer */
        for (int i = 0; i < sc->sc_bufsize; i++)
                sc->sc_buffer[i] = (uint8_t)(i + 1);

        return (uiomove(sc->sc_buffer, sc->sc_bufsize, uio));
}
```

缓冲区是每softc的，跨所有打开共享。两个线程同时调用`read`写入同一缓冲区；第二个线程在第一个完成复制之前覆盖了第一个的内容。第一个线程的`uiomove`然后部分从第一个线程的模式读取，部分从第二个线程的读取。结果是混乱的序列。

**验证。** 读者在read的开头和结尾添加一个`DPRINTF`，捕获线程的tid：

```c
DPRINTF(sc, MYF_DBG_IO, "read start: tid=%d\n", curthread->td_tid);
/* ... 生成和uiomove ... */
DPRINTF(sc, MYF_DBG_IO, "read end: tid=%d\n", curthread->td_tid);
```

当在损坏事件期间检查日志时，在一个`start`及其`end`之间出现了两个`start`消息。竞态被确认。

**修复。** 缓冲区必须是每次调用分配的（在每次read时分配）或访问必须用互斥锁序列化：

```c
static int
myfirst_read(struct cdev *dev, struct uio *uio, int ioflag)
{
        struct myfirst_softc *sc = dev->si_drv1;
        uint8_t *buf;
        int error;

        buf = malloc(sc->sc_bufsize, M_MYFIRST, M_WAITOK);
        for (int i = 0; i < sc->sc_bufsize; i++)
                buf[i] = (uint8_t)(i + 1);

        error = uiomove(buf, sc->sc_bufsize, uio);
        free(buf, M_MYFIRST);
        return (error);
}
```

每次调用分配消除了共享状态，错误消失。基于锁的修复对小缓冲区更快，但一次只允许一个读取者；权衡取决于驱动程序的用例。

**经验教训。** 损坏微妙正是因为它不产生崩溃或错误。`WITNESS`无法帮助，因为没有要排序的锁。`INVARIANTS`无法帮助，因为共享状态不触发断言。找到此错误的唯一方法是：测试输出，观察损坏，形成关于竞态的假设，添加仪器化以确认假设，然后修复。DPRINTF模式是仪器化；有纪律的测试工具是重现。两者一起解决问题。

此类错误是对驱动程序作者从一开始就编写测试工具的最强论证。一个从设备读取并与预期输出比较的小程序可以在几分钟内捕获许多错误，否则这些错误会在现场存活数月。

### 五个案例的总结

五个案例研究覆盖了刻意多样化的错误集。案例1是日志正确性错误，一旦回读sysctl就在几秒内找到。案例2是由分页路径触发的锁递归，通过结合崩溃追踪和DPRINTF找到。案例3是中断处理程序中的性能错误，用DTrace性能分析诊断。案例4是`cdev`路径中的分离后使用，用崩溃栈诊断并用正确的FreeBSD原语修复。案例5是read路径中的共享状态竞态，用测试工具和DPRINTF找到。

三个模式在案例中反复出现：

1. **工具找到症状；阅读代码找到原因。** 没有工具直接指向错误行。每个工具缩小搜索：`vmstat -m`、WITNESS、DTrace、DDB。然后读者打开源代码找到错误。

2. **重现比工具更重要。** 可靠重现的错误可以用几乎任何工具诊断。不能重现的错误必须先被重现，通常通过编写锻炼可疑路径的工作负载脚本来实现。

3. **调试基础设施的回报远超其成本。** `DPRINTF`宏、SDT探针、`sc_debug` sysctl：每一个添加花了五分钟，但每一个节省了数小时的调试。从一开始就编写仪器化代码的读者就是避免漫长痛苦调试会话的读者。

带着这些模式，读者准备好将本章的方法向前推进。后续章节中添加到`myfirst`驱动程序的每个新部分都将在`DPRINTF`调用、SDT探针和测试工具就位的情况下开发；从第24章开始，调试痛苦的下降将是明显的。

## 附录：驱动程序工作的高级DTrace技术

本附录以聚焦的DTrace技术导览扩展第5节，这些技术在驱动程序开发中经常出现。第5节介绍了基础；本附录展示如何高效地组合它们。发现DTrace有用的读者可能希望将本附录保留为速查表。

### 谓词和选择性追踪

谓词将探针缩小到匹配条件的事件。它出现在探针描述和操作之间的`/ ... /`中：

```sh
dtrace -n 'fbt::myfirst_read:entry /execname == "myftest"/ {
               @ = count(); }'
```

只有名为`myftest`的进程的读取被统计。谓词可以引用`pid`、`tid`、`execname`、`zonename`、`uid`和探针参数`arg0`、`arg1`等。

常见的模式是按softc指针筛选以聚焦单个设备实例：

```sh
dtrace -n 'fbt::myfirst_read:entry /args[0] == 0xfffff8000a000000/ {
               @ = count(); }'
```

softc的字面地址来自`devctl`或检查驱动程序的sysctl。当存在多个`myfirst`设备时，这可以一次隔离一个。

### 告诉故事的聚合

聚合是DTrace的统计主力。第5节介绍了`count()`和`quantize()`；其他聚合函数同样有用：

- `sum(x)`：`x`的运行总和
- `avg(x)`：`x`的运行平均值
- `min(x)`：所见`x`的最小值
- `max(x)`：所见`x`的最大值
- `lquantize(x, low, high, step)`：有界的线性量化
- `llquantize(x, factor, low, high, steps)`：有界的对数量化

典型的每函数延迟报告使用`quantize`，因为它揭示分布：

```sh
dtrace -n 'fbt::myfirst_read:entry { self->t = timestamp; }
           fbt::myfirst_read:return /self->t/ {
               @ = quantize((timestamp - self->t) / 1000);
               self->t = 0; }'
```

除以1000将纳秒转换为微秒，使桶有意义。添加一个按缓冲区大小索引的`@buf`聚合可以看到按缓冲区大小的延迟：

```sh
dtrace -n '
fbt::myfirst_read:entry {
    self->t = timestamp;
    self->len = args[1]->uio_resid;
}
fbt::myfirst_read:return /self->t/ {
    @["read", self->len] = quantize((timestamp - self->t) / 1000);
    self->t = 0;
}'
```

这对每个不同的`uio_resid`分别聚合，揭示大读取是否不成比例地慢。

### 线程局部变量

`self->`前缀引入线程局部变量：每个线程看到自己的副本。这就是第5节延迟测量的工作方式：入口探针将时间戳存储在`self->t`中，返回探针读取它。因为两个探针在同一线程上触发，变量是明确的。

常见用途：

```sh
self->start    /* 特定函数入口的时间 */
self->args     /* 保存的参数用于返回探针 */
self->error    /* 从返回中捕获的错误码 */
```

线程局部变量默认为零。自由使用它们；它们的成本是每个活跃线程一个字的内存。

### 数组和关联访问

DTrace数组是稀疏关联数组，由任何表达式索引。聚合隐式使用它们（`@[key]`语法），但普通变量也可以使用：

```sh
dtrace -n 'fbt::myfirst_read:entry {
               counts[args[0], execname]++; }'
```

虽然实际上，聚合更可取，因为它们在所有CPU上正确累积。

### 推测追踪

对于大多数情况无趣的高频事件，DTrace的推测追踪缓冲输出直到做出决定：

```sh
dtrace -n '
fbt::myfirst_read:entry {
    self->spec = speculation();
    speculate(self->spec);
    printf("read entry: pid=%d", pid);
}
fbt::myfirst_read:return /self->spec && args[1] == 0/ {
    speculate(self->spec);
    printf("read return: bytes=%d", arg1);
    commit(self->spec);
    self->spec = 0;
}
fbt::myfirst_read:return /self->spec && args[1] != 0/ {
    discard(self->spec);
    self->spec = 0;
}'
```

只有成功返回（`args[1] == 0`）的读取提交其输出；失败的读取被丢弃。这在重负载下产生安静、集中的日志。

推测追踪是DTrace最有效的特性之一，对于小型驱动程序很少需要。但对于每秒数千事件的驱动程序，它使有用输出和不可读洪流之间的区别成为可能。

### 用于窄追踪的谓词链

常见需求是从特定入口点追踪一个函数调用树。谓词可以使用线程局部标志链接：

```sh
dtrace -n '
syscall::ioctl:entry /pid == $target/ { self->trace = 1; }
fbt::myfirst_*: /self->trace/ { printf("%s: %s", probefunc, probename); }
syscall::ioctl:return /self->trace/ { self->trace = 0; }'
```

用`-p <pid>`运行以定向特定进程。追踪标志在系统调用入口设置并在返回时清除，因此只有ioctl期间调用的驱动程序函数被记录。此模式对于隔离特定用户操作是基础的。

### 内核性能分析

`profile`提供者以规律速率采样内核。有用的探针包括：

- `profile-97`：每秒的第97分之一采样，在每个CPU上
- `profile-1001hz`：以1001 Hz采样，略微偏离任何整秒工作负载
- `profile-5ms`：每5毫秒采样

每个样本记录当前栈。按栈聚合揭示时间花在哪里：

```sh
dtrace -n 'profile-1001hz /arg0/ { @[stack(8)] = count(); }
           tick-10s { exit(0); }'
```

`arg0`谓词过滤掉空闲CPU。`stack(8)`记录前8帧。`tick-10s`在10秒后结束运行。

输出如下：

```text
  kernel`myfirst_process_data+0x120
  kernel`myfirst_intr+0x48
  kernel`ithread_loop+0x180
  kernel`fork_exit+0x7f
  kernel`fork_trampoline+0xe
         15823
```

该栈有15 823个样本，约占总共约100 000个（10秒 * 1001 Hz * 几个CPU）样本。驱动程序的中断路径主导。结合上面的延迟测量，这足以证明重构的合理性。

### 美化打印结构

启用CTF的内核为DTrace提供每个内核结构的类型。这使`print()`能按名称而非原始内存显示字段：

```sh
dtrace -n 'fbt::myfirst_read:entry { print(*args[0]); }'
```

输出显示read操作的`struct myfirst_softc`的每个字段，按字段名称，带人类可读的值。

该特性取决于已生成CTF（`makeoptions WITH_CTF=1`）。没有CTF，DTrace回退到基于地址的打印，可读性差得多。

### 环形缓冲输出

对于非常长的DTrace运行，默认输出填满内存。`-b`标志设置一个环形缓冲区，满时丢弃旧记录：

```sh
dtrace -b 16m -n 'myfirst::: { trace(timestamp); }'
```

16 MB环形缓冲保存大约最后16 MB的追踪输出。旧数据在新数据到达时被覆盖，这通常对诊断来说没问题：感兴趣的窗口是问题前的最后几秒，而不是前一小时。

### 干净退出

DTrace在内存中累积聚合直到脚本退出。对于长时间运行，读者必须干净退出才能看到聚合：

```sh
dtrace -n 'fbt::myfirst_*:entry { @[probefunc] = count(); }
           tick-60s { printf("%Y\n", walltimestamp); printa(@);
                      clear(@); }'
```

每60秒，聚合被打印并清除。输出是运行报告，而不是退出时的单一报告。此模式对于通宵运行很理想：输出包含每分钟快照，读者可以看到驱动程序的行为如何演变。

### 总结

这些技术一起覆盖了驱动程序作者需要的DTrace技能的大约80%。剩下的20%来自DTrace指南和阅读`/usr/share/dtrace/toolkit`中的现有脚本。

两条经验法则指导高级使用：

1. **从可能回答问题的最简单脚本开始。** 仅在脚本产生太多输出时添加谓词。仅在计数和时间重要时添加聚合。

2. **对照已知工作负载验证DTrace输出。** 首先运行简单测试用例（比如一次open、一次read、一次close），确认DTrace输出匹配，然后才在复杂工作负载上信任工具。

有了这些规则，DTrace是FreeBSD武器库中最强大的调试工具，是每投入一小时学习就能节省数月调试时间的工具。


## 附录：WITNESS追踪解读

WITNESS往往是读者第一次接触解读内核诊断追踪。本附录演练WITNESS违规消息的格式并解释每个字段的意思。

### 一般格式

WITNESS违规看起来像：

```text
lock order reversal: (non-sleepable after sleepable)
 1st 0xfffff8000a000000 myfirst_lock (myfirst, sleep mutex) @ myfirst.c:123
 2nd 0xfffff8000a001000 myfirst_intr_lock (myfirst, spin mutex) @ myfirst.c:234
lock order myfirst_intr_lock -> myfirst_lock established at:
#0 0xffffffff80abcdef at kdb_backtrace+0x1f
#1 0xffffffff80abcdee at _witness_debugger+0x4f
#2 0xffffffff80abcdaf at witness_checkorder+0x21f
#3 0xffffffff80c00000 at _mtx_lock_flags+0x8f
#4 0xffffffff80ffff00 at myfirst_intr+0x30
#5 ...
panic: witness
```

消息有三个部分：标题行、锁描述部分和反向顺序首次确立位置的反向追踪。

### 标题

```text
lock order reversal: (non-sleepable after sleepable)
```

这说在已持有睡眠（可睡眠）互斥锁时获取了自旋（不可睡眠）互斥锁。反向顺序（先自旋再睡眠）是允许的；此顺序（先睡眠再自旋）不允许。

常见标题变体：

- `lock order reversal: (sleepable after non-sleepable)`：声明的顺序后来被反向获取打破
- `spin lock recursion`：尝试获取当前线程已持有的自旋锁，除非是递归自旋锁不允许
- `spin lock held too long`：自旋锁被持有超过阈值（通常是数秒）
- `blocking on condition variable with spin lock`：在持有自旋锁时尝试睡眠

### 锁描述

```text
 1st 0xfffff8000a000000 myfirst_lock (myfirst, sleep mutex) @ myfirst.c:123
 2nd 0xfffff8000a001000 myfirst_intr_lock (myfirst, spin mutex) @ myfirst.c:234
```

- **0xfffff8000a000000**：互斥锁在内存中的地址
- **myfirst_lock**：传给`mtx_init`的人类可读名称
- **(myfirst, sleep mutex)**：类（通常是驱动程序名）和类型
- **@ myfirst.c:123**：`mtx_init`调用的源位置（当WITNESS设置有DEBUG_LOCKS时）

两个锁都出现，先是已持有的那个，然后是正在获取的那个。

### 反向追踪

反向追踪显示反向顺序首次记录的位置。WITNESS记住它看到的每个(A, B)对的第一个实例；后续违规引用更早的出现。如果首次出现当时是有效的（例如两个锁都正确获取）但后来的路径反向它们，反向追踪指向更早的正确获取。读者可能需要搜索源代码寻找特定的反向路径，通常这是当前栈追踪（在此片段中未显示）。

要看的两帧：

1. 当前持有第一个锁并尝试获取第二个的函数（或反之）。
2. 首次以原始顺序获取两个锁的函数。

比较两者揭示冲突。

### 阅读真实示例

新驱动程序中的常见示例：

```text
lock order reversal: (non-sleepable after sleepable)
 1st 0xfff0 myfirst_lock (sleep mutex) @ myfirst.c:100
 2nd 0xfff1 ithread_lock (spin mutex) @ kern_intr.c:234
```

新驱动程序在`myfirst.c:100`的代码获取`myfirst_lock`（睡眠互斥锁）。在代码路径深处，中断调度函数尝试获取`ithread_lock`（中断子系统使用的自旋互斥锁）。这是`睡眠后自旋`违规，这总是错误：持有睡眠锁时绝不能获取自旋锁。

修复：重构代码使自旋锁（或使用它的子系统）在持有`myfirst_lock`时不被调用。常见方法包括在调用前释放`myfirst_lock`，然后在需要时重新获取。

### 断言锁所有权

独立于WITNESS，在每个假设持有锁的函数顶部放置`mtx_assert(&lock, MA_OWNED)`是强防御性编码模式。设置`INVARIANTS`时，断言在任何违规时触发。结合WITNESS，这两个检查在首次发生时捕获绝大多数锁相关错误，远早于它们损坏生产数据。

### 当WITNESS太吵时

偶尔WITNESS在实际上是正确的代码中产生违规，通常因为锁获取由WITNESS无法看到的运行时检查保护。对于这些情况，`mtx_lock_flags(&lock, MTX_DUPOK)`或`MTX_NEW`告诉WITNESS允许获取。谨慎使用这些标志；大多数WITNESS违规是真实的。

## 附录：构建驱动程序特定的调试工具包

第8节添加到`myfirst`的调试宏和SDT探针是起点。在驱动程序的生命周期中，工具包增长：更多类、更多探针、自定义sysctl，也许一个调试专用ioctl转储内部状态。本附录概述扩展工具包的模式。

### 调试专用Sysctl

`dev.myfirst.0.debug.*`下的sysctl是运行时调试控制的自然位置。自由添加节点：

- `dev.myfirst.0.debug.mask`：类位掩码
- `dev.myfirst.0.debug.loglevel`：DPRINTF输出的syslog优先级
- `dev.myfirst.0.debug.trace_pid`：聚焦追踪的PID
- `dev.myfirst.0.debug.count_io`：处理的I/O事件计数器
- `dev.myfirst.0.debug.dump_state`：只写，触发一次性状态转储

每个在大小上成本很小，在现场调试期间极其有用。规则是：如果一块状态在错误搜索期间有用，通过sysctl暴露它。

### 调试Ioctl

对于太复杂或太大无法作为sysctl暴露的状态，调试专用ioctl是好选择。定义：

```c
#define MYFIRST_IOC_DUMP_STATE  _IOR('f', 100, struct myfirst_debug_state)

struct myfirst_debug_state {
        uint64_t        open_count;
        uint64_t        read_count;
        uint64_t        write_count;
        uint64_t        error_count;
        uint32_t        current_mask;
        /* ... */
};
```

ioctl处理程序将当前状态复制到用户缓冲区。一个小用户程序读取并打印它：

```c
struct myfirst_debug_state s;
int fd = open("/dev/myfirst0", O_RDONLY);
ioctl(fd, MYFIRST_IOC_DUMP_STATE, &s);
printf("opens=%llu, reads=%llu, writes=%llu, errors=%llu\n",
    s.open_count, s.read_count, s.write_count, s.error_count);
```

调试ioctl在感兴趣的状态不是单个数字而是结构时特别有用，当sysctl的类型约束笨拙时也是如此。

### 关键事件计数器

驱动程序的softc获得一组计数器，每个重要事件一个。计数器是简单的`uint64_t`字段，用`atomic_add_64`或在softc锁下增加。它们通过sysctl或调试ioctl暴露，可由读者请求清除：

```c
sc->sc_stats.opens++;       /* 在open中 */
sc->sc_stats.reads++;       /* 在read中 */
sc->sc_stats.errors++;      /* 在任何错误路径上 */
```

每个驱动程序十个计数器，内存成本不到100字节，对驱动程序行为的可见性巨大。生产监控系统可以定期轮询计数器并对异常发出警报。

### 自测Ioctl

对于复杂驱动程序，自测ioctl可以无价。它运行一系列内部测试（每个都是驱动程序中的函数）并报告哪些通过哪些失败。结果以小结构返回。操作员在现场问题调试时可以运行自测并立即知道驱动程序的任何子系统是否损坏。

自测ioctl不是单元测试或集成测试的替代品。它是现场的诊断捷径。

### 与rc脚本集成

感知调试的rc脚本（在`/usr/local/etc/rc.d/myfirst_debug`）可以在启动时设置调试掩码：

```sh
#!/bin/sh
# PROVIDE: myfirst_debug
# REQUIRE: myfirst
# KEYWORD: shutdown

. /etc/rc.subr

name="myfirst_debug"
rcvar="myfirst_debug_enable"
start_cmd="myfirst_debug_start"
stop_cmd="myfirst_debug_stop"

myfirst_debug_start()
{
        echo "Enabling myfirst debug mask"
        sysctl dev.myfirst.0.debug.mask=${myfirst_debug_mask:-0x0}
}

myfirst_debug_stop()
{
        echo "Disabling myfirst debug"
        sysctl dev.myfirst.0.debug.mask=0
}

load_rc_config $name
run_rc_command "$1"
```

操作员在`/etc/rc.conf`中设置`myfirst_debug_enable=YES`和`myfirst_debug_mask=0x3`，重启，调试基础设施自动启用。此模式是生产系统跨许多设备管理调试详细度的方式。

### 总结

第8节的基础设施足够本书的运行示例。真实驱动程序超出它，随时间获得十多个调试导向特性。模式总是一样的：使仪器化关闭时便宜、开启时丰富，并由操作员普遍可控。有此纪律，驱动程序在其整个生命周期中保持可诊断。

### myfirst_debug.h

```c
/*
 * myfirst_debug.h - myfirst驱动程序的调试和追踪基础设施
 *
 * 此头文件从驱动程序源文件包含。它提供：
 *   - 调试详细度类的位掩码
 *   - 条件device_printf的DPRINTF宏
 *   - 驱动程序在关键点触发的SDT探针声明
 *
 * 匹配的SDT_PROVIDER_DEFINE和SDT_PROBE_DEFINE调用位于
 * 驱动程序源中，它拥有探针条目的存储。
 */

#ifndef _MYFIRST_DEBUG_H_
#define _MYFIRST_DEBUG_H_

#include <sys/param.h>
#include <sys/systm.h>
#include <sys/sdt.h>

/*
 * 调试详细度类。每个类是sc->sc_debug中的单个位。
 * 操作员设置sysctl dev.myfirst.0.debug.mask为这些位
 * 的组合以启用相应类别的输出。
 *
 * 当驱动程序增长新子系统时在此添加新类。使用
 * 下一个未使用的位并相应更新DEBUG.md。
 */
#define MYF_DBG_INIT    0x00000001  /* probe/attach/detach */
#define MYF_DBG_OPEN    0x00000002  /* open/close生命周期 */
#define MYF_DBG_IO      0x00000004  /* read/write路径 */
#define MYF_DBG_IOCTL   0x00000008  /* ioctl处理 */
#define MYF_DBG_INTR    0x00000010  /* 中断处理程序 */
#define MYF_DBG_DMA     0x00000020  /* DMA映射/同步 */
#define MYF_DBG_PWR     0x00000040  /* 电源管理事件 */
#define MYF_DBG_MEM     0x00000080  /* malloc/free追踪 */
/* 位0x0100..0x8000为未来驱动程序子系统保留 */

#define MYF_DBG_ANY     0xFFFFFFFF
#define MYF_DBG_NONE    0x00000000

/*
 * DPRINTF - 当给定类位在softc的调试掩码中设置时，
 * 通过device_printf条件性地记录消息。
 *
 * 用法: DPRINTF(sc, MYF_DBG_OPEN, "open: pid=%d\n", pid);
 *
 * 当位清除时，成本是一次加载和一个分支，
 * 实际上可以忽略。当位设置时，成本等于
 * 正常device_printf调用。
 */
#ifdef _KERNEL
#define DPRINTF(sc, m, ...) do {                                        \
        if ((sc)->sc_debug & (m))                                        \
                device_printf((sc)->sc_dev, __VA_ARGS__);                \
} while (0)
#endif

/*
 * SDT探针声明。匹配的SDT_PROBE_DEFINE调用在
 * myfirst_debug.c中（或在主驱动程序源中如果更喜欢）。
 *
 * 探针参数约定：
 *   open  (softc *, flags)            -- 入口，访问检查前
 *   close (softc *, flags)            -- 入口，状态更新前
 *   io    (softc *, is_write, resid, off) -- 入口，进入read或write
 */
SDT_PROVIDER_DECLARE(myfirst);
SDT_PROBE_DECLARE(myfirst, , , open);
SDT_PROBE_DECLARE(myfirst, , , close);
SDT_PROBE_DECLARE(myfirst, , , io);

#endif /* _MYFIRST_DEBUG_H_ */
```

头文件有三部分：类位掩码、DPRINTF宏和SDT探针声明。每个都很小、自包含，并设计为随驱动程序增长。

### myfirst_debug.c中的SDT定义

```c
/*
 * myfirst_debug.c - SDT探针条目的存储。
 *
 * 此文件存在以保存SDT_PROVIDER_DEFINE和SDT_PROBE_DEFINE
 * 声明。按myfirst驱动程序的约定，这些位于
 * 专用源文件中以保持主驱动程序整洁。
 */

#include <sys/param.h>
#include <sys/systm.h>
#include <sys/sdt.h>
#include "myfirst_debug.h"

/*
 * 提供者"myfirst"将我们的所有静态探针暴露给DTrace。
 * 脚本用"myfirst:::<name>"选择探针。
 */
SDT_PROVIDER_DEFINE(myfirst);

/*
 * open: 每次成功或尝试的设备打开时触发。
 *   arg0 = struct myfirst_softc *
 *   arg1 = int flags 来自open调用
 */
SDT_PROBE_DEFINE2(myfirst, , , open,
    "struct myfirst_softc *", "int");

/*
 * close: 每次设备关闭时触发。
 *   arg0 = struct myfirst_softc *
 *   arg1 = int flags
 */
SDT_PROBE_DEFINE2(myfirst, , , close,
    "struct myfirst_softc *", "int");

/*
 * io: 每次read或write调用时触发，在函数入口。
 *   arg0 = struct myfirst_softc *
 *   arg1 = int is_write (0表示read, 1表示write)
 *   arg2 = size_t resid (请求的字节数)
 *   arg3 = off_t offset
 */
SDT_PROBE_DEFINE4(myfirst, , , io,
    "struct myfirst_softc *", "int", "size_t", "off_t");
```

文件很短，有意如此。其唯一目的是保存探针条目的存储。如果驱动程序增长更多探针，读者在此添加。

### 在驱动程序中触发探针

在`myfirst.c`中（或实现每个方法的文件），探针在适当的调用点触发：

```c
/*
 * myfirst_open: 设备open方法。
 * 用户进程打开/dev/myfirst0时调用。
 */
static int
myfirst_open(struct cdev *dev, int flags, int devtype, struct thread *td)
{
        struct myfirst_softc *sc = dev->si_drv1;
        int error = 0;

        DPRINTF(sc, MYF_DBG_OPEN, "open: pid=%d uid=%d flags=0x%x\n",
            td->td_proc->p_pid, td->td_ucred->cr_uid, flags);

        /* 在入口处触发SDT探针，在任何状态改变前。 */
        SDT_PROBE2(myfirst, , , open, sc, flags);

        mtx_lock(&sc->sc_mtx);
        if (sc->sc_open_count >= MYFIRST_MAX_OPENS) {
                error = EBUSY;
                goto out;
        }
        sc->sc_open_count++;
out:
        mtx_unlock(&sc->sc_mtx);

        if (error != 0)
                DPRINTF(sc, MYF_DBG_OPEN, "open failed: error=%d\n", error);
        else
                DPRINTF(sc, MYF_DBG_OPEN, "open ok: count=%d\n",
                    sc->sc_open_count);

        return (error);
}

/*
 * myfirst_close: 设备close方法。
 * 当对设备的最后引用被释放时调用。
 */
static int
myfirst_close(struct cdev *dev, int flags, int devtype, struct thread *td)
{
        struct myfirst_softc *sc = dev->si_drv1;

        DPRINTF(sc, MYF_DBG_OPEN, "close: pid=%d flags=0x%x\n",
            td->td_proc->p_pid, flags);

        SDT_PROBE2(myfirst, , , close, sc, flags);

        mtx_lock(&sc->sc_mtx);
        if (sc->sc_open_count > 0)
                sc->sc_open_count--;
        DPRINTF(sc, MYF_DBG_OPEN, "close ok: count=%d\n", sc->sc_open_count);
        mtx_unlock(&sc->sc_mtx);

        return (0);
}

/*
 * myfirst_read: 设备read方法。
 */
static int
myfirst_read(struct cdev *dev, struct uio *uio, int ioflag)
{
        struct myfirst_softc *sc = dev->si_drv1;
        int error;

        DPRINTF(sc, MYF_DBG_IO, "read: pid=%d resid=%zu off=%jd\n",
            curthread->td_proc->p_pid,
            (size_t)uio->uio_resid, (intmax_t)uio->uio_offset);

        SDT_PROBE4(myfirst, , , io, sc, 0,
            (size_t)uio->uio_resid, uio->uio_offset);

        error = myfirst_read_impl(sc, uio);

        if (error != 0)
                DPRINTF(sc, MYF_DBG_IO, "read failed: error=%d\n", error);
        return (error);
}

/*
 * myfirst_write: 设备write方法。
 */
static int
myfirst_write(struct cdev *dev, struct uio *uio, int ioflag)
{
        struct myfirst_softc *sc = dev->si_drv1;
        int error;

        DPRINTF(sc, MYF_DBG_IO, "write: pid=%d resid=%zu off=%jd\n",
            curthread->td_proc->p_pid,
            (size_t)uio->uio_resid, (intmax_t)uio->uio_offset);

        SDT_PROBE4(myfirst, , , io, sc, 1,
            (size_t)uio->uio_resid, uio->uio_offset);

        error = myfirst_write_impl(sc, uio);

        if (error != 0)
                DPRINTF(sc, MYF_DBG_IO, "write failed: error=%d\n", error);
        return (error);
}
```

每个方法遵循入口-退出-错误模式：

1. 入口处的`DPRINTF`，显示谁调用以及用了什么参数。
2. 入口日志后的`SDT_PROBE`，在真正工作开始前。
3. 任何非零返回上的错误路径`DPRINTF`。
4. 有用的成功路径`DPRINTF`显示结果。

在`myfirst_attach`顶部附近，版本被记录：

```c
DPRINTF(sc, MYF_DBG_INIT, "attach: %s loaded\n", myfirst_version);
```

设置了包含`MYF_DBG_INIT`的掩码后，操作员在每个attach上看到干净的版本报告行。

### 启用所有类时的示例dmesg

当所有类启用且设备被使用时，`dmesg`显示全面的日志：

```text
myfirst0: attach: myfirst 1.6-debug loaded
myfirst0: open: pid=1234 uid=1001 flags=0x0
myfirst0: open ok: count=1
myfirst0: read: pid=1234 resid=4096 off=0
myfirst0: close: pid=1234 flags=0x0
myfirst0: close ok: count=0
```

驱动程序关心的每个事件都可见，标记设备名称、PID和相关参数。接收此日志的现场工程师可以重构事件序列，无需源代码访问。

所有类禁用时，`dmesg`只显示attach行（如果`MYF_DBG_INIT`启用）或什么都没有（如果掩码为0）。驱动程序以全速运行，没有可见性成本。

### 使用探针的示例DTrace会话

SDT探针允许机器可读分析：

```sh
$ sudo dtrace -n 'myfirst::: { @[probename] = count(); } tick-10s { exit(0); }'
dtrace: description 'myfirst::: ' matched 3 probes
CPU     ID                    FUNCTION:NAME
  0  49012                        :tick-10s

  close                                                     43
  open                                                      43
  io                                                       120
```

在10秒采样期间，驱动程序处理了43个open/close周期和120个I/O事件。聚合在没有任何驱动程序修改和基本无运行时成本的情况下生成，因为没有脚本附加意味着探针是惰性的。

### 重构总结

1.6-debug重构增加大约100行代码，分布如下：

- `myfirst_debug.h`：50行（类、DPRINTF、SDT声明）
- `myfirst_debug.c`：30行（SDT提供者和探针定义）
- 驱动程序源更改：20行（sysctl注册、DPRINTF调用、探针触发）

投资很小；回报巨大。本书后续每章继承此基础设施无需额外成本，驱动程序未来的每个现场问题都有清晰的诊断路径。

## 附录：驱动程序崩溃时内核做什么

跟随本章到目前为止的读者理解调试工具良好。最后的附录覆盖灾难性错误发生时的情况：崩溃。内核在崩溃期间的行为对于驱动程序作者来说理解不是可选的，因为任何严重到足以导致崩溃的错误会产生驱动程序必须正确理解的一系列特定事件。

### 崩溃序列

当任何内核代码调用`panic()`时，以下事件按顺序发生：

1. 崩溃消息打印到控制台，通常通过`printf`。
2. CPU将其优先级提升到最大，停止大多数普通内核活动。
3. 如果内核构建了`options KDB`且`kdb_unattended`未设置，控制转移到内核调试器。操作员看到`db>`提示。
4. 如果操作员（或`kdb_unattended=1`）继续崩溃，内核启动崩溃转储。转储将当前内存镜像写入崩溃转储设备（通常是交换分区）。
5. 内核重启（除非设置了`panic_reboot_wait_time`）。

下次启动时，`savecore(8)`从交换设备提取转储，写入`/var/crash/vmcore.N`，并重新编号计数器。读者然后可以运行`kgdb`针对转储检查事后状态。

### 驐动程序职责

崩溃路径对驱动程序有特定要求：

1. **在分离期间不要崩溃。** 在`detach`期间崩溃的驱动程序阻止内核其余部分的干净关闭。每个分离路径应无崩溃。

2. **不要在崩溃处理程序中分配内存。** 如果驱动程序注册在崩溃期间运行的关闭钩子，钩子绝不能调用`malloc`或任何可能睡眠的函数。

3. **不要在崩溃期间依赖用户空间。** 用户程序在崩溃期间冻结；驱动程序无法向它们发送消息。

4. **不要在崩溃期间等待中断。** 崩溃路径期间中断可能被禁用；等待一个的驱动程序会挂起。

这些规则适用于关闭钩子和内核在崩溃期间遍历的罕见路径。正常操作中，驱动程序不在崩溃路径中运行，但某些回调确实运行。

### 阅读崩溃转储

崩溃转储是崩溃时刻内核的完整内存镜像。如果内核有调试信息，`kgdb`以完整符号信息打开它：

```sh
sudo kgdb /boot/kernel/kernel /var/crash/vmcore.0
```

在`kgdb`内，最有用的命令是：

```console
(kgdb) bt            # 崩溃线程的回溯
(kgdb) info threads  # 列出所有线程
(kgdb) thread N      # 切换到线程N
(kgdb) frame N       # 显示帧N（在当前回溯中）
(kgdb) list          # 显示当前帧的源代码
(kgdb) info locals   # 显示局部变量
(kgdb) print var     # 显示变量
(kgdb) print *sc     # 显示softc指针内容
(kgdb) ptype struct myfirst_softc  # 显示结构类型
```

常见调查步骤：

1. `bt`看什么崩溃了。
2. `list`看源代码行。
3. `info locals`看局部状态。
4. `print`感兴趣的字段。
5. `thread apply all bt`看所有线程，以防崩溃由一个线程与另一个线程的交互触发。

### 组合内核和驱动程序符号

如果驱动程序作为模块构建，`kgdb`也需要加载其符号：

```console
(kgdb) add-symbol-file /boot/kernel/myfirst.ko <address>
```

`<address>`是模块的加载地址，`kldstat -v`在运行系统中显示，或出现在崩溃输出中。一旦加载，`kgdb`可以解码模块内的帧，带完整源可见性。

### 崩溃转储大小

完整崩溃转储大小等于崩溃时的内核内存使用量，通常是GB级。转储设备（通常是交换）必须有足够空间。为快速崩溃恢复调优的系统经常使用专用转储设备而非交换，以避免交换空间约束。

对于仅驱动程序调试，小型转储仅捕获内核自己的内存而非用户内存。在`loader.conf`中设置`dumpdev=none`或`dumpdev=mini`调整此设置。小型转储很小（几十MB）并快速加载到`kgdb`。

### 总结

崩溃路径在生产中很少但对理解至关重要。干净处理错误的驱动程序很少触发崩溃；崩溃的驱动程序产生作者可以用`kgdb`分析的崩溃转储。`INVARIANTS`、`WITNESS`、`DEBUG_MEMGUARD`和功能的崩溃转储管道的组合使崩溃变成有价值的诊断事件而非灾难。

### 常用DTrace一行脚本

```sh
# 统计驱动程序中每个函数入口
dtrace -n 'fbt::myfirst_*:entry { @[probefunc] = count(); }'

# 测量特定函数中花费的时间
dtrace -n 'fbt::myfirst_read:entry { self->t = timestamp; }
           fbt::myfirst_read:return /self->t/ {
               @ = quantize(timestamp - self->t); self->t = 0; }'

# 统计驱动程序触发的SDT探针
dtrace -n 'myfirst::: { @[probename] = count(); }'

# 显示命名进程的系统调用频率
dtrace -n 'syscall:::entry /execname == "myftest"/ { @[probefunc] = count(); }'

# 聚合通过驱动程序的I/O大小
dtrace -n 'myfirst:::io { @ = quantize(arg2); }'
```

```sh
# 在ktrace下记录进程
sudo ktrace -di -f trace.out ./myprogram

# 以人类可读形式转储

# 仅筛选系统调用

```

### 调试检查清单（简化）

1. `dmesg`干净吗？如果不，读取问题前最后一条消息。
2. 驱动程序池的`vmstat -m`稳定吗？如果不，寻找泄漏。
3. `WITNESS`在工作负载下崩溃吗？如果是，锁顺序错误。
4. `fbt::myfirst_*:entry`计数匹配预期活动吗？如果不，检查路径。
5. 用户侧`ktrace`显示预期系统调用序列吗？如果不，检查ioctl或命令设置。
6. 驱动程序处于预期版本吗？`kldstat -v | grep myfirst`并确认`MODULE_VERSION`。

## 参考：第23章术语表

**CTF**：紧凑C类型格式。DDB和DTrace用来打印类型化结构的内核类型信息格式。需要`makeoptions WITH_CTF=1`。

**DDB**：内置FreeBSD内核调试器。交互式、基于控制台。通过`options DDB`激活。

**DDB_CTF**：让调试器使用CTF类型信息打印类型化值的DDB选项。

**device_printf**：FreeBSD标准的驱动程序消息日志函数。输出带设备名前缀。

**DPRINTF**：以详细度掩码为条件的传统驱动程序条件日志宏。

**DTrace**：动态追踪框架。使用`fbt`、`syscall`、`sdt`、`io`、`sched`、`lockstat`和其他提供者。

**fbt**：函数边界追踪。追踪每个内核函数入口和返回的DTrace提供者。

**INVARIANTS**：启用`KASSERT`、`MPASS`和相关断言的内核构建选项。调试构建的标准。

**KASSERT**：仅在设置`INVARIANTS`时计算的断言宏。条件为假时崩溃。

**KDB**：内核调试器框架。`DDB`是一个后端。

**KDTRACE_HOOKS**：启用DTrace探针的内核构建选项。

**kdump**：读取`ktrace`输出文件并以人类可读形式打印的用户空间工具。

**ktrace**：记录给定进程的系统调用和其他事件的用户空间工具。

**log**：以优先级级别发送消息到syslog的内核函数。

**MPASS**：类似`KASSERT`但仅有编译时检查。发布构建中零成本。

**myfirst_debug.h**：第8节介绍的调试头文件，声明详细度类和SDT探针结构。

**sc_debug**：驱动程序softc中控制哪些`DPRINTF`类别活跃的`uint32_t`字段。

**SDT**：静态定义追踪。DTrace可以附加的编译时探针。

**SDT_PROBE_DEFINE**：在内核或驱动程序中注册新SDT探针的宏族。

**syslog**：BSD日志子系统；消息优先级按`/etc/syslog.conf`映射到目的地。

**vmstat -m**：显示每池内核内存统计的用户空间工具。

**WITNESS**：内核锁顺序验证系统。设置`options WITNESS`时在锁顺序违规上崩溃。

**1.6-debug**：第8节后的驱动程序版本。携带详细度掩码、sysctl、DPRINTF模式和SDT探针。
