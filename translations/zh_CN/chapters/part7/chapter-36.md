# 在无文档情况下创建驱动程序（逆向工程）

## 简介

到目前为止，本书中我们编写的驱动程序都是针对那些行为至少有部分文档记录的设备。有时文档非常详尽，提供了完整的程序员参考手册，命名了每个寄存器，定义了每个位，描述了每条命令。有时文档很稀少，只有厂商提供的一个头文件和简短的操作码列表。然而，即使在文档稀少的情况下，我们也总有一个起点：一个提示、一份不完整的数据手册、来自相关设备系列的示例，或者一个告诉我们要让设备做什么的手册页面。

本章改变了这一假设。在这里，我们将学习如何为文档缺失、丢失或被故意隐瞒的设备编写驱动程序。硬件是存在的。它在其他操作系统下可以工作，或者曾经可以工作，或者有人用逻辑分析仪捕获了它几秒钟的行为。但没有寄存器参考，没有命令列表，没有数据格式的描述。我们需要每一个事实都要自己去发现。

如果这听起来令人生畏，请深呼吸一下。逆向工程硬件设备是一门严肃的技术，但它不是魔法。它就是你在本书中一直实践的同样的工程学科，只是应用方向略有不同。我们不是阅读数据手册并编写实现它的代码，而是观察设备，形成关于它做什么的假设，用小型实验测试这些假设，然后一次验证一个事实地编写驱动程序。最终结果与我们在前几章中编写的驱动程序是同一类型，具有相同的 `cdevsw`、相同的 `device_method_t` 表、相同的 `bus_space(9)` 调用和相同的模块构建。唯一的区别是我们如何得出这些内容。

逆向工程值得专门写一章，原因有几个。首先是这种情况比新手意识到的更常见。厂商已经倒闭的旧硬件；从未被正确记录的消费者设备；定制板上的嵌入式外设，制造商只是提供一个二进制代码块，数据手册随着合同到期而失效；专业工业、科学或医疗设备，文档存在一张没人能找到的光盘上。所有这些都是真正的 FreeBSD 开发者遇到的真实情况，几个重要的 FreeBSD 驱动程序之所以存在，只是因为有人有足够的耐心去做这项工作。

第二个原因是逆向工程是一种特别需要纪律的工作，即使你永远不编写完全逆向工程的驱动程序，这种纪律也值得学习。将观察与假设分开的习惯，在测试之前写下每个假设的习惯，拒绝在可能损坏硬件的地方猜测的习惯：这些习惯使普通的驱动程序工作变得更好，而不仅仅是逆向工程工作。

第三个原因是这项工作发生在一个编写软件和做实验科学之间区别变得非常模糊的前沿领域。逆向工程会话看起来更像是一本实验室笔记本，而不是编程会话。你将运行一个实验，观察结果，写下你看到的，提出解释，设计下一个实验，并慢慢建立起未知系统的图景。这种工作有自己的节奏、自己的步调和自己的一小组专业习惯，本章就是我们要学习这些的地方。

我们将首先探讨为什么逆向工程有时是必要的，以及法律和伦理的界限在哪里。然后，我们将搭建一个小型、安全、受控的实验室，使工作可以在不对生产系统或昂贵硬件造成风险的情况下进行。我们将研究用于观察设备的标准 FreeBSD 友好工具，从 `usbconfig(8)` 和 `pciconf(8)` 到 `usbdump(8)`、`dtrace(1)`、`ktr(9)` 以及 `bus_space(9)` API 本身。我们将看到如何捕获设备的初始化序列，如何比较运行结果以隔离单个比特的含义，以及如何通过实验组装寄存器映射。

我们将研究如何识别硬件喜欢使用的常见结构：环形缓冲区、命令队列、描述符环、状态和控制寄存器对、固定格式的命令包。我们将从头开始编写一个最小驱动程序，一次验证一个功能，从复位开始，慢慢添加更多行为。我们将在冒险使用真实硬件之前，在模拟器中验证每一个我们能验证的假设。我们将研究 FreeBSD 社区如何处理这项工作，在哪里可以找到先前的成果，以及如何以有助于他人的形式发布你的发现。

最后，因为这是新手最常低估的部分，我们将在安全性上花费大量时间。一些猜测可能会损坏硬件。一些随意探索未知寄存器的模式可能会擦除非易失性存储器、使电路板变砖，或者使设备进入一种只有厂商专用修复工具才能恢复的状态。本章将向你展示如何思考这种风险，如何设计限制风险的保护包装器，以及如何识别那些在没有充分证据表明安全的情况下永远不应该执行的操作。

本章的配套代码位于 `examples/part-07/ch36-reverse-engineering/`，包含一组小工具，你可以在普通的 FreeBSD 14.3 开发虚拟机上构建和运行：一个用于识别和转储目标 USB 或 PCI 设备信息的脚本，一个在指定内存区域执行安全寄存器探测的内核模块，一个可在没有真实硬件时用于验证驱动程序代码的模拟设备，以及一个 Markdown 模板，用于编写逆向工程会话最终应产生的伪数据手册。实验室中的任何内容都不会以可能损坏硬件的方式接触硬件，每个示例都可以安全地在虚拟机内运行。

到本章结束时，你将拥有一套清晰的、可重复的方法来处理无文档设备。你将知道如何搭建实验室、如何观察、如何提出假设、如何测试以及如何记录。你将了解美国和欧盟中界定这项工作的法律界限，你将了解保护你自己和硬件的专业习惯。你不会在本章结束时成为逆向工程专家，因为这种专业知识需要多年实践积累，但你将知道足够多的知识来开始工作，并且你将知道在学习过程中如何保持自身和硬件的安全。

## 读者指南：如何使用本章

本章位于本书第 7 部分"精通主题"，紧跟在异步 I/O 之后。它假设你已经阅读了异步 I/O 章节、高级调试章节和性能章节，因为这些章节中的工具和习惯与本章将要使用的工具和习惯相同。如果这些章节的内容感觉不够熟悉，快速回顾一下将会在本章中带来数倍的回报。

你不需要任何特殊的硬件来跟随本章。实际示例使用的要么是一个小型内核模块（在操作员指定的内存区域进行探测），要么是一个在软件中模拟未知硬件的模拟设备。两者都可以在普通的 FreeBSD 14.3 虚拟机上运行。如果你有一台想要研究的真实未知硬件，本章将为你提供开始所需的技术和安全习惯，但实验本身并不需要它。

本章有意写得较长，因为逆向工程是一个部分知识就很危险的领域。一个只学习了这门技术中光鲜的部分而跳过安全部分的读者，很可能会把昂贵的东西变砖。实验部分和安全部分应该与技术部分得到同等的重视。

一个合理的阅读计划是这样的。第一次阅读前三节。它们是概念基础：为什么逆向工程很重要、法律环境如何、以及如何搭建实验室。休息一下。第二次阅读第 4 到 6 节。它们涵盖了这门技术的核心技术：构建寄存器映射、识别缓冲区以及编写最小驱动程序。再休息一下。第三次阅读第 7 到 11 节。它们涵盖了扩展、验证、协作、保持安全和记录的规范工作。把实验留到周末或几个短晚上，因为如果你有时间坐下来仔细查看捕获的数据，实验的效果会好得多。

本章中的一些技术故意设计得很慢。例如，捕获设备的初始化序列可能需要运行十次相同的启动过程，并对捕获结果进行差异比较以隔离变化的比特。差异比较本身就是课程的一部分。如果你发现自己想跳到驱动程序能工作的部分，请记住，只有当你先完成了缓慢、仔细的观察工作之后，驱动程序才能正常工作。逆向工程对耐心的回报是系统编程中少有的。

本章中的几个小型内核模块被有意编写为探索脚手架，而不是生产驱动程序。它们已被注释说明。不要在生产系统上加载它们。一个内核崩溃只需重启的开发虚拟机才是进行这种工作的正确环境。

如果在阅读完本章后你有想要研究的硬件，请慢慢来。从最安全的观察工具开始。在你对写入操作应该做什么以及最坏情况是什么有清晰的书面假设之前，抵制向任何东西写入的冲动。如果某个猜测可能触发闪存擦除，就不要做这个猜测。本章将详细说明哪些操作需要特别谨慎以及为什么。

## 如何充分利用本章

本章遵循一种反映真实逆向工程会话工作流程的模式。每一节要么教授适合该工作流某一阶段的技术，要么展示该技术如何与底层学科联系。如果你将工作流作为一个整体来学习，各个技术将自然而然地各就各位。

几个习惯将帮助你吸收这些材料。

在键盘旁放一本打开的笔记本。如果可以的话，用真正的笔记本，而不是文本文件。逆向工程是一门做笔记的学科，最优秀的实践者会记录他们观察到了什么、假设了什么、测试了什么以及学到了什么。书写的行为能让你的思维慢下来以发现错误，而且纸质笔记本能抵抗事后重新整理记录的诱惑。如果纸质笔记本不实际，可以使用 Git 仓库中的 Markdown 文件，这样你可以看到你的理解是如何随时间变化的。

保持一个终端连接到你的 FreeBSD 开发机，另一个终端打开 `/usr/src/`。本章引用了几个真实的 FreeBSD 源文件，包括 `/usr/src/usr.sbin/` 和 `/usr/src/sys/dev/` 下的驱动程序和工具。阅读这些文件是课程的一部分。FreeBSD 源码本身就是逆向工程工作的成果，因为树中的许多驱动程序之所以存在，是因为有人足够仔细地观察了一个无文档设备，从而为其编写了代码。

第一次看到示例模块和脚本时，请自己输入它们。`examples/part-07/ch36-reverse-engineering/` 中的配套文件作为安全网和参考，供你想将代码与已知正确版本进行比较时使用，但第一次自己输入代码才是建立直觉的部分。整个章节都是关于为未知系统建立直觉的，这没有捷径。

请密切注意我们用来描述已知内容和猜测内容的语言。逆向工程的写作在观察、假设和已验证事实之间划出了明确的界限。观察是你看到的东西。假设是你认为观察结果的含义。已验证事实是一个经受了故意证伪尝试后仍然存活的假设。不同类型的陈述应该有不同的置信度，本章将仔细示范这种语言，以便你在自己的笔记中采用同样的精确性。

请认真对待安全建议。在这种工作中，最令人痛苦的错误是摧毁了你需要研究的那个硬件。这里描述的几种技术如果粗心使用，可能会写入闪存、永久更改设备 ID，或者使电路板进入厂商无法恢复的状态。本章将告诉你哪些模式需要最大的谨慎。请像对待化学实验室的警告标志一样对待这些建议。

最后，允许自己慢慢来。逆向工程不是短跑。一个严肃外设的完整寄存器映射是数周或数月耐心观察的结果，社区项目发布的成果通常在简洁的摘要背后隐藏着大量的细致工作。如果某个特定设备长时间抵抗你的理解，那不是失败。这就是这项工作通常的感觉。

带着这些习惯，让我们从这项工作为何必要这个问题开始。

## 1. 为什么可能需要逆向工程

当一位新驱动程序作者第一次听说逆向工程时，最直接的问题通常是"为什么这会是必要的？"如果硬件厂商想让他们的设备有用，为什么要隐藏其编程接口？如果设备很有名，为什么会有文档问题？

诚实的回答是，硬件和操作系统的世界比文档完善的标准世界更混乱。存在许多真实情况：一个可以工作的设备存在，某个操作系统已经支持它，但对于想从头开始编写驱动程序的程序员来说，不存在公开的、机器可读的、可再分发的文档。本章第一节将介绍这些情况中最常见的几种，以便你在遇到它们时能够识别，并知道每种情况需要什么样的调查。

### 1.1 无厂商支持的遗留硬件

最常见的逆向工程情况是制造商已不存在的旧设备。一块 1990 年代的小型仪器板，一张来自被收购并关闭公司的网卡，一个来自运行几年后就结束的研究项目的嵌入式控制器。硬件可以工作。硬件在出售时是有文档的。但文档是一本装订成册的纸质手册，或者是随设备附带的光盘上的 PDF，二十年后，无论是手册还是光盘都不复存在了。

在这种情况下，逆向工程是让设备重新投入使用的唯一方法。有时社区已经完成了部分工作并发布了部分笔记。有时存在 Linux 或 NetBSD 驱动程序可供阅读以获取线索；这两种情况并不等同，在法律和技术上这种区别都很重要。OpenBSD 或 NetBSD 驱动程序是 BSD 许可的，可以在保留署名的情况下直接阅读、引用和移植。Linux 驱动程序几乎总是 GPL 许可的，这意味着可以阅读它来理解，但根本不能复制到 BSD 许可的驱动程序中。即使撇开许可证不谈，直接从 Linux 移植很少能成功，因为 Linux 内核的锁原语、内存分配器和设备模型与 FreeBSD 的差异深入到每一行代码。我们将在第 12 节和第 13 节的案例研究中回到跨平台阅读的法律框架和技术陷阱。有时设备太冷门，根本不存在先前的工作，整个工作就落在想要让它工作的人身上。

FreeBSD 有支持这类设备的悠久历史。仔细阅读 `/usr/src/sys/dev/` 下的源码，会发现许多驱动程序的注释中提到"无数据手册"、"基于观察"或类似的语言。驱动程序作者完成了这项工作，社区受益。这不是边缘活动；这是 FreeBSD 长期以来支持被原始厂商抛弃的大量设备的一部分。

遗留硬件情况下的挑战往往是技术性的，而不是法律性的。硬件足够老，原始厂商要么已不存在，要么不再关心。专利已经过期。商业秘密（如果有的话）已不再受保护。风险主要是文档确实已经消失，再多的礼貌询问也无法恢复它。

### 1.2 在其他操作系统上仅有二进制驱动的设备

第二种常见情况是设备的厂商只发布一个或多个操作系统的二进制驱动程序，并拒绝发布允许其他操作系统支持该设备的文档。许多专有外设都属于这种情况：某些厂商的显卡、专业音频或视频捕获设备、只有 Windows 或 Linux 驱动的科学仪器、笔记本电脑中的指纹读取器、某些无线芯片组等等。

在这种情况下，设备仍在积极生产中。它的文档存在，但厂商要么将其视为商业秘密，要么限制签署保密协议的公司才能获取，要么根本没有看到发布它的商业理由。厂商的官方立场可能是 FreeBSD 用户不应期望获得支持，即使底层硬件在正确编写的 FreeBSD 驱动程序下完全能够工作。

这种情况下的逆向工程比较微妙，因为法律和伦理环境比遗留硬件更复杂。厂商可能持有驱动程序二进制文件的版权。运行在设备上的固件本身也可能受版权保护。驱动程序的分发和使用可能受最终用户许可协议的约束，该协议限制某些类型的分析。我们将在本节末尾回到法律问题。现在，只需注意这种情况确实存在，并且它是逆向工程重要的反复出现的原因之一。

这种情况下的技术挑战通常比遗留情况更丰富，因为有更多的材料可供使用。你有一个正在运行的驱动程序可以观察。你有一个可以工作的设备，其行为可以捕获。你可能有可以静态分析的固件映像。调查可能非常有成效，但也需要更仔细地关注工作的法律背景。

### 1.3 文档稀少或无文档的定制嵌入式系统

第三种情况在工业和嵌入式工作中越来越常见，即带有定制芯片的定制板。一家小公司为特定应用设计仪器或控制器。他们委托定制集成电路，或者用专有固件编程标准微控制器，或者用现成部件组装成从未在其他地方使用过的配置的板。他们只在自己的操作环境中支持该设备，通常是定制的 Linux 发行版或小型实时操作系统。

当该公司要求承包商将设备集成到运行 FreeBSD 的更大系统中，或者当最终用户购买硬件并想在 FreeBSD 上使用时，唯一可用的信息可能是一页机械图纸、一个简短的 README 文件和二进制固件。没有寄存器映射，没有命令集，没有设备启动方式的描述。

这种情况在某些方面是最困难的，因为设备专属于一家公司和一个客户。没有社区，因为没有其他人拥有它。没有先前的驱动程序可供阅读，因为没有其他人编写过。调查者真正独自面对硬件、捕获的流量以及从观察现有固件中可以推断出的任何内容。我们将在实验中看到如何系统地处理这种调查，我们将在第 9 节中看到如何充分记录你自己的发现，以便其他人以后可以在此基础上继续。

### 1.4 共同线索

这些情况都有一个共同的潜在属性：设备存在且可以工作，但缺少能够让人根据规范编写新驱动程序的描述。编写驱动程序的技术工艺与我们在本书其余部分练习的工艺相同。不同的是编写之前的工作形式。我们必须发现通常我们会查阅的内容。这就是本章要教授的部分。

值得注意的是逆向工程不是什么。它不是猜测。它不是随机地探查寄存器并希望发生有趣的事情。它不是试图绕过复制保护或破解加密或做任何其他会进入不同伦理领域的事情。它是一个耐心、结构化、书面记录的过程，通过观察硬件做什么和产生什么来推断其工作原理，然后编写正确与其交互的软件。

一个做得好的逆向工程项目，看起来更像是一门实验室科学，而不是黑客刻板印象。有假设、实验、测量和结论，重复数百次，直到积累了足够的结论来编写一个可工作的驱动程序。某人疯狂打字、屏幕文字揭示"秘密"的场景浪漫只存在于电影中。现实更接近于一个漫长、缓慢、系统的理解构建过程，一次一个寄存器、一次一个比特。

### 1.5 法律与伦理考量

在我们接触任何工具之前，我们必须谈谈围绕这项工作的法律环境。情况并不复杂，但它是真实存在的，忽视它的初学者可能会陷入麻烦，而本章技术部分没有任何内容可以帮助解决这些麻烦。

出于互操作性目的的逆向工程——这也是本章关心的目的——在美国和欧盟法律中都得到广泛接受。互操作性工作的目的是允许设备、格式或接口与原始厂商未提供的软件一起使用。为随 Windows 驱动程序一起提供的 USB Wi-Fi 适配器编写 FreeBSD 驱动程序是一个教科书式的互操作性案例。驱动程序让 FreeBSD 与用户已经拥有的硬件通信。它不复制厂商的驱动程序。它不以违反许可证的形式再分发厂商的固件。它不绕过任何保护版权内容的安全措施。它产生一个独立的软件，执行与厂商驱动程序相同的功能，基于对底层接口的清晰理解而编写。

在美国，相关的法律框架是版权法下的合理使用原则，有一系列法院判例认定出于互操作性目的的逆向工程是合法的合理使用。1992 年的 Sega 诉 Accolade 案、2000 年的 Sony 诉 Connectix 案以及几个类似的先例已经确立了反汇编代码以理解其接口是合理使用，前提是目的是合法的互操作性，且最终产品不包含原始受版权保护的代码。偶尔使情况复杂化的相关法规是《数字千年版权法》(DMCA)，该法禁止规避保护版权作品的"技术措施"。DMCA 包含互操作性研究的具体豁免，但这些豁免比基本的合理使用权更窄。对于普通的驱动程序工作，DMCA 很少成为问题，但如果你需要破解加密来读取固件，法律情况会变得更加复杂，咨询真正的律师是值得的。

在欧盟，相关框架是软件指令，最初是指令 91/250，2009 年更新为指令 2009/24。软件指令第 6 条明确允许为了实现与独立创建的程序互操作性而进行反编译，前提是满足几个条件：进行反编译的人有权使用该程序、互操作性所需信息不易获得、反编译仅限于程序中互操作性所需的部分。这是任何主要司法管辖区中对互操作性逆向工程最明确的法律认可之一。

在这两个体系之外，情况各不相同。许多国家在实践中遵循类似的原则。少数国家则不然。如果你在一个法律不明确或执法不可预测的司法管辖区工作，请咨询真正了解当地软件版权法律的律师。一小时的 法律咨询费用与艰难发现问题的代价相比是微不足道的。

互操作性工作与可能损害厂商或用户的工作之间存在明确的伦理界限。互操作性工作产生一个新程序，让硬件在新操作系统下完成其预期工作。它不分发受版权保护的代码。它不剥离购买产品的许可证限制。它不绕过保护除厂商在接口本身商业利益之外内容的安全措施。如果你发现自己想做任何这些事情，你就不再是在做互操作性工作，本章其余部分不适用于你的情况。

第二条伦理界限在于观察和篡改之间。观察设备的行为方式是观察。捕获设备与厂商驱动程序之间的流量是观察。读取厂商以可分发形式分发的固件是观察。根据你观察到的内容编写自己的驱动程序是合法的最终产品。编写修改后的固件来替换设备上的厂商固件，或分发此类修改后的固件，是不同类别的工作，会带来不同的法律和伦理考量。本章不会涵盖该活动。本章是关于编写一个干净、原始的驱动程序，与原始配置下的原始硬件通信。

第三个伦理考量是对你工作的诚实。记录你的信息来源。如果某个寄存器定义来自阅读厂商的开源驱动程序，请说明。如果数据包格式来自已发布的规范，请引用。如果某种行为是从你自己的捕获中推断出来的，请描述捕获内容。这种诚实部分是法律问题，因为它可以让你证明你做了净室工作，部分是社区问题，因为它可以让其他人在你的工作基础上继续，而不必重做你已经完成的部分。

### 1.6 第1节小结

本节为本章其余部分奠定了基础。我们已经看到了逆向工程必要的三种最常见情况：遗留硬件、只有厂商在其他操作系统上支持的设备，以及没有文档的定制嵌入式系统。我们已经看到，每种情况的潜在属性都是相同的：一个可以工作但其编程接口没有被描述的设备。我们还看到了美国和欧盟围绕这项工作的法律框架，以及合法的互操作性工作与本章不会涉及的其他更受法律限制的活动之间的明确区别。

下一节从为什么转向如何。我们将搭建一个进行逆向工程工作的小型实验室，并介绍你需要的工具。实验室是后续所有内容的基础，花几个小时正确设置它将在以后节省许多小时的困惑。

## 2. 准备逆向工程流程

逆向工程会话本质上是一种实验活动。你将在硬件上运行实验，捕获结果，分析它们，并设计后续实验。像任何实验活动一样，它受益于装备齐全的实验室。实验室不需要很昂贵。我们需要的大部分软件已经包含在 FreeBSD 基本系统中，其余可以从免费或低成本工具的简短列表中组装。投资主要集中在正确设置设备，以便你的捕获可靠且实验可重复。

本节将介绍完整的工具包。它从实验室正在做什么的思维模型开始，然后列举软件工具，然后讨论可选的硬件工具，然后描述如何在同一台机器上启动另一个操作系统下的厂商驱动程序以便观察其行为，最后建议一种保持实验室有条理的工作流程。

### 2.1 心理模型：实验室的用途

在列举工具之前，想象一下完成的实验室会是什么样子会有所帮助。实验室是你将进行以下活动的地方：

1. **识别**设备，记录关于它的每一个公开事实。
2. **观察**处于已知操作状态的设备。
3. **捕获**设备与现有驱动程序之间的流量。
4. **实验**通过读写来发现寄存器行为。
5. **记录**每一个观察结果。
6. **验证**每一个假设，理想情况下在冒险使用真实设备之前先使用模拟器进行验证。

因此，实验室是一个小型系统，旨在支持驱动整个技术的观察-假设-测试-记录循环。你添加的每个工具都应该以某种可识别的方式服务于该循环。看起来令人印象深刻但不服务于该循环的工具是杂物，杂物会拖慢你的速度。

第一次接触的读者有时会假设逆向工程需要非常昂贵的专业设备。其实不需要。一台普通的 FreeBSD 开发机，一个可以运行厂商驱动程序的目标系统，以及购买电缆和适配器的小预算，就能让你完成大多数项目的大部分工作。昂贵的工具确实不错，我们会提到它们，但核心工作是用你已经可以获得的软件完成的。

### 2.2 FreeBSD基本系统中的软件工具包

FreeBSD 的基本系统包含一套出色的工具集合，它们共同覆盖了驱动程序作者在实验室软件端所需的大部分内容。让我们按照逆向工程会话通常使用它们的顺序逐一介绍。

**`pciconf(8)`** 是任何 PCI 或 PCI Express 设备的起点。它是内核通过 `/dev/pci` 暴露的 `pci(4)` ioctl 的接口。以 root 身份运行时，它会列出内核已枚举的每个 PCI 设备，包括没有驱动程序附加的设备。逆向工程最重要的调用方式是：

```sh
$ sudo pciconf -lv
```

这会生成系统中每个 PCI 设备的一行摘要，如果已知数据库能识别，后面还会跟上人类可读的厂商和设备字符串。没有驱动程序的设备显示为 `noneN@pci0:...`。对于每个未知设备，这一行告诉你 PCI 总线地址、厂商标识符、设备标识符、子系统标识符、类代码和修订版本。这些标识符是你将记录的关于设备的第一批公开信息。

```sh
$ sudo pciconf -lvc
```

`-c` 标志添加设备 PCI 能力列表，如 MSI、MSI-X、电源管理、厂商特定能力和 PCI Express 链路训练数据。重复 `-c -c` 会增加某些能力类型的详细程度。对于 PCI Express 设备，这也是你将看到链路状态信息的地方，告诉你设备是否协商了它应该协商的链路宽度和速度。大量的"此设备不工作"问题竟然是链路训练问题，而这一个命令就能揭示。

```sh
$ sudo pciconf -r device addr:addr2
```

`-r` 形式直接读取 PCI 配置寄存器值，返回配置空间中某个偏移处的原始字节。这是检查设备最安全的方式，因为配置空间被设计为可以无副作用地读取。互补的 `-w` 形式写入配置寄存器；使用时要格外小心，因为某些配置寄存器确实会永久改变设备行为。

**`devinfo(8)`** 打印内核眼中的 FreeBSD 设备树。`pciconf` 显示的是总线级别，而 `devinfo` 显示完整的层次结构：设备挂在哪个总线上、父控制器是什么、分配了什么资源以及内核给它起了什么名字。详细形式 `devinfo -rv` 在早期特别有用，因为它显示分配给设备的内存和 I/O 端口的确切范围，这些范围是所有总线空间实验的活动场所。

**`devctl(8)`** 是设备控制实用程序，让你可以从设备上分离驱动程序、附加不同的驱动程序、列出事件以及在内核级别禁用特定设备。在逆向工程过程中，最有用的调用是 `devctl detach deviceN`（从设备上移除内核树内驱动程序）和 `devctl attach deviceN`（把它放回去）。有时需要分离驱动程序以便你的实验驱动程序可以声明设备，能够在不重启的情况下放回内核树内驱动程序可以节省大量时间。

**`usbconfig(8)`** 是 USB 等同于 `pciconf` 的工具。它枚举 USB 设备、转储其描述符并更改其状态。逆向工程最重要的调用方式是：

```sh
$ sudo usbconfig
$ sudo usbconfig -d ugen0.3 dump_device_desc
$ sudo usbconfig -d ugen0.3 dump_curr_config_desc
$ sudo usbconfig -d ugen0.3 dump_all_config_desc
```

第一种形式列出系统看到的每个 USB 设备及其单元号和地址。`dump_device_desc` 形式打印 USB 设备描述符：bDeviceClass、bDeviceSubClass、bDeviceProtocol、bMaxPacketSize0、idVendor、idProduct、bcdDevice、制造商和产品字符串以及配置数量。`dump_curr_config_desc` 和 `dump_all_config_desc` 形式遍历配置描述符并打印其中包含的接口、备用设置和端点。这三个命令一起为你提供了 USB 设备声称内容的几乎完整的静态图像，这个图像是每个 USB 调查的起点。

**`usbdump(8)`** 是 FreeBSD 等同于 Linux `usbmon` 的工具。它通过打开 `/dev/bpf` 并绑定到由 `usbpf` 数据包过滤模块创建的克隆 `usbusN` 接口来捕获 USB 数据包，并将捕获的数据包以与 libpcap 兼容的格式写入文件。捕获可以使用 `-w file` 保存，使用 `-r file` 重放，并使用标准 BPF 表达式语言进行过滤。对于逆向工程，工作流程通常是：

```sh
$ sudo usbdump -i usbus0 -w session1.pcap
```

这会将命名 USB 总线上的所有内容捕获到文件中。捕获后，可以使用 `usbdump -r session1.pcap` 读回该文件，在 Wireshark 中打开，或使用自定义脚本进行处理。捕获的文件格式记录每个 USB 传输，包括 SETUP 数据包、IN 和 OUT 数据、状态响应和计时信息。在不同操作期间捕获的多个会话可以相互比较以隔离负责特定行为的数据包，这种差异比较是工具包中最有效的技术之一。

**`dtrace(1)`** 是我们在前面章节中使用过的动态跟踪工具。对于逆向工程，DTrace 对于跟踪内核与驱动程序交互的点特别有用：哪个 `device_probe` 被调用于哪个设备、中断处理程序何时触发、内核树内驱动程序执行哪些 `bus_space` 操作。几个精心选择的 DTrace 探针可以详细告诉你现有驱动程序正在做什么，即使不存在其他文档。

**`ktr(9)`** 是用于细粒度事件跟踪的内核内跟踪工具。它比 DTrace 更具侵入性，因为它在构建时需要内核选项，但它为你提供每个被跟踪事件的高分辨率日志。对于逆向工程，`ktr` 在添加到你自己的实验驱动程序时最有用，这样可以精确重建寄存器访问的时序。

**`vmstat -i`** 和 **`procstat`**：让你观察设备正在产生的中断速率以及与设备交互的进程的小工具。两者都是基本系统的一部分。在实验过程中，观察你在操作设备时中断计数的变化很有用，因为中断速率的突然变化本身就是一个重要的观察。

**`hexdump(1)`**、**`xxd(1)`**、**`od(1)`**、**`bsdiff(1)`** 和 **`sdiff(1)`**：用于检查二进制数据和比较文件的普通用户空间实用程序。你会经常使用它们。在 `xxd` 中查看的捕获文件通常是模式首次变得可见的地方，因为眼睛可以在十六进制转储中发现重复结构，而自动化工具如果不被告知要查找什么就不会注意到。两个 `xxd` 输出的 `sdiff` 是找出两次捕获之间差异的最古老和最可靠的方法之一。

此列表并不详尽，但涵盖了你在任何项目的前几周会使用的工具。每个工具都在 FreeBSD 基本系统中，有 `man pciconf`、`man usbconfig` 等手册页面。

除了基本系统之外，当只有厂商二进制文件作为工件时，一小系列第三方反汇编器和反编译器就变得重要起来，通常是 Windows 驱动程序二进制文件、从设备提取的固件映像或从 PCI 卡拉取的选项 ROM。**Ghidra** 是美国国家安全局发布的开源逆向工程套件，是大多数 FreeBSD 开发者首先使用的工具，因为它免费、跨平台，并且可以舒适地将 x86、ARM 和许多嵌入式架构反编译为可读的类 C 伪代码。**radare2** 及其图形伴侣 **Cutter** 是更轻量级的开源替代品，非常适合快速检查小型固件块。**IDA Pro** 是历史悠久的商业产品；它的反编译器仍然是行业的参考实现，但其成本超出了大多数个人开发者的预算。你不需要任何这些工具就可以在仅通过捕获就能重建行为的设备上做出出色的工作。当你确实需要它们时，目标始终是有限且有记录的：识别寄存器名称、识别命令缓冲区的结构、理解厂商代码编程硬件的顺序。你不要复制厂商的代码。你要写下二进制文件所暗示的硬件接口，然后丢弃反汇编结果。这就是第 1.5 节中概述的净室纪律在实践中的转化：二进制文件是硬件事实的来源，而不是要复制的代码来源。短暂使用反汇编器，写下你学到的内容，然后根据笔记构建你的驱动程序。

### 2.3 可选硬件工具

当设备与主机的交互对纯软件捕获不可见时，硬件工具就变得有价值。USB 设备的流量通过主控制器传递，可以由 `usbdump` 捕获。PCI Express 设备的事务通过根复合体传递，基本系统中没有任何东西可以捕获它们。嵌入式板上的 SPI 外设可能与主机处理器在操作系统工具无法看到的线路上通信。对于这些情况，硬件工具就派上用场了。

**逻辑分析仪**是最通用的硬件工具。它连接到线路并随时间记录每条线路上的电压，产生可以解码为线路所承载协议的数字轨迹。对于 SPI、I2C、UART 和类似的低速总线，基本的八或十六通道逻辑分析仪就足够了。Saleae Logic 系列在专业工作中广泛使用，并得到 `sigrok`（用于分析逻辑分析仪捕获的开源套件）的良好支持。Sigrok 的 `pulseview` GUI 让你导入捕获、将其解码为 SPI 或 I2C，并逐字节浏览总线流量。

**USB 协议分析器**是一种专用硬件，位于 USB 总线上并捕获每个数据包，包括从主机不可见的总线状态事件。Beagle 和 Total Phase 分析器是这一类别中的高端工具。它们很昂贵，但对于严肃的 USB 逆向工程，它们揭示纯软件捕获根本无法看到的行为。然而，大多数业余级别的工作使用 `usbdump` 和仔细的方法论就能完成。

**PCI Express 协议分析器**更加专用，开源世界中几乎没有东西能覆盖这个领域。对于 PCI Express 工作，通常的回退方法是使用 DTrace、`ktr(9)` 和你自己插桩的驱动程序从内核侧捕获设备的行为，并使用 `pciconf -lvc` 检查配置空间状态。Teledyne 和 Keysight 等公司的真正 PCIe 分析器确实存在，但其成本使其超出大多数个人开发者的承受范围。

对于嵌入式工作，**示波器**有时对于诊断令高级工具困惑的电气问题很有用。一个因未知原因超时的驱动程序可能超时是因为设备的时钟信号退化；示波器会在软件工具无法显示时显示这一点。一个适度的 USB 连接示波器是任何进行严肃嵌入式工作的人的合理投资。

你可以不用任何这些硬件工具就完成出色的逆向工程工作。大多数消费级 USB 和 PCI 设备完全可以仅通过软件侧捕获和内核自身的内省来访问。硬件工具在调查进入真正的低级领域时变得重要：信号完整性、总线时序、嵌入式协议、那些设计于从未预期被打开的行业设备。

### 2.4 观测平台

一旦你有了工具，下一个决定是观察装置的形态。装置是你将在其上观察现有驱动程序运行的机器和操作系统的组合。有几种常见配置，各有其优势。

最简单的配置是**一台主机，两个操作系统**。同一台物理机器启动 FreeBSD（你将在其中编写新驱动程序）或另一个已经支持该设备的操作系统（你将在其中观察它）。启动到另一个操作系统观察；启动到 FreeBSD 开发。这种配置很简单，当设备永久连接到主机时效果很好。缺点是你不能在同一会话中观察和实验，所以迭代较慢。

更灵活的配置是**两台主机**：一台运行支持设备的驱动程序的操作系统，另一台运行 FreeBSD 作为你的开发环境。设备可以连接到一台主机，被观察后移动到另一台。捕获、笔记和代码通过网络在两者之间传输。当两台机器都能放在桌面上且设备可以安全移动时，这种配置效果很好。

对于 USB 设备，**在虚拟机中运行另一个操作系统的单台 FreeBSD 主机**通常是最有效的配置。设备连接到 FreeBSD 主机，`usbdump` 可以捕获其流量。虚拟机配置为通过 USB 直通接收设备，以便 VM 内的厂商驱动程序可以看到该设备。当厂商驱动程序操作设备时，主机上的 `usbdump` 记录每个数据包。这种设置在单个会话中为你提供观察和快速迭代，因为你不需要重启任何东西来在观察和实验之间切换。

对于 PCI 设备，等效的是**使用 `ppt(4)` 驱动程序的 bhyve 直通**。FreeBSD 主机将设备从内核树内驱动程序分离，将其附加到 `ppt(4)`，并将其暴露给 bhyve 客户机，厂商驱动程序可以在其中运行。客户机中的厂商驱动程序操作设备，而主机可以使用 DTrace 和其他工具观察它们之间传递的内容。Bhyve 直通是 PCI 工作的有价值技术，具有将所有观察工具保留在单台 FreeBSD 主机中的巨大优势。

对于非常专业的硬件，**专用捕获硬件**是唯一的选择。永久连接到电路板 SPI 总线的逻辑分析仪，或插入 USB 设备与其主机之间的 USB 协议分析器，为你提供纯软件工具无法提供的观察能力。权衡在于装置更复杂，捕获的数据格式需要自己的工具来分析。

### 2.5 在其他操作系统上启动厂商驱动

如果你要观察厂商驱动程序的运行，你需要一个安装了厂商支持的操作系统的工作环境。选择取决于厂商提供什么。对于 Linux 驱动程序，最近的稳定 Linux 发行版通常是正确的选择。对于 Windows 驱动程序，标准程序是在虚拟机中安装 Windows；只要设备可以直通给客户机，这种方式效果很好。对于专用嵌入式操作系统，情况更加多变。

无论你选择哪个操作系统，都应尽量少地安装额外软件。实验室应该是安静的。其他驱动程序的背景活动、自动更新、遥测或无关进程会为你的捕获增加噪声。精简的安装让设备的流量清晰突出。

对于厂商提供 Linux 驱动程序的 USB 设备，推荐的配置是：

1. 在你的 FreeBSD 主机上的虚拟机中安装最近的稳定 Linux 发行版。
2. 配置 bhyve 将 USB 设备直通给客户机。
3. 在客户机内，安装厂商驱动程序并验证设备可以工作。
4. 在主机上，通过设备通信的 USB 总线附加 `usbdump`。
5. 在捕获主机的同时在客户机中重复设备的操作。

对于厂商提供 Windows 驱动程序的 USB 设备，配置类似，只是客户机中使用 Windows 而不是 Linux。通过 bhyve 的 Windows USB 直通一直在稳步改进，对大多数消费设备是可行的。

对于 PCI 设备，类似的配置使用通过 `ppt(4)` 驱动程序的 bhyve PCI 直通。使用 `devctl detach` 将设备从其主机驱动程序分离，使用适当的内核配置将其附加到 `ppt(4)`，并使用 `bhyve -s slot,passthru,bus/slot/function` 将其暴露给 bhyve 客户机。客户机中的厂商驱动程序然后操作设备。从主机进行软件侧观察对于 PCI 比 USB 更困难，因为一旦设备被直通，配置空间和内存空间访问对主机就不可见了。回退方法是大量插桩你自己的实验驱动程序，并从你的驱动程序所做的与厂商驱动程序似乎正在做的之间的差异中学习。

### 2.6 实验记录本

与工具同等重要的是记录你所做工作的纪律。实验室笔记本（纸质的或数字的）在逆向工程中不是可选的。没有它，你会失去对已测试内容、已观察内容和已得出结论的跟踪。有了它，你构建的制品本身就是项目输出的一部分。

笔记本应该记录：

- 获取观察时的实验室确切配置：内核版本、工具版本、观察开始前设备的状态。
- 你运行的确切命令。
- 捕获的数据，或指向捕获数据存储位置的清晰指针。
- 你对观察结果的即时解释，并明确标注"观察"或"假设"。
- 你关于下一步测试什么的任何决定，以及原因。

一条好的笔记本记录读起来像科学实验方案。它应该可以被其他有相同实验室的人复现，并且应该告诉未来的读者你在当时知道什么以及你是如何知道的。当项目完成、你编写总结所学一切的伪数据手册时，笔记本是引用的来源。当后来发现某些内容是错误的时候，笔记本是你去查明错误观念何时引入以及哪些其他结论可能受到影响的地方。

在测试之前写下假设的纪律特别重要。没有这种纪律，你很容易在事后说服自己你预测了一个你实际上没有预测的结果。有了它，你可以准确判断哪些实验证实了你的理解，哪些实验让你感到惊讶。惊讶是最有价值的观察，因为它们是你的模型错误的地方，但只有当预测在结果已知之前被写下来时，它们才能清晰地显现出来。

### 2.7 示例实验室布局

作为具体示例，以下是一种在许多 FreeBSD 逆向工程项目中效果良好的配置。

主机是一台至少有 16 GB 内存的小型 FreeBSD 14.3 台式机。它运行驱动程序的开发版本、观察工具和 bhyve 管理程序。其 `/usr/src/` 目录填充了 FreeBSD 源代码，以便源代码级别浏览快速。

bhyve 内的虚拟机运行最近的 Linux 发行版。VM 安装了厂商驱动程序，并根据需要配置了 USB 或 PCI 直通。

主机上的独立 Git 仓库保存项目笔记本、捕获的 pcap 文件、实验驱动程序代码和不断增长的伪数据手册。每次提交都有日期和描述，因此项目理解的历史得以保存。

主机上的第二个终端始终运行 `tail -F /var/log/messages`，以便实验驱动程序产生的任何内核消息立即可见。

这种布局不是唯一可行的，但它是合理的起点。关键特征是：干净的 FreeBSD 开发环境、运行厂商驱动程序的方式、观察设备的方式以及随项目增长的 Git 跟踪笔记本。

### 2.8 第2节小结

我们现在有了工具包和实验室。基本系统为我们提供了 `pciconf`、`usbconfig`、`usbdump`、`devinfo`、`devctl`、`dtrace` 和 `ktr`。可选的硬件工具在纯软件捕获不够时为我们提供更深入的可见性。Bhyve 和 `ppt(4)` 为我们提供了在虚拟机内运行厂商驱动程序同时从 FreeBSD 主机观察的方式。书面实验室笔记本为我们提供了将即兴实验转化为可重复工程的纪律。

下一节将让实验室投入工作。我们将看到如何系统地捕获设备的行为，如何识别大多数硬件用于与其驱动程序通信的模式，以及如何将原始捕获转化为新兴思维模型的第一批迹象。我们将认真进行实验工作，本节建立的纪律在我们开始产生其他一切将基于的数据时变得至关重要。

## 3. 在受控环境中观察设备行为

有了实验室之后，我们现在转向观察。这是你收集其他一切将基于的原始数据的阶段。目标还不是理解设备。目标是尽可能高保真地捕获设备在一小组明确定义的情况下做什么。理解将在稍后通过分析产生。第一项工作是获得干净的、标记好的捕获。

本节按照项目通常应用它们的顺序介绍标准观察技术。我们从静态描述符开始，它为你提供设备身份的快照。我们转向初始化捕获，它显示设备在首次上电或附加时做什么。然后我们看功能捕获，它显示设备在执行每个有用操作时做什么。在整个过程中，我们强调结构化捕获的纪律：每次捕获都被命名、注明日期、标记其代表的操作，并与描述用户在捕获期间做了什么以及期望行为的简短笔记一起存储。

### 3.1 静态描述符与身份信息

关于任何设备首先要捕获的是其身份。对于 PCI 或 PCI Express 设备，这意味着记录以下命令的输出：

```sh
$ sudo pciconf -lv
```

对于特定设备，实际中相关的行看起来像这样。假设设备是内核看到的第三个未附加的 PCI 设备。运行 `pciconf -lv` 后，你可能会看到：

```text
none2@pci0:0:18:0:    class=0x028000 card=0x12341234 chip=0xabcd5678 \
    rev=0x01 hdr=0x00
    vendor     = 'ExampleCorp'
    device     = 'XYZ Wireless Adapter'
    class      = network
    subclass   = network
```

这一行记录了未来分析需要的六个事实：总线位置（`0:18:0`）、类代码（`0x028000`，FreeBSD 类代码表将其标识为无线网络控制器）、子系统标识符（`0x12341234`）、芯片标识符（`0xabcd5678`，厂商 ID 为 `0xabcd`，设备 ID 为 `0x5678`）、修订版本（`0x01`）和头类型（`0x00`，标准端点）。这些事实中的每一个都可能在以后很重要。厂商和设备 ID 是内核找到你的驱动程序的方式。子系统 ID 有时是区分共享芯片但使用不同布局的两个设备的唯一方法。类代码告诉你设备属于哪个类别。修订版本区分可能行为不同的硅片版本。请全部记录下来。

添加能力列表：

```sh
$ sudo pciconf -lvc none2@pci0:0:18:0
```

这将附加 PCI 能力列表，每行包含一个名称、一个 ID 和配置空间中的位置。现代 PCI Express 设备的典型列表包括电源管理、MSI 或 MSI-X、PCI Express、厂商特定能力以及一个或多个扩展能力。厂商特定能力特别有趣，因为它们是厂商隐藏非标准功能的地方，并且它们通常是芯片特定配置的入口点。

对于 USB 设备，等效的捕获是：

```sh
$ sudo usbconfig
$ sudo usbconfig -d ugen0.5 dump_device_desc
$ sudo usbconfig -d ugen0.5 dump_curr_config_desc
```

这三个命令一起产生设备描述符、当前配置描述符和所有配置的列表。简单 USB 设备的典型输出如下：

```text
ugen0.5: <ExampleCorp Foo Device> at usbus0
  bLength = 0x0012
  bDescriptorType = 0x0001
  bcdUSB = 0x0210
  bDeviceClass = 0x00
  bDeviceSubClass = 0x00
  bDeviceProtocol = 0x00
  bMaxPacketSize0 = 0x0040
  idVendor = 0x1234
  idProduct = 0x5678
  bcdDevice = 0x0102
  iManufacturer = 0x0001  <ExampleCorp>
  iProduct = 0x0002  <Foo Device>
  iSerialNumber = 0x0003  <ABC123>
  bNumConfigurations = 0x0001
```

每个字段都是一个问题的答案。`bcdUSB` 值告诉你设备声称实现的 USB 协议版本。`bDeviceClass`、`bDeviceSubClass` 和 `bDeviceProtocol` 是 USB 类系统，有时将设备标识为标准类的成员（HID、大容量存储、音频、视频等），有时将三者都设为零，意味着类在配置描述符内按接口确定。`idVendor` 和 `idProduct` 是唯一的数字标识符；组合起来，它们是 USB 驱动程序绑定到设备的方式。`iManufacturer`、`iProduct` 和 `iSerialNumber` 是设备字符串表的索引；`usbconfig` 为你解析它们并打印字符串。

也转储配置描述符：

```sh
$ sudo usbconfig -d ugen0.5 dump_curr_config_desc
```

这显示了接口描述符和端点描述符。对于每个接口，你将看到其编号、备用设置、类、子类和协议，以及端点列表。对于每个端点，你将看到其地址（编码了端点号和方向）、其属性（编码了传输类型：控制、等时、批量或中断）、其最大数据包大小以及如果是中断端点的轮询间隔。

这些静态信息是 USB 设备的整个编程可见身份。它确切地告诉你设备公开了什么类型的管道、在哪个方向、什么类型和什么大小。仅凭这些信息，你已经可以做出一些有根据的猜测。公开单个批量 IN 和单个批量 OUT 端点的设备可能是某种应用特定协议的传输通道。公开两个中断 IN 端点的设备可能是某种事件源。带有等时端点的设备几乎肯定在处理时间敏感数据，如音频或视频。

将这些转储保存在你的笔记本中。它们是设备的静态身份，不会在捕获之间改变。它们是你将编写的关于设备的每份报告的标题。

### 3.2 第一次捕获：初始化

一旦你有了静态身份，下一个捕获是初始化序列。这是厂商驱动程序在设备附加和设备准备就绪之间执行的操作序列。

初始化序列是你能够捕获的信息量最大的内容之一，因为它通常会触及设备的每个寄存器。驱动程序写入初始值、设置配置选项、分配缓冲区、设置中断、启用数据流并报告成功。设备公开的几乎每个寄存器都会在此序列中至少被触及一次，许多寄存器将以符合标准初始化模式的方式被触及，从而揭示其一般用途。

对于 USB 设备，初始化捕获是：

```sh
$ sudo usbdump -i usbus0 -w init.pcap
```

启动 `usbdump`，然后附加设备。设备完全初始化后，停止捕获。生成的 pcap 文件包含设备和其驱动程序之间通过总线的每个 USB 传输，从 USB 枚举序列开始（应该与静态描述符转储匹配），继续进行驱动程序用于配置设备的类特定或厂商特定控制传输，并在驱动程序将设备置于就绪状态时结束。

在 Wireshark 中打开文件以交互方式查看它，或使用 `usbdump -r init.pcap` 以文本形式打印它。Wireshark 具有特别好的 USB 解析器，会自动解码许多标准类特定传输。对于厂商特定传输的解析，你需要自己读取原始字节。

对于 PCI 设备，初始化的软件侧捕获更困难，因为一旦不使用直通，配置空间和内存空间写入从设备外部不可见。通常的技术是插桩你自己的实验驱动程序，或者如果存在内核树内驱动程序，则向其副本添加跟踪。我们将在第 4 节讨论寄存器映射时回到这个主题。目前，PCI 设备的"首次捕获"等效的是以下命令的输出：

```sh
$ sudo devinfo -rv
```

限制到你的设备。这向你显示内核为设备分配的资源：内存范围、I/O 端口范围、中断分配。这些资源告诉你将要探索的活动场所的范围。它们不会告诉你活动场所内发生了什么，但它们是后续一切内容的边界条件。

### 3.3 功能性捕获

一旦你有了初始化，下一个捕获是功能性的。对于设备可以做的每件事，你进行单独的捕获。每次捕获应尽可能干净地隔离一个操作。

对于网络设备，你可能需要对"发送一个 ping"、"接收一个主动流量数据包"、"设置 MAC 地址"、"更改信道"进行单独捕获。对于传感器，你可能需要对"读取一次"、"设置采样率"、"启用连续模式"、"校准"进行单独捕获。对于打印机，你可能需要捕获"打印一页纯文本"、"打印一张图像"、"查询状态"。

隔离操作的纪律不是可选的。如果你的捕获文件包含一百个不同的操作，理清哪些数据包对应哪个操作将是不可能的。如果你的捕获文件恰好包含一个操作，其中的数据包就是该操作的数据包，你的工作就容易得多。

每次捕获应包括：

1. 正在捕获的确切操作，用简单的英语描述。
2. 触发操作的确切用户动作。
3. 用户开始操作的确切时刻，记录为文件名中的时间戳或附注中的时间戳。
4. 设备的预期行为。
5. 观察到的实际行为。

附注是必不可少的。六个月后，你不会记得哪个捕获对应哪个操作，仅靠文件名并不总是足够的。

一个在许多项目中效果良好的示例命名方案是：

```text
init-001-attach-cold.pcap
init-002-attach-hot.pcap
op-001-set-mac-address-aa-bb-cc-dd-ee-ff.pcap
op-002-send-icmp-echo-request.pcap
op-003-receive-broadcast-arp.pcap
err-001-attach-with-no-power.pcap
```

`init-`、`op-` 和 `err-` 前缀按目的对捕获进行分组。编号后缀是唯一的。文件名中的英文描述足以让你在不打开文件的情况下找到特定捕获。附注作为 `.txt` 或 `.md` 文件与文件相邻，具有相同的基本名称。

### 3.4 差异比较：隔离有意义的位

逆向工程中最重要的是差异比较技术。类似操作的两次捕获可能大部分相同，只有少数差异对应于所做操作的差异。这些差异是重要的部分，因为它们是最可能显现含义的部分。

假设你有"设置信道为 1"和"设置信道为 6"的捕获。视觉上，两次捕获看起来几乎相同。它们将以相同的设置开始，执行相同的预备操作，并以相同的包装结束。然而，在中间某处，两次捕获之间会有少量字节不同。这些字节几乎肯定与信道号有关。通过仔细比较它们，你可以推断出：哪个传输携带信道值、该值位于传输中的什么位置、使用什么数字编码（直接数值、表索引、位字段）以及该值是以小端还是大端发送。

差异比较技术在两次捕获仅在一个变量上不同时效果最好。如果你比较"设置信道为 1"和"设置信道为 6"，且信道差异是捕获之间唯一的差异，差异比较是干净的。如果你比较"设置信道为 1，模式 A"和"设置信道为 6，模式 B"，你有两个变量在变化，分析会更困难。让捕获一次只改变一个变量。

对于文本格式捕获，`sdiff` 是最简单的工具。对于二进制捕获，`bsdiff` 生成紧凑的补丁，可以检查以查看确切哪些字节发生了变化。对于 pcap 文件，`tshark -r file.pcap -T fields -e ...` 提取特定字段，然后使用 `diff`，给你一种可编程的方式来比较捕获的特定方面。

一种更复杂的技术是捕获同一操作的多个实例并进行比较。名义上相同操作的捕获之间的差异揭示了哪些字节实际上是常量（协议的部分），哪些字节在操作没有变化的情况下在运行之间变化（序列号、时间戳、随机值）。常量字节是你想要推断含义的那些；变量字节是你目前可以忽略含义的那些。

保存每次捕获。存储很便宜，差异比较技术随着捕获越多而越有用。一个有五十次操作捕获的项目可以回答比只有一次捕获的项目多得多的问题，即使两个项目都"捕获了该操作"。

### 3.5 Wireshark与USB解析器

特别是对于 USB 工作，Wireshark 是不可或缺的工具。Wireshark 将 USB 数据包流解析为结构化树状视图，比原始字节更容易阅读，并且它有显示过滤器让你隔离一个设备、一个端点或一个方向的流量。

最有用的过滤器是：

- `usb.device_address == 5` 将视图限制为总线上的一个特定设备。
- `usb.endpoint_address == 0x81` 将视图限制为一个特定端点和方向（此处为 IN 端点 1）。
- `usb.transfer_type == 2` 将视图限制为批量传输（1 = 等时，2 = 批量，3 = 中断）。
- `usb.bRequest == 0xa0` 将视图限制为特定控制请求，在逆向工程厂商特定控制传输时很有用。

这些过滤器的组合让你准确隔离你关心的捕获部分。Wireshark 中的"统计"菜单还提供有用的聚合视图，例如看到的每个端点列表和每个端点上的数据包数量。对于 USB，"端点"视图通常是你在打开新捕获后首先检查的内容。

如果你有一个 Wireshark 解析为类特定内容的捕获（例如，解码为 SCSI 命令的 USB 大容量存储捕获），解析器实际上已经为你完成了最困难的工作。如果你有一个 Wireshark 仅解析为原始批量传输的捕获，你需要自己读取字节。

### 3.6 需要识别的观测模式

即使在你理解特定设备之前，某些模式在几乎每个设备的协议中都会重复出现，学会识别它们可以加速每个项目。在查看捕获时注意这些模式。

**重复写入后跟单次读取。** 这是经典的"写入命令，然后读取结果"模式。重复写入通常是在设置请求：命令代码、参数、长度。读取是在获取响应。许多设备对任何返回数据的操作使用这种形式。

**事件前后变化的状态标志。** 某个状态寄存器中的一位在设备完成工作时被设置，或在开始工作时被清除，这是设备传达进度最常见的方式之一。寻找那些随用户触发的操作可靠变化的位。

**对递增地址的写入序列，都是四的倍数。** 这通常是按顺序写入的寄存器块。该操作是复位后跟配置：清除所有寄存器，然后将它们设置为新值，然后通过写入末尾的"go"寄存器来触发操作。

**从同一地址的相同读取直到值改变。** 这是轮询。驱动程序正在等待设备做某事，并重复检查状态寄存器。被轮询的地址几乎肯定是状态寄存器。结束轮询的值告诉你该寄存器中的哪一位是"就绪"指示器。

**相同长度的批量 OUT 后跟批量 IN。** USB 批量端点上命令-响应的常见模式是在 OUT 端点上发送固定大小的命令，然后从 IN 端点读取固定大小的响应。两个端点一起作为请求-响应通道工作。

**时间戳线性增加的周期性中断 IN 数据包。** 这是心跳或周期性状态模式。设备以固定速率报告其状态，无论主机做什么。数据包通常包含带有状态位和计数器的小型固定结构。

**长序列写入后没有可观察的响应。** 这通常是固件下载。设备有一个可写代码区域，驱动程序正在向其中加载新指令。此类捕获与其他操作相比通常非常大，并且它们通常以标识固件映像的固定大小头部开始。

这些模式并不详尽，但它们如此频繁地出现，以至于一眼识别它们可以节省大量时间。使用新捕获文件的第一个小时几乎总是花在识别捕获展示了这些模式中的哪些。

### 3.7 第3节小结

我们建立了一组捕获。每次捕获都被命名、注明日期、标记其代表的操作，并附有描述用户动作和预期设备行为的简短笔记。我们使用 `pciconf` 和 `usbconfig` 记录了设备的静态身份。我们使用 `usbdump` 捕获了其初始化和功能操作。我们了解到类似操作的两次捕获之间的差异是我们提取比特级含义的最佳工具。我们还学习了几乎在每个设备协议中都会出现的重复模式。

下一节开始工作的主动阶段。我们不再只是观察现有驱动程序，而是以受控方式开始自己探测设备，以发现其寄存器的含义。我们在本节进行的捕获是下一节实验将与之对比的数据。有了捕获在手，我们知道"正常行为"是什么样的，我们可以比较当我们发出自己的写入时发生什么与厂商驱动程序发出等效写入时发生什么。这种比较是寄存器映射构建的核心。

## 4. 通过实验创建硬件寄存器映射

寄存器映射是一旦完成就会使所有先前工作变得不必要的文档。它列出了设备公开的每个地址，说明该地址的内容，定义每个位的含义，并描述读取或写入的任何副作用。有了完整的寄存器映射，编写驱动程序就是一个直接的翻译练习。没有它，驱动程序根本无法编写。寄存器映射是本章其余部分在许多方面旨在产生的制品。

在没有文档的情况下，寄存器映射必须通过实验构建。你将写入地址，观察发生什么，读回地址，观察返回什么，一次更改一位，寻找行为变化，并慢慢积累关于每个地址的已验证事实。这项工作需要耐心和渐进，许多技术简单到可以用一段话来描述，但安全地执行它们并仔细记录结果的纪律是将成功项目与毁坏设备的会话区分开来的关键。

本节介绍这些技术。第 10 节将回到安全方面并详细说明你不能做什么。请一起阅读这两节；本节中的技术只有在内化了第 10 节警告的人手中才是安全的。

### 4.1 映射地址空间

在你可以探测地址之前，你必须知道存在哪些地址。对于 PCI 设备，设备的 BAR（基址寄存器）声明设备响应的内存区域和 I/O 端口范围。内核已经枚举了这些，并通过驱动程序 `attach` 例程中的 `bus_alloc_resource_any(9)` 调用使其可用。查看它们运行情况的最简单方法是从内核读回它们：

```sh
$ sudo devinfo -rv
```

限制到你的设备，此命令列出内核已分配的资源。典型的 PCI 设备可能输出如下：

```text
none2@pci0:0:18:0:
  pnpinfo vendor=0xabcd device=0x5678 ...
  Memory range:
    0xf7c00000-0xf7c0ffff (BAR 0, 64K, prefetch=no)
    0xf7800000-0xf7bfffff (BAR 2, 4M, prefetch=yes)
  Interrupt:
    irq 19
```

两个内存区域和一个中断。64K 区域最可能是寄存器块，因为寄存器块通常很小。4M 区域足够大，可以是帧缓冲区、描述符环或内存映射数据区域，但不可能是寄存器空间。这些是基于大小的有根据的猜测，尚未验证的事实。将它们作为假设记录下来。

同样的基于大小的直觉在许多情况下都有效。寄存器区域很少大于一兆字节。数据缓冲区或队列很少小于几千字节。恰好是 16 KB 或 64 KB 且在 2 的幂边界上的区域以正确的方式令人怀疑：它看起来像寄存器空间。几兆字节且可预取的区域更有可能是数据区域。

对于 USB 设备，地址空间映射的等效物是端点枚举。每个端点是一个"通道"，你可以从中读取或向其写入。端点地址、大小和类型已在前一节中由 `usbconfig dump_curr_config_desc` 捕获。从端点列表中，你已经知道设备公开了多少个通道、什么类型以及在哪些方向。USB 上没有内存映射寄存器空间的等效物；一切都是通过端点完成的，包括任何"寄存器"读取和写入（它们显示为控制传输、厂商特定请求或批量传输内的数据）。

### 4.2 先读原则

安全寄存器探索最重要的规则是在写入之前先读取。从未知寄存器的读取几乎总是无害的。硬件返回它认为该地址处的值，最多只有清除某些特定类型寄存器中某些特定状态标志的副作用。另一方面，对未知寄存器的写入可以做任何事情：触发复位、启动操作、更改配置位、写入闪存，或者在最坏的情况下使设备进入不易恢复的状态。

原则很简单：在你有证据表明写入是安全的之前，不要对写入做任何假设。证据可以来自你在第 3 节进行的捕获（厂商驱动程序执行的写入大概是设备期望的写入之一）、来自类似设备上的类似寄存器、来自已发布的头文件或相关驱动程序，或者来自你自己对寄存器在读取下的行为分析。

在进行任何写入之前，穷尽地读取整个寄存器区域。保存这些值。十分钟后再次读取。进行比较。读取之间值发生变化的寄存器是有趣的：它们要么是反映某种实时状态的状态寄存器，要么是自行递增的计数器，要么是连接到外部输入的寄存器。值稳定的寄存器要么是配置寄存器（其值在写入之前保持稳定），要么是恰好没有在观察窗口期间发生变化的数据寄存器。

这种探索的最简单工具是一个小型内核模块，它接受一个内存范围并转储它，配套代码在 `examples/part-07/ch36-reverse-engineering/lab03-register-map/` 中正好包含这样一个模块。该模块作为你指向的总线的子设备附加，将内存范围分配为资源，并公开一个 sysctl，当读取时会转储范围内的每个字。在不同时间多次读取可以产生哪些字是稳定的、哪些在变化的图景。

### 4.3 探测模块骨架

为具体起见，以下是安全探测模块的形状。完整版本在配套文件中；以下是你应该从前几章中识别出的基本结构。

```c
struct regprobe_softc {
    device_t          sc_dev;
    struct mtx        sc_mtx;
    struct resource  *sc_mem;
    int               sc_rid;
    bus_size_t        sc_size;
};

static int
regprobe_probe(device_t dev)
{
    /* Match nothing automatically; only attach when explicitly
     * told to. The user adds the device by hand with devctl(8) so
     * that the wrong device cannot be probed by accident. */
    return (BUS_PROBE_NOWILDCARD);
}

static int
regprobe_attach(device_t dev)
{
    struct regprobe_softc *sc = device_get_softc(dev);

    sc->sc_dev = dev;
    sc->sc_rid = 0;
    mtx_init(&sc->sc_mtx, "regprobe", NULL, MTX_DEF);

    sc->sc_mem = bus_alloc_resource_any(dev, SYS_RES_MEMORY,
        &sc->sc_rid, RF_ACTIVE);
    if (sc->sc_mem == NULL) {
        device_printf(dev, "could not allocate memory resource\n");
        return (ENXIO);
    }
    sc->sc_size = rman_get_size(sc->sc_mem);
    device_printf(dev, "mapped %ju bytes\n",
        (uintmax_t)sc->sc_size);
    return (0);
}
```

`probe` 函数返回 `BUS_PROBE_NOWILDCARD` 是保护性措施：此驱动程序不会附加到任何设备，除非操作员明确指向它。这可以防止探测模块意外附加到不该调查的设备。

从 sysctl 处理程序调用的读取函数如下：

```c
static int
regprobe_dump_sysctl(SYSCTL_HANDLER_ARGS)
{
    struct regprobe_softc *sc = arg1;
    char buf[16 * 1024];
    char *p = buf;
    bus_size_t off;
    uint32_t val;
    int error;

    if (sc->sc_size > sizeof(buf) - 64)
        return (E2BIG);

    mtx_lock(&sc->sc_mtx);
    for (off = 0; off < sc->sc_size; off += 4) {
        val = bus_read_4(sc->sc_mem, off);
        p += snprintf(p, sizeof(buf) - (p - buf),
            "%04jx: 0x%08x\n", (uintmax_t)off, val);
    }
    mtx_unlock(&sc->sc_mtx);

    error = sysctl_handle_string(oidp, buf, p - buf, req);
    return (error);
}
```

`bus_read_4` 是 `bus_space_read_4` 的包装器，使用资源作为标签和句柄，它是从内存映射区域读取四个字节的最简单方法。注意我们以四字节为单位、在四字节对齐的偏移处读取。大多数现代设备期望寄存器访问是对齐的且具有自然大小。未对齐的访问或错误大小的访问可能产生垃圾值，或在某些硬件上导致表现为内核崩溃的总线错误。当设备可能使用 16 位或 8 位寄存器时，分别使用 `bus_read_2` 或 `bus_read_1`；如有疑问，从 4 字节读取开始，因为这是现代外设最常见的正确选择。

此模块根本不进行写入。它纯粹是一个观察工具。在未知设备上加载它可以给你设备内存区域的快照而不改变任何内容。多次加载或多次调用转储 sysctl 可以给你一系列快照，其差异揭示哪些地址是动态的。

### 4.4 从行为推断寄存器用途

在你反复读取地址空间后，你开始注意到模式。每种模式都暗示了寄存器的用途。

**稳定的值**通常是配置寄存器、标识符寄存器或尚未更改的默认值。偏移量为 0 且始终读取相同常量值的寄存器通常是芯片标识符寄存器，有时称为"版本寄存器"或"魔术数寄存器"。许多设备专门使用这样的寄存器，以便驱动程序可以确认芯片是否符合预期。

**缓慢变化的值**通常是计数器：接收的字节、发送的数据包、检测到的错误、已用时间。缓慢递增的寄存器是计数器的标志，两次读取之间的差值告诉你计数速率。如果计数器每秒恰好增加一，你可能找到了心跳或时间戳寄存器。如果它在流量流动时每秒增加几千而在流量停止时保持平坦，你找到了实时数据路径的数据包或字节计数器。

**快速变化并循环的值**通常是生产端环形缓冲区写指针。将数据生成到环形缓冲区的设备通常会公开一个保存当前写入位置的寄存器。当数据流动时，写指针快速递增，并在达到缓冲区大小时循环。由驱动程序写入的匹配读指针告诉设备消费者在哪里。

**随操作切换的位**是状态位。当操作开始时变为 1 而完成时变为 0 的位是"忙"位。当事件发生时变为 1 并保持为 1 直到被清除的位是"待处理中断"位。当设备健康时为 0 而发生错误时为 1 的位是错误指示器。

**读回值与写入值不同的寄存器**要么被掩码（某些位未实现，无论写入什么都读为零），要么是一次性的（写入有副作用但寄存器本身不存储写入的值），要么是自清除的（写入设置的位在操作完成时由设备自动清除）。这三种变体都很常见，读回模式可以区分它们。

这些观察建立了寄存器映射的图景。每个已验证的观察都进入笔记本，包括地址、观察到的模式和提议的用途。随着时间的推移，笔记本条目固化为第 11 节将讨论的伪数据手册。

### 4.5 将捕获的写入与假设进行比较

第 3 节的捕获是观察与假设之间的桥梁。一旦你提出某个寄存器是（例如）信道选择寄存器，你可以通过比较你见过的厂商驱动程序写入该寄存器的内容与你所知道的用户执行的操作，来确认或反驳该假设。

假设你的假设是偏移 `0x40` 的寄存器是信道选择寄存器，该寄存器的值将信道号直接编码为一个小整数。你有用户选择信道 6 的捕获，以及用户选择信道 11 的捕获。如果你的假设正确，信道 6 的捕获序列应该包含向 `0x40` 等效位置写入 `0x06`，信道 11 的捕获序列应该包含向同一位置写入 `0x0b`。如果你找到了这些写入，假设得到支持。如果捕获显示写入不同的值，或根本没有向该偏移写入，假设就是错误的。

这是使工作科学化的证伪步骤。经受一次证伪尝试的假设比仅被非正式提出的十个假设更有价值。

对于 USB 设备，同样的比较发生在捕获的控制传输、批量数据包或中断数据包与假设的含义之间。如果你相信带有 `bRequest = 0xa0` 且 `wValue` 在低字节的厂商特定控制传输是"设置信道"命令，你可以通过检查信道 1、信道 6 和信道 11 选择的捕获来验证它，确认每个捕获中都恰好出现该传输，且 `wValue` 符合预期。

### 4.6 一次一个假设的纪律

在压力下很容易忘记的一个纪律是一次只测试一个假设。当多个假设悬而未决时，诱惑是设计一个可以同时测试所有假设的实验。实验运行，设备产生一些输出，输出与某个假设一致而与另一个不一致。现在你对假设 A 学到了一些东西。但因为你也改变了假设 B 的条件，你也失去了从 B 的角度解释发生了什么的能力。同样的实验需要再次运行，这次独立改变 B。

正确的纪律是：改变一件事，观察结果，得出关于该变量的结论，然后进行下一个实验。每个实验更慢但整体更快，因为每个实验留下一个清晰、孤立的事实，而不是一堆部分信息的纠缠。

牛顿的实验室笔记本充满了单变量实验。每个成功的逆向工程项目的笔记本也是如此。多变量实验属于后期阶段，当你对每个变量理解得足够好以至于可以预测它们的交互时；在发现阶段，它们主要产生混乱。

### 4.7 常见寄存器布局模式

在许多设备中，少数寄存器布局模式经常重复出现，值得提前了解。识别它们可以节省数周的工作。

**状态 / 控制 / 数据**：一组小型寄存器，其中一个寄存器保存操作的当前状态（状态），一个寄存器接受命令或配置更改（控制），一个或多个寄存器保存正在移入或移出设备的数据（数据）。许多简单外设恰好使用这种形状，更多外设将其用作设备内多个功能单元之一的形状。

**中断启用 / 中断状态 / 中断确认**：三个共同实现中断机制的寄存器。启用寄存器控制哪些条件引起中断。状态寄存器报告哪些条件当前正在引发中断。确认寄存器在中断条件被处理后清除它们。这种模式如此常见，在探索未知设备时，你应该显式寻找它：寄存器映射顶部附近三个相邻寄存器，如果能找到任何文档，带有 `INT_ENABLE`、`INT_STATUS`、`INT_CLEAR` 这样的标识名称。

**生产者指针 / 消费者指针**：两个共同实现环形缓冲区的寄存器。生产者指针由生产数据的一方更新（有时是设备，有时是驱动程序），指示下一个空闲槽位。消费者指针由消费数据的一方更新，指示下一个要处理的槽位。两者之间的差值告诉你当前有多少槽位已满。这种模式是几乎所有现代硬件高速 I/O 接口的基础，包括大多数网络控制器、大多数 USB 控制器和大多数磁盘控制器。

**窗口 / 索引 / 数据**：一种间接访问模式，设备公开少量寄存器但使用它们访问更大的内部地址空间。窗口寄存器选择当前可见的"页面"，索引寄存器选择页面内的哪个寄存器，数据寄存器读取或写入选定的寄存器。这种模式在旧设备和内部寄存器空间非常大的设备中常见。

**能力指针**：一个寄存器或一小组寄存器，指向能力描述符链的起始。PCI Express 定义了这种模式的标准形式；许多其他设备使用临时变体。能力列表让驱动程序发现芯片支持哪些可选功能，而无需查阅外部文档。

在探索未知设备时，扫描寄存器空间寻找这些重复模式是首先要做的事情之一。使用状态 / 控制 / 数据布局的设备比不使用的设备更容易驱动，早期识别布局可以节省大量时间。

### 4.8 暂时不要做的事情

在继续之前，快速提醒一下克制。当你构建了一个基本可用的寄存器映射时，诱惑是尝试写入一些东西看看会发生什么。在寄存器探索中抵制这种诱惑。只有当你有特定的假设要测试、假设预测特定结果、你有验证结果发生的方法、并且你考虑了意外解释的最坏合理结果时才写入。第 10 节将详细说明的纪律建立在这种克制之上，寄存器映射阶段正是首先应用它的地方。

### 4.9 第4节小结

我们学习了如何映射设备的地址空间、如何安全读取它、如何解释稳定值、缓慢变化值和快速变化值的模式，以及如何将捕获的写入与假设比较以确认或反驳提议的寄存器含义。我们学习了硬件喜欢使用的重复布局。我们为第 10 节的安全处理奠定了基础，这是负责任地完成所有这些工作的前提。

下一节上升一级抽象。一旦我们识别了各个寄存器，我们看看它们如何组成更大的结构：数据缓冲区、命令队列、描述符环。这些更高级的结构是设备实际移动数据的方式，识别它们是将寄存器知识转化为工作驱动程序基础的下一步。

## 5. 识别数据缓冲区和命令接口

寄存器是你与设备控制面交互的方式。数据缓冲区是设备移动真实信息的方式。你在上一节构建的寄存器映射是必要的但不充分。要完成图景，你还需要了解设备如何处理数据：其缓冲区在哪里、边界如何定义、数据在其中是什么形状以及生产者和消费者如何协调。

本节介绍硬件使用的常见缓冲区和命令结构，以及关于如何通过观察识别它们以及如何在驱动程序端设置匹配的 `bus_dma(9)` 映射的 FreeBSD 特定说明。

### 5.1 三大类型：线性、环形和描述符

现代硬件中的几乎每个数据缓冲区都属于三种形状之一。

**线性缓冲区**是驱动程序为一次操作交给设备的简单连续内存块。驱动程序用数据填充它，设备消费它，驱动程序在需要时将其收回。线性缓冲区是最简单的形状，它们出现在命令-响应模式中，每次操作有自己的专用缓冲区。它们很容易识别，因为它们在捕获中显示为单个字节块，其大小与预期的操作大小匹配。

**环形缓冲区**是带有生产者指针和消费者指针的循环缓冲区。生产者写入进入生产者指针指示的槽位，然后生产者指针前进。消费者读取来自消费者指针指示的槽位，然后消费者指针前进。当指针到达缓冲区末尾时都会循环。当生产者追上消费者时缓冲区已满；当两个指针在另一种意义上相等时缓冲区为空。环形缓冲区在高速网络和存储中无处不在，因为它们让生产者和消费者以不同的速率运行而无需协调握手。

**描述符环**是一种特殊的环形缓冲区，其中每个槽位不保存数据本身，而是保存指向数据的小型固定大小描述符。描述符通常包含内存地址（数据所在位置）、长度、状态字段（处理后由设备填写）和一些控制标志。描述符环让设备可以进行分散-聚集 DMA、接受可变大小的缓冲区并向驱动程序报告每个缓冲区的状态。它们比环形缓冲区稍复杂但灵活得多，是高性能网络和存储控制器中的主导模式。

识别特定设备使用哪种形状是第一项工作。迹象通常很明显。如果捕获显示固定大小的块一次移动一个且没有元数据，那可能是线性缓冲区安排。如果捕获显示数据连续流动且没有明显的"下一操作"边界，并且有一个看起来像生产者指针的寄存器，那可能是环形缓冲区。如果设备文档、先前的驱动程序源码或相关设备数据手册暗示"描述符"，那几乎肯定是描述符环。

### 5.2 区分不同缓冲区

当你识别出存在缓冲区时，下一个问题是：它有多大，内部如何组织？第 3 节的捕获包含数据；问题是数据有什么形状。

几个观察可以帮助。

**底层内存区域的大小。** 如果 PCI BAR 正好是 4 KB，且 sysctl 显示设备有一个设置为 64 的"环形大小"寄存器，自然结论是环有 64 个 64 字节的条目。许多设备对环和条目大小都使用 2 的幂大小，两者的乘积应该等于 BAR 大小。

**数据中的周期性结构。** 在 `xxd` 或 Wireshark 中查看的捕获有时显示明显的周期性结构：每 32 字节有一个小型固定模式，每 64 字节有一个计数器，每 16 字节有一个以可预测方式变化的状态字节。大小为 N 的周期性结构强烈建议缓冲区组织为大小为 N 的记录序列。

**指针值的对齐。** 如果你看到设备或驱动程序使用低比特始终为零的"地址"，则这些地址对齐到 2 的幂边界。如果对齐是 16 字节，条目可能是 16 字节或更大。如果对齐是一兆字节，条目非常大或设备由于其他原因施加了粗粒度对齐要求。

**头部和尾部。** 许多记录格式包括标识记录类型的头部、有效载荷和包含校验和或长度的尾部。头部通常是可以在视觉上识别的魔术值。

**跨平台线索。** 如果设备在开源树中有 Linux 驱动程序，该驱动程序的代码可能已经定义了记录结构。阅读另一个驱动程序来学习记录格式是可用的最高效研究技术之一，即使你打算以净室风格从头开始编写 FreeBSD 驱动程序。有两个注意事项值得在此内联说明，以免章节的节奏让它们溜过去。第一是法律方面的：Linux 驱动程序几乎总是 GPL 许可的，所以阅读它们是可以的，但将其表达粘贴或近乎转录到 BSD 许可的 FreeBSD 驱动程序中是不行的。第二是技术方面的：记录布局信息在内核之间可以干净地移植，因为它描述的是设备而不是主机，但周围的代码则不然。Linux 的缓冲区分配路径、锁原语和描述符环辅助函数受其自身内核塑造，试图逐行镜像它们的 FreeBSD 驱动程序会在每一步与主机内核冲突。阅读以理解格式，然后根据你自己的笔记编写 FreeBSD 代码。第 12 节详细涵盖法律框架；第 13 节展示真实驱动程序如何做得好。

### 5.3 命令队列及其排序

对于具有命令接口的设备，典型形状是命令结构队列，设备按顺序处理每个结构，通过单独的寄存器或命令结构本身的状态字段报告状态。命令队列通常是描述符环的一种特殊形式。

排序很重要。有些设备严格按照顺序处理命令；其他设备乱序处理并通过命令结构中的标签标识完成。有些设备一次处理一个命令；其他设备并行处理多个。有些设备要求驱动程序在发布下一个命令之前等待当前命令完成；其他设备接受许多并发命令。识别设备处于哪种模式是调查的一部分，第 3 节的捕获通常足够丰富以使答案可见。

一种有用的技术是故意让设备执行在时间上重叠的操作，并观察驱动程序如何响应。如果驱动程序总是在发布下一个命令之前等待当前命令完成，设备可能使用顺序处理。如果驱动程序快速连续发布多个命令然后等待多个完成，设备可能使用并发处理。如果完成以与发布不同的顺序到达，设备正在进行乱序处理。

### 5.4 DMA与总线主控

大多数移动大量数据的现代设备使用 DMA。设备作为总线主控直接读写主机内存，而不需要 CPU 复制每个字节。从驱动程序方面来看，DMA 引入了几个你必须正确处理的约束。

缓冲区必须是物理连续的，或至少以设备可以表达的方式分散。它们必须对齐到设备要求的边界。它们必须对设备可见，在 FreeBSD 上这意味着必须通过 `bus_dma(9)` 框架设置。驱动程序和设备必须使用 `bus_dmamap_sync(9)` 调用同步它们对缓冲区的视图，因为现代处理器维护的缓存可能与设备 DMA 不一致。

`bus_dma(9)` 的完整机制在第 21 章中介绍，无需在此重复。对于逆向工程，其含义是：

1. 设备几乎肯定对其缓冲区施加对齐和大小约束。显示缓冲区总是从具有相同低位比特的地址开始的捕获正在揭示对齐方式。
2. 设备可能施加最大缓冲区大小。显示大型操作被拆分为多个较小传输的捕获正在揭示最大值。
3. 设备可能使用特定的 DMA 地址位来编码元数据。例如，64 位地址字段中的 32 位指针有时会留下高 32 位用于其他用途，设备可能将这些位解释为标志或标签。

识别这些约束是任何具有 DMA 能力的设备逆向工程工作的一部分。捕获和现有驱动程序源码（如果可用）是证据的主要来源。

### 5.5 逆向分析数据包格式

当设备的数据路径使用结构化数据包时，逆向分析数据包格式是工作最有回报的部分之一。重复出现的技术是：

**寻找永不改变的固定字节。** 这些通常是魔术数、版本字节或类别代码。标记它们并探索它们的含义。

**寻找值与操作相关的字段。** 采用一组不同值的字节，每个值恰好出现在一种操作中，几乎肯定是操作码或命令类型。

**寻找长度字段。** 一个两字节或四字节的字段，其值以某种可预测编码匹配数据包其余部分的大小，是长度字段。长度字段通常靠近数据包开头，通常是让你能正确解析可变长度数据包的字段。

**寻找序列号或会话标识符。** 在会话期间单调递增或每次操作改变一次的字段通常是序列号。它们的值很少有趣，但它们的存在和位置帮助你在寻找真正内容时忽略它们。

**寻找校验和。** 其值以你无法从任何单个字节预测的方式依赖于数据包其余部分的字段可能是校验和。标准校验和（CRC-16、CRC-32、求和）有时可以通过在候选范围上计算它们并查看哪个范围匹配来识别。如果存在校验和，当驱动程序构建自己的数据包时你需要正确计算它，因此识别算法是必要的。

这些观察中的每一个都构建伪数据手册的数据包格式部分。与第 4 节的寄存器映射结合，它们形成第 11 节将组装成适当数据手册替代品的文档。

### 5.6 识别读操作与写操作

许多设备具有不对称的数据路径：某些操作导致设备从主机内存读取，其他操作导致设备写入主机内存。识别哪个是哪个对于正确设置 DMA 方向至关重要。

最清晰的证据在捕获中。例如，USB 捕获明确区分 IN 端点（设备到主机）和 OUT 端点（主机到设备）。PCI 捕获（如果你能得到的话）通过事务类型区分内存读取和内存写入。从网络控制器到主机的批量 DMA 被记录为设备写入主机内存。从主机到网络控制器的批量 DMA 被记录为设备从主机内存读取。

如果捕获不可用，一种有用的技术是在触发操作之前将缓冲区设置为可识别的模式（例如全 `0xCC` 字节），然后检查缓冲区。如果缓冲区的模式被新数据覆盖，操作是设备到主机传输（设备写入缓冲区）。如果缓冲区的模式未改变，操作是主机到设备传输（设备仅从缓冲区读取）。这种区分实验是结果告诉你难以通过其他方式学习到的事实的最清晰示例之一。

### 5.7 第5节小结

我们学会了识别缓冲区组织的三大形状：线性缓冲区、环形缓冲区和描述符环。我们了解了如何从周期性结构、对齐和底层内存区域推断缓冲区大小和条目大小。我们学会了识别命令队列形状和排序模型。我们注意了 DMA 施加的约束以及揭示它们的证据类型。我们介绍了逆向分析数据包格式的重复技术。我们还了解了当捕获本身没有直接说明时如何识别传输方向。

下一节将这种不断增长的理解转化为代码。我们将编写实际驱动程序的第一个版本，一个只实现最小有用功能的驱动程序：通常是复位，有时是复位加上单个状态读取，有时是将设备置于可以进一步驱动状态所需的绝对最小功能。原则是，正确完成某件事的小驱动程序比不确定地完成许多事的大驱动程序是更好的起点。

## 6. 重新实现最小设备功能

观察之后是重构。有了捕获在手、寄存器假设得到确认，并且至少对缓冲区结构的粗略形状有了了解，你就可以开始编写驱动程序了。关键词是"开始"。逆向工程驱动程序的第一个版本不需要做设备能做的所有事情。它甚至不需要做设备能做的大部分事情。它需要做的是最小有用的部分，并且正确地完成这一部分。

本节介绍从小处着手的原则以及编写第一个最小驱动程序的实际步骤。这里的工作建立在本书前面每一章的基础上，因为"编写驱动程序"的机制与我们一直在练习的机制完全相同。变化的是首先实现哪些功能以及如何获得每个功能正确的信心方面的纪律。

### 6.1 始终从复位开始

你实现的第一个功能几乎应该总是复位。复位操作是其他每个操作的基础，原因有几个。

复位通常是设备最简单的操作。它通常只需要对单个寄存器的单次写入，或者三到四次写入序列，将设备置于已知状态。假设很小且易于测试。

复位是其结果最容易验证的操作。在复位之前，设备的寄存器可能保存任意值。成功复位后，它们保存可预测的默认值：零配置、无待处理中断、无活动操作、标识符寄存器读取其常量魔术值。如果你执行复位然后读取寄存器，你应该看到"刚上电"状态。

复位是其失败模式最良性的操作。如果你的复位尝试实际上没有复位设备，设备将处于未知状态，你可以通过卸载驱动程序并让内核重新探测设备来恢复。失败的复位很少损坏任何东西；它只是让你不比以前更糟。

复位也是几乎所有其他操作的先决条件。不能将设备置于已知状态的驱动程序不能依赖任何后续操作。通过首先实现复位，你构建了每个后续功能测试将依赖的基础。

### 6.2 最小驱动程序骨架

对于 PCI 设备，骨架驱动程序看起来与你在书中见过的其他每个 PCI 驱动程序非常相似。以下是基本形状：

```c
struct mydev_softc {
    device_t          sc_dev;
    struct mtx        sc_mtx;
    struct resource  *sc_mem;
    int               sc_rid;
    struct resource  *sc_irq;
    int               sc_irid;
    void             *sc_ih;
    bus_size_t        sc_size;
    /* Driver-specific state goes here as it accumulates. */
};

static int
mydev_probe(device_t dev)
{
    if (pci_get_vendor(dev) == 0xabcd &&
        pci_get_device(dev) == 0x5678) {
        device_set_desc(dev, "ExampleCorp XYZ Reverse-Engineered");
        return (BUS_PROBE_DEFAULT);
    }
    return (ENXIO);
}

static int
mydev_attach(device_t dev)
{
    struct mydev_softc *sc = device_get_softc(dev);
    int error;

    sc->sc_dev = dev;
    mtx_init(&sc->sc_mtx, "mydev", NULL, MTX_DEF);

    sc->sc_rid = PCIR_BAR(0);
    sc->sc_mem = bus_alloc_resource_any(dev, SYS_RES_MEMORY,
        &sc->sc_rid, RF_ACTIVE);
    if (sc->sc_mem == NULL)
        goto fail;
    sc->sc_size = rman_get_size(sc->sc_mem);

    sc->sc_irid = 0;
    sc->sc_irq = bus_alloc_resource_any(dev, SYS_RES_IRQ,
        &sc->sc_irid, RF_SHAREABLE | RF_ACTIVE);
    if (sc->sc_irq == NULL)
        goto fail;

    error = mydev_reset(sc);
    if (error != 0)
        goto fail;

    error = mydev_verify_id(sc);
    if (error != 0)
        goto fail;

    device_printf(dev, "attached, mapped %ju bytes\n",
        (uintmax_t)sc->sc_size);
    return (0);

fail:
    mydev_detach(dev);
    return (ENXIO);
}
```

结构应该看起来很熟悉。驱动程序为其寄存器区域分配内存资源，为其中断分配 IRQ 资源。然后执行复位和标识符检查。如果任一失败，驱动程序干净地退出。复位和标识符检查是值得编写的前两个函数。

### 6.3 实现复位

复位函数在最简单的情况下是三行代码：

```c
static int
mydev_reset(struct mydev_softc *sc)
{
    bus_write_4(sc->sc_mem, MYDEV_REG_CONTROL, MYDEV_CTRL_RESET);
    pause("mydev_reset", hz / 100);
    bus_write_4(sc->sc_mem, MYDEV_REG_CONTROL, 0);
    return (0);
}
```

我们将复位位写入控制寄存器，等待短时间让设备对其做出反应，然后清除控制寄存器让设备处于静默状态。常量 `MYDEV_REG_CONTROL` 和 `MYDEV_CTRL_RESET` 来自你在第 4 节构建的寄存器映射。

在实践中，复位很少这么简单。许多设备有特定的复位协议，涉及按特定顺序的多个寄存器写入，或者涉及轮询状态寄存器直到复位完成，或者涉及等待特定持续时间才能继续。第 3 节的捕获向你展示厂商驱动程序在附加时做什么。复制该序列是最安全的复位实现。

更现实的复位可能如下：

```c
static int
mydev_reset(struct mydev_softc *sc)
{
    int i;
    uint32_t status;

    /* Disable any pending interrupts. */
    bus_write_4(sc->sc_mem, MYDEV_REG_INT_ENABLE, 0);

    /* Trigger reset. */
    bus_write_4(sc->sc_mem, MYDEV_REG_CONTROL, MYDEV_CTRL_RESET);

    /* Wait for reset complete, with timeout. */
    for (i = 0; i < 100; i++) {
        status = bus_read_4(sc->sc_mem, MYDEV_REG_STATUS);
        if ((status & MYDEV_STATUS_RESETTING) == 0)
            break;
        pause("mydev_reset", hz / 100);
    }
    if (i == 100) {
        device_printf(sc->sc_dev, "reset timed out\n");
        return (ETIMEDOUT);
    }

    /* Clear any pending interrupt status. */
    bus_write_4(sc->sc_mem, MYDEV_REG_INT_STATUS, 0xffffffff);

    return (0);
}
```

此版本处理复位需要一些时间才能完成且设备暴露一个报告"正在复位"状态位的情况。从一开始就值得养成添加超时（带超时到期时的显式错误路径）的习惯。一个无法区分"设备今天很慢"和"设备已经失败"的驱动程序最终会挂起内核，而添加超时的代价是微不足道的。

### 6.4 验证标识符

每个逆向工程的驱动程序都应该验证附加的设备是否是驱动程序所期望的设备。验证通常是一两次寄存器读取，将值与已知常量进行比较：

```c
static int
mydev_verify_id(struct mydev_softc *sc)
{
    uint32_t id, version;

    id = bus_read_4(sc->sc_mem, MYDEV_REG_ID);
    if (id != MYDEV_ID_MAGIC) {
        device_printf(sc->sc_dev,
            "unexpected ID 0x%08x (expected 0x%08x)\n",
            id, MYDEV_ID_MAGIC);
        return (ENODEV);
    }

    version = bus_read_4(sc->sc_mem, MYDEV_REG_VERSION);
    if ((version >> 16) != MYDEV_VERSION_MAJOR) {
        device_printf(sc->sc_dev,
            "unsupported version 0x%08x\n", version);
        return (ENODEV);
    }

    device_printf(sc->sc_dev, "ID 0x%08x version 0x%08x\n",
        id, version);
    return (0);
}
```

此检查有两个目的。首先，它确认驱动程序对寄存器布局的理解是正确的：如果标识符寄存器在预期偏移处保存预期值，则布局至少与我们相信的一致。其次，它为你提供了一条快速失败路径，如果驱动程序以某种方式附加到了错误的设备，可能是因为两个设备共享一个 PCI ID 前缀，或者因为设备有多个硅片版本。

### 6.5 添加日志

日志是你在逆向工程中的眼睛。每个寄存器写入、每个值重要的寄存器读取、驱动程序启动序列中的每个里程碑：都要记录下来。内核的 `device_printf` 输出到 `dmesg`，你可以在驱动程序加载时实时查看。

日志不是稍后删除的调试辅助工具。在逆向工程项目中，日志是驱动程序的一部分，它会留在驱动程序中，直到驱动程序被充分理解。成本很小：少量 `device_printf` 调用几乎不占用 CPU 时间，只产生几行输出。收益很大：当出现问题时，你有驱动程序试图执行的操作的完整记录。

常见的模式是将寄存器访问包装在记录访问的小型内联辅助函数中：

```c
static inline uint32_t
mydev_read(struct mydev_softc *sc, bus_size_t off)
{
    uint32_t val = bus_read_4(sc->sc_mem, off);
    if (mydev_log_reads)
        device_printf(sc->sc_dev,
            "read  off=0x%04jx val=0x%08x\n",
            (uintmax_t)off, val);
    return (val);
}

static inline void
mydev_write(struct mydev_softc *sc, bus_size_t off, uint32_t val)
{
    if (mydev_log_writes)
        device_printf(sc->sc_dev,
            "write off=0x%04jx val=0x%08x\n",
            (uintmax_t)off, val);
    bus_write_4(sc->sc_mem, off, val);
}
```

`mydev_log_reads` 和 `mydev_log_writes` 标志可以是可调 sysctl，这样你可以在不重新构建驱动程序的情况下启用或禁用日志。`examples/part-07/ch36-reverse-engineering/lab03-register-map/` 中的配套代码使用的正是这种模式。

### 6.6 将日志输出与捕获进行比较

有了日志，你可以将驱动程序所做的与厂商驱动程序所做的进行比较。在观察装置中运行厂商驱动程序并捕获结果。在开发环境中运行你的驱动程序并捕获其日志输出。比较它们。

如果两个序列匹配，你的驱动程序正在做厂商驱动程序所做的事情，至少对于你已经实现的功能来说是如此。如果它们不同，差异正在告诉你一些东西。也许你的驱动程序缺少厂商驱动程序执行的写入。也许你的驱动程序写入的值与厂商驱动程序写入的值不同。也许你的驱动程序读取了厂商驱动程序没有读取的寄存器。每个差异都是一个需要调查的假设。

这种比较技术是你拥有的最有效的技术之一。它将捕获从被动证据转变为主动规范：驱动程序应该产生与捕获显示的相同操作序列。驱动程序偏离的地方，偏离就是错误或缺失部分，捕获告诉你添加什么。

### 6.7 编写模块样板代码

最小驱动程序还需要标准模块样板代码。在本书的这个阶段，样板代码应该已经很熟悉了：

```c
static device_method_t mydev_methods[] = {
    DEVMETHOD(device_probe,   mydev_probe),
    DEVMETHOD(device_attach,  mydev_attach),
    DEVMETHOD(device_detach,  mydev_detach),
    DEVMETHOD_END
};

static driver_t mydev_driver = {
    "mydev",
    mydev_methods,
    sizeof(struct mydev_softc),
};

DRIVER_MODULE(mydev, pci, mydev_driver, NULL, NULL);
MODULE_VERSION(mydev, 1);
```

`DRIVER_MODULE` 中的 `pci` 父级使内核在探测期间向此驱动程序提供每个 PCI 设备，驱动程序的 `probe` 方法决定是否声明每个设备。`MODULE_VERSION` 宏是对任何使用 `kldstat -v` 检查已加载模块的人的一种礼貌；如果其他模块想声明依赖于你的模块，它也是必需的。

Makefile 是标准的 FreeBSD 内核模块 Makefile：

```makefile
KMOD=   mydev
SRCS=   mydev.c device_if.h bus_if.h pci_if.h

SYSDIR?= /usr/src/sys

.include <bsd.kmod.mk>
```

`SRCS` 中列出的 `device_if.h`、`bus_if.h` 和 `pci_if.h` 是生成的头文件。构建系统按需创建它们，列出它们告诉 `make` 在编译 `mydev.c` 之前这样做。如果驱动程序变得更复杂并拆分为多个源文件，请分别列出每个源文件。

### 6.8 第一次成功加载

当你第一次加载最小驱动程序时，成功标准是适度的。驱动程序应该：

1. 被 `kldload` 成功加载。
2. 通过 `probe` 识别设备。
3. 通过 `bus_alloc_resource_any` 分配其内存和 IRQ 资源。
4. 复位设备。
5. 验证标识符。
6. 在 `dmesg` 中报告成功并无错误地保持加载状态。
7. 被 `kldunload` 成功卸载而不会崩溃。

满足这些标准的驱动程序已经完成了有用的工作。它已证明寄存器布局至少部分正确，复位序列有效，标识符寄存器在你认为的位置。与你开始的地方相比，这是实质性的进展。

如果任何步骤失败，失败是有信息价值的。`bus_alloc_resource_any` 失败通常意味着 BAR 布局不是你所期望的。ID 验证失败通常意味着寄存器偏移错误，或设备的标识符值不是你所假设的。卸载时崩溃通常意味着资源没有正确释放；这是分离路径中的错误，需要在进行进一步工作之前修复。

### 6.9 干净卸载

分离路径至少与附加路径同等重要。可以加载但不能干净卸载的驱动程序是每次想测试更改时都需要重启的驱动程序，重启很快就会成为你工作的瓶颈。尽早花时间在干净的分离路径上会在开发速度上产生巨大回报。

```c
static int
mydev_detach(device_t dev)
{
    struct mydev_softc *sc = device_get_softc(dev);

    if (sc->sc_ih != NULL)
        bus_teardown_intr(dev, sc->sc_irq, sc->sc_ih);
    if (sc->sc_irq != NULL)
        bus_release_resource(dev, SYS_RES_IRQ,
            sc->sc_irid, sc->sc_irq);
    if (sc->sc_mem != NULL)
        bus_release_resource(dev, SYS_RES_MEMORY,
            sc->sc_rid, sc->sc_mem);
    mtx_destroy(&sc->sc_mtx);
    return (0);
}
```

分配的每个资源都必须释放。注册的每个中断处理器都必须拆除。每个互斥锁都必须销毁。顺序是分配顺序的逆序，这是一条有用的通用规则。

### 6.10 第6节小结

最小驱动程序是一个基础。它做的事情很少，但正确地完成了那很少的事情。复位有效，标识符验证有效，资源分配正确，分离路径干净。从这个基础开始，每个后续功能都可以增量添加，并确信底层脚手架是健全的。

下一节将采用最小驱动程序并展示如何一次添加一个功能，在添加下一个之前总是确认每个新功能是正确的。该技术是增量开发的纪律应用于没有规范可参考的领域：我们不是根据规范进行验证，而是根据捕获和测量行为进行验证。结果是一样的：一个做我们相信它所做的事情的驱动程序，由证据而非希望支持。

## 7. 迭代扩展驱动程序并确认行为

第6节的最小驱动程序是真正驱动程序的种子。要将其成长为有用的东西，你需要一次添加一个功能，对于每个功能，你要确认运行驱动程序中该功能所做的与你在厂商驱动程序下观察到的设备行为相匹配。过程简单但要求严格：每个步骤都必须验证，每次验证都必须记录，每条记录都成为第11节将组装的伪数据手册的一部分。

本节介绍增量扩展的方法论。它不引入新的底层技术，因为底层技术与第4和第6节相同。它增加的是在不确定性下负责任地成长驱动程序的纪律。

### 7.1 一次一个功能的规则

规则就在名字里。添加一个功能，验证它有效，将结果提交到你的仓库，然后添加下一个功能。不要一次添加三个功能然后试图调试组合。不要在不验证的情况下添加功能并继续下一个。不要跳过提交，因为提交是让你在后续更改破坏以前有效的东西时可以回滚的东西。

这种纪律与测试驱动开发相同，有一个重要的适应：测试通常是第3节捕获的比较，而不是传统意义上的单元测试。"测试"是"当被调用执行操作 X 时，驱动程序产生与厂商驱动程序相同的寄存器访问序列"。比较是机械的，但它是真正的测试：序列要么匹配要么不匹配，失败告诉你某些东西已经改变。

网络控制器的典型功能序列可能是：

1. 读取 MAC 地址。
2. 设置 MAC 地址。
3. 设置接收描述符环。
4. 接收一个数据包。
5. 设置发送描述符环。
6. 发送一个数据包。
7. 启用中断。
8. 实现中断处理器。
9. 启动链路。
10. 关闭链路。
11. 复位并从错误中恢复。

每个项目是一个离散的功能。每个项目建立在前面的项目之上。每个项目有明确的成功标准（操作要么有效要么无效）。每个项目都有第3节中对应的捕获，驱动程序的行为可以与之进行比较。

### 7.2 验证循环

对于每个新功能，验证循环是：

1. 形成假设：此功能通过执行 X 来工作。
2. 在驱动程序中实现该功能，带有记录每个寄存器访问的日志。
3. 使用小型用户空间程序、sysctl 或任何你已连接的接口触发该功能。
4. 捕获内核日志输出。
5. 与第3节中相关的捕获进行比较。
6. 如果它们匹配，声明功能已实现。
7. 如果它们不匹配，识别差异并决定是你的实现错误还是你的假设错误。

循环很慢。单个功能可能需要循环几次，中间还有假设修订。但慢就是重点。每次迭代都是一个有意识的测试，带有有意识的预测和有意识的观察。经过多次迭代，已验证事实的主体稳步增长，错误信念存活到最终驱动程序中的机会迅速缩小。

### 7.3 与厂商比较的模式

与厂商驱动程序的比较对于这项工作如此核心，以至于它值得自己的讨论。模式如下。

你有第3节中厂商驱动程序执行操作 X 的捕获。捕获包含寄存器访问列表（对于内存映射设备）或 USB 传输（对于 USB 设备），每个都标有时间戳、地址或端点、方向和值。

你有你的驱动程序，启用日志，执行相同的操作 X。内核日志包含寄存器访问或 USB 传输列表，每个都标有时间戳、地址或端点、方向和值。

你比较两个列表。访问应该以相同的顺序、相同的地址或端点、相同的方向，值要么完全匹配，要么以你可以解释的方式不同（因为，例如，值包括序列号或时间戳，它们在运行之间自然不同）。

两个列表一致的地方，你的驱动程序正在做厂商驱动程序所做的。它们不一致的地方，你有差异需要调查。有些差异是良性的：你的驱动程序可能省略了厂商驱动程序出于某种内部原因执行的冗余寄存器读取，省略对设备行为没有影响。其他差异是真正的错误：你写入的值不正确、你错过的序列步骤、你没有意识到操作需要的寄存器。

与厂商比较的模式本质上是一种示例规范。操作 X 的规范是"厂商驱动程序在执行操作 X 时进行的访问序列"。当你的访问序列匹配时，你的实现符合规范。该技术并不完美：它无法检测影响设备内部状态但不影响访问序列的错误，它无法检测你写入值中的错误，如果这些值在厂商捕获中也在运行之间变化。但它捕捉绝大多数实现错误，并且它正好在它们发生的详细程度上捕捉它们。

### 7.4 文档化每个发现的寄存器

随着你添加功能，你发现寄存器。你在任何功能中写入或读取的每个寄存器都是寄存器映射的一部分，每个都需要添加到你正在构建的伪数据手册中。

一种有用的技术是将伪数据手册作为驱动程序源代码旁边的单独 Markdown 文件维护，并在每次发现新事实时更新它。第11节将详细讨论伪数据手册的结构；现在，规则很简单：你学到的任何事实都不应该只留在驱动程序代码中。驱动程序代码是实现；伪数据手册是文档。它们服务于不同的目的，阅读其中任何一个的人都应该能够理解设备。

一个常见的错误是将文档推迟到项目"完成"。逆向工程项目很少在这个意义上完成。它们随时间达到越来越高的完整度水平，文档应该与实现一起增长。一个有二十个功能和一段话伪数据手册的驱动程序是等待发生的维护灾难，因为没有人可以在不重新发现你已经知道的一切的情况下扩展它。

### 7.5 当厂商驱动程序与自身不一致时

当厂商驱动程序在不同场合以不同方式执行同一操作时，会出现一个微妙的复杂情况。有时这是因为驱动程序对同一操作有多条代码路径，取决于设备状态或用户配置。有时是因为驱动程序有在特定条件下激活的可选行为。

当你遇到这种变化时，正确的反应是捕获更多，而不是放弃。在几种不同的起始条件下捕获操作并比较它们。如果你能识别出什么条件选择每个变体，你就学到了关于驱动程序逻辑的一些东西。如果变体在功能上等效（它们都达到相同的最终状态），你可以选择最简单的一个作为规范实现。如果它们在功能上不等效，设备可能有多种需要支持的操作模式。

有时变化只是非确定性：厂商驱动程序从计数器选择序列号，或使用时间戳，结果捕获在运行之间略有不同。非确定性是良性的，学习识别它（这样你可以忽略它）是比较捕获技能的一部分。

### 7.6 在假设分歧时分支驱动程序

当你有两个相互竞争的假设和一个可以区分它们的实验时，值得在 git 中分支你的驱动程序。在一个分支上实现假设 A，在另一个分支上实现假设 B，对两者运行实验，看看哪个产生与厂商捕获匹配的结果。获胜的假设成为主线；失败的假设被丢弃或保留作为参考（以防你后来发现它部分正确）。

分支纪律在实验可能破坏现有功能时特别有价值。如果你有一个可工作的复位实现并想尝试一种新的初始化序列，在分支上尝试。如果新序列有效，合并它。如果它失败，你不会失去复位功能的可工作版本。

### 7.7 第7节小结

我们学习了如何将最小驱动程序成长为更强大的驱动程序。一次一个功能的规则将混乱的开发过程变成了有序、可验证的序列。验证循环将每个功能变成了一个假设测试。与厂商比较的模式给了我们一个无规范设备的具体规范。记录每个发现的寄存器的纪律确保文档与代码一起增长。处理厂商不一致和假设分歧的技术给了我们处理复杂情况的方法。

下一节转向一个稍微不同的主题：如何构建和使用模拟工具来验证假设而不冒真实硬件的风险。模拟让我们在驱动程序接触真实设备之前在软件中测试许多假设，大幅降低破坏东西的风险。

## 8. 使用仿真工具验证假设

逆向工程工作在两种风险之间摇摆。第一种是错误风险：你可能误解寄存器的作用，将错误的值写入错误的地方，或遗漏必要的序列步骤。第二种是损坏风险：某些错误写入可能擦除闪存、使设备变砖或永久改变其配置。第一种风险可以用耐心和仔细的验证管理。第二种风险需要不同的工具：模拟。

本节介绍三种类型的模拟：在软件中实现硬件的模拟设备、在软件中包装硬件的安全包装器，以及让你将真实硬件暴露给虚拟机的直通技术。每种都有其用途，每种都是管理逆向工程风险的工具箱的一部分。

### 8.1 最简单的模拟：模拟内核模块

最简单的模拟形式是一个小型内核模块，在软件中实现设备的寄存器接口。不是通过 `bus_space` 访问真实硬件，你的驱动程序调用模拟模块，后者返回预编程的值或模拟简单的状态机。

对于内存映射设备，模拟是这样工作的。真实设备有一个内存区域，驱动程序对其进行读写。模拟模块分配一个内存区域（或使用你给它的内存区域）并用一个行为类似设备寄存器的软件状态机填充它。当驱动程序写入"复位"寄存器时，模拟将内部状态设置为"已复位"。当驱动程序读取"标识符"寄存器时，模拟返回预编程的魔术值。

模拟很小，通常只有几百行代码，对于一个其真实驱动程序有几千行的设备。它不需要实现设备的全部行为；它只需要实现你正在测试的行为。如果你正在测试复位序列，模拟只需要实现复位寄存器和状态寄存器。如果你稍后想测试数据传输，你扩展模拟以支持数据寄存器。

`examples/part-07/ch36-reverse-engineering/lab04-mock-device/` 中的配套文件包含一个用于假设网络设备的完整模拟模块。它实现标识符、复位、状态、MAC 地址和数据环寄存器。它不传输真实数据；它只是记录驱动程序请求它做什么并返回合理的响应。你可以将它用作自己的模拟的模板。

### 8.2 硬件包装器

第二种模拟是软件包装器，在真实硬件上拦截和验证操作。不是直接访问硬件，你的驱动程序通过一个包装模块，后者记录每次访问，检查其是否安全，并可选地在允许它传递到硬件之前修改它。

当你想测试一个已经适用于简单操作的驱动程序，但现在想向其添加更复杂的操作，并且你想确保复杂操作不会破坏设备时，此技术很有用。包装器可以限制写入的地址范围，过滤掉潜在危险的写入，或将写入值限制为已知安全的范围。

当你想准确记录驱动程序在涉及捕获和重放的复杂序列中做什么时，此技术也很有用。包装器产生一个完整、带时间戳的寄存器访问日志，你可以稍后分析。

实现这种拦截比简单模拟更复杂，因为包装器需要知道哪些地址是安全的、哪些值是安全的，以及如何在不改变其语义的情况下过滤或修改操作。一个天真的包装器可能拒绝一个本意安全的写入，因为它不理解上下文。一个复杂的包装器可能允许一个危险的写入，因为它没有识别出上下文。

实现复杂包装器的最简单方法是使其可调：使用 sysctl 控制其行为。启用地址过滤、启用值过滤、启用日志。对简单操作禁用过滤以建立基线，然后对复杂操作启用它以验证新行为保持在安全范围内。

### 8.3 USB 模拟模板

对于 USB 设备，FreeBSD 包含一个异常强大的模拟框架：USB 模板系统。模板是一个内核模块，实现一个符合标准类或自定义协议的 USB 设备。模板用于实现 USB 设备模式的硬件（如实现 USB 大容量存储的移动电话或实现 USB 以太网的嵌入式板）。对于逆向工程，模板可以作为 USB 协议的软件模拟。

USB 模板位于 `/usr/src/sys/dev/usb/template/`。框架包括一个通用模板控制器和几个特定模板：大容量存储、以太网（CDC）、串行（CDC）、 MIDI 和通用 HID。每个模板是一个可编程的 USB 设备描述：它声明其接口、端点和协议，内核的 USB 设备栈将模板暴露给任何连接到 USB 主机端口的东西。

`/usr/src/sys/dev/usb/template/usb_template_msc.c` 下的几个附加模板覆盖其他设备类。每个都是一个可编程的 USB 设备描述：它声明其配置、接口、端点、类特定描述符以及端点背后的行为。

对于逆向工程工作，模板有两个用途。第一个是作为 USB 协议本身的参考实现。如果你正在逆向工程一个 USB 大容量存储设备，MSC 模板告诉你大容量存储设备应该如何行为。你可以将捕获与模板实现进行比较，看看厂商驱动程序偏离标准的地方。

第二个用途是作为软件模拟的基础，该模拟复现你正在研究的设备的行为。你可以编写一个自定义模板，实现设备的厂商特定协议。模板接收来自主机（可能是运行厂商驱动程序的虚拟机）的 USB 传输，解释它们，并返回设备应该返回的响应。这是一个强大的测试环境，因为你可以在暴露真实设备之前对模拟设备验证你的驱动程序。

### 8.4 bhyve 直通模拟

bhyve PCI 直通是一种模拟形式，将真实硬件暴露给虚拟机，同时保持宿主机操作系统可用进行分析。宿主机使用 `ppt(4)` 驱动程序声明设备，bhyve 将设备的 PCI 配置空间和内存空间映射到客户机。在客户机内部，设备行为像普通 PCI 设备。在客户机中运行的厂商驱动程序可以像在任何物理机器上一样访问它。

对于逆向工程，bhyve 直通的价值在于你可以运行厂商驱动程序（在客户机中），同时从宿主机观察和分析其行为。宿主机的 DTrace 探针可以观察内核与 `ppt(4)` 驱动程序的交互。宿主机的 `pciconf` 可以检查设备的配置空间。宿主机可以在直通启用之前和之后对配置空间进行差异比较。

直通工作流程是：

1. 在宿主机上，识别设备的 PCI 地址。
2. 使用 `devctl detach` 将任何现有驱动程序从设备分离。
3. 确保 `ppt` 驱动程序在内核中可用（`kldload ppt` 或在内核配置中添加 `device ppt`）。
4. 使用 `devctl attach` 将设备附加到 `ppt`。
5. 配置 bhyve 客户机，使用 `-s slot,passthru,bus/slot/function` 直通设备。
6. 启动客户机并在其中安装厂商驱动程序。

在客户机内部，设备行为像普通 PCI 设备。客户机中运行的厂商驱动程序可以像在任何物理机器上一样访问它。宿主机可以看到配置空间读取和写入（通过 DTrace 观察 `ppt`），但看不到内存空间访问，因为那些直接从客户机内存传递到硬件。

对于逆向工程，bhyve 直通的价值在于你可以运行厂商驱动程序（在客户机中），同时保持宿主机操作系统可用于分析。宿主机的 DTrace 探针可以观察内核与 `ppt(4)` 驱动程序的交互。宿主机的 `pciconf` 可以检查设备的配置空间。宿主机可以在直通启用之前和之后对配置空间进行差异比较。

### 8.5 验证循环：模拟然后硬件

模拟和硬件一起使用时最强大。一种有效的工作流程是模拟验证循环：

1. 在驱动程序中实现一个功能，起初使用模拟设备。
2. 在模拟上验证功能有效。
3. 如果功能有效，切换到真实硬件并验证功能仍然有效。
4. 如果功能在模拟上有效但在真实硬件上失败，差距揭示了模拟没有捕获的一些硬件行为。
5. 更新模拟以捕获缺失的行为。
6. 重复直到功能在两者上都有效。

例如，假设你想知道当驱动程序将两个数据包背靠背写入发送描述符环时驱动程序做什么。你可以在模拟中运行该场景，模拟记录它接收的描述符并报告它将"发送"的数据包。如果模拟显示驱动程序正确格式化了描述符，按正确顺序写入它们，并在每次写入后正确敲响了门铃寄存器，你可以对基本逻辑有信心。

然后你在真实硬件上运行相同场景。如果硬件行为符合模拟——数据包实际发送，中断实际触发——你对实现有信心。如果硬件行为不同——也许第二个数据包不发送，也许中断有奇怪的延迟——差距告诉你模拟遗漏了真实硬件的一些行为。

### 8.6 错误处理的验证

验证循环技术对于错误处理特别有价值。错误路径是驱动程序中最难测试的部分，因为产生真实错误需要真实硬件真正失败。模拟可以按需产生失败。

你可以在模拟中实现失败模式：超时、错误响应、损坏的数据、意外状态转换。然后你可以在驱动程序中演练错误处理代码，而无需等待真实硬件失败。错误处理在模拟中有效后，你对它在真实硬件上有效有信心——同时你也知道代码至少处理了你能想到的失败模式。

### 8.7 模拟的局限

模拟价值巨大，但它有局限。模拟是根据你对设备的理解构建的。如果你的理解是错的，模拟就是错的，在错误模拟上验证驱动程序并不能证明驱动程序对真实硬件有效。

正确的态度是将模拟视为发现错误的一种方式，而非正确性的证明。模拟测试捕获驱动程序中与模拟不一致的错误。它不捕获驱动程序和模拟都共享的错误，或模拟没有建模的硬件部分中的错误。

使模拟有价值的是它与硬件测试的组合。模拟发现你可以轻易产生的错误——格式错误、序列错误、逻辑错误。硬件测试发现你无法轻易产生的错误——时序敏感行为、电气边缘情况、模拟遗漏的厂商特定怪癖。组合是答案。模拟测试捕获模拟覆盖的错误；硬件测试捕获其余的。组合覆盖的范围比单独任一种都大得多。

### 8.8 第8节小结

我们看到了补充逆向工程验证工作的模拟工具。最小内核模块可以在软件中模拟设备的寄存器接口。硬件包装器可以在真实设备上拦截和验证操作。USB 模板提供了 USB 协议的参考实现和自定义模拟的基础。bhyve 直通将真实硬件暴露给虚拟机，同时保持宿主机可用于分析。模拟-然后-硬件验证循环在暴露真实设备之前在模拟中捕获错误。

下一节从技术工作后退一步，看看逆向工程如何与 FreeBSD 社区互动。逆向工程的技术部分是可见部分，但社区部分是大部分成果最终归宿的地方。

## 9. 逆向工程中的社区协作

逆向工程很少是真正的单独活动，即使当它是一个人独自在桌前执行的。对于几乎所有值得研究的设备，存在先前的工作；做得好的新工作会贡献给持续的社区知识。知道如何找到先前工作、如何评估它以及如何贡献回是逆向工程技能的一部分。

本节讨论如何找到先前工作、如何评估其可信度、如何记录你的来源以及如何发布你自己的发现。

### 9.1 寻找先前工作

当你开始调查一个设备时，首先要做的是检查其他人是否已经做了部分工作。搜索项目笔记本、Google、GitHub、FreeBSD 和 Linux 源码树、OpenBSD 和 NetBSD 源码树以及任何相关的邮件列表档案。

最有价值的来源，按大致有用程度排序：

- **FreeBSD 源码树**。如果你的设备或密切相关的设备有 FreeBSD 驱动程序，这是最好的可能来源。驱动程序是 FreeBSD 许可的、有文档并且针对 FreeBSD 内核编写。你可以阅读它、引用它并在此基础上构建。

- **OpenBSD 和 NetBSD 源码树**。BSD 许可的驱动程序，来自其他 BSD，是出色的来源。它们可能需要移植工作，因为内核 API 不同，但法律框架清晰且技术质量通常很高。

- **Linux 源码树**。Linux 驱动程序通常有 GPL 许可证，这意味着你可以阅读它理解设备但不可以复制代码。不过，你可以从阅读中推断寄存器定义和协议序列，并写出自己的实现。法律细节在第12节讨论。

- **GitHub 和其他公共仓库**。独立开发者有时发布不在任何操作系统源码树中的驱动程序。这些驱动程序质量各异，但有时它们是某些设备的唯一来源。

- **邮件列表档案**。`freebsd-hackers@`、`freebsd-drivers@`、`linux-kernel@` 和类似的列表经常有讨论设备的线程。搜索档案可能找到部分笔记、失败的尝试或指向其他来源的指针。

- **厂商开源驱动程序**。一些厂商发布其驱动程序的源代码，有时有宽松许可证，有时有 GPL。如果可用，厂商源是权威来源——它可能正确，因为它来自编写设备的人。

- **学术论文**。少数设备——特别是网络接口和存储控制器——出现在论文中，研究人员描述其内部工作。这些论文很少给出完整的寄存器映射，但它们经常给出高层架构描述。

任何项目的第一个小时应该花在搜索这些来源上。找到先前工作可以节省数周的重新发现。

### 9.2 评估先前工作

并非所有先前工作都同等可信。一个可工作的开源驱动程序可能是一个你可以几乎完全信任的来源。一个论坛帖子描述某人尝试编写驱动程序并失败，可能是错误信息的有用来源，但不是你可以信任的技术细节的来源。

评估来源的技能与历史学家评估历史来源的技能相同。询问：

- **作者是否有第一手知识？** 厂商工程师或成功编写驱动程序的人有第一手知识。复制他人笔记的人没有。

- **信息是否可验证？** 如果来源声称寄存器 X 在偏移 Y 持有值 Z，你能验证吗？如果驱动程序实际使用寄存器 X，你可以通过阅读代码验证。如果来源只是文本笔记，验证更难。

- **信息是否与其他来源一致？** 如果三个独立来源都同意一个寄存器偏移，置信度高。如果一个来源声称一件事，另一个声称不同的事，你需要解决差异。

- **来源是否有偏见？** 厂商文档有时略过错误或边缘情况。论坛帖子有时反映单个用户与单个设备的经验，可能不推广。

### 9.3 记录你的来源

当你从先前工作采用一个事实时，记录它来自哪里。笔记本条目应该说"来自 Linux ath5k 驱动程序，偏移 0x1000 处的寄存器 0x1234 持有固件版本"。来源引用让你稍后追溯事实，如果事实证明是错误的，它让你证明你做了净室工作。

净室纪律是使用外部信息的标准方法论，同时保持法律清晰。在严格净室形式中，涉及两个人。一个人"脏"——他们阅读 GPL 许可的代码、厂商二进制文件或任何其他不能直接复制的东西。他们编写笔记、寄存器映射和高层描述。另一个人"净"——他们只阅读脏团队写的笔记，并根据笔记编写新驱动程序。净团队从不接触原始代码，因此新驱动程序法律上独立于原始代码。

在常用于单独项目的宽松形式中，同一个人扮演两个角色。你阅读外部代码（GPL 驱动程序、厂商二进制文件），编写笔记，然后关闭外部代码并根据笔记编写你的驱动程序。你从不将外部代码复制粘贴到你的驱动程序中。

宽松形式在法律上不如严格形式那么无懈可击，因为证明你实际上没有复制比证明你不能复制更难。但它广泛用于社区项目，对于大多数实际目的足够。

在所有情况下，新驱动程序必须不包含复制代码、复制注释或复制非 BSD 兼容许可证的文件结构。如果你阅读 GPL 驱动程序，你可以根据你学到的内容编写 BSD 驱动程序，但你不能复制任何东西。这不仅仅是法律要求；这也是工程纪律。复制代码会继承你不理解的错误；编写新代码会迫使你理解你在做什么。

### 9.4 发布你的工作

逆向工程工作完成后，应该发布。发布方式取决于你的目标和工作的完整性。

最小有用发布是一个包含以下内容的 Git 仓库：

- 驱动程序源代码，带有注释解释寄存器和序列。
- 伪数据手册，作为单独 Markdown 文件。
- 笔记本，至少作为捕获和实验的摘要。
- 如果有来自硬件捕获，一些代表性捕获文件。

许可证很重要。对于打算包含在 FreeBSD 中的驱动程序，BSD-2-Clause 许可证是标准。对于打算作为独立项目的驱动程序，BSD 或 MIT 许可证让他人最容易使用你的工作。GPL 许可证阻止 FreeBSD 包含你的驱动程序，即使它很优秀。

发布后，在人们会看到的地方宣布工作：

- 如果驱动程序是某个 FreeBSD 已经有相关支持的子系统中的设备驱动程序，`freebsd-drivers@` 那是合适的地方。
- 如果驱动程序是全新的或高度实验性的，`freebsd-hackers@` 可能是更好的受众。
- 如果工作主要是文档（你部分逆向工程了设备并发布了寄存器映射），`freebsd-hackers@` 和相关论坛都适合。

发布让他人受益于你的工作。他们可以修复错误、添加功能或将其用作类似设备的起点。发布也让你受益：用户报告错误、建议改进、有时贡献补丁。一个发布的驱动程序会改进；一个私人驱动程序停滞。

### 9.1 寻找已有成果

当你开始调查一个设备时，首先要做的就是搜索已有成果。即使看起来不起眼的设备有时也是大量先前研究的主题，找到那些研究可以节省大量时间。

最有价值的来源，按有用性大致排序：

**FreeBSD 源码树。** 在 `/usr/src/sys/dev/` 中搜索设备的厂商 ID、设备 ID、USB 描述符或厂商名称。FreeBSD 树中的驱动程序是已有成果的黄金标准，因为它已经过验证、针对当前内核构建并遵循 FreeBSD 约定。即使不存在驱动程序，同一厂商的相关驱动程序通常也共享适用的模式。

**OpenBSD 和 NetBSD 源码树。** 其他 BSD 有自己的驱动程序集合，它们经常有 FreeBSD 尚未支持设备的驱动程序。代码通常易于阅读和移植。许可证通常与 FreeBSD 兼容（大多数 BSD 许可代码都是）。

**Linux 源码树。** Linux 内核拥有任何开源项目中最广泛的设备驱动程序集合。大多数是 GPL 许可的，这意味着你不能将代码复制到 FreeBSD 驱动程序中，但你可以阅读它们来理解并从空白开始编写自己的实现。阅读 Linux 驱动程序来理解设备、然后从该理解编写原创代码的净室方法，在互操作性案例中已建立并被法律接受；安全执行此操作的完整框架，包括净室辩护实际依赖的书面证据痕迹，出现在本章后面的第 12 节。即使法律问题已解决，由于纯技术原因，从 Linux 到 FreeBSD 的直接移植很少可行。Linux 驱动程序依赖没有直接 FreeBSD 等效物的内核 API：其自旋锁和互斥原语有不同的唤醒语义，其内存分配器区分 FreeBSD 不同表达方式的上下文，其设备模型（带有 `struct device`、sysfs 和驱动核心探测流程）不映射到 Newbus。使用 Linux 驱动程序的正确方法是从中挖掘设备级事实（寄存器布局、魔法值、初始化顺序、quirk 表），然后针对 FreeBSD 自己的原语重建控制流。第 13 节展示了两个正是这样做的 FreeBSD 驱动程序。

**硬件特定社区站点。** 对于许多设备，爱好者建立了收集特定系列所有已知信息的专门站点。Wireshark wiki 有关于 USB 协议的详尽文档。OpenWrt 项目有关于许多嵌入式设备的文档。许多特定硬件系列有自己的社区 wiki 或论坛。

**厂商邮件列表、技术支持档案、应用笔记。** 一些厂商在技术支持上下文中发布的比其公共文档中更多。搜索他们的支持档案或开发者邮件列表有时可以找到官方文档中没有的信息。

**专利。** 专利申请通常包含设备如何工作的详细描述，因为描述发明的法律要求强制某种程度的披露。专利数据库可按公司和年份搜索，搜索正确厂商的专利有时可以揭示大量细节。

**学术论文。** 当设备处于专业领域（科学仪器、工业控制、某些网络设备类）时，该领域的学术论文可能已经出于研究目的记录了设备的接口。

任何项目的第一个小时应该花在搜索这些来源上。即使不成功的搜索也是有信息价值的：知道没有已有成果存在会显著改变项目的范围。

### 9.2 评估可信度

并非所有已有成果都同样可信。一个可工作的开源驱动程序是高度可信的，因为作者的主张有可证明工作的代码支持。爱好者编写的社区 wiki 页面可能可信也可能不可信，取决于来源的过往记录。厂商白皮书在厂商想表达的要点上通常可信，但在厂商不愿讨论的要点上可能沉默或误导。

评估来源的技能与历史学家和记者发展的技能相同。你要问：这是谁写的、什么时候写的、基于什么证据、出于什么动机？你跨多个来源比较各种主张。你对由代码或直接测量支持的主张给予最大权重。你对仅依赖作者声誉的主张给予较少权重。

当你从已有成果采纳一个事实时，记录它来自哪里。笔记本条目应该写"来自 Linux ath5k 驱动程序，偏移 N 处的寄存器 XYZ 是通道选择器"。一年后，当事实证明是错误时，你会知道错误来自哪里以及同一来源的其他事实可能也是错误的。

### 9.3 净室纪律

净室纪律是在不侵犯版权的情况下使用已有成果的标准方法论。该纪律简单但需要谨慎。

在严格的净室形式中，涉及两个人。一个人阅读已有成果（厂商驱动程序、泄露的规范、反汇编的二进制文件）并产生设备做什么的描述。描述使用自然语言，不包含任何受版权保护的材料。第二个人只阅读描述并编写新驱动程序。第二个人永远看不到原始作品。因为新驱动程序是在不参考原始作品的情况下编写的，根据版权法它不可能是其衍生作品。

在宽松形式中（通常用于单独项目），同一个人执行两个角色，但注意在时间上将它们分开并记录这种分离。阅读已有成果。做笔记。将已有成果放在一边。等待。然后，只从你的笔记工作，编写驱动程序。笔记是桥梁；在你编写新代码时，原始代码永远不在你面前。

宽松形式在法律上不如严格形式严密，但这是大多数个人逆向工程师实际做的，互操作性工作的判例法通常支持它。关键是将阅读与编写分开的纪律，这样你可以证明（并向自己证明）新代码是独立的。

在所有情况下，新驱动程序不得包含从原始作品复制的代码、复制的寄存器名称宏、复制的数据结构或复制的注释。它可以包含想法、结构模式和设备接口本身，因为这些不受版权保护。任何受版权保护的东西都必须用你自己的语言和自己的结构重新表达。

### 9.4 分享你的发现

逆向工程工作完成后应该发布。发布服务几个目的：它让其他人使用驱动程序，让其他人扩展驱动程序，在你停止工作的情况下保护知识，并贡献关于设备的公共知识体。

最小有用发布是包含以下内容的 Git 仓库：

- 驱动程序源代码，带有清晰的许可证。
- 解释驱动程序做什么以及如何构建的 README。
- 记录设备的伪数据手册（第 11 节）。
- 使用驱动程序所需的测试程序和任何其他工具。
- 笔记本或其清理版本，以便其他人可以追溯到任何主张背后的推理。

更完善的发布还添加：

- 描述方法论的文档，以便其他人可以复现工作。
- 未解决问题的列表。
- 已验证事实与剩余假设之间的明确区分。
- 所有已有成果来源的署名，带链接。

许可证很重要。对于打算纳入 FreeBSD 的驱动程序，BSD 两条款许可证是标准选择。许可证必须使作品与 FreeBSD 源码树的许可证要求兼容。如果你使用了来自 GPL 许可项目的材料，即使是净室风格的，也要仔细记录来源，以便任何关于衍生作品的疑虑可以通过检查你的笔记来解决。

一旦发布，在感兴趣的人能看到的地方宣布工作：嵌入式工作用 FreeBSD-arm 邮件列表，网络驱动程序用 FreeBSD-net 邮件列表，相关的项目特定社区和你个人渠道。公告应该简短但有信息量：驱动程序做什么、完整度如何、仓库在哪里以及欢迎谁贡献。

### 9.5 在Markdown或Git中维护伪数据手册

伪数据手册本身值得特别关注。它是你产生的最有价值的单一制品，通常比驱动程序代码更有价值。驱动程序代码是你对设备理解的一个实现；伪数据手册是理解本身，从中可以编写任意数量的驱动程序（FreeBSD、Linux、NetBSD、自定义嵌入式系统）。

在社区项目中最有用的格式是 Git 跟踪的 Markdown 文件（或一小组文件），围绕设备的逻辑组件构建：身份、寄存器映射、缓冲区布局、命令集、错误代码、初始化序列、编程示例。我们将在第 11 节详细查看其结构。

Git 跟踪是必不可少的。伪数据手册是一个活文档；随着新事实的发现、旧事实的纠正、额外逆向工程工作扩展其覆盖范围，它将被更新。Git 历史是理解如何演变的记录。当一个事实被纠正时，Git 日志显示何时被纠正以及由谁纠正，以便其他读者可以知道他们旧的打印输出何时已过时。

`examples/part-07/ch36-reverse-engineering/lab06-pseudo-datasheet/` 中的配套代码包含一个伪数据手册的 Markdown 模板，适合作为你自己项目的起点。模板对结构有自己的主张，因为一致的结构使文档更易读、更易与其他伪数据手册比较、更易于自动提取到其他格式。

### 9.6 第9节小结

逆向工程的社区协作方面是赋予工作持久性的方面。孤立编写且从未发布的驱动程序可能很好地服务其作者，但它会随作者一起消失。与已有成果对话并清晰发布的驱动程序服务更广泛的受众并持续更长时间。

下一节转向工作的安全方面，我们已经反复引用过。安全值得专门讨论，因为做错的后果可能包括对昂贵硬件的永久损坏，避免此类损坏的技术足够具体，可以明确教授。

## 10. 避免设备变砖或硬件损坏

这是本章一直在推迟的一节。粗心进行的逆向工程可能会损坏硬件。有些损坏可以通过努力恢复。其他损坏会将工作设备变成一块无用的砖头，除了专门的厂商恢复工具（如果存在的话）之外，什么都无法修复。不了解哪些模式是危险的就进行工作的逆向工程师最终会毁坏某些东西。本节的目标是具体告诉你避免哪些模式，哪些模式是安全的。

本节中的建议适用于本章中的其他每一节。第 4 节中的探测技术只有与这些警告结合才是安全的。第 6 节中的最小驱动程序只有当其写入受到此处描述的约束时才是安全的。第 7 节中的迭代扩展只有当每个新功能在被允许写入设备之前根据这些标准进行评估时才是安全的。请完整阅读本节；不要在没有阅读它的情况下跳到技术部分。

### 10.1 总体原则

一般原则很简单，通过练习可以直接应用。

> **永远不要在没有有力证据表明操作安全的情况下，向未知寄存器写入或发送未知命令。**

充分的证据可以来自多个来源，每个都值得单独讨论。

**厂商的驱动程序执行了相同的操作。** 如果你捕获了厂商驱动程序向特定寄存器写入特定值，并且结果设备行为健康，那么在你的驱动程序中执行相同的写入几乎肯定是安全的。厂商的驱动程序大概是经过充分测试的，不会随意做会让设备变砖的事情。

**操作是有文档的标准操作。** 某些操作如此通用，可以假设是安全的。PCI 配置空间读取根据设计是安全的。PCI 标准控制寄存器写入（清除待处理中断、启用总线主控、设置内存访问启用）在标准范围内是安全的。在标准端点上使用标准请求的 USB 控制传输是安全的。协议的标准化部分可以信任；厂商特定部分在没有证据的情况下不能信任。

**操作是可逆的。** 你可以立即撤销的写入比你不能撤销的写入安全得多。设置配置位、观察效果、然后恢复原始值比执行具有永久后果的操作安全得多。

**操作已在模拟中测试。** 你首先在模拟设备或模拟器中测试过的写入，其中最坏情况是测试 VM 中的内核崩溃，比你只在脑海中建模的写入更安全。

如果没有任何这些形式的证据，操作就是未知的，规则是：不要执行它。

### 10.2 需要特别谨慎的操作

某些类别的操作危险到值得明确关注。这些是粗心写入可能永久损坏设备的操作，它们是永远不应该在没有充分证据的情况下执行的操作。

**闪存写入。** 许多设备包含用于固件、配置、校准或设备标识符的闪存。对闪存的写入是不可逆的（或者充其量可以通过恢复备份来逆转）。意外写入闪存的驱动程序可能会损坏固件（使设备无法启动）、覆盖校准数据（使设备不准确或不可用）或更改设备标识（使其自身的固件无法识别）。触发闪存写入的模式因设备而异，但它们通常涉及特定的"解锁"序列，随后是对明确为闪存区域的地址的写入。如果你的捕获显示这样的序列出现在用户明确执行"固件更新"操作的上下文中，不要在你的驱动程序中复制该序列，除非你明确正在实现固件更新。

**影响非易失性状态的硬复位。** 大多数复位是将设备返回到初始内存状态的软复位。然而，有些复位也会清除非易失性状态：校准数据、配置、标识符编程。这些"恢复出厂设置"操作有时由触发软复位的同一个寄存器触发，但使用不同的位模式。只使用厂商驱动程序在附加时使用的软复位模式，不要实验你未见厂商驱动程序使用过的位模式。

**EEPROM 写入。** 与闪存类似，EEPROM 包含长期配置。EEPROM 写入通常通过特定的寄存器写入协议进行编排，错误的协议可能会损坏 EEPROM 状态。在探索期间避免 EEPROM 写入；如果必须执行它们，只有在清楚理解最终应该在 EEPROM 中存储什么值以及有验证结果的方法的情况下才能这样做。

**修改设备标识符。** 某些设备将其厂商和设备 ID 存储在闪存或 EEPROM 中，驱动程序错误可能会覆盖它们。ID 已更改的设备将不会被任何匹配原始 ID 的驱动程序识别，恢复它可能需要使用硬件编程器进行物理重新编程。不要写入任何可能相关于设备标识的寄存器。

**电源管理状态更改。** 某些设备具有通过复杂的寄存器写入序列管理其转换的电源状态。错误的序列可能会使设备处于不易恢复的状态，有时需要重启主机甚至物理移除设备。电源管理是任何设备接口中最脆弱的领域之一。

**PHY 或 PLL 配置。** 包含自己的时钟生成的设备（特别是 PCI Express 设备）具有 PLL 配置，如果设置错误，可能会使设备完全无法通信。网络设备上的 PHY 配置有类似的陷阱。这些是必须完全遵循厂商驱动程序的子系统，因为通过任何进一步配置从错误设置恢复没有好方法。

**在 DMA 地址配置错误时启用总线主控。** 如果你在设备上启用总线主控，而设备的 DMA 地址指向一个无效的内存位置，设备可能会读取或写入该位置并损坏主机内存。这是驱动程序错误通过硬件操作导致主机崩溃的少数方式之一。始终在启用总线主控之前设置有效的 DMA 映射，如果你还没有准备好管理 DMA，永远不要启用总线主控。

这些类别并不详尽，但它们涵盖了损坏设备的最常见方式。一般教训是：尊重你尚未完全理解其后果的操作，不要对它们进行实验。

### 10.3 软复位与看门狗定时器

安全讨论的建设性方面是当出现问题时该怎么办。第一种技术是软复位。

软复位是设备的"我已经进入糟糕状态，请重新启动"机制。模式与你在第 6 节中实现的复位相同：写入控制寄存器将设备放回其已知初始状态。保守使用时，软复位可以从许多类型的问题中恢复而无需干预。

代码中的模式很简单：

```c
static void
mydev_recover(struct mydev_softc *sc)
{
    device_printf(sc->sc_dev, "recovering device\n");
    mydev_reset(sc);
    /* Reapply any necessary configuration. */
    mydev_init_after_reset(sc);
}
```

恢复例程应该从任何检测到问题的代码路径中调用：超时、错误状态、意外的寄存器值。不必要的复位代价很小；未恢复错误的代价可能大得多。

看门狗定时器是下一层防御。驱动程序定期检查设备是否在取得进展，如果进展停滞，则触发恢复。模式如下：

```c
static void
mydev_watchdog(void *arg)
{
    struct mydev_softc *sc = arg;
    uint32_t counter;

    mtx_lock(&sc->sc_mtx);
    counter = bus_read_4(sc->sc_mem, MYDEV_REG_COUNTER);
    if (counter == sc->sc_last_counter) {
        sc->sc_stall_ticks++;
        if (sc->sc_stall_ticks >= MYDEV_STALL_LIMIT) {
            device_printf(sc->sc_dev, "device stalled, resetting\n");
            mydev_recover(sc);
            sc->sc_stall_ticks = 0;
        }
    } else {
        sc->sc_stall_ticks = 0;
    }
    sc->sc_last_counter = counter;
    callout_reset(&sc->sc_watchdog, hz, mydev_watchdog, sc);
    mtx_unlock(&sc->sc_mtx);
}
```

看门狗读取设备在正常操作期间应该递增的计数器。如果计数器在几次迭代中没有变化，看门狗假设设备已停滞并触发恢复。该模式对瞬态减速（容忍单次递增丢失）具有健壮性，但能在几秒钟内捕获真正的停滞。

### 10.4 在进行任何冒险操作之前备份固件

对于带有固件的设备（大多数现代设备都有），固件本身应该在任何冒险操作之前备份。如果你的捕获显示固件是在初始化期间由宿主机加载的，那么固件映像大概已经作为文件存在某处，你可以保留一份副本。如果固件存在于设备上的闪存中而不是从宿主机加载的，你必须通过设备暴露的任何读取闪存的机制将其读回，并存储结果。

备份是防止某些实验覆盖固件情况的保险。有备份，你可以恢复原始。没有备份，你可能有一块砖。

冒险操作前备份的纪律与风险系统管理前备份的纪律相同。成本很小，当它重要时的价值巨大，跳过它的人迟早会后悔跳过了它。

### 10.5 尽可能使用只读探测

只要实验可以只读执行，就只读执行。获得的信息通常比写入而不是读取节省的时间有价值得多。第4节的寄存器映射工作几乎完全只读是有原因的：读取是安全的，写入不是。

当你必须调查写入效果时，一个有用的模式是读取值、计算你期望新值是什么、执行写入、再次读取值，并验证结果符合你的预期。验证步骤让你在基于错误假设进行进一步工作之前，捕获写入没有产生你预期效果的情况。

```c
static int
mydev_set_field(struct mydev_softc *sc, bus_size_t off,
    uint32_t mask, uint32_t value)
{
    uint32_t old, new, readback;

    old = bus_read_4(sc->sc_mem, off);
    new = (old & ~mask) | (value & mask);
    bus_write_4(sc->sc_mem, off, new);
    readback = bus_read_4(sc->sc_mem, off);
    if ((readback & mask) != (value & mask)) {
        device_printf(sc->sc_dev,
            "set_field off=0x%04jx mask=0x%08x value=0x%08x "
            "readback=0x%08x mismatch\n",
            (uintmax_t)off, mask, value, readback);
        return (EIO);
    }
    return (0);
}
```

辅助函数在一个地方执行读-改-写-验证循环，带有不匹配日志。一致使用时，它捕获写入未生效的情况（因为寄存器是只读的，因为写入编码值的方式与你预期的不同，或者因为设备此时状态不允许更改），在它们导致下游混乱之前。

### 10.6 安全探测包装器

`examples/part-07/ch36-reverse-engineering/lab05-safe-wrapper/` 中的配套代码包含一个小型安全探测包装器库，结合了这里讨论的几种技术：每个操作的超时、检测到停滞时自动恢复、只读模式和每次操作的日志。包装器给每个操作增加几百微秒，这在探索期间无关紧要，它们在大多数安全问题变得有害之前捕获它们。

该模式推荐用于所有探索性驱动程序。生产驱动程序（当它最终存在时）可以放弃包装器转而使用更高效的直接访问，但在探索期间，包装器在安全方面的价值是值得其成本的。

### 10.7 知道何时停止

最困难的安全决策有时是停止的决策。当实验表现不符合预期，当设备产生你无法解释的值，当你的假设全部错误而你还没有新的假设时，诱惑是继续探索直到某些东西变得清晰。抵制这种诱惑。停下来，休息一下，重新查看捕获，与其他人讨论你观察到的内容，在恢复之前让情况明朗起来。

原因是困惑的时刻正是意外损坏最可能发生的时刻。清醒的头脑会小心谨慎地判断哪些写入是安全的；沮丧的头脑会尝试看起来有希望的操作，然后太晚才发现其中一个是破坏性的。

短暂休息也让你的潜意识工作。许多最有用的逆向工程洞察在你做其他事情时到来：散步、睡觉、处理无关问题。大脑在被给予机会时有将观察整合为连贯理解的非凡能力。当工作停滞时强迫自己呆在键盘前通常产生不了有用的东西，有时会产生灾难。

### 10.8 第10节小结

逆向工程中的安全建立在一小组纪律之上：没有有力证据绝不写入、识别需要特别谨慎的操作、内置软复位恢复和看门狗定时器、冒险操作前备份固件、尽可能使用只读实验、使用安全探测包装器、知道何时停止。这些纪律都不复杂。所有都是必要的。遵循它们的逆向工程师不太可能损坏硬件。忽略它们的逆向工程师迟早肯定会损坏。

下一节通过讨论如何将前几节建立的工作体转变为可维护的驱动程序和可用的伪数据手册来结束本章的技术部分。从观察和假设开始的逆向工程过程以驱动程序和文档结束，这些最终制品如何构建决定了项目对其未来读者的有用程度。

## 11. 重构和文档化逆向工程的设备

逆向工程项目的结束不是驱动程序工作的时刻。它是驱动程序和伪数据手册一起可以交给另一个工程师、他可以阅读并理解设备和实现的时刻。直到那个时刻，项目是不完整的，即使驱动程序看起来可以工作。本节介绍将项目发现整合为可维护形式的工作。

工作有两部分。驱动程序本身需要清理、重构以遵循正常驱动程序编写约定，并以任何 FreeBSD 驱动程序应有的方式文档化。在整个项目期间作为事实笔记本增长的伪数据手册需要重新组织为从未见过项目的人可以阅读和学习的独立文档。

### 11.1 驱动程序清理

逆向工程驱动程序，在其可工作但未清理的状态下，通常带有其历史的痕迹：引用"疑似控制寄存器"的注释、用条件编译测试多个假设的代码块、在特定调查期间添加的调试 sysctl、当时有用但现在不再需要的日志消息。清理意味着在保留实质性内容的同时删除历史噪声。

清理检查清单：

1. 删除每个用于测试替代方案的条件的编译块。选定的替代方案应该保留；替代方案应该删除。
2. 用确认的名称替换推测性名称。如果寄存器最初命名为 `MYDEV_REG_UNKNOWN_40`，现在知道是通道选择寄存器，将其重命名为 `MYDEV_REG_CHANNEL`。
3. 用解释性注释替换调查注释。说"我认为这可能是轮询延迟"的注释是调查阶段的假设注释；一旦假设被确认，用说"等待设备完成复位"的注释替换它。
4. 删除不再有用的调试 sysctl，保留操作员或维护者想要的那些。
5. 删除在调查期间有价值但现在在生产中是噪声的 `device_printf` 调用。
6. 验证分离路径是干净的。可以干净加载但不能干净卸载的驱动程序是维护问题。
7. 验证所有错误路径正确释放其资源。错误时泄漏资源的驱动程序是一个最终会暴露的问题。

`examples/part-07/ch36-reverse-engineering/` 中的配套代码包括调查阶段的脚手架和清理后的形式，这样你可以看到区别。清理后的形式是适合包含在 FreeBSD 源码树中的形式（我们将在第 37 章讨论包含）。

### 11.2 驱动程序文档

FreeBSD 驱动程序应该至少有一个第 4 节的手册页，理想情况下还有内核开发者文档中的一节。手册页是为想知道驱动程序做什么以及如何配置它的用户准备的。开发者文档是为想要内部理解驱动程序的内核黑客准备的。

手册页遵循标准 FreeBSD 风格。`style(4)` 的相关手册页记录了约定，类似驱动程序的手册页是很好的参考示例。典型驱动程序手册页涵盖：

- 驱动程序名称和一行描述。
- 显示如何在 `loader.conf` 中加载驱动程序的概要。
- 描述驱动程序支持什么硬件和提供什么功能的说明部分。
- 硬件支持部分，列出驱动程序适用的特定设备。
- 配置部分，描述驱动程序暴露的任何 sysctl 或 loader 变量。
- 诊断部分，列出驱动程序可能产生的消息及其含义。
- 另见部分，带交叉引用。
- 历史部分，说明驱动程序何时出现。
- 作者部分。

对于逆向工程驱动程序，描述部分应对驱动程序的来源诚实：它通过逆向工程开发，支持的功能集是通过该过程可实现的，设备的某些角落可能行为与驱动程序预期不同。操作员欣赏诚实的文档；模糊声称完全支持的文档在用户遇到未实现行为时只会造成困惑。

### 11.3 伪数据手册结构

伪数据手册是捕获你关于设备所学一切的独立文档，以从未见过你驱动程序的人可以阅读和理解的形式呈现。结构良好的伪数据手册通常成为任何使用设备的项目中被引用最多的文档，因为它回答了驱动程序源代码不仔细阅读就无法回答的问题。

实用的伪数据手册结构如下：

```text
1. 身份
   1.1 厂商和设备 ID
   1.2 USB 描述符（如适用）
   1.3 类代码（如适用）
   1.4 子系统标识符（如适用）
   1.5 覆盖的硬件修订版

2. 来源
   2.1 咨询的来源
   2.2 使用的方法论
   2.3 每个事实的验证状态（高/中/低）

3. 资源
   3.1 内存区域及其大小
   3.2 I/O 端口（如适用）
   3.3 中断分配

4. 寄存器映射
   4.1 带有偏移和简短描述的寄存器列表
   4.2 每个寄存器的详细信息：大小、访问类型、复位值、字段

5. 缓冲区布局
   5.1 环形缓冲区布局
   5.2 描述符格式
   5.3 数据包格式

6. 命令接口
   6.1 命令排序
   6.2 带有格式和响应的命令列表
   6.3 错误报告

7. 初始化
   7.1 冷附加序列
   7.2 热复位序列
   7.3 操作前所需的寄存器设置

8. 操作模式
   8.1 数据流
   8.2 中断处理
   8.3 状态轮询

9. 特性和勘误
   9.1 设备中已知的错误
   9.2 驱动程序中的变通方法
   9.3 尚未完全表征的边缘情况

10. 未解决的问题
    10.1 已观察但未理解的行为
    10.2 目的尚不明确的寄存器
    10.3 尚未调查的操作

11. 参考资料
    11.1 咨询的已有成果
    11.2 其他操作系统中的相关驱动程序
    11.3 公开文档（如果有）
```

该结构是全面的，并非每个项目都会填写每一节。不相关的节可以省略；模板是"如果相关信息存在，值得记录什么"的检查清单，而不是要求编造不存在的信息。

最重要的节是来源、寄存器映射和未解决的问题。来源让读者评估文档的可信度。寄存器映射是核心参考。未解决问题告诉未来贡献者工作哪里需要更多关注。

### 11.4 来源部分的详细说明

来源部分值得特别关注，因为它是伪数据手册建立其可信度的方式。该节应该回答：

- 使用了什么信息来源？（捕获、已有代码、实验、公开文档。）
- 对每个来源应用了什么方法论？（直接阅读、差异比较、统计分析。）
- 哪些事实来自哪些来源？
- 对于每个事实，验证状态是什么？

一个有用的约定是用简短的验证标签标记寄存器映射和其他地方每个实质性事实：

- **HIGH**：由多次独立观察和实验确认。
- **MEDIUM**：由单一来源或单一实验确认。
- **LOW**：基于推理的假设，尚未直接测试。
- **UNKNOWN**：为完整性而陈述，但没有证据。

然后读者可以根据验证状态权衡每个事实，贡献者可以优先考虑在哪里投入更多调查。该约定维护成本很小，却赋予文档比将所有主张视为平等的平铺直叙更诚实的特征。

### 11.5 以表格形式记录寄存器

寄存器映射本身最好以表格形式呈现。对于每个寄存器，表格应该给出：

- 寄存器块内的偏移。
- 符号名称（驱动程序中的宏名称）。
- 大小（通常为 8、16、32 或 64 位）。
- 访问类型（只读、读写、只写、写 1 清除等）。
- 复位值。
- 一行描述。

每个寄存器的单独、更详细条目列出寄存器内的字段。例如：

```text
MYDEV_REG_CONTROL (偏移 0x10, RW, 32 位, 复位 0x00000000)

  位    名称         描述
  --------------------------------------------------
  0     RESET        写 1 触发复位。
  1     ENABLE       设置以启用正常操作。
  2-3   MODE         操作模式 (0=空闲, 1=接收, 2=发送, 3=两者)。
  4     INT_ENABLE   全局启用中断。
  31:5  reserved     读为零，写为零。
```

此表格格式是 FreeBSD 约定。它扩展性好，在 Markdown 中易于维护，是读者期望的格式。

### 11.6 伪数据手册作为活文档

伪数据手册很少是完整的。随着驱动程序发展，新行为被发现，旧事实被细化，边缘情况被表征。Git 跟踪的 Markdown 形式让文档自然演变，每次提交解释学到了什么以及何时学到的。

支持这一点的纪律是先更新伪数据手册，然后更新驱动程序以匹配。伪数据手册是规范；驱动程序是实现。当你发现寄存器的位 3 有你以前不知道的用途时，先将新的位 3 描述（带来源）写入伪数据手册，然后更新驱动程序使用它。顺序很重要，因为它迫使你将更改视为关于设备的事实，与对代码的更改分开。

随着时间的推移，伪数据手册成为可信制品，驱动程序成为文档指定约定的几种可能实现之一。新实现（在 NetBSD、Linux、自定义嵌入式代码中）可以仅从伪数据手册编写，而无需从头重新推导一切。

### 11.7 驱动程序版本管理

本书的示例使用如 `v2.5-async` 的版本字符串来标记驱动程序的主要迭代。对于逆向工程驱动程序，一个有用的约定是使用 `-rev` 后缀来指示工作的逆向工程性质，版本号跟踪实现的成熟度：

- `v0.1-rev`：最小驱动程序，仅复位和标识符。
- `v0.2-rev`：读取路径已实现并验证。
- `v0.5-rev`：大多数操作已实现，一些特性已理解。
- `v1.0-rev`：完整功能，所有已知特性已处理。
- `v2.1-rev`：稳定、成熟、重构的驱动程序，适合通用。

版本字符串可以出现在手册页、`MODULE_VERSION` 宏和伪数据手册中。它告诉操作员从任何给定构建期望什么级别的支持。

### 11.8 第11节小结

我们已经看到了如何将逆向工程项目的工作整合为可维护的驱动程序和独立的伪数据手册。驱动程序经过清理、在手册页中文档化，并加上版本号以指示其成熟度。伪数据手册以结构化形式捕获关于设备所学的内容，带来源，未来贡献者可以扩展。这两个制品共同证明了逆向工程项目所需的大量投入是合理的。

在转向动手实践之前，两个较短的节完善了理论材料。第 12 节以实践角度重新审视第 1 节介绍的法律和伦理框架，给你一套关于许可证兼容性、合同限制、净室纪律和安全港活动的紧凑规则。第 13 节然后通过 FreeBSD 源码树本身的两个实际案例研究，展示本章技术如何出现在当今生产中的驱动程序中。在这两节之后，本章剩余材料给你实验来应用所学、挑战练习来扩展理解、故障排除笔记来帮助出问题时、以及向前展望过渡到第 37 章，在那里我们将看到如何拿你刚刚构建的驱动程序并提交给 FreeBSD 源码树收录。

## 12. 实践中的法律与伦理护栏

第 1 节通过概述 FreeBSD 逆向工程发生的法律和伦理环境开始了本章。那个概述故意很宽泛，因为它必须在读者看到本章其余部分涵盖的任何技术工作之前介绍合理使用、互操作性研究和净室方法等概念。现在，在那项技术工作结束时，值得做第二次更实际的审视。本节的目标不是把你变成律师。它是给你一小组保护你、项目和代码读者的习惯，防止逆向工程工作可预测出错的方式。每个习惯都是具体的，每个都可以记录在第 11 节的伪数据手册中，每个都直接参考 FreeBSD 树过去如何处理类似问题。

### 12.1 为什么需要第二个法律章节

第 1 节回答了"这是否允许"的问题。第 12 节回答"我如何以保持允许的方式做这件事"的问题。这种区别很重要，因为许多原则上合法的活动如果毫无结构地进行，在实践中就会变得有风险。阅读另一个驱动程序来理解芯片是合法的。阅读另一个驱动程序然后凭记忆编写自己的，没有任何中间文档显示你的理解来自哪里，从外部看是一样的，但如果曾经提出问题就更难辩护。将专有数据手册作为参考是合法的。从该数据手册向驱动程序源代码注释引用长段落则不是。两种情况下的区别都是过程，而非意图。

本节其余部分通过四个具体领域进行：许可证兼容性（你被允许复制什么）、合同限制（你被允许披露什么）、净室实践（如何保持互操作性工作可辩护）和安全港活动（什么总是允许的）。下面的内容都不是特定情况下法律建议的替代品；它是有经验的 FreeBSD 开发者已经遵循的共同纪律，写在一个地方以便新作者可以采用而无需重建。

### 12.2 许可证兼容性

FreeBSD 内核使用宽松的 BSD 风格许可证。当你从另一个代码库将知识拉入你的驱动程序时，该其他代码库的许可证约束你被允许复制什么，即使它不约束你被允许学习什么。实践中出现的类别有 BSD、GPL、CDDL 和专有，每个都有不同的规则。

**BSD 许可来源。** 已经存在于 OpenBSD 或 NetBSD 树中的驱动程序，或同一设备较旧的 FreeBSD 移植，使用与 FreeBSD 内核兼容的宽松许可证。你可以直接复制代码，在版权块中适当署名。`/usr/src/sys/dev/usb/wlan/if_zyd.c` 的 `zyd` 无线驱动程序是一个具体例子：文件头保留了源码顶部的原始 `$OpenBSD$` 和 `$NetBSD$` 标签，表明代码是从这些树移植的，Damien Bergamini 的 BSD 风格版权与后来的 FreeBSD 贡献者版权一起保持完整。当许可证兼容时，在树之间移动代码是在 BSD 生态系统中摊销工作的合法方式，它已经是 FreeBSD 常规实践的一部分。

**GPL 许可来源。** Linux 驱动程序几乎总是在 GPL 下许可。你被允许阅读 GPL 代码来理解设备如何工作，因为阅读不是复制。你不被允许将 GPL 代码粘贴到 BSD 许可的驱动程序中，即使是少量、即使是临时的、即使带有署名。FreeBSD 项目的立场很明确：内核不接受不兼容许可证的代码。被发现包含复制的 GPL 代码的驱动程序将被拒绝，维护者将在未来的审查中失去信誉。规则严格是因为放宽它的代价将是树的许可证完整性。

**CDDL 许可来源。** OpenSolaris 和 Illumos 树中的某些设备驱动程序在 CDDL 下发布。CDDL 是文件级 copyleft，这意味着你不能在同一文件内自由混合 CDDL 源码和 BSD 源码。FreeBSD 中的 ZFS 代码是此规则的知名适应：CDDL 文件保存在 `/usr/src/sys/contrib/openzfs/` 下的自己的目录中，其许可证头被保留，周围的 BSD 许可粘合代码位于单独文件中。对于设备驱动程序，这种结构很少是合理的工作流程，所以更安全的规则是 CDDL 代码与 GPL 代码一样是只看不复制。

**专有来源。** 厂商 SDK、参考驱动程序和示例代码通常在禁止再分发的专有许可证下分发。即使厂商鼓励你将代码用作参考，该许可也不能转移到 FreeBSD 源码树，因为厂商不能代表 FreeBSD 的每个下游用户说话。阅读专有驱动程序来理解设备在一定限度内通常是可以接受的，但你不能从中粘贴，在许多司法管辖区你也不能长篇引用它。

实际规则很简单：将任何非纯 BSD 的东西视为只读。如果你想要驱动程序中有字节，这些字节必须来自你自己的输入或来自 FreeBSD 项目已经接受其许可证的来源。当你在伪数据手册中明确此规则，在每个寄存器描述旁边写下信息是从哪里观察到的，你创建的记录在多年后新维护者需要知道文件是如何变成现在的样子时仍然有用。

### 12.3 合同限制

合同可以在版权法不适用的情境下约束你。逆向工程工作中出现的合同有三种：保密协议、最终用户许可协议和固件许可证。每种约束工作流程的不同部分。

**保密协议。** 一些厂商会分享详细文档以换取签署的 NDA。文档通常很优秀，但 NDA 通常禁止发布信息，有时是无限期的，有时带有
附带惩罚性赔偿条款。签署关于某个设备的保密协议，然后为同一设备编写开源驱动程序，是一个法律雷区：你在伪数据手册中记录的每一个寄存器、每一个字段名、每一个值，都可能后来被质疑为泄密。更安全的选择几乎总是拒绝签署保密协议，并从公开可观察的信息中开展工作。如果你已经签署了保密协议，然后又想贡献一个开放驱动程序，干净的做法是让你自己回避观察工作，让第二位作者从头构建伪数据手册，只使用你在协议下从未接触过的来源。

**最终用户许可协议。** 许多厂商提供的驱动程序和 SDK 包含明确禁止逆向工程的 EULA。此类条款在不同司法管辖区的可执行性各不相同，但更安全的路径是一开始就避免点击"我同意"。如果你只接触过作为安装镜像附带的二进制文件的厂商驱动程序，而没有接受在线协议，你的法律地位比注册开发者计划并在禁止反汇编的协议下下载 SDK 要强得多。记录你参考了哪些厂商材料以及在什么条款下参考的。该记录将成为伪数据手册中来源字段的一部分。

**固件许可证。** 许多现代设备需要驱动程序在附加时加载的二进制固件映像。厂商通常在再分发许可证下分发该映像，该许可证比周围的驱动程序代码更宽松但仍然不是完全自由的。FreeBSD 的 `ports` 树有一个成熟的模式，即将固件映像打包在专门的 `-firmware-kmod` 端口中，这样内核代码保持 BSD 许可，而映像保留其厂商许可证。你不需要逆向工程固件本身；你需要理解驱动程序如何将固件交给设备。固件的法律框架与驱动程序的法律框架是分开的，两者都必须独立满足。

### 12.4 净室实践

阅读 GPL 驱动程序以了解设备功能，然后编写做同样事情的 BSD 代码，是一种公认的互操作性技术。美国法院在 Sega 诉 Accolade 和 Sony 诉 Connectix 等案件中支持了这一做法，欧盟软件指令（2009/24/EC）第 6 条专门为互操作性目的规定了逆向工程豁免。但法律保护取决于你如何开展工作，而不仅仅取决于你做了什么。一个逐行看起来与 Linux 原版相同的驱动程序不会受到净室辩护的保护，因为记录将显示不存在净室。

良好的净室实践有两个要素：结构分离和结构文档。

结构分离意味着阅读其他平台驱动程序的人和编写 FreeBSD 驱动程序的人是不同的活动。在两人团队中，他们实际上是不同的人。在个人项目中，它们是产生不同制品的不同工作会话。阅读者的输出是一份伪数据手册：一份用自然语言描述寄存器、位字段、命令序列和特殊行为的文档，不引用其他驱动程序使用的特定标识符。编写者的输入是该伪数据手册，加上 FreeBSD 头文件和驱动程序示例。编写者在编写 FreeBSD 代码期间从不打开其他平台的驱动程序。

结构文档意味着伪数据手册记录每个事实的来源。寄存器描述可能被标注为"2026-02-14 通过 `usbdump(8)` 在线路上观察到"或"从厂商数据手册第 3.4 节推断"或"从 Linux 驱动程序推导，转写为文字"。该标注就是记录，如果需要的话，它将支持你的驱动程序是基于理解而非复制的声明。专利和版权案件有时正是基于此类记录的质量来裁决的，维护它不是偏执；它是让你其余工程工作保持诚实的同样纪律。

在实践中，一旦第 11 节的伪数据手册习惯已经成为你工作流程的一部分，净室纪律的时间成本是很小的。法律价值却很大，因为它将你从"我希望这不是问题"转移到"我可以展示它是如何完成的"。

### 12.5 安全港

了解哪些活动总是安全的很有帮助，这样你就可以通过本节的结构处理不确定的情况，而不必对每次按键都犹豫不决。以下活动在 FreeBSD 项目关注的所有司法管辖区中都是可辩护的，你可以毫不犹豫地执行它们。

**阅读代码总是安全的。** 无论其他平台的驱动程序携带什么许可证，阅读它来了解设备如何工作不是版权侵犯。它有时被称为合理使用，有时被称为互操作性研究；标签取决于司法管辖区，但原则在 FreeBSD 项目可能遇到的每个法律体系中都是稳定的。

**观察你自己的硬件总是安全的。** 在你拥有的机器上运行你拥有的设备并记录总线上发生的事情，根据版权、合同或商业秘密法是不可诉的。你从 `usbdump(8)`、`pciconf(8)`、JTAG 探针和逻辑分析仪产生的记录是你自己的作品，你可以自由描述和发布它们。

**从理解编写总是安全的。** 如果你能用自己的语言在伪数据手册中描述设备，你就可以从该文档编写 FreeBSD 驱动程序而不受限制。编写是你自己的，源自关于设备的事实而非其他作者的表达。

**发布互操作性信息通常是安全的。** 记录寄存器布局、命令序列或线协议是发布关于设备的事实，而不是发布别人的代码。即使厂商不喜欢发布，他们很少有法律依据阻止它，美国和欧盟的法院一直支持互操作性研究。

此列表之外的活动并非自动不安全，但它们值得深思熟虑。有疑问时，扩大你的净室、加强你的文档，并在提交之前在 `freebsd-hackers` 邮件列表或项目聊天频道寻求第二意见。维护者已经多次看到这些问题，询问的成本很低。

### 12.6 第12节小结

我们已经对第 1 节介绍的法律和伦理框架进行了实际的第二次审视。许可证兼容性决定你可以复制什么；合同限制决定你可以披露什么；净室实践保持互操作性研究可辩护；简短的总是安全的活动列表给你无需持续担忧的工作空间。一旦第 11 节的伪数据手册已成为你工作流程的一部分，采用这些习惯就很简单，因为伪数据手册本身就是净室的审计跟踪。

有了这个基础，我们准备看看 FreeBSD 树本身如何记录逆向工程工作。下一节展示两个实际案例研究，都直接来自当前源码，驱动程序作者在注释中记录了他们的推理，至今仍然可读。

## 13. FreeBSD源码树中的案例研究

我们现在已经涵盖了技术、文档纪律和法律框架。是时候看看在完全相同约束下编写的两个真实驱动程序，看看它们的作者是如何处理这项工作的。两个驱动程序现在都在 FreeBSD 14.3 树中。两个都有头注释和内联注释，用作者自己的话记录数据手册没有说的内容以及驱动程序对此做了什么。阅读这些注释让你直接看到逆向工程纪律在生产代码中的样子，没有润色，没有事后修饰。

### 13.1 如何阅读这些案例研究

对于每个驱动程序，我们将看四件事。首先，设备和历史的简短描述，这样你知道涉及什么类型的硬件以及工作大致何时完成。其次，作者用来解决缺失文档的方法，包括咨询了什么来源和做了什么观察。第三，编码发现的具体代码，带有确切的文件路径、函数名称以及驱动程序今天使用的寄存器或常量标识符。第四，工作完成的伦理和法律背景，包括每个来源的许可证如何塑造作者被允许编写的内容。结尾段落然后将历史方法翻译成其现代形式，这样你可以看到你今天会如何用本章前面的技术处理相同问题。

下面的历史都不是推测性的。每个主张都锚定在你可以在自己 FreeBSD 系统上打开的文件中，关于代码的每个事实都可以在当前源码中直接观察。如果这里任何内容在未来版本中过时，文件本身仍将是真实来源，阅读它们的方法仍然适用。

### 13.2 案例研究1：`umcs`驱动及其未文档化的GPIO

**设备和历史。** MosChip MCS7820 和 MCS7840 是出现在廉价多端口 RS-232 适配器中的 USB 转串行桥接芯片。MCS7820 是双端口部件，MCS7840 是四端口部件，两种情况下 USB 接口在电气上是相同的。FreeBSD 驱动程序 `umcs` 由 Lev Serebryakov 于 2010 年编写，现在位于 `/usr/src/sys/dev/usb/serial/umcs.c`。配套头文件 `/usr/src/sys/dev/usb/serial/umcs.h` 详细说明了寄存器布局。

**方法。** `umcs.c` 顶部的头注释对文档情况异常坦率。作者写道驱动程序支持 mos7820 和 mos7840 部件，并直接指出公开数据手册不包含芯片的完整编程信息。由 MosChip 技术支持在限制下分发的完整数据手册填补了一些空白，厂商提供的参考驱动程序填补了其余部分。作者的任务是编写一个原创的 BSD 驱动程序，其行为与已确认信息所指示的一致，厂商驱动程序用作观察检查而非复制来源。

了解这种纪律的最佳位置是 attach 例程中的端口计数检测。驱动 USB 转串口芯片的程序必须知道芯片有两个端口还是四个端口，因为用户可见的设备节点和内部数据结构都取决于该计数。数据手册规定了一种方法，厂商驱动程序使用了另一种方法，而在真实硬件上的实验表明数据手册的方法不可靠。

**代码。** 在 `/usr/src/sys/dev/usb/serial/umcs.c` 中，函数 `umcs7840_attach` 执行检测。内联注释以纯文本记录了问题，代码记录了选择的变通方案。相关片段如下：

```c
/*
 * Documentation (full datasheet) says, that number of ports
 * is set as MCS7840_DEV_MODE_SELECT24S bit in MODE R/Only
 * register. But vendor driver uses these undocumented
 * register & bit. Experiments show, that MODE register can
 * have `0' (4 ports) bit on 2-port device, so use vendor
 * driver's way.
 */
umcs7840_get_reg_sync(sc, MCS7840_DEV_REG_GPIO, &data);
if (data & MCS7840_DEV_GPIO_4PORTS) {
```

涉及的寄存器是 `MCS7840_DEV_REG_GPIO`，在 `/usr/src/sys/dev/usb/serial/umcs.h` 中定义为偏移 0x07。头文件对其状态同样坦诚。寄存器块整体上有一个注释，解释这些寄存器仅在完整数据手册中有文档记录，而完整数据手册需要向 MosChip 技术支持请求获取，GPIO 寄存器特别被注释为保存公开数据手册中未文档化的 GPIO_0 和 GPIO_1 位。头文件更下方的一条更长注释解释了 `GPIO_0` 在双端口板上必须接地，在四端口板上必须上拉，该约定由电路板设计者而非芯片本身强制执行。单位标志 `MCS7840_DEV_GPIO_4PORTS` 定义为 `0x01`，是 attach 例程用来与 `umcs7840_get_reg_sync` 返回的值进行测试的标志。

使这成为一个优秀案例的原因是代码记录了完整的故事。在驱动程序编写十五年后发现它的读者可以重建推理过程：数据手册说了一件事，厂商驱动程序做了另一件事，在真实硬件上进行了测试，选择的方法在注释中做了解释。没有人需要重新发现相同的死胡同。

**伦理和法律背景。** 作者有两类来源可用：受限的完整数据手册（MosChip 按请求分发）和厂商参考驱动程序（随芯片评估套件提供）。两者都是专有的，因为它们不能自由再分发，两者都不能复制到开源驱动程序中。你可以做的是从中学习，然后根据它们包含的信息编写自己的代码。`umcs` 注释正是这样做的。它命名了厂商驱动程序展示的行为，命名了在真实硬件上确认该行为的实验，它产生的代码是原创的 BSD 许可作品。源文件顶部的版权块将 Lev Serebryakov 标识为唯一作者。这是教科书式的净室结果：驱动程序受益于厂商信息而不导入厂商代码。

**现代复现。** 如果你今天要编写 `umcs`，工作流程将是相同的，只是工具更好。你将捕获厂商驱动程序 attach 序列的 `usbdump(8)` 跟踪以查看确切的 GPIO 读取，你将在已知的双端口和四端口板上运行相同的读取，并将在伪数据手册中记录数据手册与观察行为之间的差异，每个来源都有清晰的来源行。然后驱动程序将实现观察到的行为，带有与 Lev Serebryakov 所写的精神相同的注释。技术没有改变；围绕它的工具已经改进。

### 13.3 案例研究2：`axe`驱动与缺失的IPG初始化

**设备与历史。** ASIX AX88172 是一款 USB 1.1 以太网适配器芯片，AX88178 和 AX88772 是其 USB 2.0 后继产品。基于这些芯片的廉价以太网适配器自 2000 年代初就很常见，FreeBSD 驱动程序 `axe` 位于 `/usr/src/sys/dev/usb/net/if_axe.c`。原始代码由 Bill Paul 在 1997 年至 2003 年间编写，AX88178 和 AX88772 支持由 J. R. Oldroyd 于 2007 年从 OpenBSD 回移。寄存器头文件与驱动程序并列位于 `/usr/src/sys/dev/usb/net/if_axereg.h`。

**方法。** `/usr/src/sys/dev/usb/net/if_axe.c` 中文件顶部的注释块部分内容写道，芯片手册中缺少驱动程序为了让芯片正常工作必须了解的信息。作者列出了两个具体事实。必须在 RX 控制寄存器中设置某一位，否则芯片将完全无法接收数据包。三个帧间间隔寄存器都必须初始化，否则芯片将完全无法发送数据包。这两个要求都没有出现在公开数据手册中，两者都是通过阅读厂商的 Linux 驱动程序和在真实硬件上观察芯片而确立的。

IPG 寄存器的故事特别清晰。帧间间隔是以太网中的一个标准概念：发送器在帧之间必须等待最少的比特时间数，具体数量取决于链路速度。芯片设计师有多种向软件暴露 IPG 的方式，AX88172 选择暴露三个独立的寄存器，驱动程序必须在初始化期间写入它们。数据手册命名了这些寄存器，但对需要编程它们一事只字未提，因此仅遵循数据手册的朴素驱动程序会将其保持在复位值，然后发现芯片神秘地拒绝传输。

**代码。** 初始化序列出现在 `/usr/src/sys/dev/usb/net/if_axe.c` 的 `axe_init` 函数中。一个名为 `axe_cmd` 的短小辅助函数提供了发出厂商特定 USB 控制请求的标准方式，`axe_init` 为每个 IPG 写入调用一次。相关分支如下：

```c
if (AXE_IS_178_FAMILY(sc)) {
    axe_cmd(sc, AXE_178_CMD_WRITE_NODEID, 0, 0, if_getlladdr(ifp));
    axe_cmd(sc, AXE_178_CMD_WRITE_IPG012, sc->sc_ipgs[2],
        (sc->sc_ipgs[1] << 8) | (sc->sc_ipgs[0]), NULL);
} else {
    axe_cmd(sc, AXE_172_CMD_WRITE_NODEID, 0, 0, if_getlladdr(ifp));
    axe_cmd(sc, AXE_172_CMD_WRITE_IPG0, 0, sc->sc_ipgs[0], NULL);
    axe_cmd(sc, AXE_172_CMD_WRITE_IPG1, 0, sc->sc_ipgs[1], NULL);
    axe_cmd(sc, AXE_172_CMD_WRITE_IPG2, 0, sc->sc_ipgs[2], NULL);
}
```

两个分支暴露了在数据手册中不可见的第二块逆向工程信息。在较旧的 AX88172 上，三个 IPG 值中的每一个作为独立命令编程：`AXE_172_CMD_WRITE_IPG0`、`AXE_172_CMD_WRITE_IPG1` 和 `AXE_172_CMD_WRITE_IPG2`，定义在 `/usr/src/sys/dev/usb/net/if_axereg.h` 中。在较新的 AX88178 和 AX88772 上，单个命令通过将三个值打包到同一个控制请求中来一次写入全部三个：`AXE_178_CMD_WRITE_IPG012`。两个操作码具有相同的数值；AX88178 重用了 AX88172 用于写入单个 IPG 寄存器的操作码槽位，并将其语义扩展为覆盖全部三个。将两个芯片系列视为可互换的驱动程序会破坏其中一个的 IPG 编程。区分两个系列的唯一方法是宏 `AXE_IS_178_FAMILY(sc)`，而对该宏的需求本身就是逆向工程工作的发现。

**伦理和法律背景。** axe 驱动程序有着悠久的历史，其来源记录在版权块中。Bill Paul 针对 AX88172 编写了原始驱动程序，注释显示他的信息来自公开数据手册和观察工作的混合。J. R. Oldroyd 的 AX88178 支持从 OpenBSD 的姊妹驱动程序移植，该驱动程序是 BSD 许可的，与 FreeBSD 直接兼容。该移植在法律上很直接：代码来自宽松的源码树，版权块在移动过程中被保留。关于 IPG 和 RX 控制的逆向工程注释随代码一起携带，因此发现本身被永远记录在驱动程序头中。

没有发生的事情与已经发生的事情同样有信息价值。axe 驱动程序不导入 Linux `asix_usb` 驱动程序的代码，即使该驱动程序在 Bill Paul 编写 FreeBSD 等效物时已经可用。Linux 使用 GPL，从中粘贴会使 axe 无法在内核的许可证规则下发布。阅读它来理解正是 Bill Paul 所做的，而 FreeBSD 驱动程序从头到尾是他自己的编写。

**现代复现。** 如果你今天要编写 axe，你将从在 Linux 测试机上加载 Linux 驱动程序并在共享同一 USB 集线器的 FreeBSD 机器上运行 `usbdump(8)` 开始，这样你就可以捕获厂商驱动程序的 attach 序列而无需阅读一行 GPL 代码。`usbdump(8)` 输出将直接显示三个 IPG 写入，因为它们作为可见的 USB 控制传输在线路上传输。你将在伪数据手册中记录观察结果，引用 pcap 文件作为来源，并在你自己的驱动程序中实现这些写入。`axe_init` 上方的注释读起来将与 Bill Paul 最初写的非常相似。

### 13.4 共同经验

阅读两个驱动程序可以发现一些值得明确指出的共同经验。

**注释就是审计跟踪。** 在 `umcs` 和 `axe` 中，头注释和内联注释是保存逆向工程故事的地方。没有这些注释，盯着 `MCS7840_DEV_REG_GPIO` 或 `AXE_178_CMD_WRITE_IPG012` 的未来维护者将无法知道信息来自哪里或驱动程序为什么这样做。在两种情况下，作者都花了额外的一分钟写下推理过程，结果是一个在编写十五或二十年后仍然可维护的驱动程序。作者的伪数据手册在每种情况下都部分被吸收到注释中，部分保留在头文件的结构中。这种吸收正是代码今天仍然可读的原因。

**观察胜过文档。** 当观察与数据手册不一致时，两个驱动程序都信任观察而非数据手册。`umcs` 信任经过实验验证的 GPIO 读取而非数据手册的 MODE 寄存器方法，`axe` 信任芯片在没有 IPG 编程时拒绝传输的观察而非数据手册在该主题上的沉默。这不是对文档的排斥；两位作者都仔细阅读了数据手册。这是对数据手册描述的是预期行为的认识，而芯片的实际行为才是驱动程序必须匹配的。

**许可证纪律不是事后想法。** 两个驱动程序都尊重它们所借鉴来源的许可证。`umcs` 将专有参考驱动程序用作观察检查而不从中复制。`axe` 从许可证兼容的 OpenBSD 导入 AX88178 和 AX88772 支持，并远离许可证不兼容的 Linux `asix_usb` 驱动程序。两个驱动程序都不包含 FreeBSD 项目在审查期间必须剥离的代码，两个都不需要律师审查就可以提交。第 12 节的习惯已经成为作者工作方式的一部分。

**工作是可复现的。** 两个驱动程序所做的一切都可以用现代工具复现。`usbdump(8)`、`usbconfig(8)` 和本章前面的总线空间跟踪技术将为你提供与原始作者必须手工收集的相同信息，第 11 节的伪数据手册习惯将为您提供比一组内联注释更清晰的审计跟踪。你今天为冷门硬件编写的驱动程序将比 2007 年和 2010 年的驱动程序文档更完善，因为技术已经成熟。然而，工作的精神是相同的。

### 13.5 第13节小结

两个案例研究展示了有纪律的逆向工程工作被吸收到 FreeBSD 源码树后的样子。在两种情况下，作者都识别了公开文档中的特定空白，通过观察确认了正确行为，将发现记录在至今仍然可读的注释中，并在与 FreeBSD 兼容的许可证下编写了原创代码。驱动程序可以工作，代码可维护，法律记录干净。

有了第 12 节的法律框架和第 13 节的实际案例，你现在拥有了可借鉴的原则和先例。本章剩余部分为你提供了案例研究所展示技术的动手实践、扩展理解的挑战练习、常见工作失误的故障排除笔记，以及通向第 37 章的过渡桥梁。

## 动手实验

本章的实验为你提供了安全、可重复地练习本章所涵盖技术的机会。没有任何实验以可能损坏任何东西的方式接触真实的未知硬件；实验中的"未知设备"要么是软件模拟，要么是你控制的无害内存区域。将每个实验视为在将特定技术加入你的工具箱之前内化它的机会。

配套代码位于 `examples/part-07/ch36-reverse-engineering/`。每个实验子文件夹都有自己的 `README.md`，包含分步说明，代码组织使你可以独立于其他实验构建和运行每个实验。与前面章节一样，第一次完成实验时请自己输入代码；配套文件作为参考和已知正确版本供你比较。

### 实验1：识别设备并转储描述符

这个实验是最简单的逆向工程练习。你将使用 `pciconf(8)` 和 `usbconfig(8)` 枚举 FreeBSD 系统上的每个设备，并转储你选择的一个设备的静态描述符。实验的输出是一个小型文本文件，记录了一个特定设备内核能告诉你的每一个公开事实。

步骤：

1. 运行 `sudo pciconf -lvc` 并保存输出。注意任何显示为 `noneN@...` 的设备，这表示没有内核树内驱动程序声明了它。
2. 运行 `sudo usbconfig` 并保存输出。选择一个你理解的 USB 设备（例如 USB 存储盘）并且你实际拥有的。
3. 运行 `sudo usbconfig -d ugen0.X dump_device_desc` 和 `sudo usbconfig -d ugen0.X dump_curr_config_desc` 来转储所选设备的描述符，其中 `ugen0.X` 是设备的 `usbconfig` 标识符。
4. 打开捕获的输出并识别每个字段：bDeviceClass、idVendor、idProduct、bMaxPacketSize0、bNumConfigurations、端点描述符等。
5. 编写一页摘要，识别设备的身份、其类别、其端点以及任何值得注意的特性。

实验有意设计得很简单。重点是在将静态身份捕获应用于真正未知的设备之前，内化其形式。

### 实验2：捕获USB初始化序列

本实验从静态身份转向动态行为。你将使用 `usbdump(8)` 捕获 USB 设备的初始化序列，保存捕获，并在 Wireshark 中探索它。

步骤：

1. 识别一个你可以自由附加和分离的 USB 设备（USB 存储盘是一个好选择，因为你可以毫无后果地拔插它）。
2. 运行 `sudo usbdump -i usbus0 -w stick-init.pcap` 开始在设备附加的 USB 总线上捕获。使用你系统的正确总线号；`usbconfig` 会告诉你设备在哪条总线上。
3. 在 `usbdump` 运行时，插入设备。等待内核识别它。然后拔出它。
4. 用 Control-C 停止 `usbdump`。
5. 在 Wireshark 中打开生成的 pcap 文件。你应该看到一系列对应枚举的 USB 传输：设备、配置和字符串描述符的 GET_DESCRIPTOR 请求、SET_ADDRESS 请求、SET_CONFIGURATION 请求等。
6. 识别每个传输并写下它的作用。将捕获的设备描述符与实验 1 中 `usbconfig dump_device_desc` 的输出进行比较；它们应该逐字段匹配。

本实验是所有 USB 逆向工程的基础。你将来研究的每个 USB 设备在附加时都会经过枚举序列，能够阅读枚举序列是理解设备在做什么的入口。

### 实验3：构建安全的寄存器探测模块

本实验介绍了你在真实项目中用于寄存器映射工作的内核侧工具。你将构建一个名为 `regprobe` 的小型内核模块，它在你选择的设备上分配内存资源，并暴露一个转储资源内容的 sysctl。不执行任何写入。

步骤：

1. 从 `examples/part-07/ch36-reverse-engineering/lab03-register-map/` 构建 `regprobe` 模块。
2. 识别你系统中不需要用于其他用途的 PCI 设备。备用网卡或任何未使用的 PCI 设备是理想选择。（不要使用支持你控制台或存储的设备。）
3. 使用 `sudo devctl detach <device>` 从设备分离内核树内驱动程序。
4. 使用实验 README 中的操作步骤将 `regprobe` 附加到设备。
5. 多次读取转储 sysctl，每次间隔几秒。比较转储并识别哪些字是稳定的、哪些字在变化。
6. 使用 `sudo devctl attach <device>` 重新附加内核树内驱动程序。

实验演示了两件事：只读探测是安全的，而且即使是只读探测也能揭示设备寄存器空间中有趣的结构。动态变化的字最可能是计数器或状态寄存器；稳定的字最可能是配置寄存器或标识符。

### 实验4：编写和驱动模拟设备

本实验介绍了工作流程中的模拟部分。你将构建一个完全在软件中模拟小型"命令和状态"设备的内核模块，以及一个通过 sysctl 驱动模拟设备的小型用户空间测试程序。

步骤：

1. 从 `examples/part-07/ch36-reverse-engineering/lab04-mock-device/` 构建 `mockdev` 模块。
2. 使用 `sudo kldload ./mockdev.ko` 加载模块。
3. 使用测试程序（也在实验文件夹中）向模拟设备发送几个命令。
4. 观察内核日志以查看命令被处理和模拟状态更新。
5. 修改模拟设备以引入一个故意的错误：让它对某个特定命令代码报告失败。验证测试程序是否正确检测到失败。
6. 修改测试程序以使用看门狗：如果模拟设备在超时时间内未完成命令，程序应报告失败而非挂起。

实验在模拟足够小到完全理解的设置中教授基于模拟的测试结构。在真实项目中，模拟变得更复杂，但结构是相同的。

### 实验5：构建安全探测包装器

本实验整合了第 10 节的安全技术。你将构建一个小型安全探测包装器库（读-改-写-验证、超时保护操作、失败时自动恢复），并使用它们在实验 4 的模拟设备上执行实验。

步骤：

1. 打开 `safeprobe` 实验文件夹。
2. 仔细阅读包装器库。注意每个操作如何受超时保护，每个寄存器写入如何后跟回读验证，以及失败如何被清晰报告。
3. 构建包装器库和使用它的示例驱动程序。
4. 加载实验 4 的模拟设备。
5. 对模拟设备运行示例驱动程序。观察包装器如何报告它们执行的操作。
6. 修改模拟设备以注入一个失败（例如意外的回读值）。验证包装器是否检测到失败并清晰报告，而不是让驱动程序以损坏状态继续。

本实验将安全包装器教授为你可以应用于自己驱动程序的工具。使用它们的代价很小；当出现问题时它们产生的信息量很大。

### 实验6：编写伪数据手册

本实验是技术实验在文档方面的等价物。你将采用实验 4 的模拟设备并为其编写伪数据手册，遵循第 11 节的结构。

步骤：

1. 打开 `examples/part-07/ch36-reverse-engineering/lab06-pseudo-datasheet/` 中的伪数据手册模板。
2. 仔细阅读模板并理解其结构。
3. 检查模拟设备的源代码以了解其寄存器布局、命令集和行为。
4. 用模拟设备的信息填写模板，遵循结构：身份、来源、资源、寄存器映射、缓冲区布局、命令接口、初始化、操作模式、特性和勘误、未解决问题、参考资料。
5. 将结果保存为与模拟设备源代码并列的 Markdown 文件。

模拟设备足够小，伪数据手册可以在一两小时内完成。练习教你文档的结构以及每节应有的详细程度。当你后来为真实设备编写伪数据手册时，你已经知道如何组织信息，即使信息本身会更加丰富。

## 挑战练习

挑战练习将实验中的技术扩展到更大、更开放的研究中。没有任何挑战需要特殊硬件；它们都使用真实的常见设备或实验中的模拟设备。慢慢来。实验给了你技术；挑战给了你在较少引导的环境中应用它们的实践。

### 挑战1：分类未知捕获

配套文件包含一小组未知设备 USB 流量的 pcap 捕获。在 Wireshark 中打开捕获。仅通过分析，识别：

- 设备的厂商标识符和产品标识符。
- 设备类别（HID、大容量存储、厂商特定等）。
- 设备的端点布局（哪些端点、什么类型、什么方向）。
- 设备数据流的总体形状（是否使用批量传输、控制传输、中断传输）。
- 数据中任何明显的模式（周期性传输、命令和响应、连续流）。

写一段话总结捕获来自什么类型的设备以及它似乎使用什么协议。捕获附有答案标签的文件，在你写出自己的答案之前不应查看；比较并反思。

### 挑战2：扩展模拟设备

采用实验 4 的模拟设备并将其扩展为支持通过小型环形缓冲区的多字节数据传输。环形缓冲区应有一个生产者指针（模拟在"生产"数据时前进，你可以通过以固定速率生成合成数据的 callout 来模拟）、一个消费者指针（驱动程序在消费数据时前进）和一个驱动程序可以读取以了解环形容量的"队列大小"寄存器。

编写一个小型驱动程序：

1. 识别模拟设备。
2. 读取队列大小寄存器以了解环形容量。
3. 定期读取生产者指针以了解有多少新条目可用。
4. 读取每个新条目并打印到内核日志。
5. 在处理每个条目后更新消费者指针。

练习在你控制两侧的设置中练习第 5 节的环形缓冲区识别技能。挑战是编写一个能正确处理空环情况（无新条目）和满环情况（条目数超过驱动程序已消费量）的驱动程序。

### 挑战3：仅从行为检测设备身份

实验 4 的模拟设备在其配套代码中有一个"神秘模式"，禁用标识符寄存器并更改一些行为以伪装其身份。在不阅读模拟设备源代码的情况下，编写一个小型驱动程序：

1. 探测神秘模式模拟设备。
2. 执行一系列安全的寄存器读取。
3. 观察设备对一组小型测试操作的响应。
4. 仅基于观察到的行为，识别模拟设备匹配三种已知设备身份中的哪一种。

挑战教授从行为而非显式标识符指纹识别设备的高级技能。真实设备有时出于兼容性原因（伪装成更常见的芯片）隐藏其身份，通过行为识别它们是正确支持它们的唯一方法。

### 挑战4：为你拥有的设备编写文档

选择一件你拥有的、功能不太敏感的 USB 硬件。USB 游戏控制器、网络摄像头、USB 转串口适配器、USB Wi-Fi 适配器。仅使用你通过 `usbconfig`、`usbdump` 和 Wireshark 能学到的内容，为其编写伪数据手册。

挑战要求你按顺序应用本章的每项技术：识别、捕获、观察、假设、文档化。结果是一个真实的制品：一个真实设备的伪数据手册。许多这样的伪数据手册已经成长为产生生产驱动程序的社区项目；你的可能是下一个。

## 动手练习：你自己的观测

实验已经让你对软件模拟和已知设备进行了受控练习，挑战练习要求你将这些技术应用于结构稍少的目标。第 13 节的案例研究带你浏览了早已被吸收到 FreeBSD 源码树中的逆向工程工作。以上这些都没有要求你坐下来选择一个自己的设备，从零开始冷观察它，将你看到的内容映射到接口和端点结构，并从头开始勾勒驱动程序的雏形。这就是本练习要做的，也是本章能让你最接近 `umcs` 和 `axe` 作者在其项目开始时所做工作的方式。

### 步骤1：使用`usbconfig(8)`进行初始识别

将目标设备附加到你的 FreeBSD 主机并确认内核已枚举它：

```console
$ sudo usbconfig list
ugen0.1: <0x1022 XHCI root HUB> at usbus0, cfg=0 md=HOST spd=SUPER (5.0Gbps) pwr=SAVE (0mA)
ugen0.2: <Example device Example vendor> at usbus0, cfg=0 md=HOST spd=FULL (12Mbps) pwr=ON (100mA)
```

记下你设备的 `ugenB.D` 坐标。该坐标是后续所有 `usbconfig` 命令寻址设备的方式。

接下来，检查 FreeBSD 驱动程序（如果有）是否已声明设备上的某个接口：

```console
$ sudo usbconfig -d ugen0.2 show_ifdrv
```

如果内核驱动程序已经拥有该接口，输出会命名它。接口被现有驱动程序声明的设备仍然会响应只读描述符查询，但在使用 `devctl detach` 分离现有驱动程序之前，不应针对它运行你自己的实验代码。作为第一轮，选择接口未被声明或其声明驱动程序你愿意分离的设备。

最后，转储完整的描述符树并保存以备后用：

```console
$ sudo usbconfig -d ugen0.2 dump_all_desc > device-descriptors.txt
```

输出列出设备描述符、每个配置描述符、每个接口描述符和每个端点描述符。你将在步骤 3 中将此文件与数据包捕获进行比较，并将其用作练习末尾简短伪数据手册片段的原始材料。

### 步骤2：使用`usbdump(8)`进行数据包级观测

描述符告诉你设备关于自己的宣传信息。要查看设备实际做了什么，在使用设备时捕获 USB 总线：

```console
$ sudo usbdump -i usbus0 -s 2048 -w trace.pcap
```

`-i usbus0` 参数选择要监听的 USB 总线；使用你设备附加到的总线号，由 `usbconfig list` 报告。`-s 2048` 参数将每个捕获的有效载荷限制在 2048 字节，这远远高于你在此练习中可能遇到的任何全速或高速端点的最大数据包大小。`-w trace.pcap` 参数写入 Wireshark 可以用内置 USB 解析器打开和解析的二进制 pcap 文件。

在 `usbdump` 运行时，操作设备。对于 HID 键盘，按几个键。对于 USB 转串口适配器，从另一个工具打开端口并在每个方向发送少量字节。对于 USB LED 外壳，循环 LED 穿过其支持的状态。每个操作在总线上产生 USB 传输，每个传输最终都进入捕获。

用 Control-C 停止 `usbdump`。在你手动标注传输时，人类可读的文本记录也很有用，所以用输出重定向到文本文件再运行一次捕获：

```console
$ sudo usbdump -i usbus0 -s 2048 > trace.txt
```

保留两个版本。文本形式更容易逐行阅读和在纯文本编辑器中标注；pcap 形式在你想跟踪特定流时更容易在 Wireshark 中导航。

### 步骤3：将描述符映射到接口和端点结构

打开 `device-descriptors.txt` 和 `trace.txt`。遍历描述符转储，为每个接口描述符写下：

- 设备或接口类（`bInterfaceClass`）、子类（`bInterfaceSubClass`）和协议（`bInterfaceProtocol`）。标准值在 USB 类规范中列出：HID 是类 `0x03`，大容量存储是类 `0x08`，通信设备类是 `0x02`，音频是类 `0x01`，厂商特定是类 `0xFF` 等。
- 每个端点的地址（`bEndpointAddress`）、方向（地址的高位：置位表示 IN，清除表示 OUT）和传输类型（`bmAttributes` 的低两位：`0` 控制、`1` 等时、`2` 批量、`3` 中断）。
- 每个端点的最大数据包大小（`wMaxPacketSize`）。

简单设备通常暴露一个接口和少量端点。USB HID 键盘通常有一个主机轮询按键状态报告的中断 IN 端点。USB 转串口适配器通常有一个用于设备数据的大批量 IN 端点、一个用于发送设备数据的大批量 OUT 端点，并使用控制端点进行线路状态命令。

将你写下的内容与捕获进行交叉检查。你在描述符转储中记录的每个端点地址应该在 `trace.txt` 中作为至少一次传输的源或目的地出现。你记录的每种传输类型（批量、中断、控制）应该以你期望的方向出现在捕获中。如果两者不一致，在勾勒任何驱动程序代码之前再看一遍；两个观察中有一个是错误的，你想找出是哪个。

### 步骤4：勾勒Newbus风格的驱动程序骨架

## 常见错误与故障排除

逆向工程有一小群几乎每个人早期都会犯的错误，以及同样一小群即使有经验也很容易滑入的错误。提前识别它们可以显著缩短痛苦的学习曲线。

### 错误1：轻信第一个假设

最常见的错误是早期形成一个假设，然后将每个后续观察解释为支持该假设，即使更仔细的阅读会建议不同的解释。确认偏见是人之常情，它在逆向工程中特别危险，因为观察是有噪声的，假设空间很大。

防御措施是明确写下每个假设、设计一个可以证伪它的测试、运行测试并诚实接受结果的纪律。如果测试没有证伪假设，假设存活，但它尚未被"证明"；它只是"尚未被反驳"。下一个测试可能仍然会推翻它。

### 错误2：跳过记录本

第二常见的错误是因为打字更快而跳过记录本。记录本感觉像是开销。结果在当下看起来很清晰。模式很明显。没有必要写下来，因为它们会被记住。

一周后，模式不再明显。又一周后，它们被遗忘了。一个月后，项目停滞了，因为没有人（包括作者自己）能重建当时已知的内容。记录本就是本可以防止这种情况的东西。跳过它后果自负。

### 错误3：写入未知寄存器

第三常见的错误是在没有足够证据表明写入安全的情况下写入寄存器。诱惑很强：你有一个假设，你想测试它，测试涉及一次写入，最坏情况能怎样？第 10 节已经说明了可能发生什么，但值得在这里重申。粗心的写入可能永久损坏设备，从经验中学习这一点的人通常付出高昂的代价。

防御措施是证据先于写入的纪律。在执行写入之前，说出证据的来源以证明写入是安全的。如果你说不出来源，就不要执行写入。

### 错误4：将厂商驱动视为完整

第四个常见错误是假设厂商驱动程序实现了设备的全部功能，并且匹配厂商驱动程序就足以完全支持设备。这通常是错误的。厂商驱动程序有时只实现厂商产品使用的设备功能子集；完整设备可能有厂商驱动程序从未使用的功能。反过来，厂商驱动程序有时包含你可能不需要的硬件错误变通方案（如果你以不同方式实现操作的话）。

防御措施是将厂商驱动程序视为众多信息来源之一，理解它代表的是厂商的选择，而非设备的完整描述。

### 错误5：不测试卸载路径

第五个常见错误是专注于可以看到进展的附加路径而忽视分离路径。结果是驱动程序可以加载和工作，但不能干净卸载。每次测试周期然后需要重启，这将工作速度降低了一个数量级。

防御措施是早期编写分离路径、在每次构建中测试它，并将卸载时的崩溃视为阻碍进一步工作的严重错误。无法卸载的驱动程序在实际意义上是需要重启才能测试更改的驱动程序，这不是一种高效的工作方式。

### 错误6：不保存捕获

第六个错误是因为捕获"没有显示任何新东西"而丢弃它们。它们今天可能没有显示任何新东西。六周后，当你学到了足够多的知识来解释它们时，它们可能显示重要的东西。保存每个捕获。存储很便宜。捕获本身是项目历史的一部分，它们有时是重建在给定时间点已知内容的唯一方式。

### 错误7：独自工作

第七个错误在初学者中特别常见，就是独自工作。当至少有另一个人可以讨论工作时，逆向工程要快得多，即使那个人不是逆向工程师。解释你观察到的内容、你相信的内容以及你即将测试的内容的行为会强制澄清解释，而澄清正是推动工作前进的东西。

如果你没有对此感兴趣的同事，找一个感兴趣的社区。该设备系列的邮件列表、IRC 频道和论坛充满了理解这些问题的人，有时可以提供缺失的那一块。第 9 节的社区协作方面不仅仅是消费已有成果；它也是参与帮助所有参与者的讨论。

### 故障排除：驱动程序静默无操作

有时驱动程序加载、设备看起来已附加、日志说一切正常，但没有出现预期的行为。最常见的原因：

- 驱动程序认为它已设置中断处理器，但中断实际上没有被传递。检查 `vmstat -i` 看中断是否到达。如果没有，中断设置有误。
- 驱动程序读取了错误的寄存器并看到看起来合法的值。与 `pciconf -r` 或 `regprobe` 输出比较以验证你读取的值与设备中存在的值匹配。
- 驱动程序对多字节值使用了错误的字节序。这在读取看起来像整数但实际是字节字符串的值时特别常见。
- 驱动程序的序列有错误：在步骤 A 之前执行了步骤 B，或完全跳过了必要的步骤。

每种情况下的防御措施是与已知参考比较：第 3 节的捕获、`regprobe` 转储、正常工作的厂商驱动程序。当驱动程序行为不符合预期时，问题是：它的行为在哪里首次偏离了参考？

### 故障排除：驱动程序卸载时崩溃

卸载时的崩溃通常表明驱动程序过早释放了某些东西或保留了对内核已销毁内容的引用。

最常见的原因：

- callout 在 softc 被释放之前没有被排干。检查每个 `callout_init` 在分离路径中都有匹配的 `callout_drain`。
- taskqueue 任务在 taskqueue 被销毁时仍然待处理。检查每个 `taskqueue_enqueue` 在分离路径中都有匹配的 `taskqueue_drain`。
- 中断处理器在 IRQ 资源被释放之前没有被拆除。顺序很重要：先用 `bus_teardown_intr` 拆除处理器，然后用 `bus_release_resource` 释放资源。
- 字符设备节点在 softc 被释放之前没有被销毁。如果设备可能仍然处于打开状态，使用 `destroy_dev_drain`。

这些与我们第 35 章异步驱动程序讨论的模式相同；逆向工程驱动程序面临相同的问题，还有一个额外的复杂因素是你可能尚未完全理解卸载时哪些资源在使用中。

### 故障排除：行为在不同运行之间不一致

有时设备在不同运行之间行为不同，即使驱动程序没有任何变化。原因通常是以下之一：

- 设备包含在运行之间未重置的状态。调查设备的复位实际清除了什么、保留了什么。
- 驱动程序有竞态条件，根据时序产生不同结果。
- 设备对运行之间略有变化的环境条件（温度、电压）敏感。
- 第二个驱动程序或用户空间程序也在访问设备，干扰了你的调查。

区分这些原因需要控制每个变量的仔细、重复运行。对行为不确定的设备进行逆向工程比确定的情况困难得多，识别非确定性的来源本身就是工作的一部分。

## 总结

逆向工程是一门建立在耐心、纪律和仔细记录基础上的技术。本章介绍了整个过程：为什么这项工作有时是必要的、法律界限在哪里、如何搭建实验室、如何系统地观察、如何构建寄存器映射、如何识别缓冲区结构、如何编写最小驱动程序并逐步扩展、如何在模拟中验证假设、如何与社区协作、如何避免损坏硬件以及如何将结果整合为可维护的驱动程序和可用的伪数据手册。

这些技术是具体的。纪律是将它们结合在一起的东西。遵循纪律的逆向工程师将能够拿一个无文档设备并为其生成可工作的驱动程序。跳过纪律的逆向工程师将生成有时能工作、因难以诊断的原因失败、且无法在不重新发现在第一次被遗忘的所有内容的情况下维护或扩展的代码。

本章中有几个贯穿始终的主题，值得最后明确总结。

第一个主题是将观察、假设和已验证事实分开。观察是你看到的东西。假设是你认为观察结果的含义。已验证事实是一个经受了故意证伪尝试后仍然存活的假设。将三者混合会产生混乱；将它们分开会产生清晰。本章强调的笔记本纪律归根结底是保持这三个类别不同的纪律。

第二个主题是从小处着手的价值。正确完成一件事的最小驱动程序比不确定地完成许多事情的大型驱动程序是更好的基础。逐步添加并验证的每个新功能比大型多功能更改更经济。逆向工程的工作节奏应该总是感觉比正常驱动程序开发的工作节奏慢；这种慢是捕获否则会存活到最终代码中的错误的东西。

第三个主题是安全的核心地位。逆向工程是粗心错误可能永久损坏真实硬件的少数软件活动之一。第 10 节的安全技术、第 8 节的模拟技术和第 4 节的先读原则都是同一基本原则的表现：尊重未知，并通过积累证据表明操作是安全的来获得执行操作的权利。

第四个主题是文档的重要性。伪数据手册、笔记本、手册页、代码中的注释：这些不是事后想法，它们是工作输出的一部分。没有文档的驱动程序是没人能维护的驱动程序，包括六个月后的作者本人。记录所学内容的伪数据手册是赋予工作超越作者参与的耐久性的制品。

第五个主题是社区。逆向工程很少是真正的单独工作。对于几乎所有值得研究的设备都存在先前的工作；做得好的新工作会贡献给持续的社区知识。搜索、评估和贡献是这门技术的一部分。孤立工作的逆向工程师在重新发明轮子；与社区接触的逆向工程师建立在他人基础上，反过来也被他人建立。

你现在看到了这项工作的完整形态。技术触手可及。纪律需要练习。第一个严肃的项目会很慢并充满错误；第二个会更快；到第三或第四个时，节奏会开始感觉自然。本章的实验是这种练习的开始，挑战练习是下一步。真实设备上的真实项目是技能巩固的地方，FreeBSD 社区有很多设备会受益于愿意做这项工作的人。

花一点时间回顾一下你的工具包发生了什么变化。在本章之前，一个无文档设备是一个停止标志。现在它是一个项目。你学到的方法与 `/usr/src/sys/dev/` 中许多驱动程序的作者用来将那些驱动程序变为现实的方法相同。他们中有的独自工作，有的在小团队中协作，但所有人都遵循可识别的相同工作流程：观察、假设、测试、记录。你现在知道如何完成这项工作了。

## 过渡到第37章：将你的驱动程序提交给 FreeBSD 项目

你刚刚学会构建的驱动程序，无论它是完全逆向工程的设备还是某种更常规的硬件，当其他人能够找到它、构建它并依赖它时，它才最有用。到目前为止，我们一直把驱动程序当作私人项目来处理，一些你加载在自己的系统上并为自己将来参考而记录的东西。下一章将改变这一点：我们将看到如何将一个完成的驱动程序提交给 FreeBSD 源代码树本身。

这种转变意义重大。位于你私人仓库中的驱动程序只服务于你和碰巧找到你仓库的人。位于 FreeBSD 源代码树中的驱动程序会被每个 FreeBSD 版本构建，暴露给每个 FreeBSD 用户，由 FreeBSD 项目维护，并由每个触及周围代码的提交进行测试。价值的放大是巨大的，随之而来的责任也是如此。提交过程是 FreeBSD 确保驱动程序在被允许进入之前达到源代码树标准的机制。

第37章将详细讲解这一过程。我们将了解 FreeBSD 的开发模型：贡献者和提交者之间的区别、源代码树组织的作用、审查过程以及 FreeBSD 源代码树所强制执行的约定。我们将学习风格指南（代码使用 `style(9)`，手册页使用 `style(4)`，以及一些关于 makefile、头文件和命名的相关约定），如何构建驱动程序的文件以便包含在 `/usr/src/sys/dev/` 中，如何编写每个驱动程序都应该附带的手册页，以及如何以 FreeBSD 期望的形式编写提交消息。

我们还将了解贡献的社会动态：如何与审查者互动、如何回应反馈、如何迭代补丁系列，以及如何以一种建立长期声誉的方式参与项目。提交过程的技术方面很简单；社会方面是大多数首次贡献者感到意外的地方。

本章的几个主题将继续延续。干净的分离路径是必不可少的，因为审查者会检查它。第10节的安全包装器和保守行为是生产级驱动程序所展现的相同纪律，审查者会欣赏这些。伪数据手册不是提交的一部分，但它所代表的理解是证明驱动程序声明合理性的基础，也是让审查者信任实现的基础。第9节的社区参与是提交旨在建立的长期关系的基础。

逆向工程的驱动程序是 FreeBSD 收录的一个特别有趣的案例，因为该项目对此类驱动程序有长期的经验和完善的处理流程。第11节的来源纪律正是项目评估驱动程序是否清除版权问题所需要的。伪数据手册中记录的限制和未解决的问题帮助项目为用户设定适当的期望。安全包装器和保守行为帮助审查者确信驱动程序不会损坏用户硬件。

如果你跟随了本章的实验和挑战，你已经产生了至少几段小型代码，经过一些润色后，它们可能是收录的候选者。第37章将向你展示如何将这样一段代码通过提交过程。实验4的模拟设备驱动程序可能不是一个候选者（它不驱动真实硬件），但它演示的模式正是真实驱动程序应该遵循的模式，实验5的安全包装器是审查者会识别和批准的模式。

本书即将进入结尾章节。你在第1部分开始时没有任何内核知识。你在第1和第2部分学习了 UNIX 和 C。你在第3和第4部分学习了 FreeBSD 驱动程序的结构和生命周期。你在第4和第5部分学习了总线、网络、存储和伪设备模式。你在第6部分详细学习了 Newbus 框架。你现在已经完成了第7部分的精通主题：可移植性、虚拟化、安全性、FDT、性能、高级调试、异步 I/O 和逆向工程。这套技能是全面的。剩下的章节将其联系回项目、社区和为现实世界操作系统做贡献的实践。

在进入第37章之前，花点时间回顾一下你现在能做什么。你可以从头开始编写一个简单的字符驱动程序。你可以编写一个参与内核网络栈的网络驱动程序。你可以编写一个通过 Newbus 发现的设备的驱动程序。你可以编写一个支持 `poll`、`kqueue` 和 `SIGIO` 的异步驱动程序。你可以用 `dtrace`、`KASSERT` 和 `kgdb` 调试驱动程序。你可以处理一个没有文档的设备。这些能力中的每一个都是真实的，建立在前一个之上，它们共同构成了 FreeBSD 设备驱动程序开发的工作知识。

下一章将帮助你将这些知识转化为对 FreeBSD 项目的贡献。让我们看看这个过程是如何运作的。
