# 模拟硬件

## 读者指南与学习成果

第16章以一个拥有寄存器块的驱动程序结束。版本为 `0.9-mmio` 的 `myfirst` 模块持有一块64字节的内核内存区域，其形状像一个设备：十个命名的32位寄存器、一小组位掩码、一个读取值为 `0x4D594649` 的设备ID、一个编码为 `1.0` 的固件版本，以及一个 `STATUS` 寄存器，其 `READY` 位在挂载时置位，在卸载时清除。每次访问都通过 `bus_space_read_4` 和 `bus_space_write_4` 进行，由熟悉的 `CSR_READ_4`、`CSR_WRITE_4` 和 `CSR_UPDATE_4` 宏封装。驱动程序的互斥锁（`sc->mtx`）保护每次寄存器操作。访问日志记录最后64次访问，一个定时器任务按需翻转 `SCRATCH_A`，一个小型的 `HARDWARE.md` 文档记录了整个接口。

那个驱动程序已经可以做很多事情了。它可以读取寄存器。它可以写入寄存器。它可以观察自己的访问，在调试内核上捕获越界偏移，并强制执行后续每一章都将依赖的锁定规则。它还不能做的是表现得像一个真正的设备。它的 `STATUS` 位从不自主改变。它的 `DATA_IN` 寄存器不会启动多周期操作。它的 `INTR_STATUS` 不会累积事件。写入 `CTRL.GO` 不会调度任何事情。模拟中没有任何东西有自己的心跳；寄存器块的每次变化都是上一次用户空间写入的直接结果。

真正的设备不是这样的。温度传感器每几毫秒刷新一次数值，无需任何人询问。串行控制器在字节落入接收FIFO时引发中断，这发生在驱动程序最后一次写入很久之后。网络卡的描述符环在驱动程序休眠时填满数据包。设备驱动程序的有趣之处在于对变化的响应，而不是产生变化。教导读者编写这些响应路径需要一个能够产生变化的模拟设备，而第17章正是模拟学习这样做的地方。

### 第17章的内容

第17章的范围狭窄但深入。它采用第16章的静态寄存器块，赋予它四个新属性：

- **自主行为**。定时器以驱动程序控制的节奏触发，更新 `STATUS` 位、`INTR_STATUS` 位或数据寄存器，就像真实设备的内部状态机在硅片中运行一样。读者可以在两次相邻的 sysctl 读取之间看到数值变化，即使在这两次读取之间没有发生用户空间写入。

- **命令触发协议**。写入 `CTRL` 中的指定位会调度延迟的状态变化。在可配置的延迟后，`STATUS.DATA_AV` 置位，并可选地锁存 `INTR_STATUS.DATA_AV`。这是每个命令-响应设备遵循的模式：一个写入开始某事，后来的状态变化信号表示某事完成。

- **真实时序**。微秒级的延迟在适当的情况下使用 `DELAY(9)`，在可以安全休眠的情况下使用 `pause_sbt(9)`。毫秒级的延迟使用 `callout_reset_sbt(9)`，这样在模拟设备工作时不会阻塞任何线程。每种选择都有其位置；每种选择都有其代价；第17章教授这两种。

- **故障注入**。一个 sysctl 让读者要求模拟失败。失败有多种形式：下一次读取返回固定的错误值，下一次写入永不置位 `DATA_AV`，一定比例的随机操作设置 `STATUS.ERROR`，一个 `STATUS.BUSY` 位无限期保持。每种模式都测试驱动程序中不同的错误处理路径，驱动程序学习注意、恢复或报告。

本章在第16章驱动程序的基础上构建这些属性，而不是替换它。相同的寄存器映射、相同的访问器、相同的 `sc->mtx`、相同的访问日志。增长的是模拟后端。第16章的 `myfirst_hw.c` 文件获得了一个兄弟文件 `myfirst_sim.c`，其中包含模拟的定时器、故障注入钩子和使寄存器块"呼吸"的辅助函数。到本章结束时，驱动程序版本为 `1.0-simulated`，它向驱动程序展示的硬件界面足够丰富，可以测试第3部分的几乎每一课。

### 为什么自主行为值得单独一章

在这个阶段，一个自然的问题是定时器驱动的行为和故障注入是否真的值得整整一章。第16章已经在驱动程序面前放置了一个寄存器块，第18章将用真实的 PCI 硬件替换该块。模拟不正是我们应该尽快跨过的垫脚石吗？

三个答案，每个都值得认真对待。

首先，第16章的模拟是故意静态的。它的存在是为了让读者练习寄存器访问，而无需同时管理设备行为。第17章是设备行为进入画面的地方，引入它的正确位置不是在真实 PCI 章节的中间，那里读者已经在处理 BAR 分配、供应商和设备 ID 以及 newbus 粘合代码。专门讨论模拟的一章给读者空间来推理设备如何行为、延迟如何塑造协议、故障如何传播，而不会带来真实硬件带来的干扰。

其次，模拟不是第18章之后丢弃的一次性脚手架。它是驱动程序开发的永久部分。真正的驱动程序在首次在真实硬件上运行很久之后，仍会针对模拟设备进行测试，因为模拟是产生确定性故障、可重现时序和按需故障条件的唯一方法。每个严肃的 FreeBSD 子系统都有某种形式的模拟或测试工具，每个工作的驱动程序作者都学习如何构建一个。第17章在小规模上教授该技术，读者可以看到每一个移动部件。

第三，第17章的模拟本身就是一种教学设备。通过编写一个假设备，读者被迫思考真实设备做什么：何时置位一个位，何时清除一个位，何时失败，何时锁存，何时遗忘。写过模拟的读者对协议的理解方式与仅使用过模拟的读者不同。本章既是关于像硬件一样思考的纪律，也是关于实现它的代码。

### 第16章留下的驱动程序

简要回顾一下你应该处于的位置。第17章扩展了第16章第4阶段结束时的驱动程序，标记为版本 `0.9-mmio`。如果以下任何项目感觉不确定，请在开始本章之前返回第16章。

- 你的驱动程序可以干净地编译，并在 `kldstat -v` 中标识为 `0.9-mmio`。
- softc 包含一个指针 `sc->hw`，指向一个 `struct myfirst_hw`，其中包含 `regs_buf`、`regs_size`、`regs_tag`、`regs_handle` 和第16章的访问日志。
- 驱动程序中的每次寄存器访问都通过 `CSR_READ_4(sc, off)`、`CSR_WRITE_4(sc, off, val)` 或 `CSR_UPDATE_4(sc, off, clear, set)` 进行。
- 访问器在调试内核上通过 `MYFIRST_ASSERT` 断言 `sc->mtx` 被持有。
- `sysctl dev.myfirst.0` 列出每个寄存器的 `reg_*` sysctl、可写的 `reg_ctrl_set` sysctl、`access_log_enabled` 开关和 `access_log` 转储器。
- `HARDWARE.md` 文档记录了寄存器映射和 `CSR_*` API。
- `LOCKING.md` 文档记录了卸载顺序，包括新的 `myfirst_hw_detach` 步骤。

这就是第17章扩展的驱动程序。添加的内容在代码行数上仍然是适度的：一个新的 `myfirst_sim.c` 文件、四个新的定时器、两个新的 sysctl 组、一个小型的故障注入状态结构和少量协议辅助函数。心理模型的变化比代码行数暗示的要大。

### 你将学到什么

完成本章后，你将能够：

- 解释在内核空间"模拟硬件"意味着什么，为什么模拟是驱动程序开发的永久部分而不是临时拐杖，以及什么是好的模拟与误导性模拟。
- 为模拟设备设计寄存器映射，包含控制、状态、数据、中断和元数据的独立部分，并根据设备应该实现的协议论证每个选择。
- 选择寄存器宽度、偏移量、位字段布局和访问语义（只读、只写、读即清除、写一清除、读/写），这些与真实 FreeBSD 驱动程序中使用的模式相匹配。
- 在内核内存中实现模拟硬件后端，具有自主行为，包括按节奏更新寄存器的定时器和对驱动程序发出的写入做出反应的任务。
- 使用与真实驱动程序相同的 `bus_space(9)` 抽象将假设备暴露给驱动程序，使驱动程序代码不知道它是在与硅片还是内核内存中的结构通信。
- 通过写入命令、轮询状态、读取数据和在各种负载下检查不变量，从驱动程序端测试设备行为。
- 使用 `DELAY(9)`、`pause_sbt(9)` 和 `callout_reset_sbt(9)` 向模拟添加时序和延迟，并解释为什么每种工具适用于使用它的时间尺度。
- 安全地模拟错误和故障条件，通过 sysctl 驱动的故障注入路径，可以重现超时、数据错误、永久忙碌状态和随机故障，并使用这些注入的故障来测试驱动程序的错误路径。
- 重构模拟使其位于自己的文件（`myfirst_sim.c`）中，在 `SIMULATION.md` 中记录其接口，将驱动程序版本定为 `1.0-simulated`，并运行完整的回归测试。
- 认识模拟的局限性以及只有真实硬件（或虚拟机监控程序支持的虚拟硬件）才能产生可信结果的场景。

列表很长；每一项都很狭窄。本章的重点是组合。

### 本章不涵盖的内容

有几个相关主题被明确推迟，以保持第17章的专注。

- **真实 PCI 设备**。`pci(4)` 子系统、供应商和设备 ID 匹配、通过 `bus_alloc_resource_any` 进行的 BAR 分配、`pci_enable_busmaster` 以及将驱动程序绑定到真实总线的粘合代码属于第18章。第17章停留在模拟中，使用第16章的标签和句柄快捷方式。

- **中断**。模拟产生模拟中断处理程序看到的状态变化。它还不通过 `bus_setup_intr(9)` 注册实际的中断处理程序，也不在过滤器处理程序和中断线程之间分配工作。第19章涵盖这一点。第17章通过定时器和用户空间系统调用进行轮询；定时器代表中断源。

- **DMA**。描述符环、散布-聚集列表、`bus_dma(9)` 标签、反弹缓冲区和 DMA 周围的缓存刷新是第20章和第21章的内容。第17章的数据寄存器仍然是一个单一的32位槽。

- **全系统模拟器**。QEMU、bhyve、virtio 设备和虚拟机监控程序提供的模拟类型作为通往第18章的桥梁简要提及。第17章保持在内核中；模拟运行在与驱动程序相同的内核中。

- **协议验证和形式方法**。真正的硬件验证使用形式化工具来证明驱动程序的协议处理是正确的。这些工具超出了本书的范围。第17章使用 `INVARIANTS`、`WITNESS`、压力测试和故意故障注入来建立信心。

- **虚拟机监控程序支持的设备模拟**。第17章的模拟运行在内核自己的地址空间中。VM 中的 Virtio 设备、QEMU 中模拟的 NIC 和 bhyve 的设备模拟都运行在虚拟机监控程序中；驱动程序通过真实 PCI 看到它们。那条路径是第18章的关注点。

保持在这些界限内使第17章成为关于使假设备表现得像真设备的一章。词汇是可转移的；特定子系统是第18章到第22章应用词汇的内容。

### 先决条件

在开始本章之前，确认：

- 你的驱动程序源代码与第16章第4阶段（`0.9-mmio`）匹配。起点假设每个第16章的寄存器访问器、`myfirst.c` 和 `myfirst_hw.c` 之间的分离、`CSR_*` 宏和访问日志都存在。

- 你的实验机器运行 FreeBSD 14.3，磁盘上有 `/usr/src`，并与运行内核匹配。

- 一个带有 `INVARIANTS`、`WITNESS`、`WITNESS_SKIPSPIN`、`DDB`、`KDB` 和 `KDB_UNATTENDED` 的调试内核已经构建、安装并干净启动。

- 你熟悉第13章的 `callout_init_mtx`、`callout_reset_sbt` 和 `callout_drain`，以及第14章的 `taskqueue`。

- 你在阅读层面上理解 `arc4random(9)`。如果是新的，在第7节之前浏览手册页。

如果上述任何项目不稳，现在修复它，而不是在尝试从移动的基础上推理第17章时解决。模拟代码会产生时序敏感的错误，调试内核加上坚实的第3部分基础可以在第一次接触时捕获大多数错误。

### 如何从本章获得最大收益

三个习惯会很快带来回报。

首先，在你测试的机器上保持打开第二个终端。模拟使寄存器在两次 sysctl 读取之间变化，唯一能看到这一点的方法是连续两次读取同一个 sysctl。每200毫秒循环 `sysctl dev.myfirst.0.reg_status` 的 shell 是本章示例生动起来的方式。如果不能并排运行两个终端，tmux 窗格中的 `watch -n 0.2 sysctl ...` 循环也可以。

其次，每次实验后阅读访问日志。日志用时间戳、偏移量、值和上下文标签捕获每次寄存器访问。对于静态模拟，日志很无聊：用户空间读取、用户空间写入，没有别的。对于第17章的模拟，日志很密集：定时器触发、延迟事件触发、故障注入、超时过期。每个有趣的行为都留下痕迹，痕迹是本章直觉的来源。

第三，故意破坏模拟。将故障注入率设置为100%。将定时器间隔设置为零。使模拟设备永远忙碌。观察驱动程序如何反应。本章的错误处理课程在你看到失败模式上演时会清晰得多。模拟是安全的：你可以注入任意多的故障，而不危及真实硬件，最坏的情况是调试内核干净捕获的内核恐慌。

4. **将假设备暴露给驱动程序**。标签、句柄、访问器路径，以及第16章抽象如何在没有修改的情况下继续。简短的一节；重点是第16章的工作已经完成了大部分。

如果是第一次阅读，请线性阅读并按顺序完成实验。如果是重温，第3、6和7节相对独立，适合单独阅读。



## 第1节：为什么要模拟硬件？

第16章已经将模拟作为入门路线的理由。不拥有本书引用的精确设备的读者仍然可以针对模拟块练习寄存器访问。这个论点有效，在第17章仍然成立，但这不是模拟获得专门一章的唯一原因。模拟是严肃驱动程序开发的一部分，理解为什么是本节的内容。

### 工作驱动程序作者的工具包

选择任何真实的 FreeBSD 驱动程序，从提交历史向后追溯。大多数积极开发的驱动程序共享一个共同模式：最初的提交针对真实硬件构建驱动程序。后来的提交，通常是很多后来的提交，添加测试、工具、脚手架和仪表，让驱动程序可以在没有原始硬件坐在开发者桌上的情况下被测试。其中一些仪表在用户空间运行。其中一些是虚拟机监控程序支持的假设备，它说与真实设备相同的协议。其中一些是一组内核端存根，它们足够好地假装是硬件来运行驱动程序的状态机。

为什么工作的驱动程序作者要投资这些？因为真实硬件不可重现。真实设备的行为取决于其固件版本、温度、链路伙伴、物理连接的信号完整性以及月相。通过一个设备实例的测试的驱动程序可能会在稍有不同的实例上失败。今天通过测试的驱动程序明天可能会失败，因为设备的固件自动更新在一夜之间完成。失败与时序相关的驱动程序可能会通过一百次运行，而在第一百零一次失败。真实硬件是一个移动目标，而移动目标不是回归测试的良好基础。

模拟固定了目标。模拟设备在相同输入到达时每次都表现相同。模拟设备的故障条件在作者的控制之下。模拟设备可以在紧密循环中运行而不会磨损。模拟设备不需要实验室硬件、电缆、电源或你需要帮助的同事。使模拟便于学习的每个属性都使其对生产测试有价值。

结论是第17章的模拟不是一次性的脚手架。你在这里学习的模式是 FreeBSD 提交者用来测试没有人再能访问的驱动程序的相同模式，因为硅片已有二十年历史。相同的技术在到达客户机器之前捕获回归。应用于第18章真实 PCI 设备的相同规则，让驱动程序可以在没有网络的工作站上通过告诉 virtio-net 后端以受控速率丢弃数据包、损坏帧或复制描述符来测试。

### 模拟目标：行为、时序、副作用

当我们说"模拟一个设备"时，我们通常指三件事之一，值得明确我们的意图。

第一种模拟是**行为性**的。模拟暴露一个寄存器接口，写入和读取产生与真实设备相同的效果。状态机是正确的：写入 `CTRL.GO` 将设备从空闲移到忙碌再移到完成，在正确的点设置 `STATUS.DATA_AV`。数据路径是正确的：写入 `DATA_IN` 的数据经过一些处理后显示在 `DATA_OUT` 中。模拟不必匹配真实设备的时序；它只需要遵循相同的规则。

第二种模拟是**时序感知**的。模拟强制执行真实设备会强制执行的延迟。真实设备需要500微秒处理的写入在模拟中需要500微秒，或多或少，受 CPU 自身时序分辨率的限制。真实设备在命令后2毫秒置位的状态位在模拟中在命令后2毫秒置位。通过时序感知模拟的驱动程序很可能在真实硬件上工作；仅通过行为模拟的驱动程序可能会在真实设备的延迟产生驱动程序未考虑的竞争时失败。

第三种模拟是**副作用感知**的。模拟反映协议的微妙部分：清除状态的读取、无效的写入、当设备处于某种模式时返回全一的寄存器、写一清除与读即清除的位。通过这种模拟的驱动程序对低级协议细节有信心，而不仅仅是高级状态流。

三种类型叠加。现实的模拟是三种：行为、时序和副作用。第17章构建所有三种的模拟，尽管时序精度受内核调度机制所能提供的限制。目标不是逐微秒匹配硅片；目标是使模拟足够现实，以便驱动程序的重要协议路径被测试，这些路径包含的竞争条件有机会显现。

### 好的模拟教会什么和不教会什么

好的模拟教会协议。它测试驱动程序状态机的每个分支、每个错误路径、每个状态检查、每个延迟容忍度。针对好的模拟编写的驱动程序对模拟覆盖的场景是健壮的。

好的模拟不教硅片。CPU 的内存层次结构、设备的内部流水线、PCB 走线的信号完整性、与系统其余流量的交互、芯片在负载下的热行为：这些都不在模拟中。通过模拟的驱动程序仍然可能在真实硬件上失败，如果失败根源于物理。

这个区别很重要。第17章的模拟是协议的教学工具，它产生的驱动程序是学习的载体，而不是生产网络驱动程序。第20章到第22章引入 DMA、中断和真实设备性能关注点，所有这些都扩展了需要真实硬件来完全验证的模拟方向。第17章是一个更长进程的第一步：先正确处理协议，然后正确处理真实硬件细节。

还有一个微妙之处值得命名。如果模拟与真实设备不一致，模拟可能会教授*错误*的协议。学会期望模拟行为的读者，然后遇到行为分歧的真实硬件，会感到困惑。第17章的模拟故意设计为匹配真实 FreeBSD 驱动程序中常见的模式：`STATUS.READY` 在设备忙碌时清除，在准备好时设置，`STATUS.DATA_AV` 锁存直到驱动程序读取数据，`INTR_STATUS` 是读即清除，`CTRL.RESET` 是写一自清除。这些模式不是通用的；不同的设备选择不同的约定。但它们是常见的，处理这些模式良好的驱动程序已经学会了可转移的技能。

### 无硬件依赖：实际好处

对于本书，模拟最直接的好处是读者不需要特定硬件就能跟上。在第16章使用没有外部设备的笔记本电脑、没有 PCI 直通的虚拟机、只有板载外设的 ARM 单板计算机上工作的读者，仍然可以构建驱动程序并观察其行为。第17章保留了这个属性。模拟完全存在于内核内存中；驱动程序只需要一个运行 FreeBSD 14.3 的系统。

这对于教学书籍很重要。必须停下来、订购硬件、安装并等待交付才能继续的读者可能不会继续。可以在阅读章节的同一会话中构建和运行驱动程序的读者正在学习。第17章的每个实验和挑战都设计为在任何 x86 FreeBSD 系统上运行，无需外部硬件。

次要好处是模拟让书籍可以超越读者硬件的能力。第17章的驱动程序测试真实设备在正常操作下几乎不会触发的错误路径。拥有生产质量网络卡的读者很难强制卡进入第7章故障注入教学的失败模式。模拟设备可以被命令按需失败，频率任意，在任何读者想要测试的条件下。学习面更广。

### 完全控制：教学好处

模拟给读者比真实硬件更多的控制。读者可以在模拟中冻结时间，向前步进已知的刻度数，并观察每个中间状态。读者可以禁用模拟的自主行为，进行单一操作，然后重新启用。读者可以注入特定故障，观察驱动程序如何反应，然后通过切换 sysctl 修复故障。这对真实硅片是不可能的。

这种控制水平在教学上很有价值。解释"当设备忙碌时，驱动程序应该等待 `STATUS.READY`"的一章可以具体演示这个场景。第17章正是这样做的：一个 sysctl 告诉模拟进入假忙碌状态，读者可以在驱动程序耐心等待忙碌位清除时输入命令。在真实硬件上，可靠产生这个场景需要特定的工作负载（设备会拒绝）、人工测试夹具或调试器。在模拟中，它是一行 sysctl 更改。

控制还意味着读者可以无后果地实验。将寄存器设置为非法值、发出设备不期望的命令、要求驱动程序访问映射范围之外的寄存器：其中每一个在模拟中都是安全的。最坏情况是 KASSERT 触发和调试内核干净捕获的恐慌。读者可以在一小时内尝试十个坏主意，从每个中学习，并因此变得更好。真实硬件的实验带有真正的风险，读者（合理地）在那里更加谨慎。

### 安全实验：风险降低好处

"安全"这个词值得展开。来自模拟设备的内核恐慌是尴尬的；来自真实设备的内核恐慌可能更糟。向真实设备的 `CTRL` 寄存器写入错误位的驱动程序可能会变砖设备。损坏真实设备 DMA 配置的驱动程序可能会向任意 RAM 写入垃圾。错误处理真实中断的驱动程序可能会让系统处于合法中断被无限期错过的状态。这些失败模式多年来都出现在 FreeBSD 生产中，每一个都需要内核更新来修复。

模拟消除了大部分这种风险。模拟设备是内核内存中的结构；错误位进入不连接任何其他东西的内存。模拟 DMA 还不存在（它在第20章到达）。模拟中断是驱动程序可以随时停止的定时器。编写第一个驱动程序的初学者可以犯驱动程序开发生命周期中的每一个错误，而不会造成物理问题。

这值得内化。第17章的模拟给读者激进实验的许可，这是初学者建立信心需要做的。从未破坏过驱动程序的读者是还没有学会驱动程序如何破坏的读者。第17章是读者既学会如何故意破坏驱动程序，也学会如何在它自己破坏时识别的章节。

### 第17章将模拟什么

本章将引入的特定行为的简短列表。每一项都在讨论它的节中添加；这里的列表是路线图。

- **自主 `STATUS` 更新**。一个定时器每100毫秒触发并更新 `STATUS` 中的字段，就像后台硬件监视器正在采样设备的内部状态。驱动程序可以观察更新；测试套件可以断言它们。

- **命令触发延迟事件**。写入 `CTRL.GO` 调度一个定时器，在可配置延迟（默认500毫秒）后，设置 `STATUS.DATA_AV` 并将 `DATA_IN` 复制到 `DATA_OUT`。这是教科书式的"异步设备"模式。

- **读即清除 `INTR_STATUS`**。读取 `INTR_STATUS` 返回当前待处理中断位，然后原子地清除它们。写入 `INTR_STATUS` 无效（真实设备各不相同；有些允许写一清除，这是我们将讨论的常见变体）。

- **模拟传感器数据**。一个 `SENSOR` 寄存器持有模拟按节奏更新的值，模仿温度或压力读数。驱动程序可以轮询它；用户空间工具可以绘制它。

- **超时容忍**。驱动程序的命令路径等待最多可配置毫秒数的 `STATUS.DATA_AV` 置位。如果模拟配置为延迟超过超时，驱动程序报告超时并恢复。

- **随机故障注入**。一个 sysctl 控制任何操作注入故障的概率。故障类型包括：命令后设置 `STATUS.ERROR`、下一个命令完全超时、读取返回 `0xFFFFFFFF`、设备无限期报告假忙碌。每个故障测试不同的驱动程序路径。

所有这些行为都在第8节实现。本章逐一构建它们，以便读者可以在继续之前内化每一个。

### 模拟在本书中的位置

第17章位于第16章（静态寄存器访问）和第18章（真实 PCI 硬件）之间。它是抽象词汇和具体设备之间的桥梁。完成第17章的读者拥有一个已经测试了本书将讨论的几乎每个协议模式的驱动程序，针对一个行为像真实设备的模拟设备。那个读者带着在结构上已经接近生产级的驱动程序走进第18章，只需要真实 PCI 粘合代码就可以在实际硅片上运行。

模拟也是一个参考。当第18章用真实 PCI BAR 替换模拟寄存器块时，读者可以将模拟的行为与真实设备的行为进行比较，并注意差异。模拟是标尺。当第19章添加中断时，模拟的定时器驱动状态变化成为中断处理程序响应的模型。当第20章添加 DMA 时，模拟的数据寄存器成为 DMA 引擎读取和写入的模型。模拟是每一章后续扩展的教学词汇。

### 第1节总结

模拟是驱动程序开发的永久部分，不是临时脚手架。它给驱动程序作者可重现的测试、可控的故障注入和测试真实硬件很少产生的错误路径的能力。第17章在第16章驱动程序之上构建三种模拟（行为性、时序感知、副作用感知）。模拟故意设计为匹配真实 FreeBSD 驱动程序中的常见模式，以便读者的直觉稍后转移到真实硬件。

本章的具体目标包括自主 `STATUS` 更新、命令触发延迟事件、读即清除语义、模拟传感器数据、超时容忍和随机故障注入。每一项都在需要它的节中引入；它们共同产生一个足够丰富的驱动程序来测试几乎每一课第3部分的内容。

第2节迈出下一步。在我们模拟设备行为之前，我们需要一个行为操作的寄存器映射。第16章给了我们一个静态映射；第17章将其扩展为动态行为。



## 第2节：设计寄存器映射

第16章给模拟设备一个寄存器映射。第17章的工作扩展该映射，但扩展不是自动的。向真实设备添加寄存器需要一组硬件工程师、硅片重新流片和新的数据手册。向模拟设备添加寄存器在机械上很容易，但设计工作（寄存器应该做什么？谁写入？谁读取？它有什么副作用？）是一样的，做好它的纪律也是一样的。

本节通过设计，而不仅仅是结果。到第2节结束时，你将有一个支持第1节列出的每个行为的寄存器映射，文档足够好，第3到7节的实现几乎机械。练习是可转移的：每次你阅读真实数据手册时，你会认识到硬件设计师所做的决定，你会有一个评估决定是否明智的词汇。

### 什么是寄存器映射？

寄存器映射是设备寄存器接口的完整目录。它列出每个寄存器、每个偏移量、每个宽度、每个字段、每个位、每个访问类型、每个复位值、每个副作用。真实设备的寄存器映射可能有数百页；简单设备的寄存器映射可能只有一页。两者的形式相似：一个表或一组表，带有回答驱动程序作者需要问的问题的标题。

寄存器映射不是散文。它是参考文档。驱动程序作者在编写代码时查阅它，测试作者在编写测试时查阅它。模糊、不完整或模糊的寄存器映射产生难以诊断方式错误的驱动程序。精确的寄存器映射产生与设备期望匹配的驱动程序。

对于模拟设备，寄存器映射扮演一个额外的角色。它也是测试模拟行为的规范。如果模拟与寄存器映射不一致，模拟是错误的，必须修复。如果驱动程序与寄存器映射不一致，驱动程序是错误的，必须修复。映射是合同。映射的更改需要对模拟和驱动程序都进行更改。这正是真实设备开发的工作方式：数据手册是合同，硬件团队、固件团队和驱动程序团队都遵循它。

### 第16章映射作为起点

回顾第16章的寄存器映射：

| 偏移量 | 宽度  | 名称            | 访问类型    | 第16章行为                           |
|--------|--------|-----------------|-----------|------------------------------------------------|
| 0x00   | 32 位 | `CTRL`          | R/W       | 普通读写内存。                       |
| 0x04   | 32 位 | `STATUS`        | R/W       | 普通读写内存。                       |
| 0x08   | 32 位 | `DATA_IN`       | R/W       | 普通读写内存。                       |
| 0x0c   | 32 位 | `DATA_OUT`      | R/W       | 普通读写内存。                       |
| 0x10   | 32 位 | `INTR_MASK`     | R/W       | 普通读写内存。                       |
| 0x14   | 32 位 | `INTR_STATUS`   | R/W       | 普通读写内存。                       |
| 0x18   | 32 位 | `DEVICE_ID`     | 仅读    | 固定 `0x4D594649`。                            |

十个寄存器，40字节定义空间，64字节分配留出增长空间。每次访问都是直接的：读取返回上次写入存储的内容，写入直接进入内存。读取无副作用。写入无副作用。寄存器后面没有硬件状态机。

第17章改变这个映射的两件事。首先，现有寄存器增长行为语义：`STATUS.DATA_AV` 将由模拟设置，而不仅仅是驱动程序的写入；`INTR_STATUS` 变为读即清除。其次，映射添加第17章模拟需要的一些新寄存器：一个 `SENSOR` 寄存器、一个 `DELAY_MS` 寄存器、一个 `FAULT_MASK` 寄存器等。第16章的64字节分配已经有空间；不需要分配器更改。

### 寄存器访问语义

在我们编写任何寄存器之前，我们应该命名我们关心的访问语义。寄存器的访问语义是驱动程序读取或写入它时发生什么的规则。常见使用的主要类别是：

**只读（RO）**。寄存器返回设备或模拟产生的值。写入被忽略或产生错误。`DEVICE_ID` 和 `FIRMWARE_REV` 是例子。

**读写（RW）**。寄存器存储驱动程序写入的任何内容。读取返回上次写入的值。`CTRL`、`SCRATCH_A` 和 `SCRATCH_B` 是例子。

**只写（WO）**。寄存器接受写入，但读取返回固定值（通常是零，有时是垃圾，偶尔是从不同来源读取的上次值）。`DATA_IN` 在真实设备中通常是只写。

**读即清除（RC）**。读取寄存器返回其当前值然后清除它。写入通常被忽略。`INTR_STATUS` 是经典例子。

**写一清除（W1C）**。向位写入1清除该位；写入0无效。读取返回当前值。某些设备上的 `INTR_STATUS` 使用 W1C 而不是 RC。

**写一设置（W1S）**。写入1设置该位；写入0无效。比 W1C 不常见，但用于某些硬件的控制寄存器，驱动程序不应该做读-改-写。

**带副作用读写（RWSE）**。寄存器可读可写，但写入（或读取）触发超出普通内存更新的内容。`CTRL` 通常属于此类：写入 `CTRL.RESET` 触发设备复位，这是副作用。

**粘性（锁存）**。一旦被硬件设置，位保持设置直到驱动程序明确清除。`STATUS` 中的错误位通常是粘性的。`STATUS.ERROR` 在第17章映射中将是粘性的。

**保留**。位没有定义的含义。写入应该保持现有值或写入零；读取可能返回任何值。忽略保留位的驱动程序对未来硬件修订是健壮的；依赖保留位行为的驱动程序是脆弱的。

每种访问类型对驱动程序代码都有含义。读取 RC 寄存器的驱动程序应该恰好在协议要求时这样做，因为每次额外读取都消耗一个事件。读取 WO 寄存器的驱动程序正在读取垃圾，如果值曾被操作，这是一个等待出现的错误。在粘性位寄存器上做读-改-写的驱动程序必须小心不要清除它不打算触摸的位。写入 W1C 寄存器的驱动程序使用 `CSR_WRITE_4(sc, REG, mask)` 而不是 `CSR_UPDATE_4(sc, REG, mask, 0)`；两者看起来相似但在硬件上产生不同行为。

第17章的模拟使用 RO、RW、RC、粘性和 RWSE 语义。它不引入 W1C 或 W1S（尽管挑战邀请读者添加它们）。模拟的目标是彻底覆盖常见情况；不太常见的情况在理解常见情况后成为自然扩展。

### 第17章添加

第17章寄存器映射添加以下寄存器：

| 偏移量 | 宽度  | 名称            | 访问类型    | 第17章行为                                        |
|--------|--------|-----------------|-----------|-------------------------------------------------------------|
| 0x28   | 32 位 | `SENSOR`        | RO        | 模拟传感器值。由定时器每100毫秒更新。 |
| 0x2c   | 32 位 | `SENSOR_CONFIG` | RW        | 传感器更新间隔和幅度设置。              |
| 0x30   | 32 位 | `DELAY_MS`      | RW        | 设备每个命令需要的毫秒数。        |
| 0x34   | 32 位 | `FAULT_MASK`    | RW        | 哪些模拟故障被启用。位字段。               |
| 0x38   | 32 位 | `FAULT_PROB`    | RW        | 每个操作的随机故障概率（0..10000）。       |
| 0x3c   | 32 位 | `OP_COUNTER`    | RO        | 模拟设备已处理的命令计数。       |

六个新寄存器，使用偏移量 `0x28` 到 `0x3c`，完全适合第16章分配的64字节区域。无 softc 更改，无分配器更改，无 `bus_space_map` 更改。

每个寄存器有特定工作：

- **`SENSOR`** 是模拟温度、压力或电压。确切解释对模拟不重要；重要的是它自己变化，教导读者如何处理值自主产生的寄存器。值使用定时器将实现的简单公式围绕基线振荡。

- **`SENSOR_CONFIG`** 让驱动程序（或用户通过 sysctl）控制传感器的行为。低16位是以毫秒为单位的更新间隔；高16位是振荡幅度。值 `0x0064_0040` 意味着"100毫秒间隔，幅度64"。更改寄存器会更改模拟的下一次更新。

- **`DELAY_MS`** 是模拟中命令需要多长时间。写入 `CTRL.GO` 调度 `STATUS.DATA_AV` 在 `DELAY_MS` 毫秒后置位。默认值是500。设置为0使命令在下一个定时器刻度完成；设置为较大值让读者测试驱动程序的超时路径。

- **`FAULT_MASK`** 是选择哪些故障模式激活的位字段。位0启用"操作超时"故障；位1启用"读取返回全一"故障；位2启用"每个命令后设置错误位"故障；位3启用"设备永远忙碌"故障。可以同时设置多个位。

- **`FAULT_PROB`** 以基点的十分之一为单位控制任何单个操作注入故障的概率。值10000是100%（每次都故障）；5000是50%；0完全禁用随机故障。这给读者对故障注入激进度进行细粒度控制。

- **`OP_COUNTER`** 计算模拟已处理的每个命令。它是只读的；写入被忽略。驱动程序可以用它来验证命令实际到达了模拟，用户可以从用户空间读取它来检查活动。

### 新寄存器的头文件添加

添加的头文件位于 `myfirst_hw.h`，扩展第16章定义：

```c
/* 第17章向模拟设备寄存器映射添加的内容。 */

#define MYFIRST_REG_SENSOR        0x28
#define MYFIRST_REG_SENSOR_CONFIG 0x2c
#define MYFIRST_REG_DELAY_MS      0x30
#define MYFIRST_REG_FAULT_MASK    0x34
#define MYFIRST_REG_FAULT_PROB    0x38
#define MYFIRST_REG_OP_COUNTER    0x3c

/* SENSOR_CONFIG 寄存器字段。 */
#define MYFIRST_SCFG_INTERVAL_MASK  0x0000ffffu  /* 以毫秒为单位的间隔 */
#define MYFIRST_SCFG_INTERVAL_SHIFT 0
#define MYFIRST_SCFG_AMPLITUDE_MASK 0xffff0000u  /* 振荡范围 */
#define MYFIRST_SCFG_AMPLITUDE_SHIFT 16

/* FAULT_MASK 寄存器位。 */
#define MYFIRST_FAULT_TIMEOUT    0x00000001u  /* 下一个操作超时        */
#define MYFIRST_FAULT_READ_1S    0x00000002u  /* 读取返回 0xFFFFFFFF  */
#define MYFIRST_FAULT_ERROR      0x00000004u  /* 操作后 STATUS.ERROR    */
#define MYFIRST_FAULT_STUCK_BUSY 0x00000008u  /* STATUS.BUSY 锁存于   */

/* CTRL 寄存器：GO 位（第17章新增）。 */
#define MYFIRST_CTRL_GO          0x00000200u  /* 位 9：启动命令     */

/* STATUS 寄存器：BUSY 和 DATA_AV 现在是动态的（由模拟设置）。 */
/* （掩码常量已在第16章定义。）               */
```

新的 CTRL 位 `MYFIRST_CTRL_GO` 不与任何现有位重叠。`MYFIRST_CTRL_ENABLE` 是位0，`RESET` 是位1，`MODE_MASK` 覆盖位4到7，`LOOPBACK` 是位8。位9的新 `GO` 位适合间隙。

头文件中的注释故意简洁。在真实驱动程序中，此级别的注释会指向数据手册节（例如，`/* 见数据手册第4.2.1节 */`）。对于模拟，数据手册是本章；查找 `MYFIRST_FAULT_ERROR` 确切行为的读者应该查阅第17章第7节。

### 为自己的设备编写寄存器映射表

编写寄存器映射的练习是作为初学驱动程序作者可以做的最有价值的事情之一。即使你从不编写模拟或驱动程序，坐下来思考"这个设备的寄存器会是什么？"迫使你将设备视为具有协议的实体，而不是魔术盒。

有用的模板：

1. 决定设备在高层次上做什么。一句话。
2. 识别设备必须接受的命令。每个命令可能变成 `CTRL` 寄存器中的位、`COMMAND` 寄存器中的值或自己的寄存器。
3. 识别设备报告的状态。每个状态可能变成 `STATUS` 寄存器中的位、`STATE` 寄存器中的值或自己的只读寄存器。
4. 识别设备产生或消费的数据。输入数据进入 `DATA_IN` 寄存器或 FIFO。输出数据来自 `DATA_OUT` 寄存器或 FIFO。
5. 识别设备可以引发的中断。每个中断源变成 `INTR_STATUS` 中的位（或等效物）。
6. 识别驱动程序在启动时需要的任何元数据。设备ID、固件版本、特性标志、能力位。
7. 留出驱动程序可能想要调整的配置寄存器空间。在第17章，`DELAY_MS`、`FAULT_MASK` 和 `FAULT_PROB` 是配置寄存器；真实设备可能有超时阈值、错误恢复选项、电源管理设置。
8. 决定每个寄存器的宽度。32位是现代设备的常见默认值。较窄的宽度用于遗留设备或寄存器数量非常大的设备。
9. 决定每个寄存器的偏移量。尽可能保持相关寄存器连续。将偏移量对齐到寄存器宽度（32位寄存器在可被4整除的偏移量）。
10. 决定访问语义（RO、RW、RC、W1C...）。要明确；这里的模糊会产生错误。
11. 决定每个寄存器的复位值。寄存器在上电时应该保持什么？复位时？模块挂载时？
12. 记录每个非平凡寄存器的副作用。

这个列表很长，但每一步都很小。真实设备团队可能需要几周才能产生如此详细的寄存器映射。模拟设备的映射可以在一个下午产生。第17章映射是将此练习应用于教学设备的结果。

### 示例布局和设计选择

为了使纪律具体化，我们通过第17章添加背后的设计决策。

**决策1：分组相关寄存器**。新寄存器 `SENSOR`（0x28）和 `SENSOR_CONFIG`（0x2c）相邻。读取传感器代码的驱动程序可以在需要时用单个区域读取访问两者，扫描寄存器转储的读者将它们视为一组。同样的逻辑使 `FAULT_MASK`（0x34）和 `FAULT_PROB`（0x38）相邻。

**决策2：将配置放在诊断之后**。`DEVICE_ID`（0x18）和 `FIRMWARE_REV`（0x1c）是只读诊断寄存器；它们位于高偏移部分的第一个。`SENSOR` 及其配置紧随其后。`DELAY_MS`、`FAULT_MASK`、`FAULT_PROB` 是配置；它们在之后。`OP_COUNTER` 是另一个诊断；它放在最后。

**决策3：使用现有 CTRL 寄存器添加新的 GO 位**。而不是添加专用的 `COMMAND` 寄存器，`CTRL` 中的 `GO` 位是驱动程序发起命令的方式。这与第16章模式匹配（`ENABLE` 和 `RESET` 位都在 `CTRL` 中）并节省一个寄存器。

**决策4：使 `OP_COUNTER` 只读**。可写计数器会邀请驱动程序意外重置计数器 mid-operation 的错误。只读计数器始终是单调的（模32位回绕）并始终反映模拟的视图。

**决策5：在 `SENSOR_CONFIG` 中编码两个字段**。而不是为间隔和幅度使用两个单独的寄存器，一个带有两个16位字段的寄存器节省一个槽并匹配常见硬件模式。真实设备可能在寄存器的高字节中有8位命令ID，在低三个字节中有24位参数；原理相同。

**决策6：以基点的十分之一为单位使用 `FAULT_PROB`（范围0到10000）**。使用10000作为全尺度而不是100给出两个额外的小数位精度。0.5%、0.75%、1.25%的故障概率可以表示为整数而无需小数算术。

每个决定都很小；组合使映射可用。粗心设计的寄存器映射产生充满尴尬补丁的驱动程序；精心设计的映射产生自己编写的驱动程序。

### 常量命名约定

关于第17章用于位掩码的命名约定的说明，它反映了真实 FreeBSD 驱动程序所做的。

前缀 `MYFIRST_` 标记所有内容属于驱动程序。在其中，`REG_` 表示寄存器偏移量，`CTRL_` 表示 `CTRL` 中的位或字段，`STATUS_` 表示 `STATUS` 中的位，以此类推。模式是 `<驱动程序>_<寄存器>_<字段>`。例如，`MYFIRST_STATUS_READY` 是 `STATUS` 寄存器中的 `READY` 位。名称很长，但它是明确的，现代编辑器快速完成它。

覆盖多位的掩码以 `_MASK` 结尾；对齐掩码的移位以 `_SHIFT` 结尾。`MYFIRST_CTRL_MODE_MASK` 是4位 `MODE` 字段的掩码；`MYFIRST_CTRL_MODE_SHIFT` 是将字段带到位零需要右移的位数。提取字段读作 `(ctrl & MYFIRST_CTRL_MODE_MASK) >> MYFIRST_CTRL_MODE_SHIFT`；设置字段读作 `(mode << MYFIRST_CTRL_MODE_SHIFT) & MYFIRST_CTRL_MODE_MASK`。

固定值以 `_VALUE` 结尾。`MYFIRST_DEVICE_ID_VALUE` 是 `DEVICE_ID` 寄存器的常量值。此约定使值与掩码在视觉上可区分。

真实驱动程序有时偏离此约定，通常由于历史原因或因为它们的寄存器名称很长。`if_em` 驱动程序使用较短的前缀（某些地方用 `EM_` 而不是 `E1000_`）以适应合理的行长度。`if_ale` 驱动程序使用 `CSR_READ_4(sc, ALE_REG_NAME)`，以 `ALE_` 为前缀。原理在驱动程序间相同：每个常量都可见地属于一个驱动程序的命名空间，命名空间内的模式是一致的。

对于第17章，一致性比简洁更重要。读者仍在学习模式；长而明确的名称比短而模糊的名称更有教育意义。

### 在编写代码之前验证映射

在实现模拟之前，值得对映射进行一些健全性检查。

**检查1：偏移量适合分配的区域**。第16章区域是64字节。第17章映射中定义的最高偏移量是 `0x3c`，宽度为4，所以使用的最后一个字节在偏移量 `0x3f`。`0x40` 是分配区域之后的一个字节。映射恰好适合；没有空间添加更多寄存器而不扩大分配。这是一个值得知道的设计约束。

**检查2：所有偏移量4字节对齐**。每个寄存器都是32位宽，每个偏移量都是4的倍数。`bus_space_read_4` 和 `bus_space_write_4` 在大多数平台上需要4字节对齐；未对齐的访问会在某些架构上故障，在其他架构上静默错误行为。映射遵循规则。

**检查3：没有两个寄存器重叠**。每个32位寄存器占用4个连续字节。检查相邻偏移量至少相差4。它们确实如此：0x00、0x04、0x08、0x0c、0x10、0x14、0x18、0x1c、0x20、0x24、0x28、0x2c、0x30、0x34、0x38、0x3c。

**检查4：没有保留位用于多个目的**。为一个字段定义的位不能出现在另一个字段中。`MYFIRST_CTRL_ENABLE` 是位0，`RESET` 是位1，`MODE` 覆盖位4到7，`LOOPBACK` 是位8，`GO` 是位9。无重叠。

**检查5：访问语义与模拟意图一致**。`DEVICE_ID` 和 `FIRMWARE_REV` 是 RO，这意味着模拟在挂载后将拒绝更改它们。`INTR_STATUS` 将是 RC，这意味着模拟在驱动程序读取时清除它。`OP_COUNTER` 是 RO，意味着写入无效。其他都是 RW（可能带有副作用）。这与第1节描述的一致。

**检查6：复位值合理**。在挂载时，`DEVICE_ID` 是 `0x4D594649`，`FIRMWARE_REV` 是 `0x00010000`，`STATUS` 设置了 `READY`，其他都是零。`DELAY_MS` 默认为500（每个命令500毫秒）。`SENSOR_CONFIG` 默认为 0x0064_0040（100毫秒间隔，幅度64）。`FAULT_MASK` 默认为0（无故障启用）。`FAULT_PROB` 默认为0（无随机故障）。默认值描述了一个表现可靠的模拟，直到被告知否则。

这些检查是纪律性驱动程序作者在编写任何代码之前会进行的类型。它们花几分钟，在几个常见错误出现之前捕获它们。

### 寄存器映射设计中的常见错误

初学者设计第一个寄存器映射时经常犯相同的几个错误。简短目录以便避免它们。

**错误1：在代码中使用魔术数字而不是命名常量**。写入 `CSR_WRITE_4(sc, 0x00, 0x1)` 而不是 `CSR_WRITE_4(sc, MYFIRST_REG_CTRL, MYFIRST_CTRL_ENABLE)` 的驱动程序很难阅读，无法重构。规则是：每个偏移量都是常量；每个位都是常量；每个字段都是宏。

**错误2：重叠位定义**。两个名称，一个位。对 `FLAG_A` 和 `FLAG_B` 进行按位或的读者得到完全对应于第三件事的值，因为 `FLAG_A` 和 `FLAG_B` 原来是同一位。防御措施是在每个位名称旁边列出其位号的注释，加上编译时的健全性检查（如果你想要防御性，`_Static_assert((MYFIRST_CTRL_ENABLE & MYFIRST_CTRL_RESET) == 0, "bits overlap")`）。

**错误3：未对齐访问**。在偏移量 `0x06` 定义32位寄存器看起来无害，但在任何禁止未对齐访问的架构上失败。始终将偏移量对齐到寄存器宽度。

**错误4：忘记保留位**。只有位0到3定义的32位寄存器在位4到31中可能有未指定的值。写入 `CSR_WRITE_4(sc, REG, 0xF)` 的驱动程序现在没问题，但定义位5的未来修订可能会损坏，因为驱动程序实际上每次都向它写入零。防御措施是对有保留位的寄存器做读-改-写：`CSR_UPDATE_4(sc, REG, mask_of_fields_we_touch, new_field_values)`。

**错误5：驱动程序不期望的读取副作用**。驱动程序为调试目的读取的 RC 寄存器清除了驱动程序即将操作的状态。防御措施是在头文件和 `HARDWARE.md` 中明确记录每个副作用，以便读者在编写错误之前可以看到危险。

**错误6：假设写入总是成功**。写入 RO 寄存器可能被静默忽略，或引发故障，或在某些硬件上做令人惊讶的事情。防御措施是使用正确的访问语义并对驱动程序使用每个寄存器进行压力测试。

**错误7：不一致宽度**。32位寄存器旁边有8位寄存器，访问宽度从名称不明显。防御措施是使宽度成为访问器调用的一部分（`CSR_READ_4` vs `CSR_READ_1`）并在某处命名寄存器的类型。

**错误8：过度聪明的位布局**。跨越两个不相邻字节的16位字段，需要在驱动程序中收集位。真实设备有时这样做（通常由于历史约束），但模拟设备应该坚持连续字段。模拟是教学设备；清晰胜过聪明。

**错误9：缺少复位值**。未指定复位值的寄存器迫使驱动程序猜测挂载后寄存器持有什防御措施是在挂载路径中将每个寄存器初始化为文档值。

**错误10：过时文档**。代码更改，文档不更改。两个月后驱动程序工作但文档撒谎。防御措施是在更改代码的同一提交中更新 `HARDWARE.md` 的纪律。

每个错误单独很小，总体很昂贵。没有这些错误编写的驱动程序阅读良好、测试良好、老化良好。

### 第2节总结

寄存器映射是设备程序员可见接口的规范。第17章映射用六个新寄存器扩展第16章：`SENSOR`、`SENSOR_CONFIG`、`DELAY_MS`、`FAULT_MASK`、`FAULT_PROB` 和 `OP_COUNTER`。每个寄存器都有目的、宽度、访问类型、复位值和文档化的行为。每个寄存器背后的设计选择值得理解，因为它们反映了真实设备设计师所做的决定。

映射在头文件（`myfirst_hw.h`）和 `HARDWARE.md` 中文档化。头文件由编译器消费；markdown 由人类消费。随着代码演进，两者保持同步。

## 第3节：实现模拟硬件后端

第2节产生了寄存器映射。第3节使映射行为化。模拟的第一个工作是自主更新寄存器，无需驱动程序要求。这就是真实硬件的行为：传感器按时钟刷新其值，中断控制器在事件到达时锁存它们，网络卡的接收计数器在帧到达时递增。第17章的第一个模拟步骤是产生类似的心跳。

工具是读者已经知道的：一个 `callout`。第13章介绍了 `callout_init_mtx` 和 `callout_reset` 用于内部驱动程序定时器。第17章将相同的原语用于不同的目的：驱动模拟设备状态。定时器每100毫秒触发，获取驱动程序的互斥锁，更新一些寄存器，释放互斥锁，并调度其下一次触发。从驱动程序的角度看，寄存器只是变化；从模拟的角度看，定时器正在运行。

### 第一个模拟文件

### 模拟状态结构

在 `myfirst_sim.h` 内部，定义状态结构：

```c
struct myfirst_sim {
        /* 自主更新定时器。每 sensor_interval_ms 毫秒触发。 */
        struct callout       sensor_callout;

        /* 命令完成定时器。在 CTRL.GO 后 DELAY_MS 毫秒触发。 */
        struct callout       command_callout;

        /* 上次调度命令的数据。保存以便 command_cb 可以
         * 在触发时锁存 DATA_OUT。 */
        uint32_t             pending_data;

        /* 是否有命令正在执行。 */
        bool                 command_pending;

        /* 基线传感器值；定时器围绕此振荡。 */
        uint32_t             sensor_baseline;

        /* 传感器振荡算法使用的计数器。 */
        uint32_t             sensor_tick;

        /* 操作计数器。每次完成时写入 OP_COUNTER。 */
        uint32_t             op_counter;

        /* 模拟是否运行。在卸载时干净停止。 */
        bool                 running;
};
```

七个字段。两个是定时器（传感器更新和命令完成）。一个是布尔状态（命令执行中）。一个是保存的待处理数据。三个是内部计数器。一个是运行标志。

softc 增加指向模拟结构的指针：

```c
struct myfirst_softc {
        /* ... 所有现有字段，包括 hw ... */
        struct myfirst_sim   *sim;
};
```

分配和初始化发生在 `myfirst_sim_attach` 中，从 `myfirst_attach` 在 `myfirst_hw_attach` 已经设置好 `sc->hw` 之后调用。将模拟挂载放在硬件挂载之后是故意的：模拟通过第16章访问器读取和写入寄存器，所以访问器必须在模拟开始之前准备好。

### 传感器定时器：第一个自主更新

最小的有用模拟更新是传感器定时器。它每100毫秒触发，更新 `SENSOR` 寄存器，并重新调度自己。代码：

```c
static void
myfirst_sim_sensor_cb(void *arg)
{
        struct myfirst_softc *sc = arg;
        struct myfirst_sim *sim = sc->sim;
        uint32_t config, interval_ms, amplitude, phase, value;

        MYFIRST_LOCK_ASSERT(sc);

        if (!sim->running)
                return;

        /* 读取当前 SENSOR_CONFIG。 */
        config = CSR_READ_4(sc, MYFIRST_REG_SENSOR_CONFIG);
        interval_ms = (config & MYFIRST_SCFG_INTERVAL_MASK) >>
            MYFIRST_SCFG_INTERVAL_SHIFT;
        amplitude = (config & MYFIRST_SCFG_AMPLITUDE_MASK) >>
            MYFIRST_SCFG_AMPLITUDE_SHIFT;

        /* 计算简单振荡。phase 循环 0..7 并返回。 */
        sim->sensor_tick++;
        phase = sim->sensor_tick & 0x7;
        value = sim->sensor_baseline +
                ((phase < 4) ? phase : (7 - phase)) *
                (amplitude / 4);

        /* 发布。 */
        CSR_WRITE_4(sc, MYFIRST_REG_SENSOR, value);

        /* 以当前间隔重新调度。 */
        if (interval_ms == 0)
                interval_ms = 100;
        callout_reset_sbt(&sim->sensor_callout,
            interval_ms * SBT_1MS, 0,
            myfirst_sim_sensor_cb, sc, 0);
}
```

一步步来。

首先，定时器断言锁。用 `callout_init_mtx(&co, &mtx, 0)` 调度的定时器在调用回调之前自动获取 `mtx`，所以断言对于纪律良好的驱动程序是冗余的，但它很便宜并记录不变量。调试内核捕获任何违规。

其次，定时器检查 `running` 标志。如果模拟正在被拆除（卸载路径），定时器立即退出而不重新调度。然后卸载路径用 `callout_drain` 排空定时器，它等待任何正在执行的回调完成。

第三，回调读取 `SENSOR_CONFIG` 以了解间隔和幅度。每次读取配置意味着用户可以通过 sysctl 更改它，下一个回调将使用新值，除了寄存器写入之外不需要任何信号机制。

第四，回调计算振荡值。`phase < 4 ? phase : (7 - phase)` 技巧产生三角波，在连续调用中走向 0, 1, 2, 3, 4, 3, 2, 1, 0, 1, ...。乘以 `amplitude / 4`，它在 `baseline` 和 `baseline + amplitude` 之间振荡。公式故意简单；重点是产生可见变化，而不是模拟真实传感器。

第五，回调将新值写入 `SENSOR`。写入是驱动程序或用户空间观察者将看到的。

第六，回调重新调度自己。`callout_reset_sbt` 接受有符号二进制时间，所以 `interval_ms * SBT_1MS` 将毫秒转换为正确的单位。`pr` 参数（精度）为零，意味着内核可以将回调延迟最多间隔的100%以与其他定时器批处理；较低的值会强制更严格的时序。`flags` 参数为零，意味着没有像 `C_DIRECT_EXEC` 或 `C_HARDCLOCK` 这样的特殊标志；回调在普通定时器线程上运行，持有注册的互斥锁。

如果 `interval_ms` 为零，代码默认为100毫秒。零间隔会立即以零延迟重新调度，这将使内核在紧密循环中旋转；保护是防御性的。

### 初始化传感器定时器

在 `myfirst_sim_attach` 中：

```c
int
myfirst_sim_attach(struct myfirst_softc *sc)
{
        struct myfirst_sim *sim;

        sim = malloc(sizeof(*sim), M_MYFIRST, M_WAITOK | M_ZERO);

        /* 用主互斥锁初始化定时器。 */
        callout_init_mtx(&sim->sensor_callout, &sc->mtx, 0);
        callout_init_mtx(&sim->command_callout, &sc->mtx, 0);

        /* 选择基线传感器值；0x1000 是任意但可见的。 */
        sim->sensor_baseline = 0x1000;

        /* 默认配置：100毫秒间隔，幅度64。 */
        MYFIRST_LOCK(sc);
        CSR_WRITE_4(sc, MYFIRST_REG_SENSOR_CONFIG,
            (100 << MYFIRST_SCFG_INTERVAL_SHIFT) |
            (64 << MYFIRST_SCFG_AMPLITUDE_SHIFT));
        CSR_WRITE_4(sc, MYFIRST_REG_DELAY_MS, 500);
        CSR_WRITE_4(sc, MYFIRST_REG_FAULT_MASK, 0);
        CSR_WRITE_4(sc, MYFIRST_REG_FAULT_PROB, 0);
        CSR_WRITE_4(sc, MYFIRST_REG_OP_COUNTER, 0);
        MYFIRST_UNLOCK(sc);

        sc->sim = sim;
        return (0);
}
```

函数分配模拟结构，用驱动程序的互斥锁初始化其定时器（以便回调自动持有 `sc->mtx` 运行），选择基线传感器值，在锁下将默认配置写入模拟寄存器，并将指针存储在 softc 中。

定时器还没有启动。模拟准备好但空闲。单独的函数 `myfirst_sim_enable` 启动定时器；用户可以通过 sysctl 切换模拟开和关。这种分离对于调试和需要已知安静起始状态的测试很有用。

### 启用和禁用模拟

```c
void
myfirst_sim_enable(struct myfirst_softc *sc)
{
        struct myfirst_sim *sim = sc->sim;
        uint32_t config, interval_ms;

        MYFIRST_LOCK_ASSERT(sc);

        if (sim->running)
                return;

        sim->running = true;

        config = CSR_READ_4(sc, MYFIRST_REG_SENSOR_CONFIG);
        interval_ms = (config & MYFIRST_SCFG_INTERVAL_MASK) >>
            MYFIRST_SCFG_INTERVAL_SHIFT;
        if (interval_ms == 0)
                interval_ms = 100;

        callout_reset_sbt(&sim->sensor_callout,
            interval_ms * SBT_1MS, 0,
            myfirst_sim_sensor_cb, sc, 0);
}

void
myfirst_sim_disable(struct myfirst_softc *sc)
{
        struct myfirst_sim *sim = sc->sim;

        MYFIRST_LOCK_ASSERT(sc);

        if (!sim->running)
                return;

        sim->running = false;

        callout_stop(&sim->sensor_callout);
        callout_stop(&sim->command_callout);
}
```

`myfirst_sim_enable` 将 `running` 设置为 true 并调度第一个传感器定时器。`myfirst_sim_disable` 清除标志并停止两个定时器。`callout_stop` 不等待任何正在执行的回调完成；它只是取消任何待处理的重新调度。正在执行的回调将在下一次调用时注意到 `running == false` 并退出。

注意 `callout_stop` 和 `callout_reset_sbt` 都不释放互斥锁。两者在持有互斥锁时调用都是安全的。这就是为什么使用 `callout_init_mtx`：互斥锁穿过定时器子系统，以便回调总是持有它运行，控制调用从不与锁争夺。

还要注意 `myfirst_sim_disable` 不排空定时器。排空必须在卸载中发生，不在禁用中。原因是 `callout_drain` 休眠，而持有互斥锁时休眠是非法的。排空发生在卸载路径中，在互斥锁释放之后。

### 将模拟连接到挂载和卸载

在 `myfirst_attach`（主 `myfirst.c` 文件）中，在硬件挂载之后添加模拟挂载调用：

```c
int
myfirst_attach(device_t dev)
{
        struct myfirst_softc *sc;
        int error;

        sc = device_get_softc(dev);

        /* ... 现有第11-15章初始化 ... */

        /* 第16章：硬件层。 */
        error = myfirst_hw_attach(sc);
        if (error != 0)
                goto fail_hw;

        /* 第17章：模拟后端。 */
        error = myfirst_sim_attach(sc);
        if (error != 0)
                goto fail_sim;

        /* ... 现有 sysctl 和 cdev 设置 ... */

        myfirst_hw_add_sysctls(sc);
        myfirst_sim_add_sysctls(sc);

        /* 默认启用模拟。 */
        MYFIRST_LOCK(sc);
        myfirst_sim_enable(sc);
        MYFIRST_UNLOCK(sc);

        return (0);

fail_sim:
        myfirst_hw_detach(sc);
fail_hw:
        /* ... 现有回滚 ... */
        return (error);
}
```

模拟在硬件层之后挂载，在 sysctl 注册之前。模拟在挂载结束时启用，一旦其他一切都准备好。如果任何步骤失败，回滚路径以相反顺序运行：禁用已启用的，卸载已挂载的，释放已分配的。

在 `myfirst_detach` 中，模拟卸载发生在早期，在硬件卸载之前：

```c
int
myfirst_detach(device_t dev)
{
        struct myfirst_softc *sc = device_get_softc(dev);

        /* ... 现有 cdev 销毁，等待未完成的 fd ... */

        /* 第17章：停止模拟，排空其定时器。 */
        MYFIRST_LOCK(sc);
        myfirst_sim_disable(sc);
        MYFIRST_UNLOCK(sc);
        myfirst_sim_detach(sc);

        /* 第16章：卸载硬件层。 */
        myfirst_hw_detach(sc);

        /* ... 现有同步原语拆除 ... */

        return (0);
}
```

注意顺序。禁用在持有锁时发生；卸载在锁释放后发生。卸载是排空发生的地方：

```c
void
myfirst_sim_detach(struct myfirst_softc *sc)
{
        struct myfirst_sim *sim;

        if (sc->sim == NULL)
                return;

        sim = sc->sim;
        sc->sim = NULL;

        callout_drain(&sim->sensor_callout);
        callout_drain(&sim->command_callout);

        free(sim, M_MYFIRST);
}
```

`callout_drain` 等待任何正在执行的回调完成。如果回调在调用 `callout_drain` 时在另一个 CPU 上运行，`callout_drain` 阻塞直到回调完成并返回。在 `callout_drain` 返回后，该定时器的回调将永远不再运行，所以释放模拟状态是安全的。

模式是第13章教授的定时器通用模式。唯一的微妙之处是在调用 `callout_drain` 之前必须释放互斥锁；如果回调在等待获取互斥锁而我们持有它，我们就会死锁。解锁-在-排空之前的顺序是必不可少的。

### 第一次测试：传感器更新可见

构建并加载驱动程序。然后：

```text

```

这将间隔设置为 `0x0100 = 256` 毫秒，幅度设置为 `0x0040 = 64`。传感器现在每256毫秒更新一次。你可以在访问日志中看到变化：

这就是模拟产生的。驱动程序在此期间什么都不做；模拟自己运行，由传感器定时器驱动。

### 第二个定时器：命令完成

传感器定时器是周期性的。命令定时器是一次性的：它在命令完成时触发一次，在发出下一个命令之前不做任何事情。

```c
static void
myfirst_sim_command_cb(void *arg)
{
        struct myfirst_softc *sc = arg;
        struct myfirst_sim *sim = sc->sim;
        uint32_t status;

        MYFIRST_LOCK_ASSERT(sc);

        if (!sim->running || !sim->command_pending)
                return;

        /* 完成命令：将待处理数据复制到 DATA_OUT，
         * 设置 STATUS.DATA_AV，清除 STATUS.BUSY，递增 OP_COUNTER。 */
        CSR_WRITE_4(sc, MYFIRST_REG_DATA_OUT, sim->pending_data);

        status = CSR_READ_4(sc, MYFIRST_REG_STATUS);
        status &= ~MYFIRST_STATUS_BUSY;
        status |= MYFIRST_STATUS_DATA_AV;
        CSR_WRITE_4(sc, MYFIRST_REG_STATUS, status);

        sim->op_counter++;
        CSR_WRITE_4(sc, MYFIRST_REG_OP_COUNTER, sim->op_counter);

        sim->command_pending = false;
}
```

回调做四件事。它将待处理数据复制到 `DATA_OUT`（模拟的"结果"）。它更新 `STATUS`：`BUSY` 清除，`DATA_AV` 设置。它递增 `OP_COUNTER`。它清除 `command_pending` 标志。

命令由 `myfirst_sim_start_command` 发起，当驱动程序写入 `CTRL.GO` 时调用：

```c
void
myfirst_sim_start_command(struct myfirst_softc *sc)
{
        struct myfirst_sim *sim = sc->sim;
        uint32_t data_in, delay_ms;

        MYFIRST_LOCK_ASSERT(sc);

        if (!sim->running)
                return;

        if (sim->command_pending) {
                /* 命令已在执行。真实设备可能会拒绝
                 * 这个，设置错误位，或排队命令。对于
                 * 模拟，最简单的行为是将其视为
                 * 无操作。驱动程序不应该这样做。 */
                device_printf(sc->dev,
                    "sim: overlapping command; ignored\n");
                return;
        }

        /* 现在快照 DATA_IN；定时器稍后触发。 */
        data_in = CSR_READ_4(sc, MYFIRST_REG_DATA_IN);
        sim->pending_data = data_in;

        /* 设置 STATUS.BUSY，清除 STATUS.DATA_AV。 */
        {
                uint32_t status = CSR_READ_4(sc, MYFIRST_REG_STATUS);
                status |= MYFIRST_STATUS_BUSY;
                status &= ~MYFIRST_STATUS_DATA_AV;
                CSR_WRITE_4(sc, MYFIRST_REG_STATUS, status);
        }

        /* 读取配置的延迟。 */
        delay_ms = CSR_READ_4(sc, MYFIRST_REG_DELAY_MS);
        if (delay_ms == 0) {
                /* 零延迟：在下一个刻度完成。 */
                delay_ms = 1;
        }

        sim->command_pending = true;

        callout_reset_sbt(&sim->command_callout,
            delay_ms * SBT_1MS, 0,
            myfirst_sim_command_cb, sc, 0);
}
```

函数将 `DATA_IN` 快照到模拟状态，设置 `STATUS.BUSY`，清除 `STATUS.DATA_AV`，读取配置的 `DELAY_MS`，并调度命令定时器。命令定时器在 `delay_ms` 毫秒后触发并完成命令。

一个微妙的点：为什么要快照 `DATA_IN` 而不是在命令定时器中读取它？因为驱动程序可能在定时器触发之前再次写入 `DATA_IN`。模拟应该使用写入 `CTRL.GO` 时 `DATA_IN` 中的值，而不是定时器运行时恰好在那里的任何值。在开始时快照对大多数真实设备也是正确行为：设备在发出命令时读取命令寄存器，在执行期间不会重新读取它。

### 拦截 `CTRL.GO`

当驱动程序写入 `CTRL.GO` 时调用 `myfirst_sim_start_command` 函数。写入拦截位于现有的 `myfirst_ctrl_update` 辅助函数中（第16章引入）：

```c
void
myfirst_ctrl_update(struct myfirst_softc *sc, uint32_t old, uint32_t new)
{
        /* 第16章行为：记录 ENABLE 转换。 */
        if ((old & MYFIRST_CTRL_ENABLE) != (new & MYFIRST_CTRL_ENABLE)) {
                device_printf(sc->dev, "CTRL.ENABLE now %s\n",
                    (new & MYFIRST_CTRL_ENABLE) ? "on" : "off");
        }

        /* 第17章：如果设置了 GO，在模拟中启动命令。 */
        if (!(old & MYFIRST_CTRL_GO) && (new & MYFIRST_CTRL_GO)) {
                myfirst_sim_start_command(sc);
                /* 清除 GO 位；它是一次性触发器。 */
                CSR_UPDATE_4(sc, MYFIRST_REG_CTRL, MYFIRST_CTRL_GO, 0);
        }
}
```

扩展监视 `GO` 位从0到1的转换。当它发生时，调用 `myfirst_sim_start_command` 并立即清除 `GO` 位，因为 `GO` 是一次性触发器：它置位，命令开始，位回到零。这是真实硬件中"启动"位的常见模式；一旦命令被接受，硬件自动清除该位。

自清除行为意味着驱动程序不必记住自己清除该位。它写入 `CSR_UPDATE_4(sc, MYFIRST_REG_CTRL, 0, MYFIRST_CTRL_GO)` 来启动命令，模拟处理其余部分。

### 第二次测试：命令路径

命令路径代码到位后，模拟现在支持命令周期。尝试：

```text
# sysctl dev.myfirst.0.reg_status
dev.myfirst.0.reg_status: 1

# sysctl dev.myfirst.0.reg_ctrl_set_datain=0x12345678   # （DATA_IN 的新 sysctl）
# sysctl dev.myfirst.0.reg_ctrl_set=0x200                # 位 9 = CTRL.GO

# sysctl dev.myfirst.0.reg_status
dev.myfirst.0.reg_status: 3    # BUSY | READY

# sleep 1
# sysctl dev.myfirst.0.reg_status
dev.myfirst.0.reg_status: 9    # READY | DATA_AV

# sysctl dev.myfirst.0.reg_data_out
dev.myfirst.0.reg_data_out: 305419896   # 0x12345678

# sysctl dev.myfirst.0.reg_op_counter
dev.myfirst.0.reg_op_counter: 1
```

在 `CTRL.GO` 之后立即读取状态显示 `BUSY|READY`。等待500毫秒延迟并再次读取显示 `READY|DATA_AV`。读取 `DATA_OUT` 返回写入 `CTRL.GO` 时 `DATA_IN` 中的值。`OP_COUNTER` 已增加一。

示例中暗示了一个新的 sysctl `reg_ctrl_set_datain`；它是镜像 `reg_ctrl_set` 但写入 `DATA_IN` 而不是 `CTRL` 的简单添加。模式与第16章 `reg_ctrl_set` 相同：一个带有可写处理程序的 `SYSCTL_ADD_PROC`。为用户可能想要直接操作的每个寄存器添加一个 sysctl。

### 防止重叠命令

`myfirst_sim_start_command` 中对 `command_pending` 的检查防止重叠命令。如果驱动程序在命令已在执行时写入 `CTRL.GO`，模拟忽略第二个命令并记录警告。这对每个真实设备来说不现实（有些排队命令，有些以错误拒绝它们），但对于教学模拟来说是最简单的正确行为。

驱动程序方面不应该在命令执行时发出命令。驱动程序可以轮询 `STATUS.BUSY` 来查看设备是否准备好接受新命令，并在写入 `CTRL.GO` 之前等待 `BUSY` 清除。第5节在驱动程序中教授这个模式。

### 模拟控制的 Sysctl

模拟寄存器已经通过第16章寄存器 sysctl 暴露（`reg_sensor`、`reg_sensor_config`、`reg_delay_ms`、`reg_fault_mask`、`reg_fault_prob`、`reg_op_counter`）。第17章添加一些不映射到寄存器的顶级 sysctl：

```c
void
myfirst_sim_add_sysctls(struct myfirst_softc *sc)
{
        struct sysctl_ctx_list *ctx = &sc->sysctl_ctx;
        struct sysctl_oid *tree = sc->sysctl_tree;

        SYSCTL_ADD_BOOL(ctx, SYSCTL_CHILDREN(tree), OID_AUTO,
            "sim_running", CTLFLAG_RD,
            &sc->sim->running, 0,
            "模拟定时器是否活动");

        SYSCTL_ADD_UINT(ctx, SYSCTL_CHILDREN(tree), OID_AUTO,
            "sim_sensor_baseline", CTLFLAG_RW,
            &sc->sim->sensor_baseline, 0,
            "SENSOR 围绕振荡的基线值");

        SYSCTL_ADD_UINT(ctx, SYSCTL_CHILDREN(tree), OID_AUTO,
            "sim_op_counter_mirror", CTLFLAG_RD,
            &sc->sim->op_counter, 0,
            "OP_COUNTER 值的镜像（用于可观察性）");

        /* 为映射到寄存器的字段添加可写 sysctl。 */
        SYSCTL_ADD_PROC(ctx, SYSCTL_CHILDREN(tree), OID_AUTO,
            "reg_delay_ms_set",
            CTLTYPE_UINT | CTLFLAG_RW | CTLFLAG_MPSAFE,
            sc, MYFIRST_REG_DELAY_MS, myfirst_sysctl_reg_write,
            "IU", "命令延迟以毫秒为单位（可写）");

        SYSCTL_ADD_PROC(ctx, SYSCTL_CHILDREN(tree), OID_AUTO,
            "reg_sensor_config_set",
            CTLTYPE_UINT | CTLFLAG_RW | CTLFLAG_MPSAFE,
            sc, MYFIRST_REG_SENSOR_CONFIG, myfirst_sysctl_reg_write,
            "IU", "传感器配置：间隔|幅度（可写）");
}
```

顶级 `sim_running`、`sim_sensor_baseline` 和 `sim_op_counter_mirror` 暴露不映射到寄存器的内部状态。其余为第17章寄存器添加可写 sysctl，使用第16章现有的 `myfirst_sysctl_reg_write` 处理程序。

添加保持用户视图一致：每个有趣的状态片段通过 `sysctl dev.myfirst.0` 可见，可写 sysctl 遵循驱动程序已使用的 `_set` 后缀模式。

### 第1阶段完成的内容

第3节结束时，驱动程序有：

- 一个带有模拟后端的新文件 `myfirst_sim.c`。
- 一个带有模拟 API 的新头文件 `myfirst_sim.h`。
- 一个由 `sc->sim` 指向的新模拟状态结构。
- 两个定时器：一个用于周期性传感器更新，一个用于一次性命令完成。
- 对 `myfirst_ctrl_update` 的扩展，拦截 `CTRL.GO`。
- 用于应该可配置的新寄存器的可写 sysctl。

版本标签变为 `1.0-sim-stage1`。驱动程序仍然做第16章教的一切；它现在还有两个产生自主寄存器行为的定时器。

构建、加载并测试：

```text

# make clean && make
# kldload ./myfirst.ko
# sysctl dev.myfirst.0.sim_running
# sysctl dev.myfirst.0.reg_sensor
# sleep 1
# sysctl dev.myfirst.0.reg_sensor
# sysctl dev.myfirst.0.reg_op_counter

```

你应该看到 `sim_running` 是1，传感器值变化，`op_counter` 保持为零直到你发出命令。如果 `kldunload` 干净返回，卸载路径是正确的。

### 第1阶段不是什么

第1阶段让模拟运行，但驱动程序的系统调用路径还不知道新行为。写入 `/dev/myfirst0` 仍然写入环形缓冲区，而不是模拟设备。进入环形缓冲区的字节从不出现在 `DATA_IN` 中，从不导致命令，从不出现在 `DATA_OUT` 中。模拟准备好了；驱动程序还没有真正使用它。

第4节迈出下一步：它将模拟连接到驱动程序的数据路径，所以用户空间写入导致模拟命令，命令的结果是用户空间读取看到的。在此之前，模拟是附带表演。

### 第3节总结

模拟设备的行为来自一小部分原语。用于周期性更新的定时器，用于一次性定时事件，以及两个定时器都查询的共享状态结构。第17章的模拟使用其中三个：一个总是运行的传感器定时器，一个每命令运行的命令定时器，以及一个防止重叠命令的内部 `command_pending` 标志。

模拟位于自己的文件 `myfirst_sim.c` 中，它依赖于第16章的 `myfirst_hw.h` 来获取寄存器定义。分离使模拟代码隔离；后续用真实 PCI 面向代码替换模拟的章节可以删除 `myfirst_sim.c` 并用 `myfirst_pci.c` 替换它，而不触摸驱动程序。

第4节开始真正的集成：驱动程序的数据路径开始通过模拟发送和接收，模拟开始从驱动程序方面看起来像一个真正的设备。



## 第4节：将假设备暴露给驱动程序

第3节构建了一个在定时器上运行并通过第16章访问器写入寄存器的模拟后端。第4节问一个不同的问题：驱动程序如何看到模拟？从驱动程序的角度看，访问路径是什么？代码是什么样子的，它与和真实硬件通信的代码有什么不同？

简短回答是访问路径不变。第16章 `bus_space(9)` 抽象的全部意义在于驱动程序不知道也不关心寄存器块是真实的还是模拟的。第4节故意是一个较短的节，因为繁重的工作在第16章已完成。剩下的是仔细审视抽象如何在添加动态行为后存活，以及驱动程序需要让模拟驱动自己的行为的少数小钩子。

### 标签和句柄，仍然

第16章在 `myfirst_hw_attach` 中设置了模拟的 `bus_space_tag_t` 和 `bus_space_handle_t`：

```c
#if defined(__amd64__) || defined(__i386__)
        hw->regs_tag = X86_BUS_SPACE_MEM;
#else
#error "第16章模拟仅支持 x86"
#endif
        hw->regs_handle = (bus_space_handle_t)(uintptr_t)hw->regs_buf;
```

第17章该代码不变。模拟通过 `CSR_WRITE_4` 写入寄存器，展开为 `myfirst_reg_write`，展开为 `bus_space_write_4(hw->regs_tag, hw->regs_handle, offset, value)`。相同的标签；相同的句柄；相同的访问器。第3节引入的定时器使用与系统调用路径完全相同的 API。

这不是意外。`bus_space` 存在的原因正是隐藏真实和模拟寄存器访问之间的差异。正确使用 `bus_space` 的驱动程序可以指向真实的 PCI BAR 或 `malloc(9)` 分配，而无需更改任何访问代码。模拟演练这一属性。

此时的一个有用练习：在一个窗口中打开 `/usr/src/sys/dev/ale/if_ale.c`，在另一个窗口中打开 `myfirst_hw.c`。比较寄存器访问代码。ALE 驱动使用 `CSR_READ_4(sc, ALE_REG_XYZ)`；您的驱动使用 `CSR_READ_4(sc, MYFIRST_REG_XYZ)`。展开是相同的。唯一区别是标签和句柄：ALE 从 `rman_get_bustag(sc->ale_res[0])` 和 `rman_get_bushandle(sc->ale_res[0])` 获取它们；您的驱动从第16章模拟设置获取它们。用真实硬件替换模拟（第18章将做）是 `myfirst_hw_attach` 中的一个函数更改。

### 为什么抽象现在更重要

在第16章，使用 `bus_space` 的论证是前瞻性的："当您最终切换到真实硬件时，驱动程序代码不必更改"。在第17章，论证变得具体：寄存器块现在自主变化，驱动程序无法知道变化是来自模拟定时器还是来自真实设备的内部状态机。抽象在真正工作。

考虑一个小例子。一个轮询 `STATUS.DATA_AV` 的驱动函数：

```c
static int
myfirst_wait_for_data(struct myfirst_softc *sc, int timeout_ms)
{
        int i;

        MYFIRST_LOCK_ASSERT(sc);

        for (i = 0; i < timeout_ms; i++) {
                uint32_t status = CSR_READ_4(sc, MYFIRST_REG_STATUS);
                if (status & MYFIRST_STATUS_DATA_AV)
                        return (0);

                /* 释放锁，短暂睡眠，重新获取。 */
                MYFIRST_UNLOCK(sc);
                pause_sbt("mfwait", SBT_1MS, 0, 0);
                MYFIRST_LOCK(sc);
        }

        return (ETIMEDOUT);
}
```

此函数读取 `STATUS`，检查一个位，要么返回要么睡眠一毫秒后重试。它不知道该位是被第17章模拟的命令定时器、真实设备的内部状态机还是其他东西设置的。它只是轮询。

`pause_sbt` 调用值得检查。函数在睡眠前释放驱动锁，之后重新获取。持有锁睡眠会阻塞每个需要锁的其他上下文，包括模拟定时器；在睡眠前释放锁让模拟运行。`pause_sbt` 接受睡眠标识符（"mfwait"）、有符号二进制时间（此处为1毫秒）、精度和标志。精度为零意味着内核可以将此睡眠与其他睡眠合并，减少定时器中断。

第6节将更深入地重新审视 `DELAY(9)`、`pause_sbt(9)` 和 `callout_reset_sbt(9)` 之间的选择。简短版本：每秒轮询寄存器一千次并在每次轮询间睡眠不是最高效的模式，但它是正确的、简单的、可读的。第17章的驱动使用它因为协议很小且轮询开销无关紧要。真正的高性能驱动使用中断（第19章）替代。

### 命令路径集成

驱动程序当前的数据路径（第10章的 `myfirst_write`）将字节写入环形缓冲区。第17章希望通过模拟路由这些字节：用户写入一个字节，驱动程序将其写入 `DATA_IN`，设置 `CTRL.GO`，等待 `STATUS.DATA_AV`，读取 `DATA_OUT`，将结果写入环形缓冲区。这是完整的命令周期。

对于第17章驱动程序的第2阶段，我们添加命令周期代码路径。写入函数变为：

```c
static int
myfirst_write_cmd(struct myfirst_softc *sc, uint8_t byte)
{
        int error;

        MYFIRST_LOCK_ASSERT(sc);

        /* 等待设备准备好接受新命令。 */
        error = myfirst_wait_for_ready(sc, 100);
        if (error != 0)
                return (error);

        /* 将字节写入 DATA_IN。 */
        CSR_WRITE_4(sc, MYFIRST_REG_DATA_IN, (uint32_t)byte);

        /* 发出命令。 */
        CSR_UPDATE_4(sc, MYFIRST_REG_CTRL, 0, MYFIRST_CTRL_GO);

        /* 等待命令完成（STATUS.DATA_AV）。 */
        error = myfirst_wait_for_data(sc, 2000);
        if (error != 0)
                return (error);

        /* 检查错误。 */
        {
                uint32_t status = CSR_READ_4(sc, MYFIRST_REG_STATUS);
                if (status & MYFIRST_STATUS_ERROR) {
                        /* 清除错误锁存。 */
                        CSR_UPDATE_4(sc, MYFIRST_REG_STATUS,
                            MYFIRST_STATUS_ERROR, 0);
                        return (EIO);
                }
        }

        /* 读取结果。DATA_OUT 将保存模拟回显的字节（或者，如果回环激活，我们写入的相同字节）。 */
        (void)CSR_READ_4(sc, MYFIRST_REG_DATA_OUT);

        /* 清除 DATA_AV 以便下一个命令可以设置它。 */
        CSR_UPDATE_4(sc, MYFIRST_REG_STATUS, MYFIRST_STATUS_DATA_AV, 0);

        return (0);
}
```

六个步骤。等待就绪，写入 `DATA_IN`，设置 `CTRL.GO`，等待 `DATA_AV`，检查错误，读取 `DATA_OUT`。清除 `DATA_AV` 以便下一次迭代。

`myfirst_wait_for_ready` 辅助函数轮询 `STATUS.BUSY`：

```c
static int
myfirst_wait_for_ready(struct myfirst_softc *sc, int timeout_ms)
{
        int i;

        MYFIRST_LOCK_ASSERT(sc);

        for (i = 0; i < timeout_ms; i++) {
                uint32_t status = CSR_READ_4(sc, MYFIRST_REG_STATUS);
                if (!(status & MYFIRST_STATUS_BUSY))
                        return (0);
                MYFIRST_UNLOCK(sc);
                pause_sbt("mfrdy", SBT_1MS, 0, 0);
                MYFIRST_LOCK(sc);
        }

        return (ETIMEDOUT);
}
```

与 `myfirst_wait_for_data` 结构相同；只是轮询的位不同。两个辅助函数都是共享实现的合理候选：

```c
static int
myfirst_wait_for_bit(struct myfirst_softc *sc, uint32_t mask,
    uint32_t target, int timeout_ms)
{
        int i;

        MYFIRST_LOCK_ASSERT(sc);

        for (i = 0; i < timeout_ms; i++) {
                uint32_t status = CSR_READ_4(sc, MYFIRST_REG_STATUS);
                if ((status & mask) == target)
                        return (0);
                MYFIRST_UNLOCK(sc);
                pause_sbt("mfwait", SBT_1MS, 0, 0);
                MYFIRST_LOCK(sc);
        }

        return (ETIMEDOUT);
}
```

然后 `wait_for_ready` 调用 `wait_for_bit(sc, MYFIRST_STATUS_BUSY, 0, timeout_ms)`，`wait_for_data` 调用 `wait_for_bit(sc, MYFIRST_STATUS_DATA_AV, MYFIRST_STATUS_DATA_AV, timeout_ms)`。一旦有两个用户，抽象就值得提取；第6节重新审视此模式。

### 命令与写入系统调用的集成位置

现有的第10章 `myfirst_write` 将字节推入环形缓冲区。第17章不替换该路径；它增强它。对于第2阶段，我们添加一个每字节命令周期钩子，除了将每个字节推入环外还通过模拟发送。

```c
static int
myfirst_write(struct cdev *dev, struct uio *uio, int ioflag)
{
        struct myfirst_softc *sc = dev->si_drv1;
        uint8_t buf[64];
        size_t n;
        int error;

        /* ... 现有验证 ... */

        while (uio->uio_resid > 0) {
                n = MIN(uio->uio_resid, sizeof(buf));
                error = uiomove(buf, n, uio);
                if (error != 0)
                        return (error);

                MYFIRST_LOCK(sc);
                for (size_t i = 0; i < n; i++) {
                        /* 通过模拟设备发送每个字节。 */
                        error = myfirst_write_cmd(sc, buf[i]);
                        if (error != 0) {
                                MYFIRST_UNLOCK(sc);
                                return (error);
                        }

                        /* 现有第10章路径：推入环。 */
                        error = cbuf_put(sc->cbuf, buf[i]);
                        if (error != 0) {
                                MYFIRST_UNLOCK(sc);
                                return (error);
                        }
                }
                MYFIRST_UNLOCK(sc);
        }

        return (0);
}
```

循环为了清晰而逐字节进行。每个字节通过命令周期，然后进入环。真正的驱动程序通常会流水线处理（将多个字节写入FIFO，发出单个命令，读回多个字节），但第17章的教学点是关于单个命令；流水线是后续的改进。

注意锁定。外锁覆盖整个批次（最多64字节）。每次 `myfirst_write_cmd` 调用在睡眠时内部释放和重新获取锁。需要锁的其他上下文（特别是模拟定时器）在命令之间有机会运行。

### 第2阶段测试

构建、加载并测试：

```text
# kldload ./myfirst.ko
# echo -n "hi" > /dev/myfirst0
# sysctl dev.myfirst.0.reg_op_counter
dev.myfirst.0.reg_op_counter: 2

# dd if=/dev/myfirst0 bs=1 count=2 2>/dev/null
hi
```

写入两个字节；`OP_COUNTER` 增加两个。读取返回相同的两个字节，因为模拟将 `DATA_IN` 回显到 `DATA_OUT`。测试验证端到端流程：用户空间写入，驱动程序命令，模拟处理，驱动程序读取，用户空间接收。

整个周期大约需要1秒（每字节500毫秒，两个字节）。这很慢但真实：具有类似延迟的真实设备（例如，需要500毫秒产生读数的SPI传感器）会感觉完全一样。加速意味着减少 `DELAY_MS`（尝试 `sysctl dev.myfirst.0.reg_delay_ms_set=10` 每命令10毫秒）。

### 不变的内容

值得列举第16章驱动程序在第2阶段不变的几个属性。

访问日志继续工作。驱动程序执行的每次寄存器读取和写入都显示在日志中。模拟自己的寄存器写入也显示（因为它们通过相同的访问器）。日志很密集，这是好事：您可以看到模拟的心跳以及驱动程序的活动。

sysctl 接口继续工作。每个寄存器可读；每个可写寄存器可写。第16章的便利功能（读取每个寄存器、写入特定寄存器）不变。

锁定规则继续工作。每次寄存器访问在 `sc->mtx` 下发生；模拟定时器被设计为持有 `sc->mtx` 运行（通过 `callout_init_mtx`），因此无需担心锁顺序反转。

卸载路径继续工作。模拟卸载被添加到现有卸载顺序，模拟定时器的排空在释放硬件区域之前发生。

### 仍需注意的内容

第5节和第6节将收紧的一些松散结尾。

`myfirst_wait_for_ready` 和 `myfirst_wait_for_data` 中的超时是硬编码的（分别是100毫秒和2000毫秒）。可配置的超时（可能通过sysctl暴露）会更好；第5节添加一个。

1毫秒的 `pause_sbt` 睡眠是合理的默认值，但并非每种情况都最优。在快速模拟上（如 `DELAY_MS=0`），每1毫秒轮询是浪费的；在慢速模拟上（如 `DELAY_MS=5000`），每1毫秒轮询是过度的。第6节讨论更好的策略。

命令周期写入路径不与模拟的故障注入交互。如果模拟被配置为使命令失败（第7节的工作），驱动程序需要识别并从失败中恢复。第5节添加基本错误恢复；第7节添加故障注入基础设施。

驱动程序的读取路径尚未集成。读取当前从环形缓冲区拉取，其中包含写入命令推送的任何内容。更真实的模式是读取也导致命令（一个"请给我另一个字节"的命令），但这是第5节的话题。

### 第4节总结

将模拟设备暴露给驱动程序不是第17章的新机制。第16章 `bus_space(9)` 抽象已经处理了这一点；模拟的写入和驱动程序的读取通过相同的标签和句柄，驱动程序既不知道也不关心寄存器块是模拟的。第2阶段添加的是命令周期代码路径：驱动程序写入 `DATA_IN`，设置 `CTRL.GO`，等待 `DATA_AV`，读取 `DATA_OUT`。两个辅助函数（`myfirst_wait_for_ready` 和 `myfirst_wait_for_data`）使轮询显式；一个总括函数（`myfirst_wait_for_bit`）提取共享结构。驱动程序现在以真实的命令-响应模式演练模拟。

第5节深化测试故事。它添加错误恢复、可配置超时、读取和写入路径之间更清晰的分离，以及第一次足以暴露竞争条件的负载测试。



## 第5节：集成读取路径并添加负载测试

第4节连接了写入路径：用户空间写入导致模拟命令，命令结果写入环形缓冲区。读取路径仍然从环形缓冲区拉取而不与模拟交互。第5节修复这一不对称：读取也导致命令，驱动程序的两个数据路径现在都演练模拟。

此外，第5节添加第一次负载测试。单个命令正确是第4节的问题；并发命令正确是第5节的问题。负载测试暴露竞争条件和锁定漏洞，第5节修复它们。

### 第5节的结构

第5节分三个阶段：

1. **阶段1：读取路径集成。** `myfirst_read` 调用模拟命令从设备拉取数据，而不是从预先填充的环拉取。
2. **阶段2：写入路径细化。** 第4节的写入路径将数据通过模拟发送并推入环。第5节使环成为可选的：对于纯命令-响应协议，驱动程序写入，设备响应，没有中间存储。
3. **阶段3：负载测试。** 并发读写者在驱动程序上压力测试，暴露竞争条件和锁定漏洞。

三个阶段共同使驱动程序成为命令-响应设备的完整模拟。

### 读取路径命令

第4节的读取函数从环形缓冲区拉取：

```c
static int
myfirst_read(struct cdev *dev, struct uio *uio, int ioflag)
{
        /* ... 从 cbuf_get 拉取 ... */
}
```

第5节将其改为命令驱动。如果环为空，驱动程序向模拟发出一个"采样"命令：设置 `CTRL.GO`，等待 `DATA_AV`，将 `DATA_OUT` 推入环，然后返回字节给用户空间。模式是"用户空间读取触发模拟命令，命令结果填充环，环数据返回给用户空间"。

采样命令与第4节的写入命令略有不同。写入命令接受来自用户空间的输入字节；采样命令不接受输入——它询问模拟"您有什么？"并将响应放入环。寄存器协议反映了这一区别：

- **写入命令**：写入 `DATA_IN`，设置 `CTRL.GO`，等待 `DATA_AV`，读取 `DATA_OUT`。
- **采样命令**：设置 `CTRL.GO`（`DATA_IN` 为零或不使用），等待 `DATA_AV`，读取 `DATA_OUT`，将 `DATA_OUT` 推入环。

模拟以相同方式处理两者：等待 `DELAY_MS` 毫秒，用结果设置 `DATA_AV`。区别纯粹在驱动程序侧：写入命令携带用户数据；采样命令不携带。

### `myfirst_sample_cmd` 辅助函数

一个新的辅助函数发出采样命令：

```c
static int
myfirst_sample_cmd(struct myfirst_softc *sc)
{
        int error;

        MYFIRST_LOCK_ASSERT(sc);

        error = myfirst_wait_for_ready(sc, sc->rdy_timeout_ms);
        if (error != 0)
                return (error);

        /* 不写入 DATA_IN；采样命令不携带输入。 */
        CSR_UPDATE_4(sc, MYFIRST_REG_CTRL, 0, MYFIRST_CTRL_GO);

        error = myfirst_wait_for_data(sc, sc->cmd_timeout_ms);
        if (error != 0)
                return (error);

        /* 读取结果并推入环。 */
        {
                uint32_t data_out = CSR_READ_4(sc, MYFIRST_REG_DATA_OUT);
                error = cbuf_put(sc->cbuf, (uint8_t)data_out);
                if (error != 0)
                        return (error);
        }

        CSR_UPDATE_4(sc, MYFIRST_REG_STATUS, MYFIRST_STATUS_DATA_AV, 0);
        return (0);
}
```

结构与 `myfirst_write_cmd` 相同，除了缺少 `DATA_IN` 写入和 `cbuf_put` 调用。采样命令产生一个字节放入环。

### 将采样集成到读取路径

读取函数在环为空时调用采样命令：

```c
static int
myfirst_read(struct cdev *dev, struct uio *uio, int ioflag)
{
        struct myfirst_softc *sc = dev->si_drv1;
        uint8_t byte;
        int error;

        /* ... 验证 ... */

        MYFIRST_LOCK(sc);
        while (uio->uio_resid > 0) {
                /* 如果环为空，发出一个采样命令。 */
                if (cbuf_is_empty(sc->cbuf)) {
                        error = myfirst_sample_cmd(sc);
                        if (error != 0) {
                                MYFIRST_UNLOCK(sc);
                                return (error);
                        }
                }
                error = cbuf_get(sc->cbuf, &byte);
                if (error != 0) {
                        MYFIRST_UNLOCK(sc);
                        return (error);
                }
                MYFIRST_UNLOCK(sc);
                error = uiomove(&byte, 1, uio);
                MYFIRST_LOCK(sc);
                if (error != 0) {
                        MYFIRST_UNLOCK(sc);
                        return (error);
                }
        }
        MYFIRST_UNLOCK(sc);
        return (0);
}
```

现在从 `/dev/myfirst0` 读取在环为空时触发模拟命令，有效地从模拟每个命令拉取一个字节。结合第4节的写入路径（推送式），驱动程序现在演练两个方向。

### 第一次负载测试

有了两个路径集成，驱动程序准备好负载测试。一个简单的测试脚本：

```sh
#!/bin/sh
# cmd_load.sh: 在负载下演练命令路径。

set -e

# 用于负载测试的快速模拟。
sysctl dev.myfirst.0.reg_delay_ms_set=10

# 并行启动4个写入者和4个读取者。
for i in 1 2 3 4; do
        (for j in $(seq 1 100); do echo -n "X" > /dev/myfirst0; done) &
done
for i in 1 2 3 4; do
        (dd if=/dev/myfirst0 of=/dev/null bs=1 count=100 2>/dev/null) &
done
wait

# 显示结果计数器。
sysctl dev.myfirst.0.reg_op_counter

# 重置为默认。
sysctl dev.myfirst.0.reg_delay_ms_set=500
```

在 `DELAY_MS=10` 和4个写入者各发送100字节加4个读取者各接收100字节的情况下，测试应在约10秒内完成（800个命令，每个10毫秒，通过模拟串行化）。`OP_COUNTER` 应至少达到800。

关于此测试有两点值得注意。

首先，命令被串行化。模拟拒绝重叠命令（第3节的 `command_pending` 检查）。驱动程序等待 `STATUS.BUSY` 清除后再发出下一个命令。有多个用户空间进程争夺驱动程序时，`sc->mtx` 上的锁将它们串行化。有效速率是每个 `DELAY_MS` 毫秒一个命令，无论运行多少写入者。

其次，驱动程序无需更改即可适应并发。第11章的互斥锁、第14章的任务、第16章的访问器和第4节的命令周期代码自然组合。第3部分建立的规则在此回报：无需额外同步即可将模拟硬件添加到已正确协调自身的驱动程序。

### 错误恢复细化

第4节的命令周期代码清除 `STATUS.ERROR` 并返回 `EIO`。第5节细化此点以区分几种错误情况。

```c
static int
myfirst_write_cmd(struct myfirst_softc *sc, uint8_t byte)
{
        uint32_t status;
        int error;

        MYFIRST_LOCK_ASSERT(sc);

        error = myfirst_wait_for_ready(sc, sc->rdy_timeout_ms);
        if (error != 0) {
                sc->stats.cmd_rdy_timeouts++;
                return (error);
        }

        CSR_WRITE_4(sc, MYFIRST_REG_DATA_IN, (uint32_t)byte);
        CSR_UPDATE_4(sc, MYFIRST_REG_CTRL, 0, MYFIRST_CTRL_GO);

        error = myfirst_wait_for_data(sc, sc->cmd_timeout_ms);
        if (error != 0) {
                sc->stats.cmd_data_timeouts++;
                /* 清除模拟中的任何部分状态。 */
                myfirst_recover_from_stuck(sc);
                return (error);
        }

        status = CSR_READ_4(sc, MYFIRST_REG_STATUS);
        if (status & MYFIRST_STATUS_ERROR) {
                CSR_UPDATE_4(sc, MYFIRST_REG_STATUS,
                    MYFIRST_STATUS_ERROR, 0);
                CSR_UPDATE_4(sc, MYFIRST_REG_STATUS,
                    MYFIRST_STATUS_DATA_AV, 0);
                sc->stats.cmd_errors++;
                return (EIO);
        }

        (void)CSR_READ_4(sc, MYFIRST_REG_DATA_OUT);
        CSR_UPDATE_4(sc, MYFIRST_REG_STATUS, MYFIRST_STATUS_DATA_AV, 0);
        sc->stats.cmd_successes++;
        return (0);
}
```

与第4节相比有三点变化。

首先，每种错误类型的计数器。`sc->stats` 是一个 `uint64_t` 计数器结构，驱动程序在每个结果上更新。`cmd_successes`、`cmd_rdy_timeouts`、`cmd_data_timeouts` 和 `cmd_errors` 各有自己的计数器。一个 sysctl 暴露它们。在负载下，计数器成为主要诊断："驱动程序发出800个命令，5个超时，0个有错误，795个成功"比"驱动程序运行了"有用得多。

其次，`myfirst_recover_from_stuck`。当命令超时时，模拟可能仍在处理中途。清除陈旧状态很重要：

```c
static void
myfirst_recover_from_stuck(struct myfirst_softc *sc)
{
        uint32_t status;

        MYFIRST_LOCK_ASSERT(sc);

        /* 清除模拟中的任何待处理命令标志。下一个
         * CTRL.GO 将被接受，无论任何陈旧状态。 */
        if (sc->sim != NULL) {
                sc->sim->command_pending = false;
                callout_stop(&sc->sim->command_callout);
        }

        /* 强制清除 BUSY 和 DATA_AV。 */
        status = CSR_READ_4(sc, MYFIRST_REG_STATUS);
        status &= ~(MYFIRST_STATUS_BUSY | MYFIRST_STATUS_DATA_AV);
        CSR_WRITE_4(sc, MYFIRST_REG_STATUS, status);
}
```

注意 `myfirst_recover_from_stuck` 直接访问 `sc->sim->command_pending`。这是一个合法的仅模拟路径：在真实硬件上，没有这样的变量；相反，驱动程序会向设备发出复位命令。该函数是模拟特有的恢复路径，代码中应有注释说明。当第18章用真实硬件替换模拟时，此函数完全改变。

第三，错误时的状态清除。错误路径清除 `ERROR` 和 `DATA_AV`。留下任一个设置会导致下一个命令的 `wait_for_data` 立即返回（如果 `DATA_AV` 仍设置）或观察意外错误（如果 `ERROR` 仍设置）。防御性清理确保模拟处于下一个命令的干净状态。

第2阶段将模拟与驱动程序的数据路径集成。写入和读取路径现在都演练模拟命令。可配置超时让测试针对特定行为。每种错误类型的统计计数器暴露驱动程序与模拟的交互。恢复路径在命令超时时清除卡住状态。负载测试大量演练驱动程序并确认第3部分同步与第17章模拟的组合。

### 第5节总结

针对模拟测试驱动程序不是单独的活动；它是驱动程序命令路径的自然扩展。添加可配置超时，将模拟命令集成到读取和写入中，跟踪每种结果的统计，并运行足够的负载以暴露常见失败模式。

动手测试验证往返：用户空间输入通过驱动程序、通过模拟、完好返回用户空间。计数器确认无错误。访问日志按顺序显示每次寄存器接触。

第6节深化时序讨论。`myfirst_wait_for_bit` 中1毫秒的 `pause_sbt` 是合理的默认值，但不是唯一选择。`DELAY(9)` 对非常短的等待更好；`callout_reset_sbt` 对即发即弃调度更好。理解何时每种适当是第6节的主题。



## 第6节：添加时序和延迟

时序是真实硬件变得有趣的地方。在5微秒内处理命令的设备需要与在500毫秒内处理命令的设备非常不同的驱动程序模式。在错误时间尺度使用错误时序原语的驱动程序在短等待上消耗CPU或在长等待上积累延迟，最坏情况下在争夺下死锁。

FreeBSD 为驱动程序代码提供三种主要时序原语：`DELAY(9)`、`pause_sbt(9)` 和 `callout_reset_sbt(9)`。每种在特定时间尺度适当，每种有特定成本，每种有特定正确性要求。第6节按顺序教授它们并展示如何为第17章模拟中的每种情况选择正确原语。

### 三种时序原语

原语在三个维度不同：它们是忙等待还是睡眠、是否可取消、以及它们成本多少。

**`DELAY(n)`** 对 `n` 微秒忙等待。它不睡眠、不释放CPU、不释放任何锁。可以从任何上下文安全调用，包括自旋互斥锁、中断过滤器或 callout。其成本是它消耗的CPU时间：100微秒的 `DELAY` 是CPU不能做其他任何事情的100微秒。其精度通常很好（个位数微秒），受CPU时间戳计数器分辨率限制。它是非常短等待的正确选择，其中睡眠不切实际。

**`pause_sbt(wmesg, sbt, pr, flags)`** 使调用线程睡眠 `sbt` 个有符号二进制时间单位。线程睡眠期间，CPU可用于其他工作，线程未持有的任何锁可用于其他上下文。如果线程持有锁，`pause_sbt` 不释放它；释放睡眠期间不应持有的锁是调用者的责任。精度约为内核定时器滴答（典型系统上1毫秒）。它是短到中等等待的正确选择，线程无事可做。`pause_sbt` 不能通过任何比睡眠本身更短的机制取消；线程调用后，至少睡眠请求的时间（除非异常事件）。

**`callout_reset_sbt(c, sbt, pr, fn, arg, flags)`** 安排回调在 `sbt` 个有符号二进制时间单位后的未来运行。调用线程不等待；它立即返回。时间到期时回调在 callout 线程上运行。可以用 `callout_stop` 或 `callout_drain` 取消 callout。精度与 `pause_sbt` 相似。它是即发即弃调度、可能提前取消的超时和周期性工作的正确选择。

三种原语组合。驱动程序可能用 `DELAY(10)` 等待10微秒让寄存器位稳定，用 `pause_sbt(..., SBT_1MS * 50, ...)` 睡眠50毫秒等待命令完成，用 `callout_reset_sbt(..., SBT_1S * 5, ...)` 安排5秒后的看门狗。每种选择反映时间尺度和上下文。

### 时间尺度决策表

三种原语的快速参考：

| 等待时间      | 上下文                       | 首选原语                       |
|---------------|------------------------------|--------------------------------|
| < 10 us       | 任何                         | `DELAY(9)`                     |
| 10-100 us     | 非中断，非自旋               | `DELAY(9)` 或 `pause_sbt(9)`   |
| 100 us - 10 ms| 非中断，非自旋               | `pause_sbt(9)`                 |
| > 10 ms       | 非中断，非自旋               | `pause_sbt(9)` 或 callout      |
| 即发即弃      | 任何（callout 状态机）       | `callout_reset_sbt(9)`         |
| 中断/自旋     | 任何时间                     | 仅 `DELAY(9)`                  |

表是起点，不是硬规则。真正问题是：调用者能承受睡眠吗？如果是，pause 或 callout。如果否（持有自旋锁、中断处理程序、过滤器处理程序、临界区），DELAY。

### 第17章驱动程序已使用每种的地方

第4节的轮询循环使用带1毫秒延迟的 `pause_sbt`。这对时间尺度适当：模拟的默认 `DELAY_MS` 是500，所以我们预期驱动程序在命令完成前睡眠多次。睡眠让其他线程在等待时使用CPU。`DELAY(1000)` 也可行但会浪费CPU；驱动程序目前每个命令约500次轮询，轮询间无有用CPU工作，`DELAY` 会全部500次空转。

第3节的模拟 callout 使用可变间隔的 `callout_reset_sbt`。这适当因为模拟需要在未来做工作，而没有任何线程主动等待。`pause_sbt` 需要每个 callout 有专用内核线程，这会浪费。`DELAY` 需要线程在整个间隔期间保持内核模式，这不可接受。

驱动程序中还没有 `DELAY`。第6节在适当处引入它。

### `DELAY` 的好用法

考虑第5节的恢复路径。超时后，驱动程序调用 `myfirst_recover_from_stuck`，清除一些状态并返回。代码可行，但有微妙竞争：模拟的命令 callout 可能在超时检测和恢复调用之间触发，以意外方式改变 `STATUS`。一个小修复：在清除状态前用 `DELAY(10)` 等待10微秒让任何进行中的 callout 排空。

但这错了。`DELAY(10)` 这里无有用作用。我们持有 `sc->mtx` 时 callout 无法运行（因为它通过 `callout_init_mtx` 注册），所以没有竞争。在此添加 `DELAY` 是迷信：因为可能出问题而添加延迟，没有延迟防止什么的清晰模型。

一个更好的例子。一些真实硬件要求两个寄存器写入间有小延迟让设备内部逻辑稳定。数据手册可能说："写入 `CTRL.RESET` 后，至少等待5微秒再写入任何其他寄存器。"第17章模拟的 `CTRL.RESET` 位实际未实现，但我们可以想象。如果实现，复位代码会用 `DELAY(5)`：

```c
static void
myfirst_reset_device(struct myfirst_softc *sc)
{
        MYFIRST_LOCK_ASSERT(sc);

        /* 置位复位。 */
        CSR_UPDATE_4(sc, MYFIRST_REG_CTRL, 0, MYFIRST_CTRL_RESET);

        /* 数据手册：等待5 us 让内部状态清除。 */
        DELAY(5);

        /* 清除复位。 */
        CSR_UPDATE_4(sc, MYFIRST_REG_CTRL, MYFIRST_CTRL_RESET, 0);
}
```

5微秒太短，`pause_sbt` 无法高效（内核定时器分辨率通常1毫秒，所以 `pause_sbt` 5微秒实际睡眠到下一个定时器滴答，浪费近一毫秒）。`DELAY(5)` 是正确原语。

### `pause_sbt` 的好用法（释放锁）

第4节的轮询循环：

```c
for (i = 0; i < timeout_ms; i++) {
        uint32_t status = CSR_READ_4(sc, MYFIRST_REG_STATUS);
        if ((status & mask) == target)
                return (0);
        MYFIRST_UNLOCK(sc);
        pause_sbt("mfwait", SBT_1MS, 0, 0);
        MYFIRST_LOCK(sc);
}
```

锁在 `pause_sbt` 前释放，睡眠后重新获取。此模式至关重要，值得仔细关注。

如果锁在 `pause_sbt` 期间持有，模拟 callout（需要锁）会在睡眠期间阻塞。由于 callout 正是驱动程序等待的东西，这会死锁。释放锁让 callout 运行，让模拟更新 `STATUS`，让循环下一次迭代观察新值。

正确性要求是：锁释放期间驱动程序关心的任何东西不能以不可恢复方式改变。我们的情况，驱动程序的循环变量是本地的（`i`、`status`、`mask`、`target`）；它们不在其他上下文接触的内存中。其他上下文接触的状态（寄存器块）正是我们要它们更新的。所以释放锁安全。

此模式的一个变体使用可睡眠条件变量而非轮询：

```c
MYFIRST_LOCK_ASSERT(sc);

while ((CSR_READ_4(sc, MYFIRST_REG_STATUS) & mask) != target) {
        int error = cv_timedwait_sbt(&sc->data_cv, &sc->mtx,
            SBT_1MS * timeout_ms, 0, 0);
        if (error == EWOULDBLOCK)
                return (ETIMEDOUT);
}
return (0);
```

条件变量等待直到目标位改变（由其他上下文信号通知）或超时到期。`cv_timedwait_sbt` 自动释放互斥锁、睡眠、返回前重新获取互斥锁。无需显式 `UNLOCK`/`LOCK` 包装。

为此工作，改变 `STATUS` 的模拟 callout 必须在改变后 `cv_signal` 条件变量。`myfirst_sim_command_cb` 中的代码变为：

```c
/* 设置 DATA_AV 后： */
cv_broadcast(&sc->data_cv);
```

驱动程序的等待循环和模拟的更新通过条件变量协作：驱动程序睡眠直到发生有趣的事情，模拟在发生时唤醒它。这是比用 `pause_sbt` 轮询高效得多的模式。驱动程序整个等待期间睡眠；只需一次唤醒；无空转。

第17章在主轮询循环使用 `pause_sbt` 因教学简单（您可以确切看到睡眠何时发生），但条件变量模式在此提及因为它更接近生产驱动程序的做法，特别是对于长等待。

### `callout_reset_sbt` 的好用法

第3节的模拟 callout 是经典示例。传感器 callout 每100毫秒触发，更新传感器寄存器，重新武装。命令 callout 在 `DELAY_MS` 毫秒后触发一次以完成命令。两者都用 `callout_reset_sbt`：

- `sbt` = SBT 单位的间隔
- `pr` = 0（无精度提示；内核可合并）
- `flags` = 0（无特殊行为）

对于更精确的调度，`pr` 可以设置为限制合并：

```c
callout_reset_sbt(&sim->sensor_callout,
    interval_ms * SBT_1MS,
    SBT_1MS,                   /* 精度：在1毫秒内 */
    myfirst_sim_sensor_cb, sc, 0);
```

`pr = SBT_1MS` 时，callout 不会比请求晚超过1毫秒触发。对于我们的模拟，精度不关键；默认0即可。

对于必须在特定CPU上运行的 callout，`C_DIRECT_EXEC` 标志安排回调直接从 hardclock 中断运行而非分发到 callout 线程。这更高性能但更受限：回调不能睡眠或获取睡眠锁。第17章模拟不需要此；它使用普通的基于互斥锁的 callout。

对于与 hardclock 边界对齐的 callout，`C_HARDCLOCK` 标志适当。我们的模拟也不需要对齐；默认即可。

### 现实示例：模拟慢设备

将原语放在一起。假设模拟配置为 `DELAY_MS=2000`（慢设备，每命令两秒）。驱动程序发出写入命令。每种原语做什么？

1. 驱动程序写入 `DATA_IN`，设置 `CTRL.GO`，调用 `myfirst_wait_for_data(sc, 3000)`（3秒超时）。
2. `myfirst_wait_for_data` 读取 `STATUS`，发现 `DATA_AV` 未设置，释放锁，调用 `pause_sbt("mfwait", SBT_1MS, 0, 0)`，睡眠约1毫秒。
3. 同时，模拟的 `myfirst_sim_command_cb` 安排在约2000毫秒后触发。尚未运行。
4. 驱动程序的循环醒来，重新获取锁，再次读取 `STATUS`，看到 `DATA_AV` 仍未设置，释放锁，再次睡眠。
5. 步骤2-4在两秒内重复约2000次。
6. 模拟的 callout 触发。它读取 `STATUS`，设置 `DATA_AV`，写入命令完成值。
7. 驱动程序循环下一次迭代时，驱动程序看到 `DATA_AV` 设置并返回。

驱动程序等待消耗的总CPU时间：极小。每次 `pause_sbt` 是睡眠，不是空转。CPU在2000次睡眠的每次中都可用于其他工作。

现在假设驱动程序用 `DELAY(1000)` 替代 `pause_sbt(..., SBT_1MS, ...)`。每次迭代时间尺度相同；CPU行为完全不同。每次 `DELAY(1000)` 消耗1000微秒CPU时间。2000次迭代 = 2秒纯CPU。驱动程序可行，但整个命令期间完全占用一个核心。如果其他进程需要CPU，它们等待。

差异说明为什么 `pause_sbt` 是毫秒级等待的正确选择：睡眠免费；空转消耗CPU。

现在假设驱动程序用 `callout_reset_sbt` 替代轮询。这更难：`callout_reset_sbt` 不使当前线程等待。线程立即返回调用者。调用者需要知道睡眠直到 callout 触发。这正是条件变量所做的。代码变为：

```c
MYFIRST_LOCK_ASSERT(sc);

while (!(CSR_READ_4(sc, MYFIRST_REG_STATUS) & MYFIRST_STATUS_DATA_AV)) {
        int error = cv_timedwait_sbt(&sc->data_cv, &sc->mtx,
            SBT_1MS * timeout_ms, 0, 0);
        if (error == EWOULDBLOCK)
                return (ETIMEDOUT);
}
```

模拟 callout 在设置 `DATA_AV` 后调用 `cv_broadcast(&sc->data_cv)`。驱动程序线程睡眠一次（非2000次）；CPU开销极小；响应时间由模拟自己的时序限制，而非驱动程序的轮询率。

对于第17章，轮询模式先教因为它更易读：`for` 循环显式，`pause_sbt` 显式，`CSR_READ_4` 显式。条件变量模式更高效但需要理解第12章全部三个原语。我们用轮询模式作为阶梯，并指出条件变量模式作为生产首选。

### 模拟本身的时序

模拟自己的时序选择值得审查。

传感器 callout 以 `SENSOR_CONFIG` 寄存器指定的间隔触发。默认100毫秒。用户可更改。内核在请求间隔几毫秒精度内尊重。这对教学传感器可行；真实设备可能要求亚微秒精度，这需要 `C_DIRECT_EXEC` 回调甚至非标准时序源。

命令 callout 在命令发出后 `DELAY_MS` 毫秒触发。同样，毫秒精度可行。驱动程序的超时对此校准：默认命令超时2000毫秒，远高于默认 `DELAY_MS` 500毫秒。

驱动程序的轮询间隔1毫秒。对于500毫秒的命令，意味着约500次轮询，可接受。对于5毫秒的命令，意味着约5次轮询，足够快。对于5秒的命令，意味着5000次轮询，浪费。轮询间隔可以自适应（长命令更长轮询），但教学驱动的实现复杂度不值得。

### 现实约束：500毫秒后就绪

为使时序故事具体，配置模拟为每命令500毫秒延迟并运行压力测试：

```text
# sysctl dev.myfirst.0.reg_delay_ms_set=500
# time echo -n "test" > /dev/myfirst0
```

预期输出：

```text
real    0m2.012s
user    0m0.001s
sys     0m0.005s
```

四个字节，每个500毫秒，总共2秒。几乎所有时间是内核睡眠时间；CPU使用可忽略。即使命令慢，驱动程序也高效。

现在用并发写入者压力测试：

```text
# for i in 1 2 3 4; do
    (echo -n "ABCD" > /dev/myfirst0) &
  done
# time wait
```

四个进程，每个4字节，每字节500毫秒。总命令数：16。每个命令500毫秒（通过驱动锁串行化），总时间：8秒。预期：

```text
real    0m8.092s
```

驱动锁将命令串行化，模拟自己的 `command_pending` 检查防止重叠。有效吞吐量是每个 `DELAY_MS` 毫秒一个命令，无论写入者数量。

### 优雅处理超时

命令超时时驱动程序的行为值得单独演练。命令路径：

1. 驱动程序写入 `DATA_IN`，设置 `CTRL.GO`。
2. 驱动程序等待最多 `cmd_timeout_ms` 等待 `STATUS.DATA_AV`。
3. 如果等待超时：
   a. 增加 `cmd_data_timeouts` 计数器。
   b. 调用 `myfirst_recover_from_stuck` 清除陈旧模拟状态。
   c. 向调用者返回 `ETIMEDOUT`。
4. 如果等待成功，继续正常路径。

恢复很重要。实际慢（非卡住）的模拟最终会设置 `DATA_AV`，但驱动程序已放弃。如果驱动程序不清除模拟的待处理状态，后续命令可能看到旧命令残留的 `DATA_AV` 并认为新命令立即完成。恢复函数停止待处理 callout，清除 `command_pending` 标志，清除相关 `STATUS` 位。

在真实硬件上，等效恢复是设备复位。驱动程序写入 `CTRL.RESET`，短暂等待，重启。一些设备有通过PCI配置空间访问的单独复位机制。模式相同：超时后，在做其他任何事情前将设备恢复到已知状态。

### 调试技术：人为减速

一种对时序敏感代码有用的调试技术：人为减慢模拟以拉伸竞争窗口。

```text
# sysctl dev.myfirst.0.reg_delay_ms_set=3000
# sysctl dev.myfirst.0.cmd_timeout_ms=1000
```

现在每个命令需要3秒但驱动程序只等待1秒。每个命令超时。驱动程序的恢复路径在每个命令上运行。恢复路径中的错误立即浮现：

- 如果恢复路径崩溃，内核在第一个命令上恐慌。
- 如果恢复路径留下陈旧状态，第二个命令以某种异常方式失败。
- 如果恢复路径泄漏内存，`vmstat -m | grep myfirst` 随时间增长。

这是合法的调试技术。真正驱动程序难以这样压力测试，因为真实硬件难以按需变慢。相反，模拟距离非常慢只有一个寄存器。使用此能力演练正常操作很少到达的代码路径。

### 常见时序错误

避免时序错误的简短目录。

**错误1：用 `DELAY` 等待毫秒。** `DELAY(1000000)` 对整秒忙等待，消耗一个CPU核心。用 `pause_sbt(..., SBT_1S, ...)` 替代。

**错误2：在 `pause_sbt` 期间持有锁。** 阻塞每个需要锁的上下文。释放锁，睡眠，重新获取。

**错误3：在依赖其他上下文条件上空转不释放。** 如果您的代码等待 callout 设置位，callout 在您空转时无法运行。用 `pause_sbt`（释放锁）或用条件变量释放。

**错误4：用 `callout_reset_sbt` 对应立即完成的即发即弃动作。** 如果动作快，直接做。Callout 有设置成本和调度开销；延迟有意义时才值得。

**错误5：设置比预期操作时间更短的超时。** 每个命令超时。驱动程序不做有用工作。让超时慷慨但有限。

**错误6：设置比预期操作时间更长的超时。** 合法慢操作在全部超时期间挂起用户空间调用。用户失去耐心。为特定操作设置合理超时。

**错误7：释放内存前不排空 callouts。** Callout 在内存释放后触发，回调解引用释放的内存。释放前排空。

**错误8：从回调内重新武装 callout 而不检查取消。** 分离路径请求 callout 停止；回调无论如何重新武装自己。无限循环；分离永不完成。重新武装前检查 `running` 标志（如第3节 `myfirst_sim_sensor_cb` 所做）。

每种错误在初学驱动代码中常见。第17章各阶段通过遵循第13章对 callout 的模式、第12章对条件变量的模式和第6节现在明确化的模式避免所有这些。

### 第6节总结

三种时序原语，三种时间尺度，三种成本特征。`DELAY(9)` 对非常短等待（< 100 us）。`pause_sbt(9)` 对中等等待（100 us 到几秒）当线程无事可做。`callout_reset_sbt(9)` 对即发即弃调度和周期性工作。选择正确原语是根据时间尺度、上下文和线程其他可做之事的判断。

第17章驱动程序为其轮询循环使用 `pause_sbt`，为模拟自己的定时器使用 `callout_reset_sbt`。复位路径在适当处使用 `DELAY(5)`。驱动程序的时序故事一致：它不浪费CPU，不在睡眠期间持有锁，不阻塞其他上下文。

第7节添加第17章模拟的最后主要部分：故障注入框架。时序现在可靠，驱动程序可以在各种失败条件下演练，验证错误恢复路径。



## 第7节：模拟错误和故障条件

只通过快乐路径测试且从未演练错误路径的驱动程序最终会在生产中失败。错误路径是发生意外事情时运行的代码，"意外"是对真实硬件在糟糕一天所做事情最诚实的描述。PHY失去链路信号的网络卡。命令队列溢出的磁盘控制器。校准超出范围的传感器。固件在特定命令序列后死锁的设备。

真实硬件很少产生这些条件，通常在最糟糕的时刻。等待真实硬件产生这些条件的驱动作者会让客户发现错误。使用模拟按需产生这些条件的驱动作者可以在错误发布前修复。

第7节在第17章模拟中构建故障注入框架。该框架让读者请求模拟以特定、可控的方式异常行为：超时、数据损坏、卡住忙状态、随机失败。每种模式演练不同的驱动代码路径。到第7节末尾，驱动程序已见过它设计处理的每个错误，开发者确信错误路径工作。

### 故障注入哲学

在写代码前，短暂哲学停顿。故障注入框架应该是什么样的，不应该是什么样的？

好的故障注入框架：

- 针对特定失败模式，非随机混乱。说"下次读取返回 0xFFFFFFFF"的错误有用；说"可能发生坏事"的错误没用。
- 细粒度可控。打开、关闭或部分注入故障。注入一个、多个、概率注入。
- 可观察。错误触发时，框架记录（或提供计数器）。测试者可以看到发生了什么并与驱动行为关联。
- 与其他测试正交。负载测试不应关心错误是否启用；错误测试不应要求禁用负载。
- 配置时确定。配置为100%概率的错误每次触发。配置为50%概率的错误触发一半时间，伪随机决策可用种子重现。

坏的故障注入框架要么无用（从不触发需要的路径），要么危险（产生驱动即使理论上也无法恢复的失败）。目标是有效的不可靠，非破坏性混乱。

第17章框架用 `FAULT_MASK` 中的一小组标志、`FAULT_PROB` 中的概率和跟踪触发错误数量的计数器实现这些目标。模拟在每次操作开始时查询这些，要么正常进行要么根据配置注入错误。

### 错误模式

第17章实现的四种错误模式：

**`MYFIRST_FAULT_TIMEOUT`**（位0）。下一个命令永不完成。命令 callout 未调度；`STATUS.DATA_AV` 永不置位。驱动程序的 `myfirst_wait_for_data` 超时，触发恢复路径。

**`MYFIRST_FAULT_READ_1S`**（位1）。下次 `DATA_OUT` 读取返回 `0xFFFFFFFF` 而非真实值。这模拟许多真实硬件总线失败方式的总线读取（断开或断电设备的读取通常返回全1）。

**`MYFIRST_FAULT_ERROR`**（位2）。下次命令完成但设置 `STATUS.ERROR`。预期驱动检测错误、清除锁存并向调用者报告错误。

**`MYFIRST_FAULT_STUCK_BUSY`**（位3）。`STATUS.BUSY` 锁定且永不清除。驱动程序的 `myfirst_wait_for_ready` 在任何命令可发出前超时。这模拟卡住需复位的设备。

每种错误演练不同路径。超时错误演练 `cmd_data_timeouts` 和恢复。读取全1错误测试驱动是否注意损坏数据。错误位错误演练 `cmd_errors` 和错误清除逻辑。卡住忙错误演练 `cmd_rdy_timeouts`。

多种错误可同时激活：在 `FAULT_MASK` 中设置多个位。每种错误在模拟相关点独立应用。

### 错误概率字段

`FAULT_MASK` 选择哪些错误是候选；`FAULT_PROB` 控制候选错误实际触发频率。概率表示为0到10000的整数，其中10000意味着100%（每次错误），0意味着从不。这给出四位小数精度而无需分数算术。

错误是否触发的测试：

```c
static bool
myfirst_sim_should_fault(struct myfirst_softc *sc)
{
        uint32_t prob, r;

        MYFIRST_LOCK_ASSERT(sc);

        prob = CSR_READ_4(sc, MYFIRST_REG_FAULT_PROB);
        if (prob == 0)
                return (false);
        if (prob >= 10000)
                return (true);

        r = arc4random_uniform(10000);
        return (r < prob);
}
```

`arc4random_uniform(10000)` 返回 `[0, 10000)` 范围内的伪随机整数。如果小于 `prob`，错误触发。`prob = 5000`（50%）时，错误约一半时间触发。`prob = 100`（1%）时，约百分之一操作。`prob = 10000`，每次操作。

函数在每次操作开始时调用。如果返回真，操作从 `FAULT_MASK` 应用错误（模拟逻辑决定应用哪个或哪些错误，通常是第一个匹配的）。

### 实现超时错误

超时错误最简单。在 `myfirst_sim_start_command` 中，调度命令 callout 前检查错误掩码：

```c
void
myfirst_sim_start_command(struct myfirst_softc *sc)
{
        struct myfirst_sim *sim = sc->sim;
        uint32_t data_in, delay_ms, fault_mask;
        bool fault;

        MYFIRST_LOCK_ASSERT(sc);

        if (!sim->running)
                return;

        if (sim->command_pending) {
                sc->stats.cmd_rejected++;
                device_printf(sc->dev,
                    "sim: overlapping command; ignored\n");
                return;
        }

        data_in = CSR_READ_4(sc, MYFIRST_REG_DATA_IN);
        sim->pending_data = data_in;

        {
                uint32_t status = CSR_READ_4(sc, MYFIRST_REG_STATUS);
                status |= MYFIRST_STATUS_BUSY;
                status &= ~MYFIRST_STATUS_DATA_AV;
                CSR_WRITE_4(sc, MYFIRST_REG_STATUS, status);
        }

        delay_ms = CSR_READ_4(sc, MYFIRST_REG_DELAY_MS);
        if (delay_ms == 0)
                delay_ms = 1;

        sim->command_pending = true;

        /* 错误注入检查。 */
        fault_mask = CSR_READ_4(sc, MYFIRST_REG_FAULT_MASK);
        fault = (fault_mask != 0) && myfirst_sim_should_fault(sc);

        if (fault && (fault_mask & MYFIRST_FAULT_TIMEOUT)) {
                /* 不调度完成 callout。命令
                 * 永不完成；驱动程序将超时。
                 * （模拟不清除 command_pending；
                 * 驱动程序的恢复路径负责。） */
                sc->stats.fault_injected++;
                device_printf(sc->dev,
                    "sim: injecting TIMEOUT fault\n");
                return;
        }

        /* 为 callout 保存错误状态以遵守。 */
        sim->pending_fault = fault ? fault_mask : 0;

        callout_reset_sbt(&sim->command_callout,
            delay_ms * SBT_1MS, 0,
            myfirst_sim_command_cb, sc, 0);
}
```

与第3节版本相比四处变化。

首先，`cmd_rejected` 统计计数器替换裸 printf。

其次，函数读取 `FAULT_MASK` 并调用 `myfirst_sim_should_fault` 决定此操作是否注入错误。

第三，如果选择了超时错误，函数返回而不调度完成 callout。`command_pending` 设为真，所以重叠命令仍被拒绝，但无 callout 触发完成命令。

第四，模拟在 `sim->pending_fault` 中保存错误状态，这样 callout（当触发时，对于非超时错误）可以遵守错误。

`pending_fault` 添加到模拟状态结构：

```c
struct myfirst_sim {
        /* ... 现有字段 ... */
        uint32_t   pending_fault;   /* 完成时应用的 FAULT_MASK 位 */
};
```

### 实现错误位错误

错误位错误在命令完成时触发。在 `myfirst_sim_command_cb` 中：

```c
static void
myfirst_sim_command_cb(void *arg)
{
        struct myfirst_softc *sc = arg;
        struct myfirst_sim *sim = sc->sim;
        uint32_t status, fault;

        MYFIRST_LOCK_ASSERT(sc);

        if (!sim->running || !sim->command_pending)
                return;

        fault = sim->pending_fault;

        /* 总是清除 BUSY。 */
        status = CSR_READ_4(sc, MYFIRST_REG_STATUS);
        status &= ~MYFIRST_STATUS_BUSY;

        if (fault & MYFIRST_FAULT_ERROR) {
                /* 设置 ERROR，不设置 DATA_AV。驱动程序应
                 * 在下次 STATUS 读取时检测错误。 */
                status |= MYFIRST_STATUS_ERROR;
                sc->stats.fault_injected++;
                device_printf(sc->dev,
                    "sim: injecting ERROR fault\n");
        } else if (fault & MYFIRST_FAULT_READ_1S) {
                /* 正常完成，但 DATA_OUT 被损坏。 */
                CSR_WRITE_4(sc, MYFIRST_REG_DATA_OUT, 0xFFFFFFFF);
                status |= MYFIRST_STATUS_DATA_AV;
                sc->stats.fault_injected++;
                device_printf(sc->dev,
                    "sim: injecting READ_1S fault\n");
        } else {
                /* 正常完成。 */
                CSR_WRITE_4(sc, MYFIRST_REG_DATA_OUT, sim->pending_data);
                status |= MYFIRST_STATUS_DATA_AV;
        }

        CSR_WRITE_4(sc, MYFIRST_REG_STATUS, status);

        sim->op_counter++;
        CSR_WRITE_4(sc, MYFIRST_REG_OP_COUNTER, sim->op_counter);

        sim->command_pending = false;
        sim->pending_fault = 0;
}
```

三个分支。错误位错误设置 `STATUS.ERROR` 而非 `STATUS.DATA_AV`，保持 `DATA_OUT` 不变。读取全1错误写 `0xFFFFFFFF` 到 `DATA_OUT` 并正常设置 `DATA_AV`。正常分支写真实数据并设置 `DATA_AV`。

模拟适当增加 `op_counter` 和 `fault_injected`。驱动程序通过寄存器值看到效果；计数器让测试验证错误已触发。

### 实现卡住忙错误

卡住忙错误与命令周期正交：它独立于任何命令保持 `STATUS.BUSY` 设置。添加一个监控 `FAULT_MASK` 并在位设置时保持 `STATUS.BUSY` 锁定的 callout：

```c
static void
myfirst_sim_busy_cb(void *arg)
{
        struct myfirst_softc *sc = arg;
        struct myfirst_sim *sim = sc->sim;
        uint32_t fault_mask, status;

        MYFIRST_LOCK_ASSERT(sc);

        if (!sim->running)
                return;

        fault_mask = CSR_READ_4(sc, MYFIRST_REG_FAULT_MASK);
        if (fault_mask & MYFIRST_FAULT_STUCK_BUSY) {
                status = CSR_READ_4(sc, MYFIRST_REG_STATUS);
                if (!(status & MYFIRST_STATUS_BUSY)) {
                        status |= MYFIRST_STATUS_BUSY;
                        CSR_WRITE_4(sc, MYFIRST_REG_STATUS, status);
                }
        }

        /* 每50毫秒重新武装。 */
        callout_reset_sbt(&sim->busy_callout,
            50 * SBT_1MS, 0,
            myfirst_sim_busy_cb, sc, 0);
}
```

在 `myfirst_sim_attach` 和 `myfirst_sim_enable` 中初始化并启动 callout：

```c
callout_init_mtx(&sim->busy_callout, &sc->mtx, 0);
/* 在 enable 中： */
callout_reset_sbt(&sim->busy_callout, 50 * SBT_1MS, 0,
    myfirst_sim_busy_cb, sc, 0);
/* 在 disable 中： */
callout_stop(&sim->busy_callout);
/* 在 detach 中： */
callout_drain(&sim->busy_callout);
```

现在，当设置 `FAULT_STUCK_BUSY` 时，忙 callout 每50毫秒持续重新置位 `STATUS.BUSY`。驱动程序尝试发出的任何命令将发现 `BUSY` 设置，等待其清除，在 `wait_for_ready` 上超时。清除 `FAULT_STUCK_BUSY` 位停止重新置位，`BUSY` 恢复自然状态（命令路径留下的任何状态）。

### 驱动程序的错误处理路径

错误连线后，驱动程序的错误处理路径被演练。现在审查以确认它们正确工作。

**`wait_for_ready` 超时**（来自第5节）：

```c
error = myfirst_wait_for_ready(sc, sc->rdy_timeout_ms);
if (error != 0) {
        sc->stats.cmd_rdy_timeouts++;
        return (error);
}
```

设置 `FAULT_STUCK_BUSY` 时，此路径触发。调用者看到 `ETIMEDOUT`。驱动程序不尝试发出命令因为无法。计数器反映事件。

**`wait_for_data` 超时**：

```c
error = myfirst_wait_for_data(sc, sc->cmd_timeout_ms);
if (error != 0) {
        sc->stats.cmd_data_timeouts++;
        myfirst_recover_from_stuck(sc);
        return (error);
}
```

设置 `FAULT_TIMEOUT` 时，此路径触发。驱动程序调用 `myfirst_recover_from_stuck`，清除模拟的 `command_pending` 标志和 `STATUS.BUSY`。下一个命令看到干净状态。计数器反映事件。

**`STATUS.ERROR` 检测**：

```c
status = CSR_READ_4(sc, MYFIRST_REG_STATUS);
if (status & MYFIRST_STATUS_ERROR) {
        CSR_UPDATE_4(sc, MYFIRST_REG_STATUS,
            MYFIRST_STATUS_ERROR, 0);
        CSR_UPDATE_4(sc, MYFIRST_REG_STATUS,
            MYFIRST_STATUS_DATA_AV, 0);
        sc->stats.cmd_errors++;
        return (EIO);
}
```

设置 `FAULT_ERROR` 时，此路径触发。驱动程序清除 `STATUS.ERROR` 和 `STATUS.DATA_AV`，增加计数器，返回 `EIO`。

**损坏 `DATA_OUT` 处理** 需要添加。设置 `FAULT_READ_1S` 时，`DATA_OUT` 是 `0xFFFFFFFF`，但驱动程序当前不检查此值。驱动程序是否应检查取决于协议：对于传感器，`0xFFFFFFFF` 通常是合法的"读取错误"标记；对于其他设备，可能是合理的数据值。第17章驱动程序为了演示将其视为潜在错误：

```c
if (data_out == 0xFFFFFFFF) {
        /* 可能是总线读取错误。真实设备很少产生此
         * 值作为合法数据；视为错误。 */
        sc->stats.cmd_errors++;
        return (EIO);
}
```

此行为是判断调用。真实设备真实驱动程序会查阅数据手册中设备记录的错误值并适当响应。

### 逐一测试错误

基础设施就位后，每种错误可单独测试：

**测试1：100%超时错误。**

```text
# sysctl dev.myfirst.0.reg_fault_mask_set=0x1    # MYFIRST_FAULT_TIMEOUT
# sysctl dev.myfirst.0.reg_fault_prob_set=10000  # 100%
# echo -n "test" > /dev/myfirst0
write: Operation timed out
# sysctl dev.myfirst.0.cmd_data_timeouts
dev.myfirst.0.cmd_data_timeouts: 1
# sysctl dev.myfirst.0.fault_injected
dev.myfirst.0.fault_injected: 1
```

第一个字节触发超时。驱动程序恢复。后续字节也会超时直到错误被清除。计数器反映事件。

关闭错误：

```text
# sysctl dev.myfirst.0.reg_fault_mask_set=0
# sysctl dev.myfirst.0.reg_fault_prob_set=0
# echo -n "test" > /dev/myfirst0
# echo "write succeeded"
write succeeded
# sysctl dev.myfirst.0.cmd_successes
dev.myfirst.0.cmd_successes: 4   # 或更多，取决于历史
```

命令再次成功。

**测试2：25%错误位错误。**

```text
# sysctl dev.myfirst.0.reg_fault_mask_set=0x4    # MYFIRST_FAULT_ERROR
# sysctl dev.myfirst.0.reg_fault_prob_set=2500   # 25%
# sysctl dev.myfirst.0.cmd_errors
dev.myfirst.0.cmd_errors: 0
# for i in $(seq 1 40); do echo -n "X" > /dev/myfirst0; done
write: Input/output error     (发生约10次)
# sysctl dev.myfirst.0.cmd_errors
dev.myfirst.0.cmd_errors: 9    # 大约40的25%
```

约四分之一命令错误退出。驱动程序检测错误、清除状态、返回 `EIO`。计数器精确反映计数。驱动程序在测试后仍可用（后续命令成功）。

**测试3：卡住忙错误。**

```text
# sysctl dev.myfirst.0.reg_fault_mask_set=0x8    # MYFIRST_FAULT_STUCK_BUSY
# sysctl dev.myfirst.0.reg_fault_prob_set=10000  # （概率无关；锁定总是开）
# echo -n "X" > /dev/myfirst0
write: Operation timed out
# sysctl dev.myfirst.0.cmd_rdy_timeouts
dev.myfirst.0.cmd_rdy_timeouts: 1
# sysctl dev.myfirst.0.reg_status
dev.myfirst.0.reg_status: 3    # READY|BUSY
```

驱动程序无法发出命令因为 `BUSY` 锁定开。清除错误：

```text
# sysctl dev.myfirst.0.reg_fault_mask_set=0
# sleep 1     # 等待忙 callout 停止重新置位
# sysctl dev.myfirst.0.reg_status
dev.myfirst.0.reg_status: 1    # 仅 READY
# echo -n "X" > /dev/myfirst0
# echo "write succeeded"
write succeeded
```

错误锁定 `BUSY`；驱动程序无法发出命令；驱动程序在 `wait_for_ready` 上超时。清除错误并短暂等待让 `BUSY` 清除（通过命令路径，如有命令滑过，或通过无待处理命令的自然状态）。驱动程序恢复。

### 集成：负载下的随机错误

最现实的测试配置随机错误以低概率运行并在负载下运行驱动程序：

```sh
#!/bin/sh
# fault_stress.sh: 负载下的随机错误。

# TIMEOUT 或 ERROR 1% 概率（位0和2都设置）。
sysctl dev.myfirst.0.reg_fault_mask_set=0x5
sysctl dev.myfirst.0.reg_fault_prob_set=100

# 快速命令以便测试在合理时间内运行。
sysctl dev.myfirst.0.reg_delay_ms_set=10
sysctl dev.myfirst.0.cmd_timeout_ms=50    # 很短，以快速发现超时

# 负载：8个并行工作者，各做200次往返。
for i in $(seq 1 8); do
    (for j in $(seq 1 200); do
        echo -n "X" > /dev/myfirst0 2>/dev/null
        dd if=/dev/myfirst0 bs=1 count=1 of=/dev/null 2>/dev/null
    done) &
done
wait

# 报告。
sysctl dev.myfirst.0 | grep -E 'cmd_|fault_|op_counter'

# 清理。
sysctl dev.myfirst.0.reg_fault_mask_set=0
sysctl dev.myfirst.0.reg_fault_prob_set=0
sysctl dev.myfirst.0.reg_delay_ms_set=500
sysctl dev.myfirst.0.cmd_timeout_ms=2000
```

运行后预期计数器：

- `cmd_successes`：约3168（~3200次预期往返）
- `cmd_data_timeouts`：约16（一半操作的1%）
- `cmd_errors`：约16（一半操作的1%）
- `fault_injected`：约32（两者之和）
- `cmd_recoveries`：约16（每次超时一个）

精确数字每次运行不同（错误是随机的），但比例应大致为每种错误类型1%。如果比例差异巨大，要么是错误注入错误，要么是驱动错误计数错误。无论哪种，测试通过暴露异常做其工作。

### 常见陷阱：从不全部清除的恢复

错误测试中出现的错误是恢复逻辑不清除一切。假设 `myfirst_recover_from_stuck` 忘记清除 `STATUS.BUSY`：

```c
/* 错误版本： */
static void
myfirst_recover_from_stuck(struct myfirst_softc *sc)
{
        MYFIRST_LOCK_ASSERT(sc);
        if (sc->sim != NULL) {
                sc->sim->command_pending = false;
                callout_stop(&sc->sim->command_callout);
        }
        /* 缺少：STATUS 清理。 */
}
```

运行压力测试会显示：

- 第一次超时：恢复运行，`command_pending` 清除。
- 第二次命令尝试：`STATUS.BUSY` 从第一次尝试仍设置。命令无法发出。`cmd_rdy_timeouts` 攀升。
- 第三次尝试：相同。
- ... 永远。

计数器 `cmd_successes` 停止增加。`cmd_rdy_timeouts` 无限攀升。仔细的测试注意到此点并标记。粗心的测试只看到"大量错误"并忽略。

教训是故障注入暴露不完整恢复。在无错误条件下工作的驱动仍可能有只在错误发生时出现的错误。压力测试是暴露这些的方式。

### 组合错误

第17章框架允许多种错误同时激活：

```text
# sysctl dev.myfirst.0.reg_fault_mask_set=0xf    # 全部四种错误
# sysctl dev.myfirst.0.reg_fault_prob_set=1000   # 10%
```

每次操作有10%概率触发错误。如果触发，模拟选择启用的错误之一（实践中，代码按顺序检查：TIMEOUT、ERROR、READ_1S、STUCK_BUSY）。驱动响应必须对所有错误健壮。

更严酷的测试：

```text
# sysctl dev.myfirst.0.reg_fault_prob_set=10000  # 100%
```

每次操作都失败。驱动永不成功完成命令。每条错误路径运行。在此设置下运行驱动一分钟。如果驱动崩溃，错误路径有错误。如果驱动泄漏内存（检查 `vmstat -m | grep myfirst`），错误路径有泄漏。如果驱动死锁（用 `procstat -kk` 检查），错误路径有死锁。

在100%错误下持续运行一段时间是故障注入框架的标准测试。这是故意敌对的；通过的驱动对真实世界失败条件健壮。

### 错误测试的可观察性

驱动现有的可观察性（访问日志、统计计数器）承载了错误测试所需的大部分。有三处添加值得做。

**错误注入日志条目。** 访问日志记录每次寄存器访问；它应记录错误已注入。添加新访问日志条目类型：

```c
#define MYFIRST_CTX_FAULT   0x10   /* 注入错误 */
```

并在错误触发时记录：

```c
myfirst_access_log_push(sc, offset, 0, 4, true, MYFIRST_CTX_FAULT);
```

现在访问日志中，与正常访问交错，显示"此处注入错误X"的条目。测试者可将错误注入与后续驱动行为关联。

**每种错误类型的计数器。** 而非单一 `fault_injected` 计数器，跟踪每种类型：

```c
struct myfirst_stats {
        /* ... 现有计数器 ... */
        uint64_t        fault_timeout;
        uint64_t        fault_read_1s;
        uint64_t        fault_error;
        uint64_t        fault_stuck_busy;
};
```

并根据错误增加正确的计数器：

```c
if (fault & MYFIRST_FAULT_TIMEOUT)
        sc->stats.fault_timeout++;
else if (fault & MYFIRST_FAULT_ERROR)
        sc->stats.fault_error++;
else if (fault & MYFIRST_FAULT_READ_1S)
        sc->stats.fault_read_1s++;
```

sysctl 暴露每个计数器。测试者一目了然看到哪些错误类型已触发哪些未触发。

**错误注入摘要 sysctl。** 单一 sysctl 报告"错误掩码是X，概率是Y，总注入是Z"。交互调试期间快速检查有用：

```c
static int
myfirst_sysctl_fault_summary(SYSCTL_HANDLER_ARGS)
{
        struct myfirst_softc *sc = arg1;
        char buf[128];
        uint32_t mask, prob;
        uint64_t injected;

        MYFIRST_LOCK(sc);
        mask = CSR_READ_4(sc, MYFIRST_REG_FAULT_MASK);
        prob = CSR_READ_4(sc, MYFIRST_REG_FAULT_PROB);
        injected = sc->stats.fault_injected;
        MYFIRST_UNLOCK(sc);

        snprintf(buf, sizeof(buf),
            "mask=0x%x prob=%u/10000 injected=%ju",
            mask, prob, (uintmax_t)injected);
        return (sysctl_handle_string(oidp, buf, sizeof(buf), req));
}
```

调用：

```text
# sysctl dev.myfirst.0.fault_summary
dev.myfirst.0.fault_summary: mask=0x5 prob=100/10000 injected=47
```

这是适合一行并讲述当前错误配置完整故事的摘要。

### 安全的错误注入

"安全"一词需要限定。第17章模拟中的错误注入是安全的因为模拟与驱动在同一内核中运行，错误是局部的：损坏的 `DATA_OUT` 是 `malloc(9)` 内存中损坏的四字节值，非覆盖随机内存的损坏DMA描述符。卡住的 `BUSY` 位是模拟寄存器中卡住的位，非硬件队列阻塞到系统挂起。

真实硬件错误注入是不同故事。故意超时的真实设备可能无限期持有总线锁，阻止其他设备进展。故意损坏的DMA事务可能写入任意物理地址。生产系统上的真实错误注入需要小心设置、专用测试硬件，通常需要管理程序或虚拟设备模拟。

第17章模拟通过在内存中运行避开了所有这些。行为不端的错误最坏情况是内核恐慌（如果 KASSERT 触发）或死锁（如果恢复代码不正确）。两者都可存活：调试内核干净捕获恐慌，死锁驱动可通过重启杀死。无物理硬件风险。

这就是为什么错误注入属于模拟，而不仅仅是生产测试。生产测试捕获真实硬件实际产生的错误；模拟测试捕获真实硬件有时产生且总是有害的错误。

### 动手：错误注入检查清单

开发期间有效使用错误注入的程序：

1. 先写驱动的错误路径。启用错误前，阅读代码并识别恢复的样子。
2. 在100%概率短暂启用错误。验证驱动的错误路径运行。检查计数器。
3. 以低概率（1-10%）启用错误。运行压力测试。验证驱动整体仍在进展。
4. 压力测试前后检查 `vmstat -m | grep myfirst`。内存使用不应增长。
5. 检查 `WITNESS` 输出的锁顺序警告。触发不寻常路径的错误可能暴露潜在锁顺序错误。
6. 检查 `INVARIANTS` 输出的断言失败。损坏状态的错误可能触发不变量。
7. 关闭错误。运行快乐路径。验证一切恢复正常。
8. 在 `HARDWARE.md` 或 `SIMULATION.md` 中记录错误。

此程序将错误注入变为有纪律的活动，非混乱实验。每步有清晰目标，每步验证驱动的特定属性。

### 反思：错误注入教会了我们什么

错误注入是更大驱动开发规则的缩影。驱动必须处理其无法防止的错误、来自其无法控制的源、在其无法预测的时间。错误注入给开发者按需演练这三个属性的工具。

不仅如此，错误注入揭示了驱动错误处理哲学的结构。将每种错误视为同等可恢复的驱动处理某些真实世界失败很差。区分"瞬态"（重试）和"灾难性"（复位设备）的驱动对设备在做什么有更清晰模型。记录每个错误并有足够上下文稍后诊断的驱动是可通过日志分析找到错误的驱动。

第17章驱动不到此级别的精密；它有单一错误处理路径将所有错误视为瞬态且仅在调用者重试时重试。但基础已就位。章末的挑战邀请读者用每种错误类型处理扩展驱动，模拟有演练每种类型的钩子。

### 第7节总结

错误注入是驱动作者在真正失败到达前演练错误处理代码的方式。第17章框架提供四种错误模式（超时、读取全1、错误位、卡住忙），一个概率字段，和足够的可观察性以关联注入错误与驱动行为。

驱动的错误路径通过单独启用每种错误并确认预期计数器增加和预期恢复来验证。组合错误并在负载下以低概率运行演练错误路径的组合。在100%概率运行同时压力所有错误路径。

版本标签变为 `1.0-sim-stage4`。驱动现在已见过它设计处理的每种错误，开发者已观察每条错误路径正确运行。剩余工作是有组织的：重构、版本化、文档化，和通往第18章的桥梁。

第4阶段产出了一个在快乐路径和错误注入条件下都正确工作的驱动。第5阶段（第8节）使驱动可维护。变更是组织性的：清理文件边界、更新文档、版本升级，以及验证沿路没有破坏任何东西的回归通过。

### 模拟特有字段移入模拟结构

第17章逐步引入模拟状态。到第7节末尾，一些模拟相关字段在 softc 而非 `struct myfirst_sim` 中。第5阶段整理此事。

应放入 `struct myfirst_sim` 的字段：

- 三个 callout（`sensor_callout`、`command_callout`、`busy_callout`）。
- 模拟内部状态（`pending_data`、`pending_fault`、`command_pending`、`running`、`sensor_baseline`、`sensor_tick`、`op_counter`）。

应保留在 softc 或 `struct myfirst_hw` 中的字段：

- 寄存器块和 `bus_space` 状态（`regs_buf`、`regs_tag`、`regs_handle`）：`myfirst_hw`。
- 驱动自己的超时（`cmd_timeout_ms`、`rdy_timeout_ms`）：softc。
- 统计（`sc->stats`）：softc（由 hw 和 sim 路径共享）。

推理：硬件状态属于硬件层；模拟状态属于模拟层；驱动可见行为属于驱动。第18章将用真实 PCI 后端替换模拟，最干净的替换是 `myfirst_sim.c` 和 `myfirst_sim.h` 完全消失，被 `myfirst_pci.c` 和（可选）`myfirst_pci.h` 替换。模拟结构字段无需在此过渡中存活；驱动字段需要。

### `SIMULATION.md` 文档

新的 markdown 文件 `SIMULATION.md` 捕获模拟接口。章节：

5. **Callout 映射。** 每个 callout 的表格、其目的、其默认间隔、其与驱动锁的关系。

错误模式表格的示例条目：

```text
| 错误              | 位  | 触发器                          | 效果                        | 恢复            |
|--------------------|------|----------------------------------|-------------------------------|---------------------|
| MYFIRST_FAULT_TIMEOUT | 0  | should_fault() 返回 true      | 命令 callout 未调度 | 驱动超时路径 |
| MYFIRST_FAULT_READ_1S | 1  | should_fault() 返回 true      | DATA_OUT = 0xFFFFFFFF          | 驱动错误检查  |
| MYFIRST_FAULT_ERROR   | 2  | should_fault() 返回 true      | STATUS.ERROR 设置而非 DATA_AV | 驱动错误路径 |
| MYFIRST_FAULT_STUCK_BUSY | 3 | FAULT_MASK 位设置             | STATUS.BUSY 持续锁定 | 清除 FAULT_MASK  |
```

这种表格使阅读代码容易得多：读者在代码中看到 `MYFIRST_FAULT_READ_1S`，可在一处查找完整故事。

### 更新的 `HARDWARE.md`

第16章的 `HARDWARE.md` 描述了静态寄存器块。第17章用动态寄存器和驱动它们的行为扩展它。变化的章节：

**寄存器映射。** 表现在包含第17章寄存器：`SENSOR`（0x28）、`SENSOR_CONFIG`（0x2c）、`DELAY_MS`（0x30）、`FAULT_MASK`（0x34）、`FAULT_PROB`（0x38）、`OP_COUNTER`（0x3c）。每个有其访问类型、默认值和一行描述。

**CTRL 寄存器字段。** 新位：`MYFIRST_CTRL_GO`（位9）。表格添加行："位9：`GO`，写1触发命令；硬件自清除"。

**STATUS 寄存器字段。** 表格获得第17章中哪些位动态的注释：`READY` 在 attach 时设置并保持开；`BUSY` 由 `start_command` 设置并由 `command_cb` 清除；`DATA_AV` 由 `command_cb` 设置并在读取 `DATA_OUT` 后由驱动清除；`ERROR` 由错误注入设置并由驱动清除。

**SENSOR_CONFIG 字段。** 新小节解释双字段布局（低16位间隔，高16位幅度）。

**FAULT_MASK 字段。** 新小节列出每个错误位及其效果。

**动态行为摘要。** 新小节解释什么自主变化：`SENSOR`（默认每100毫秒）、`STATUS.BUSY`（由命令周期或 `FAULT_STUCK_BUSY` callout 设置/清除）、`STATUS.DATA_AV`（由 `command_cb` 设置）、`OP_COUNTER`（每次命令增加）。

**动态行为的锁定规则。** 一句话："所有动态更新在 `sc->mtx` 下发生，callout 通过 `callout_init_mtx` 自动获取。驱动命令路径获取互斥锁并仅在 `pause_sbt` 调用期间释放，模拟 callout 可在那里运行。"

文档从约80行（第16章版本）增长到约140行。它保持可扫描。新章节有组织，使查找第17章信息的读者快速找到。

### 更新的 `LOCKING.md`

第15章的 `LOCKING.md` 描述锁顺序 `sc->mtx -> sc->cfg_sx -> sc->stats_cache_sx` 和排空多个原语的分离顺序。第16章添加硬件层的分离到列表。第17章添加模拟层的分离。

更新的分离顺序，从最外到最内：

1. 销毁 `sc->cdev` 并等待任何打开的文件描述符关闭。
2. 停止并排空所有驱动级 callout（心跳、看门狗、tick_source）。
3. 禁用并排空模拟的 callout（传感器、命令、忙）。
4. 释放模拟状态。
5. 分离硬件层（释放 `regs_buf`，释放 `hw`）。
6. 销毁驱动的同步原语（mutex、sx、cv、sema）。

步骤3在第17章新。模拟 callout 在硬件层拆除前停止，因为模拟的 callout 通过硬件层访问器接触寄存器块。在 callout 仍在执行时释放寄存器块会产生释放后使用。

步骤3和步骤2的顺序重要。驱动级 callout（心跳）可能读寄存器，所以它们必须在模拟排空前排空。有意识排序：驱动 callout 先（它们是最外层消费者），然后模拟 callout（最内层），然后寄存器本身。

### 使用相同模式的真实 FreeBSD 驱动

本章演练的模式不限于模拟硬件。FreeBSD 树中有三处特别值得与第17章一起打开，因为每处以真实子系统使用相似形状，阅读它们将技巧从"我如何写模拟"转化为"内核实际如何操作"。

第一是 `/usr/src/sys/dev/watchdog/watchdog.c` 中的 **`watchdog(4)`** 子系统。该文件顶部的核心例程 `wdog_kern_pat()` 是一个小状态机，由用户空间或其他内核子系统的周期性"拍"驱动；如果拍在配置超时内未到达，子系统触发预超时处理器并最终系统复位。与第17章模拟的直接平行：滴答单位的超时值、在后台推进状态的 callout、从用户空间改变间隔的 ioctl 表面（`WDIOC_SETTIMEOUT`），和暴露最后配置超时以供观察的 sysctl 表面。它也足够短可从头读到尾，这对于生产子系统很罕见。

第二是 `/usr/src/sys/dev/random/random_harvestq.c` 中的 **`random_harvestq`**，熵收集路径。函数 `random_harvestq_fast_process_event()` 及其周围的队列规则是内核版本的"从许多源接受事件并在后台处理它们"模式，本章用模拟传感器演练。收获队列使用环形缓冲区、工作线程，和消费者落后时的显式背压，是树中更干净的例子之一，展示必须永不阻塞供给代码路径的类驱动子系统。第17章后阅读展示当"传感器"是一次系统中所有熵源时自主更新模式的样子。

第三，简单提及 `/usr/src/sys/dev/random/randomdev.c` 中的 **伪随机设备**。它使用 cdev 表面、可配置 sysctl，和收获侧与输出侧的仔细分离。此分离与第17章在 `myfirst_sim.c` 和 `myfirst.c` 间引入的分离相同，看 `randomdev.c` 如何安排文件是规则的有用第二示例。

这些都不是"玩具"子系统。它们是已发布多年的活内核代码。指针的意义不是让您现在掌握它们，而是标记您刚在模拟中练习的技巧存在于生产代码的位置，以便稍后打开 `/usr/src/sys` 时，文件看起来熟悉，您已见过模式。

## 动手实验

第17章的实验侧重于从多个角度演练模拟过程：观察自主行为、在负载下运行命令、注入故障以及观察驱动程序的反应。每个实验需要20到60分钟。

### 实验 1：观察传感器的呼吸

加载驱动程序，并在没有任何驱动程序活动的情况下观察传感器值的变化。
```text
# kldload ./myfirst.ko
# sysctl dev.myfirst.0.sim_running
dev.myfirst.0.sim_running: 1

# while true; do
    sysctl dev.myfirst.0.reg_sensor
    sleep 0.5
  done
```
你应该会看到该值在振荡。默认基线为 `0x1000 = 4096`，振幅为 64，因此该值会在几秒钟内从大约 4096 到大约 4160 之间来回变化。

更改配置以加快振荡速度：
```text
# sysctl dev.myfirst.0.reg_sensor_config_set=0x01000020
```
这将间隔设置为 `0x0100 = 256` ms，并将幅度设置为 `0x0020 = 32`。传感器每 256 ms 更新一次，量程更小。观察输出的变化。

再次更改它以使其大幅减速：
```text
# sysctl dev.myfirst.0.reg_sensor_config_set=0x03e80100
```
这将间隔设置为1000毫秒，幅度设置为256。传感器每秒变化一次，范围更大。

停止模拟：
```text
# sysctl dev.myfirst.0.sim_running=0   # (requires a writeable sysctl, added in the lab examples)
```
传感器数值冻结。重新启用：
```text
# sysctl dev.myfirst.0.sim_running=1
```
它恢复运行。本实验端到端地练习了自主更新路径。

### 实验 2：运行单个命令

手动发出一条命令，并观察每一个寄存器的变化。
```text
# sysctl dev.myfirst.0.access_log_enabled=1
# sysctl dev.myfirst.0.reg_delay_ms_set=200

# echo -n "A" > /dev/myfirst0
# sysctl dev.myfirst.0.access_log | head -20
# sysctl dev.myfirst.0.access_log_enabled=0
```
日志应显示：

1. 对 `STATUS` 的读取（在命令开始时轮询等待 BUSY 清除）。
2. 对 `DATA_IN` 的写入（字节 'A' = 0x41）。
3. 对 `CTRL` 的读取，随后是对 `CTRL` 的写入（设置 GO 位）。
4. 对 `CTRL` 的写入（清除 GO 位；自动清除逻辑）。
5. 对 `STATUS` 的读取（轮询等待 DATA_AV）。
6. 最终 `DATA_AV` 被置位的转换过程（来自仿真的调用）。
7. 对 `DATA_OUT` 的读取（返回 0x41）。
8. 对 `STATUS` 的写入（清除 DATA_AV）。

尝试将每个条目与 `myfirst_sim.c` 和 `myfirst_write_cmd` 中的源代码进行匹配。如果有任何条目不合理，说明你还有知识漏洞需要填补。

### 实验 3：压力测试命令路径

运行多个并发命令并验证正确性。
```text
# sysctl dev.myfirst.0.reg_delay_ms_set=20
# sysctl dev.myfirst.0.cmd_successes     # note value

# for i in 1 2 3 4 5 6 7 8; do
    (for j in $(seq 1 50); do
        echo -n "X" > /dev/myfirst0
     done) &
  done
# wait

# sysctl dev.myfirst.0.cmd_successes     # should have grown by 400
# sysctl dev.myfirst.0.cmd_errors        # should still be 0
# sysctl dev.myfirst.0.reg_op_counter    # should match the growth in cmd_successes
```
8 个写入者，每个 50 条命令，总计 400 条。每条命令耗时 20 ms，测试运行约 8 秒（在驱动程序锁处串行化）。`cmd_successes` 应增加 400。`cmd_errors` 应保持为零（未启用故障）。`reg_op_counter` 应与 `cmd_successes` 的增量匹配。

### 实验 4：注入超时故障

### 实验 6：注入 Stuck-Busy 状态并观察驱动程序等待

启用 stuck-busy 故障，尝试执行一个命令，观察其超时，清除故障，确认恢复正常。
```text
# sysctl dev.myfirst.0.reg_fault_mask_set=0x8     # STUCK_BUSY bit
# sleep 0.1                                         # let the busy callout assert

# sysctl dev.myfirst.0.reg_status
dev.myfirst.0.reg_status: 3                         # READY|BUSY

# sysctl dev.myfirst.0.cmd_rdy_timeouts            # note value

# echo -n "X" > /dev/myfirst0
write: Operation timed out                           # after sc->rdy_timeout_ms

# sysctl dev.myfirst.0.cmd_rdy_timeouts            # should have grown by 1

# sysctl dev.myfirst.0.reg_fault_mask_set=0
# sleep 0.2                                          # let the busy callout stop

# sysctl dev.myfirst.0.reg_status
dev.myfirst.0.reg_status: 1                         # READY only

# echo -n "X" > /dev/myfirst0
# echo "write succeeded"
write succeeded
```
故障锁存 `BUSY`；驱动程序无法发出命令；驱动程序在 `wait_for_ready` 上超时。清除故障并短暂等待可以让 `BUSY` 清除（通过命令路径，如果有任何命令通过，或者通过无未完成命令的自然状态）。驱动程序恢复。

### 实验 8：观察高负载命令下的传感器更新

传感器 callout 和命令 callout 共享同一个互斥锁。如果锁被持续占用，长时间运行的负载测试可能会导致传感器无法获得更新。请检查这一点。
```text
# sysctl dev.myfirst.0.reg_delay_ms_set=5          # very fast commands
# # Start a long load test in the background.
# (for i in $(seq 1 5000); do
    echo -n "X" > /dev/myfirst0 2>/dev/null
  done) &

# while kill -0 $! 2>/dev/null; do
    sysctl dev.myfirst.0.reg_sensor
    sleep 0.2
  done
```
每个轮询的传感器值应与上一次不同（传感器标注正在运行）。如果在负载期间值冻结（多次读取完全相同），说明锁被持有时间过长，传感器处于饥饿状态。正确的驱动程序即使在繁重的命令负载下也能看到平滑更新的传感器值，因为命令路径在 `pause_sbt` 期间会释放锁。

## 挑战练习

挑战练习通过可选的深入工作扩展了第 17 章的内容。每项挑战需要一到四个小时，锻炼的是判断力，而不仅仅是敲击键盘。

### 挑战 1：读即清除 `INTR_STATUS`

第 17 章的 `INTR_STATUS` 仍然是 RW（读写），而不是 RC（读即清除）。实现 RC 语义：当驱动程序读取 `INTR_STATUS` 时，返回当前值，然后将其清除。在访问器路径中添加一个钩子，用于识别 RC 寄存器并对其进行特殊处理。更新 `HARDWARE.md`。

思考一下：驱动程序如何确保读取操作在锁的保护下进行？如果 sysctl 读取了 `INTR_STATUS`（无意中清除了驱动程序所需的状态），会发生什么？

### 挑战 2：添加命令队列

当前的模拟会拒绝重叠的命令。真实设备通常会将它们排队。实现一个小型命令队列：如果在另一个命令待处理时到达了一个新命令，则将其添加到队列中；当当前命令完成时，启动下一个命令。将队列限制为 4 个条目。

思考一下：驱动程序如何等待排队的命令？驱动程序应该维护自己的队列视图，还是依赖模拟的视图？

### 挑战 3：模拟采样率漂移

传感器回调（callout）以固定的时间间隔触发。真实的传感器经常发生漂移：由于温度或电压的变化，间隔会随着时间的推移发生轻微偏移。修改传感器回调，为每个间隔添加一个小的随机扰动（例如，+/- 10%）。观察这会对驱动程序产生什么影响。

思考一下：对于只读取传感器值的驱动程序来说，漂移重要吗？对于关心采样时间的驱动程序来说，它重要吗（第一个驱动程序不关心；但后面的驱动程序可能会关心）？

### 挑战 4：写一清除的 `INTR_STATUS`

W1C（写一清除）语义是 RC 的常见变体。为 `INTR_STATUS` 实现 W1C：向某一位写入 1 会清除该位；写入 0 没有效果；读取操作返回当前值且无副作用。将其与挑战 1 中的 RC 实现进行对比。

思考一下：对驱动程序来说，哪种方式更方便（读时清除还是写时清除）？哪种方式更能防止意外的调试读取？

### 挑战 5：不重启的错误恢复

第 17 章驱动程序的恢复路径直接清除了特定于模拟的状态。真实硬件无法做到这一点；驱动程序只能使用寄存器级别的操作。重写 `myfirst_recover_from_stuck`，使其仅使用寄存器写入（例如，使用 `CTRL.RESET` 位）。调整模拟以响应 `CTRL.RESET`。

思考一下：驱动程序如何等待重置完成？重置会影响其他寄存器吗（例如，它会清除故障掩码吗）？

### 挑战 6：用户空间调试器

编写一个小型用户空间程序 `myfirstctl`，它打开 `/dev/myfirst0`，发出一系列 ioctl 来控制模拟，并显示寄存器状态。使其具有交互性：用户可以输入 `status`、`go X`、`delay 100`、`fault timeout 50%` 等命令。

思考一下：用户空间程序如何与驱动程序通信（ioctl、sysctl、读/写）？什么是合理的命令语法？

### 挑战 7：DTrace 驱动的故障注入

## 故障排除参考

第 17 章模拟最可能暴露的问题的快速参考。

### 传感器不变化

- `sim_running` 为 0。请使用可写的 sysctl 切换它。
- `SENSOR_CONFIG` 的间隔字段为零。请将其设置为正值。
- 传感器回调未调度。请确认 `myfirst_sim_enable` 已被调用。检查 `dmesg` 以获取错误消息。
- `sc->sim` 为 NULL。请确认 `myfirst_sim_attach` 已成功运行。

### 命令总是超时

- `DELAY_MS` 设置得非常高。请减小它或增加命令超时时间。
- `FAULT_MASK` 中设置了 `FAULT_TIMEOUT` 位。请将其清除。
- `FAULT_STUCK_BUSY` 位已设置。请将其清除，并等待大约 100 毫秒，直到 busy 回调停止重新断言。
- 模拟已禁用（`sim_running=0`）。请启用它。
- 上一个命令卡住且未恢复。检查 `command_pending`（通过 sim sysctl），并在需要时手动重置。

### `WITNESS` 警告锁顺序

- detach 路径以错误的顺序获取锁。将堆栈跟踪与 `LOCKING.md` 进行比较。
- 一段新代码在持有休眠锁之后获取了 `sc->mtx`。必须保留第 15 章中的顺序。

### 在 `kldunload` 时内核发生 panic

- 在 `free` 之前未耗尽模拟回调。检查 `callout_drain` 是否在 `free(sim)` 之前运行。
- 模拟状态在回调处于执行状态时被释放。耗尽回调是解决办法。
- 硬件层在模拟层之前被释放。必须遵循第 17 章中的顺序（先 sim detach，然后 hw detach）。

### 内存使用量随时间增长

- `vmstat -m | grep myfirst` 显示驱动程序的分配大小在攀升。这是一处泄漏。检查故障注入的错误路径；它们通常忽略了释放某些东西。
- 访问日志在某些配置中是动态分配的。确认它在 detach 时被释放。

### 访问日志显示了意外的条目

- 回调按照其自身的节奏运行；第 17 章中所有的回调在日志中都是可见的。如果日志显示回调触发得比预期快，请检查间隔。
- 用户空间工具（`watch -n 0.1 sysctl ...`）轮询非常快。每次 sysctl 访问都会经过访问器。

### 故障注入概率与观察结果不符

- `FAULT_PROB` 的范围为 0 到 10000。值为 50 表示 0.5%，而不是 50%。要实现 50%，请使用 5000。
- 故障仅适用于触发 `myfirst_sim_should_fault` 的操作。并非每个操作都是候选对象；例如，传感器更新不经过该路径。

### `sim_running` 切换被忽略

- sysctl 处理程序可能未在锁的保护下调用 `myfirst_sim_enable` 或 `myfirst_sim_disable`。确认处理程序在委托之前获取了 `sc->mtx`。

### 命令成功但花费的时间比预期长得多

- `cmd_timeout_ms` 设置得非常高；任何停滞的命令都会使超时预算变得极其庞大。
- 故障掩码设置了 `STUCK_BUSY`，一个成功的命令意外地变成了 `wait_for_ready` 停滞，直到下一次故障清除。

### 负载测试比预期慢

- 命令通过驱动程序互斥锁进行序列化。N 个写入者各自发出 M 个命令所需的总时间为 N*M * DELAY_MS。
- 减小 `DELAY_MS` 会加速模拟；减小 `cmd_timeout_ms` 不会加速任何操作，除非实际发生了超时。

## 参考：时序原语备忘单

| 原语                | 开销                 | 可取消 | 适用范围       |
|--------------------------|----------------------|------------|-------------------------|
| `DELAY(us)`              | 忙等待，完全占用 CPU  | 否         | < 100 us                |
| `pause_sbt(..., sbt, ...)` | 休眠，让出 CPU  | 否（此形式下不可中断） | 100 us - 秒        |
| `callout_reset_sbt(...)` | 调度的回调   | 是        | 即发即弃、周期性 |
| `cv_timedwait_sbt(...)`  | 条件休眠   | 是（通过 cv_signal） | 可以被缩短的等待 |

各自合法的上下文：

- `DELAY`: 任何上下文（包括过滤器中断、自旋互斥锁）。
- `pause_sbt`: 进程上下文，未持有自旋锁。
- `callout_reset_sbt`: 任何上下文（回调在回调上下文中运行）。
- `cv_timedwait_sbt`: 进程上下文，持有特定的互斥锁。



## 参考：模拟回调剖析

用于添加新模拟行为的模板，以第 17 章的传感器回调为模型。

### 步骤 1：在模拟状态中声明回调
```c
struct myfirst_sim {
        /* ... existing fields ... */
        struct callout   my_new_callout;
        int              my_new_interval_ms;
};
```
### 第 2 步：在 attach 时初始化标注
```c
/* In myfirst_sim_attach: */
callout_init_mtx(&sim->my_new_callout, &sc->mtx, 0);
sim->my_new_interval_ms = 200;   /* default */
```
### 步骤 3：编写回调函数
```c
static void
myfirst_sim_my_new_cb(void *arg)
{
        struct myfirst_softc *sc = arg;
        struct myfirst_sim *sim = sc->sim;

        MYFIRST_LOCK_ASSERT(sc);

        if (!sim->running)
                return;

        /* Do the work: update a register, signal a condition, ...  */
        CSR_UPDATE_4(sc, MYFIRST_REG_SENSOR, 0, 0x100);

        /* Re-arm. */
        callout_reset_sbt(&sim->my_new_callout,
            sim->my_new_interval_ms * SBT_1MS, 0,
            myfirst_sim_my_new_cb, sc, 0);
}
```
### 第4步：在 enable 时启动 callout
```c
/* In myfirst_sim_enable: */
callout_reset_sbt(&sim->my_new_callout,
    sim->my_new_interval_ms * SBT_1MS, 0,
    myfirst_sim_my_new_cb, sc, 0);
```
### 第5步：在 disable 时停止它，在 detach 时排空
```c
/* In myfirst_sim_disable: */
callout_stop(&sim->my_new_callout);

/* In myfirst_sim_detach, after releasing the lock: */
callout_drain(&sim->my_new_callout);
```
### 步骤 6：记录到 SIMULATION.md

在标注映射表中添加一条记录：
```text
| Callout           | Interval         | Purpose                       |
|-------------------|------------------|-------------------------------|
| my_new_callout    | my_new_interval_ms | ... explain ...              |
```
六个步骤。每个步骤都是机械性的。这种模式对于每种模拟行为都是一样的；一旦你内化了模板，添加新行为既快速又可靠。

## 参考：第 17 章简化内容的如实说明

教授一个大主题中的一部分的章节不可避免地会进行简化。为了对读者诚实，这里列出了第 17 章简化了什么以及完整的情况是怎样的。

### Callout 的互斥锁交互

第 17 章使用 `callout_init_mtx`，因此每个模拟回调都在持有 `sc->mtx` 的情况下运行。这是最简单且最安全的模式，但不是唯一的模式。真实的驱动程序有时使用 `callout_init`（无关联互斥锁）并在回调内部获取锁；这提供了更精细的锁控制，但更容易出错。一些真实的驱动程序使用 `callout_init_rm`（用于读写锁），甚至在回调必须在硬中断上下文中运行的情况下使用带有自旋互斥锁的 `callout_init_mtx`。

完整情况：FreeBSD 的 callout API 支持多种锁风格，每种都适合不同的并发模型。第 17 章选择 `callout_init_mtx` 在教学上是简洁的；生产环境的驱动程序会选择匹配其特定锁定图的风格。`/usr/src/sys/dev/e1000/if_em.c` 为其 tick callout 使用 `callout_init_mtx`，为其受自旋锁保护的 DMA 轮询路径使用 `callout_init`，说明了其中的区别。

### 故障注入框架

第 17 章的故障注入仅限于四种模式、一个概率以及在每个操作开始时的简单 `should_fault` 检查。真实的故障注入框架（例如 FreeBSD 自带的 `fail(9)`）支持更丰富的词汇：调用点特定的注入、随时间变化的概率曲线、第 N 次调用的确定性注入，以及可以通过名称从用户空间触发的注入路径。

完整情况：`/usr/src/sys/kern/kern_fail.c` 中的 `fail(9)` 框架实现了一个生产级的故障注入系统。第 17 章的框架是一个简化版本，专注于模拟的特定需求。章末的一个挑战邀请读者用基于 `fail(9)` 的框架替换本章的临时框架。

### 随机数来源

第 17 章使用 `arc4random_uniform(10000)` 作为故障概率决策源。这是非确定性的：同一次测试的两次运行会产生不同的故障模式。为了进行可复现的测试，驱动程序作者会使用确定性源（由固定值种子的简单 LCG，或计数器）并通过 sysctl 暴露种子。

完整情况：可复现的故障注入对于回归测试很重要。如果一个驱动程序在某一天通过了种子为 `0x12345` 的所有故障，而在下一周同样的种子却失败了，那么它就有一个以前不存在的 bug。第 17 章的非确定性对于教学是可以接受的，但对于严肃的回归套件来说是不够的。

### 命令队列

第 17 章一次只允许一条命令在处理中。真实设备通常会对命令进行排队；一个网络驱动程序可能有数千个正在处理的描述符。单命令限制简化了模拟的状态机和驱动程序的锁定逻辑，但它限制了教程驱动程序的吞吐量。

完整情况：排队命令模型需要每条命令的状态（状态、到达时间、结果）、具有自身同步机制的队列数据结构，以及处理命令合并、批量完成和乱序完成的驱动程序模式。第 20 章的 DMA 描述符环介绍了基于 DMA 设备的模式；真实的命令队列驱动程序将其扩展到了非 DMA 上下文。

### 传感器模型

第 17 章的传感器使用简单的算术公式产生三角波值。真实传感器产生的值受物理噪声、温度漂移、采样抖动和设备特定非线性特性的影响。更现实的模拟将包括高斯噪声、慢漂移项和偶尔的异常值。

完整情况：传感器建模本身就是一门学科，真实的传感器设备驱动程序测试通常使用通过模拟层回放的捕获现实世界数据。第 17 章的算术传感器在教学上是足够的；生产环境的模拟将实质性地更加丰富。

### 错误分类

第 17 章区分了四种故障类型：超时、读全 1、错误、忙卡死。真实硬件产生范围更广的故障模式，包括部分写入、中断位卡死、字段反转、寄存器值偏差 1 以及瞬态电噪声。这四种章节故障只是一个起点。

完整情况：特定设备系列的 FreeBSD 驱动程序作者随着时间的推移会开发出更详尽的故障分类。e1000 驱动程序的错误处理代码本身就有几百行，处理几十种不同的故障模式。第 17 章教授了模式；将其应用于真实硬件必然会产生更丰富的分类。

### 缺乏中断

第 17 章的驱动程序通过在循环中读取寄存器来轮询 `STATUS`（在轮询之间使用 `pause_sbt`）。真实驱动程序使用中断：当 `STATUS` 改变时设备引发中断，驱动程序的中断处理程序唤醒等待线程，线程读取新状态。与中断相比，轮询是浪费的。

完整情况：第 19 章介绍中断。第 17 章的驱动程序将在第 19 章获得一条由 `INTR_STATUS` 驱动的唤醒路径，轮询循环将变得短得多（唤醒一次而不是每毫秒唤醒一次）。第 17 章的轮询是一个教学上的垫脚石。

### 计时精度

第 17 章的 callout 以大约配置的间隔运行，精度约为 1 毫秒（内核的默认计时器分辨率）。真实设备有时需要亚微秒级的计时。依赖这种计时的驱动程序必须使用 `DELAY(9)` 进行短时间等待，而不能依赖 `callout` 精度。

完整情况：驱动程序中的高精度计时需要特定平台的机制（TSC、ACPI-PM 计时器、HPET 或嵌入式平台上的硬件特定计时器）。第 17 章的毫秒级模拟不涉及亚毫秒路径；对于有此类要求的真实设备驱动程序，将使用 `DELAY` 或硬件辅助机制。

### 总结

第 17 章是一个教学模拟。它所做的每一个简化都是故意的、命名的，并由后续章节或真实驱动程序扩展所补充。第 17 章教授的模式是每一后续章节扩展的模式；第 17 章建立的纪律是每一后续章节所依赖的纪律。本章有意略去了完整的模拟故事。后续章节和现实世界的实践填补了其余部分。

## 参考：第 17 章术语表

简要列出了第 17 章介绍或广泛使用的术语。

**自主更新。** 在没有驱动程序发起的情况下发生的寄存器更改。在第 17 章中，传感器 callout 产生自主更新。

**命令周期。** 从“驱动程序决定发出命令”到“命令完成且驱动程序观察到结果”的完整事件序列。在第 17 章中：写入 `DATA_IN`，设置 `CTRL.GO`，等待 `STATUS.DATA_AV`，读取 `DATA_OUT`。

**Callout。** 一种 FreeBSD 原语，用于调度回调在未来的某个时间运行。回调在 callout 线程上运行。使用 `callout_init_mtx` 创建，使用 `callout_reset_sbt` 启动，使用 `callout_stop` 取消，使用 `callout_drain` 排空。

**故障注入。** 为了测试目的，故意向系统引入故障条件。第 17 章的框架支持四种故障模式，可通过 `FAULT_MASK` 和 `FAULT_PROB` 配置。

**锁存。** 一旦置位就保持置位直到显式清除的寄存器位。真实设备中的错误位通常是锁存的。第 17 章的 `STATUS.ERROR` 是锁存的。

**轮询。** 重复读取寄存器以检测变化。第 17 章的 `myfirst_wait_for_bit` 在两次读取之间休眠 1 毫秒来轮询 `STATUS`。

**待处理命令。** 已启动但尚未完成的命令。第 17 章的模拟使用 `sim->command_pending` 跟踪此状态。

**协议。** 驱动程序在与设备通信时必须遵守的规则。在第 17 章中：在 `CTRL.GO` 之前写入 `DATA_IN`；在读取 `DATA_OUT` 之前等待 `STATUS.DATA_AV`；读取后清除 `DATA_AV`。

**读清除 (RC)。** 一种寄存器语义，读取寄存器会返回其当前值，然后将其清除。许多真实设备上的 `INTR_STATUS` 是 RC 的。

**模拟后端。** 提供模拟设备行为的内核代码。在第 17 章中，这是 `myfirst_sim.c`。

**副作用。** 寄存器访问触发的超出明显读取或写入的行为。`CTRL.RESET` 写入有副作用（它们重置设备）。`INTR_STATUS` 读取有副作用（在 RC 语义下，它们清除待处理位）。

**粘性。** 类似于锁存。一种持续存在直到被故意清除的位。

**超时。** 一种有界等待，如果预期事件未及时发生则返回错误。第 17 章的 `wait_for_bit` 在配置的毫秒数后超时。

## 参考：阅读 Callout 驱动的驱动程序

引导您浏览一个真实的 FreeBSD 驱动程序，其设计与第 17 章的模拟类似：一个通过 callout 调度周期性工作，且主要工作是响应状态变化的驱动程序。该驱动程序是 `/usr/src/sys/dev/led/led.c`，即 LED 伪设备驱动程序。

在一个终端中打开它。请跟随阅读。

### LED 驱动程序的结构

`led.c` 大约 400 行。它的工作是暴露一个接口，供其他驱动程序通过 `/dev/led/NAME` 公告 LED 设备，并允许用户空间通过写入模式来闪烁 LED。实际的硬件工作委托给注册 LED 的驱动程序；`led.c` 负责调度和模式匹配。

关键数据结构：

- `struct led_softc`：每个 LED 的状态，包括一个 `struct callout led_ch`，一个保存闪烁模式的 `struct sbuf *spec`，以及一个指向驱动程序“设置状态”回调的函数指针。
- `struct mtx led_mtx`：一个保护所有 LED 列表的全局互斥锁。
- 所有 LED 的链表。

### Callout 回调

闪烁模式解释器在 callout 回调中运行：
```c
static void
led_timeout(void *p)
{
        struct ledsc *sc = p;
        char c;
        int count;

        if (sc->spec == NULL || sc->ptr == NULL || sc->count == 0) {
                /* no pattern; stop blinking */
                sc->func(sc->private, sc->on);
                return;
        }

        c = *(sc->ptr)++;
        if (c == '.') {
                /* Pattern complete, restart. */
                sc->ptr = sbuf_data(sc->spec);
                c = *(sc->ptr)++;
        }

        if (c >= 'a' && c <= 'j') {
                sc->func(sc->private, 0);   /* LED off */
                count = (c - 'a') + 1;
        } else if (c >= 'A' && c <= 'J') {
                sc->func(sc->private, 1);   /* LED on */
                count = (c - 'A') + 1;
        } else {
                count = 1;
        }

        callout_reset(&sc->led_ch, count * hz / 10, led_timeout, sc);
}
```
（为了便于展示，该代码相对于真实源码作了少量删减。）

### 该模式

观察回调函数的结构。

首先，它检查是否有事可做。如果 `spec == NULL`，则表明模式已被清除；该函数会将 LED 关闭（切换到已知状态）并在不重新设定的情况下返回。这与第17章的 callout 中的 `if (!sim->running) return;` 模式完全一致。

其次，它在模式缓冲区中逐个字符地向前推进。模式语言使用小写字母 'a' 到 'j' 表示“LED 关闭 100 毫秒到 1000 毫秒”，大写字母 'A' 到 'J' 表示“LED 开启”。这是一个由 callout 驱动的小型状态机。

第三，它调用 `sc->func(sc->private, state)` 来实际切换 LED 状态。这是硬件层面的工作，被委托给注册该 LED 的驱动程序。`led_timeout` 回调并不关心该 LED 到底是 GPIO 引脚、通过 I2C 连接的 LED 控制器，还是发送给远程设备的消息；它只负责调用该函数指针。

第四，它使用由当前模式字符决定的时间间隔重新设定 callout。这是第17章的 `callout_reset_sbt` 模式，尽管 `led.c` 使用的是较老的基于滴答数间隔的 `callout_reset`。

### 第17章的经验教训

`led.c` 驱动程序阐明了第17章中的几个模式：

- callout 每次调用将状态机向前推进一步。
- callout 根据当前状态以可变的时间间隔重新设定自身。
- “无事可做”的分支直接返回而不再重新设定；分离路径正是依赖于这一点来最终排空任务。
- 回调使用函数指针来执行委托的工作，从而使 callout 逻辑与特定硬件的工作分离开来。

第17章的模拟以略微不同的表现形式使用了相同的模式。传感器回调更新寄存器；命令回调触发完成事件；忙回调重新断言状态位。每一个都是由 callout 推进的小型状态机。

从头到尾阅读 `led.c`。它是 `/usr/src/sys/dev/` 中最短的驱动程序之一，展示了一个清晰的 callout 驱动设计。阅读之后，写一段话总结 `led_timeout` 如何与 `led_drvinit`、`led_destroy` 和 `led_write` 进行交互。如果你能清晰地表达出它们之间的关系，说明你已经内化了这种 callout 驱动驱动程序的模式。


## 参考：“模拟中”与“现实中”的区别

第17章模拟了一个设备。一个自然的问题是：模拟与真实设备有多大区别，当两者互换时，驱动程序具体需要做哪些改变？

简短的比较如下。

### 模拟与现实的共同点

- 寄存器映射：相同的偏移量、相同的宽度、相同的访问类型。
- CSR 宏：`CSR_READ_4`、`CSR_WRITE_4`、`CSR_UPDATE_4` 展开为相同的 `bus_space_*` 调用。
- 驱动程序的命令周期逻辑：写入 `DATA_IN`，设置 `CTRL.GO`，等待 `DATA_AV`，读取 `DATA_OUT`。
- 加锁规则：`sc->mtx` 在两种情况下都用于保护寄存器访问。
- 统计数据：`cmd_successes`、`cmd_errors` 等无论如何都会跟踪真实的事件。

### 不同之处

- 标签和句柄：模拟使用 `X86_BUS_SPACE_MEM` 和一个 `malloc(9)` 地址；现实使用由 `rman_get_bustag` 和 `rman_get_bushandle` 为已分配的 PCI BAR 返回的标签和句柄。
- 寄存器块的生命周期：模拟的生命周期从 `myfirst_sim_attach` 到 `myfirst_sim_detach`；现实的生命周期从 `bus_alloc_resource_any` 到 `bus_release_resource`。
- 自主行为：模拟通过 callout 产生；现实通过设备内部逻辑产生。驱动程序看不出区别。
- 故障模式：模拟的故障是刻意且受控的；现实的故障则是自然发生的。
- 计时：模拟的 `DELAY_MS` 是一个寄存器；现实的计时由设备的设计决定且无法更改。
- `CTRL.GO` 的自动清除：模拟的自动清除由写入拦截实现；现实的则在硅片中实现。驱动程序期望得到相同的行为。
- 错误恢复：模拟的 `myfirst_recover_from_stuck` 直接操纵模拟状态；现实的恢复路径仅使用寄存器操作（通常是复位）。

这个列表很短。这就是抽象的意义所在：驱动程序的大部分代码是相同的，而不同的部分都被很好地局部化了。

### 第18章的改变

当第18章用真实的 PCI 替换模拟时，驱动程序的改动如下：

1. `myfirst_hw_attach` 发生改变：不再使用 `malloc` 分配寄存器块，而是调用 `bus_alloc_resource_any` 来分配 PCI BAR。
2. `myfirst_hw_detach` 发生改变：不再使用 `free` 释放寄存器块，而是调用 `bus_release_resource` 来释放 BAR。
3. 不再编译模拟文件（`myfirst_sim.c`、`myfirst_sim.h`）；Makefile 会将它们剔除。
4. 驱动程序的探测逻辑获得了一个 `probe` 方法，用于匹配真实设备的供应商和设备 ID。
5. `DRIVER_MODULE` 的注册从伪设备风格转变为 PCI 附加风格。

其他所有内容（命令周期、统计数据、加锁、文档）均保持不变。第17章建立的规范使其成为可能。


## 参考：当模拟不够时

关于模拟局限性的客观视角。第17章将模拟作为一项主要技术进行讲解，但它并不能替代所有类型的测试。有几种场景必须要使用真实硬件或由虚拟机监控程序支持的虚拟设备。

### 当错误源于真实硅片的计时时

仅在特定的总线时钟频率比、特定的内存流量模式或特定的设备固件版本下才会出现的竞态条件，是无法在模拟中重现的。模拟以内核线程的速度运行，而不是硬件速度；其时机由内核的调度器决定，而不是总线架构。

### 当错误源于特定平台代码时

需要处理 IOMMU 重映射、特定缓存属性或特定于体系结构的内存屏障的驱动程序，无法在模拟中得到完全验证。模拟的内核内存具有与真实设备内存不同的缓存行为，并且只有在涉及真实设备内存时，才会执行特定于体系结构的路径。

### 当错误依赖于实际硬件资源时

需要配置 PCIe 链路宽度、管理电源状态或与设备固件交互的驱动程序，无法在模拟中进行验证。模拟没有 PCIe 链路，没有电源状态，也没有固件。

### 当错误出在测试本身时

如果测试模拟的方式与真实设备的使用方式不对应，就会产生虚假的自信。模拟可能会通过测试；而真实设备在现实使用中仍然可能会失败。只要条件允许，针对模拟的测试必须与针对现实的测试配对进行。

### 应对措施

正确的做法是分层测试。模拟可以捕获协议错误、加锁顺序错误、错误处理错误以及绝大多数逻辑错误。由虚拟机监控程序支持的虚拟设备（bhyve 或 qemu 中的 virtio）可以捕获 PCI 特定的错误、总线主控问题以及一些时序问题。实验室系统中的真实硬件能捕获其他方法都无法重现的“最后一公里”错误。

第17章的重点在于模拟。第18章的重点将转向基于虚拟机监控程序的 PCI。后续关于 DMA 和中断的章节将引入额外的测试层面。在专用的实验室系统上进行真实硬件测试，是驱动程序开发者在长期实践中养成的一种规范；本书可以为你提供相关的词汇，但无法为你提供硬件。

## 参考：实战示例：完整的 `myfirst_sim.h`

这是第17章末尾完整的 `myfirst_sim.h` 文件，供快速参考。该源码也位于 `examples/part-04/ch17-simulating-hardware/stage5-final/myfirst_sim.h` 中。
```c
/*-
 * SPDX-License-Identifier: BSD-2-Clause
 *
 * Copyright (c) 2026 Edson Brandi
 *
 * myfirst_sim.h -- Chapter 17 simulation API.
 *
 * The simulation layer turns the Chapter 16 static register block into
 * a dynamic device: autonomous sensor updates, command-triggered
 * delayed completions, read-to-clear semantics for INTR_STATUS, and a
 * fault-injection framework. The API is small; most of the simulation
 * is in myfirst_sim.c.
 */

#ifndef _MYFIRST_SIM_H_
#define _MYFIRST_SIM_H_

#include <sys/callout.h>
#include <sys/stdbool.h>

struct myfirst_softc;

/*
 * Simulation state. One per driver instance. Allocated in
 * myfirst_sim_attach, freed in myfirst_sim_detach.
 */
struct myfirst_sim {
        /* The three simulation callouts. */
        struct callout       sensor_callout;
        struct callout       command_callout;
        struct callout       busy_callout;

        /* Last scheduled command's data. Saved so command_cb can
         * latch DATA_OUT when it fires. */
        uint32_t             pending_data;

        /* Saved fault state for this command. Set in start_command,
         * consumed by command_cb. */
        uint32_t             pending_fault;

        /* Whether a command is currently in flight. */
        bool                 command_pending;

        /* Baseline sensor value; the sensor callout oscillates
         * around this. */
        uint32_t             sensor_baseline;

        /* Counter used by the sensor oscillation algorithm. */
        uint32_t             sensor_tick;

        /* Local operation counter; mirrors OP_COUNTER register. */
        uint32_t             op_counter;

        /* Whether the simulation callouts are running. Checked by
         * every callout before doing work. */
        bool                 running;
};

/*
 * API. All functions assume sc->sim is valid (that is, 
 * myfirst_sim_attach has been called successfully) unless noted
 * otherwise.
 */

/*
 * Allocate and initialise the simulation state. Registers callouts
 * with sc->mtx. Does not start the callouts; call _enable for that.
 * Returns 0 on success, an errno on failure.
 */
int  myfirst_sim_attach(struct myfirst_softc *sc);

/*
 * Drain all simulation callouts, free the simulation state. Safe to
 * call with sc->sim == NULL. The caller must not hold sc->mtx (this
 * function sleeps in callout_drain).
 */
void myfirst_sim_detach(struct myfirst_softc *sc);

/*
 * Start the simulation callouts. Requires sc->mtx held.
 */
void myfirst_sim_enable(struct myfirst_softc *sc);

/*
 * Stop the simulation callouts. Does not drain (that is _detach's
 * job). Requires sc->mtx held.
 */
void myfirst_sim_disable(struct myfirst_softc *sc);

/*
 * Start a command. Called from the driver when CTRL.GO is written.
 * Reads DATA_IN and DELAY_MS, schedules the command completion
 * callout. Rejects overlapping commands. Requires sc->mtx held.
 */
void myfirst_sim_start_command(struct myfirst_softc *sc);

/*
 * Register simulation-specific sysctls on the driver's sysctl tree.
 * Called from myfirst_attach after sc->sysctl_tree is established.
 */
void myfirst_sim_add_sysctls(struct myfirst_softc *sc);

#endif /* _MYFIRST_SIM_H_ */
```

## 参考：关于模拟理念的结束语

作为本章结尾的一段话，在您完成实验之后值得再来回味一下。

模拟从本质上来说是一种建模行为。你拿到一个无法完全控制的系统（真实设备），然后构建一个你可以控制的、行为相似的较小系统（模拟）。模拟永远不会完美。它的价值不在于与真实系统的每一个细节都分毫不差，而在于保留了与你要解决的问题密切相关的那些属性。

对于第17章而言，重要的属性是协议正确性：驱动程序是否正确处理了命令周期、时序、错误情况和恢复？只要模拟保留了协议正确性，它就体现了自身的价值，即使它在每个时序细节上都存在微小的偏差。

对于后续章节以及你未来的工作而言，重要的属性可能会有所不同。性能测试需要一个保留吞吐量行为的模拟。功耗测试需要一个保留空闲和活动状态的模拟。安全测试则需要一个能够生成对抗性输入的模拟。

第17章传授的技能不是“如何模拟这个特定的设备”，而是“如何识别模拟必须保留的属性，以及如何构建一个能保留这些属性的模拟”。这项技能是可以迁移的，并且在你编写的每一个驱动程序中都将发挥作用。
